# 1. TDD、BDD、SDD、DDD：AI 时代的开发方法论选择

> AI 时代信息爆炸，开发方法论同样五花八门。TDD、BDD、SDD、DDD……这些"XX 驱动开发"的本质是什么？个人开发者和职业程序员分别该选哪些？本文从横向对比到落地建议，给出清晰的选型框架。

---

## 1.1 什么是"X-Driven Development"

所有以 "DD"（驱动开发）命名的思想，共享同一个底层逻辑：**以某要素为核心驱动，约束和引导软件开发过程**。区别仅在于**驱动的核心要素**不同——测试、行为、规范、领域、模型、数据……选择哪种，取决于你在哪个抽象层级工作、面对什么类型的问题。

可以从两个维度理解：

| 维度 | 说明 |
|------|------|
| **驱动要素** | 测试？行为？规范？领域模型？ |
| **关注层级** | 代码级 → 协作级 → 架构级 → 业务级 |
| **适用规模** | 个人工具 → 小团队 → 中大型系统 |

---

## 1.2 "DD" 谱系一览

### 1.2.1 测试/验收类（偏质量与验证）

| 缩写 | 全称 | 核心一句话 |
|------|------|-----------|
| **TDD** | Test-Driven Development，测试驱动开发 | 先写单元测试 → 再写代码 → 重构；关注**代码逻辑正确性** |
| **BDD** | Behavior-Driven Development，行为驱动开发 | 用自然语言描述系统行为（Given/When/Then）；关注**业务行为与协作** |
| **ATDD** | Acceptance Test-Driven Development，验收测试驱动开发 | 业务+开发+测试共同定义验收标准；关注**用户验收与需求对齐** |
| **PDD** | Programmer-Driven Development，程序员驱动开发（自嘲/反义） | 无规范、纯经验编码；**无明确驱动，靠个人能力** |

### 1.2.2 业务/领域类（偏复杂业务建模）

| 缩写 | 全称 | 核心一句话 |
|------|------|-----------|
| **DDD** | Domain-Driven Design，领域驱动设计 | 以领域模型为核心，统一语言、限界上下文；关注**复杂业务拆解与建模** |
| **FDD** | Feature-Driven Development，特性驱动开发 | 按客户可见特性迭代；关注**功能交付节奏与价值** |
| **UDD** | Use Case-Driven Development，用例驱动开发 | 以用例串联需求/设计/测试；关注**用户场景与功能流程** |

### 1.2.3 模型/架构/规范类（偏抽象与生成）

| 缩写 | 全称 | 核心一句话 |
|------|------|-----------|
| **MDD** | Model-Driven Development，模型驱动开发 | 先建 UML/DSL 模型 → 自动生成代码；关注**模型即核心、代码自动生成** |
| **SDD** | Specification-Driven Development，规范驱动开发 | 先写严格规范（如 API 契约） → 代码实现；关注**规范为唯一事实来源** |
| **ADD** | Architecture-Driven Development，架构驱动开发 | 先定架构（分层/微服务） → 再填业务；关注**架构先行、技术骨架** |

### 1.2.4 数据/AI/其他类（偏数据与新范式）

| 缩写 | 全称 | 核心一句话 |
|------|------|-----------|
| **DDD** (重名) | Data-Driven Development，数据驱动开发 | 以数据结构/流程为核心；关注**数据流向与处理逻辑** |
| **EDD** | Event-Driven Development，事件驱动开发 | 基于事件触发异步处理；关注**解耦与实时响应** |
| **AIDD** | AI-Driven Development，AI 驱动开发 | 用大模型辅助编码/测试/设计；关注**AI 提效与生成式开发** |

---

## 1.3 核心区别（一张表看懂）

| 缩写 | 全称 | 核心驱动 | 关注点 | 典型场景 |
|------|------|----------|--------|----------|
| TDD | 测试驱动 | 单元测试 | 代码质量、覆盖率 | 算法、核心工具类 |
| BDD | 行为驱动 | 业务行为 | 协作、自然语言描述 | 需求复杂、多方协作 |
| ATDD | 验收驱动 | 验收标准 | 需求对齐、可交付 | 合同项目、用户验收严格 |
| DDD | 领域驱动 | 领域模型 | 复杂业务、统一语言 | 金融、供应链、电商 |
| FDD | 特性驱动 | 客户特性 | 迭代节奏、功能价值 | 敏捷项目、快速交付 |
| MDD | 模型驱动 | 系统模型 | 抽象、代码生成 | 企业级、标准化系统 |
| SDD | 规范驱动 | 严格规范 | 契约、一致性 | API 平台、微服务 |

### 关键差异总结

1. **TDD vs BDD vs ATDD**
   - TDD：**dev 视角**，单元测试，关注代码
   - BDD：**业务+dev+测试视角**，行为描述，关注协作
   - ATDD：**客户视角**，验收标准，关注交付是否符合需求

2. **DDD（领域）vs FDD vs UDD**
   - DDD：**复杂业务建模**，解决"业务太乱"
   - FDD：**功能迭代**，解决"交付慢"
   - UDD：**用户场景**，解决"需求不清晰"

3. **MDD vs SDD vs ADD**
   - MDD：**模型生成代码**，抽象层级最高
   - SDD：**规范约束代码**，偏契约与一致性
   - ADD：**架构约束代码**，偏技术骨架

### 一句话记牢

| 方法论 | 一句话 |
|--------|--------|
| **TDD** | 先写测试再写代码（保质量） |
| **BDD** | 用自然语言说清楚要做什么（保协作） |
| **DDD** | 把复杂业务拆明白（保可维护） |
| **ATDD** | 先定验收标准再开发（保交付） |
| **FDD** | 按功能一个个做（保节奏） |
| **MDD** | 画模型自动生成代码（提效率） |
| **SDD** | 以规范为唯一事实来源（保一致性） |

## 1.4 AI 时代的主力方法论

AI 时代，**不是所有 DD 都同等重要**。有的被 AI 放大价值，有的被 AI 部分取代。结论如下：

### 1.4.1 SDD（规范驱动开发）—— AI 时代的"第一 DD"

**SDD = Specification-Driven Development，规范驱动开发**

- **核心**：Spec 是唯一真相源，代码是 Spec 的产物
- **规则**：**先写 Spec → 再实现代码**
- **Spec 的本质**：不是普通文档，而是**结构化、可验证、可执行**的契约
- **历史**：与 AI 无关——20 年前的"契约式设计、形式化方法"已是同样思想

**Spec Coding（AI 时代的 SDD 实践）**：2025–2026 年随大模型兴起的实践模式：

- **Spec as Code**：用结构化格式（JSON/YAML/结构化 Markdown）写 Spec
- **人写 Spec，AI 写代码**
- **全程可追溯、可校验、可回归**
- 典型工具：GitHub Spec-Kit、OpenSpec 等

一句话：**SDD 是一套"规范先行"的软件工程思想；Spec Coding 是其在 AI 时代的具落地方式**。

### 1.4.2 BDD（行为驱动开发）—— 需求对齐、少返工的利器

- **核心**：用**自然语言 + Given/When/Then** 描述用户行为
- **AI 时代价值**：
  - 个人：自己想清楚边界，避免做一半改需求
  - 团队：产品/开发/测试统一语言
  - **AI 友好**：可直接转成测试用例/验收标准

### 1.4.3 TDD（测试驱动开发）—— AI 代码质量的"方向盘"

- **核心**：**先写失败测试 → AI 写代码 → 重构**
- **AI 时代价值**：
  - 传统：质量保障
  - AI 时代：**约束 AI 输出、防止跑偏**
- **适合场景**：核心算法、工具函数、关键业务逻辑

### 1.4.4 DDD（领域驱动设计）—— 复杂业务才需要

- **核心**：统一语言、限界上下文、领域模型
- **价值**：业务复杂（电商/金融/供应链）时防混乱
- **个人小项目**：**不用硬套**，太重、收益小

### 1.4.5 AIDD 是能力，不是流程

- **AIDD**（AI-Driven Development）不是"替代 TDD/SDD"，而是**用 AI 辅助做 DDD/SDD/TDD**
- 不要把它当成独立流程，否则容易变成"vibe coding"（靠感觉写代码）

## 1.5 按角色推荐最佳组合

### 🔹 个人开发者 / 独立开发者（追求快、稳、少返工）

**首选：SDD + BDD + 轻量 TDD**

推荐流程（日常循环）：

1. **BDD（想清楚）**：用中文写 2–3 条核心场景 → AI 转 Gherkin
2. **SDD（定规范）**：写清数据结构、接口、错误码、验收条件
3. **TDD（写测试）**：核心逻辑先写测试 → AI 生成代码
4. **AI 实现**：把 Spec + 测试丢给 AI 生成完整模块
5. **人工校验**：审核关键逻辑、安全、性能

> 避坑：别上来就让 AI "写个 xx 系统"——必崩。

### 🔹 职业程序员（企业/团队，要质量+可维护+协作）

#### 1）普通业务系统（CRUD 为主）

**SDD + BDD + TDD（核心模块）**

- SDD：API 契约、数据模型、业务规则
- BDD：用户故事/验收场景
- TDD：核心逻辑、支付/权限等关键路径

#### 2）复杂业务/中大型项目（微服务/领域复杂）

**DDD（顶层）+ SDD（接口）+ BDD（场景）+ TDD（内核）**

- DDD：限界上下文、聚合根、统一语言
- SDD：服务间契约、数据模型
- BDD：跨服务流程场景
- TDD：领域服务、核心算法

### 场景选型表

| 场景 | 首选 DD | 次选 | 不推荐 |
|------|---------|------|--------|
| 个人小项目/工具 | SDD+BDD | 轻量 TDD | DDD（太重） |
| 创业产品/快速迭代 | SDD+BDD | TDD（核心） | 纯 TDD（慢） |
| 企业业务系统（CRUD） | SDD+BDD | TDD（核心） | 无 |
| 复杂领域（金融/电商） | DDD+SDD | BDD+TDD | 无 |
| 算法/底层库 | TDD | SDD | BDD（没必要） |
| 纯 AI 生成 Demo | BDD | SDD | TDD（成本高） |

## 1.6 AI 时代极简落地流程

以下流程可直接照做，从零开始构建一个 AI 辅助开发循环：

### 1.6.1 需求 → BDD（10 分钟）

用中文写：用户在什么条件下，做什么，期望什么结果。AI 转成 Gherkin（Given-When-Then）。

### 1.6.2 规范 → SDD（20 分钟）

写清：数据结构、接口 URL/参数/返回、错误码、业务规则、性能要求。

### 1.6.3 测试 → TDD（核心模块）

先写单元测试/集成测试 → AI 生成实现代码。

### 1.6.4 AI 生成 + 人工审核

把 Spec + 测试丢给 AI → 人工过一遍关键逻辑、安全、性能。

### 1.6.5 持续迭代

改需求 → **先改 BDD → 再改 SDD → 最后改代码**（文档驱动迭代）。

---

## 1.7 总结

> **个人/小团队**：SDD + BDD + 轻量 TDD（快、稳、性价比最高）
>
> **职业/企业**：复杂业务加 DDD，普通业务 SDD + BDD + TDD
>
> **AIDD 是辅助能力，不是流程**——丢了规范和测试，纯 AI 写代码必翻车。

推荐的 AI 时代开发循环可以简单概括为：

```
BDD（想清楚）→ SDD（定规范）→ TDD（写测试）→ AI 实现 → 人工校验 → 迭代
```
