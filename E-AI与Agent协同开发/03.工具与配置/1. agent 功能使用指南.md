# 1. Claude Code /agent 功能使用指南

`/agent` 是 Claude Code 中启动**子 Agent（sub-agent）** 的指令，用于将独立子任务委派给一个新的上下文窗口执行，完成后结果自动合并回主会话。

## 【核心精简提纲】

### 记忆主干

| 概念 | 说明 |
|------|------|
| `/agent` | 启动一个子 Agent，拥有独立的上下文窗口，在主会话之外并行或后台执行任务 |
| 独立上下文 | 子 Agent 的上下文（文件读取、对话历史）与主会话隔离，互不污染 |
| 结果回收 | 子 Agent 执行完毕，其最终输出自动合并回主会话的上下文 |
| 并发模式 | 支持 `run_in_background: true` 让子 Agent 在后台执行，主会话继续其他工作 |

### 核心要点清单

- **避免上下文污染**：大型项目或跨模块任务，用 `/agent` 隔离搜索/阅读/执行操作，不影响主会话的 token 预算和焦点
- **天然并发**：多个不相关的子任务可同时派发给多个子 Agent，最后合并结果
- **上下文传递**：主会话通过 prompt 向子 Agent 传递背景信息；子 Agent 不自动继承主会话的文件读取和对话历史
- **结果形态**：子 Agent 的最终消息作为普通文本返回给主会话，不是工具调用或结构数据
- **适用场景**：搜索代码库、阅读多个文件、审查代码改动、执行独立调试、生成 documentation

### 关键结论 / 流程

```plain
主会话中遇到独立子任务
    ↓
/agent + 明确的任务描述 + 必要的背景信息
    ↓
子 Agent 在独立窗口中执行（可并行）
    ↓
结果合并回主会话 → 继续主流程
```

### 相关速查

| 指令 | 与 /agent 的关系 |
|------|-----------------|
| `/compact` | 子 Agent 不继承主会话上下文，但仍可用 `/compact` 自行瘦身 |
| `/clear` | 子 Agent 内 `/clear` 只清子会话，不影响主会话 |
| `/context` | 子 Agent 内 `/context` 查看子会话的 token 占用 |

## 【完整拓展详情】

## 1.1 什么是 /agent

`/agent` 命令在 Claude Code 中创建一个**子 Agent（sub-agent）**——一个完全独立的 Claude 实例，拥有自己的上下文窗口、工具调用能力，和独立的对话历史。

核心区别：

| 维度 | 主会话 | 子 Agent（/agent） |
|------|--------|-------------------|
| 上下文 | 当前会话的全部历史 | 从零开始，只接收 prompt 中的背景 |
| 文件读取 | 累计 | 独立读取，不共享缓存 |
| 工具调用 | 共享 | 独立，可自由执行 |
| Token 消耗 | 持续增长 | 执行完毕即释放 |
| 结果 | — | 最终消息合并回主会话（纯文本） |

### 1.1.1 为什么需要子 Agent

大项目或长会话中，每一次文件读取和工具调用都会占用主会话的上下文窗口。当需要执行一个**独立的、探索性的、可能产生大量中间输出**的任务时，在主会话内完成会带来三个问题：

1. **上下文膨胀**：搜索/读取大量文件的结果占据 token，挤压了核心业务逻辑的空间
2. **信息干扰**：跨模块的文件内容混入主会话，使 AI 的注意力从当前任务偏移
3. **无法并发**：串行执行的子任务拉长了交互等待时间

`/agent` 将这类任务剥离到独立窗口，执行完毕只带回结论，不带回中间过程。

## 1.2 基本用法

### 1.2.1 内联语法

在最简单的形式中，直接在对话中输入：

```plaintext
/agent 请搜索项目中所有使用 pthread 的地方，整理出调用模式
```

子 Agent 会收到这个指令并开始独立执行。执行完成后，其最终输出会显示在主会话中。

### 1.2.2 传递背景信息

为了让子 Agent 更高效，应在 prompt 中包含充足的上下文：

```plaintext
/agent 我们的项目是一个 C++ 网络库，使用 CMake 构建。
请完成以下任务：
1. 读取 src/network/EventLoop.h 和 src/network/EventLoop.cpp
2. 分析 EventLoop 类的事件驱动机制
3. 与 src/network/Poller.h 中的 Poller 基类对比
4. 输出 EventLoop 的生命周期流程图（文字描述）

关键关注点：线程安全性、epoll 的使用方式、回调注册机制
```

> **原则**：prompt 越精确、背景越充分，子 Agent 的第一次产出质量越高。

### 1.2.3 关键参数

| 参数 | 说明 | 用法 |
|------|------|------|
| `description` | 任务描述（3-5 词） | 显示在 UI 中，便于识别多个子 Agent |
| `subagent_type` | Agent 类型 | 如 `"Explore"` 只读搜索、`"general-purpose"` 通用 |
| `run_in_background` | 后台执行 | `true` 时子 Agent 在后台执行，不阻塞主会话 |
| `isolation` | 隔离级别 | `"worktree"` 创建独立 git worktree，避免文件修改冲突 |

## 1.3 使用场景

### 1.3.1 代码库搜索（只读探索）

在长会话中，临时搜索某个函数定义或调用关系，不希望搜索过程占满上下文：

```plaintext
/agent 请在项目中搜索所有调用 Logger::Write 的地方，
列出文件路径和行号，并按调用方式分类（同步写入 / 异步写入 / 批量写入）。
```

可配合 `subagent_type: "Explore"` 限制只读操作，避免意外修改。

### 1.3.2 多文件阅读与分析

需要同时了解多个文件的逻辑关系时，交给子 Agent 整合：

```plaintext
/agent 请阅读以下文件，输出 Config 模块的架构总结：
- src/config/ConfigManager.h
- src/config/ConfigManager.cpp
- src/config/ConfigParser.h
- src/config/ConfigParser.cpp

输出格式：
1. 类关系图（文字描述）
2. 配置加载流程（步骤列表）
3. 关键设计决策（为什么选择这种方式）
```

### 1.3.3 代码审查（Code Review）

对当前分支的改动进行审查，不影响主会话中的开发思路：

```plaintext
/agent 请对当前分支的改动进行代码审查。
关注点：是否引入了线程安全问题？是否有内存泄漏风险？
接口设计与现有模块是否一致？
```

### 1.3.4 自动化测试生成

```plaintext
/agent 根据 src/parser/JsonParser.h 中的接口定义，
生成一套完整的 Google Test 测试用例：
1. Happy path：标准 JSON 输入
2. 边界条件：空对象、嵌套深度 100、超大数组
3. 错误路径：格式错误、截断输入、非 UTF-8 编码

测试文件输出到 tests/parser/json_parser_test.cpp
```

### 1.3.5 后台执行独立任务

使用 `run_in_background: true` 将任务放入后台，主会话可继续其他工作：

```plaintext
/agent 请后台分析 src/network/ 目录下所有文件的性能热点，
输出优化建议报告。
```

后台子 Agent 完成后会通知主会话，结果仍会合并回来。

### 1.3.6 隔离环境的安全操作

使用 `isolation: "worktree"` 让子 Agent 在独立的 git worktree 中操作：

```plaintext
/agent isolation: "worktree" 请在独立环境中尝试重构 TimerQueue 类，
不破坏主分支代码。
```

子 Agent 的修改与主工作区隔离，审查后可选择性地合并。

## 1.4 子 Agent 类型

| 类型 | 适用场景 | 工具 |
|------|----------|------|
| `Explore` | 只读搜索、查阅，不修改代码 | 所有除 Edit/Write 外的工具 |
| `general-purpose` | 通用任务，可读写文件 | 所有工具 |
| `Plan` | 软件架构设计，返回实施方案 | 除 Edit/Write/ExitPlanMode 外的所有工具 |
| `claude-code-guide` | Claude Code 功能问答 | Glob/Grep/Read/WebFetch/WebSearch |

选择合适类型可以限制子 Agent 的能力范围，降低误操作风险。

## 1.5 与主会话的关系

### 1.5.1 上下文隔离

子 Agent **不自动继承**主会话的：
- 已读取的文件内容
- 对话历史
- 环境状态（环境变量、工作目录变更）

需要通过 prompt 显式传递必要的背景信息。

### 1.5.2 结果回收

子 Agent 的**最终输出消息**（纯文本）合并回主会话。这意味着：
- 子 Agent 不能像工具调用那样返回结构化的数据
- 子 Agent 产生的中间文件（如生成的报告）仍在磁盘上，主会话可以后续读取
- 如果子 Agent 修改了文件，主会话可以看到修改（除非使用了 isolation worktree）

### 1.5.3 通信与协调

在主会话中可以通过 `SendMessage` 与运行中的子 Agent 通信：

```plaintext
当主 Agent 需要向特定的子 Agent 发送额外指令或信息时，使用：
to: "<agent-name>"
message: "请额外关注 ConfigManager 的单例线程安全问题"
```

子 Agent 之间**不能直接通信**——需要通过主会话中转。

## 1.6 最佳实践

### 1.6.1 任务粒度

| 任务类型 | 粒度建议 | 示例 |
|----------|----------|------|
| 搜索 | 独立发送 | "搜索所有调用某函数的地方" |
| 阅读分析 | 3-5 个文件 | "分析 A、B、C 类的继承关系" |
| 代码生成 | 单模块 | "为 ConfigParser 生成单元测试" |
| 重构 | 单一组件 | "重构 TimerQueue，保持接口不变" |

> **原则**：每个子 Agent 只做一件原子性的事情。任务过大时应进一步拆解。

### 1.6.2 Prompt 设计

一个好的子 Agent prompt 包含：

1. **项目背景**：项目类型、技术栈、构建系统
2. **具体目标**：要达成什么结果
3. **输出格式**：按什么结构输出结果
4. **约束条件**：不要做什么、注意什么

```plaintext
/agent 项目背景：C++ 网络库，使用 CMake + C++17。
目标：分析 src/buffer/Buffer.cpp 中的内存分配模式。
输出格式：按函数列出主要分配点（行号、大小、频率）。
约束：不考虑单元测试代码。
```

### 1.6.3 并行使用

同时派发多个不相关的子 Agent，最大化利用并发：

```plaintext
/agent 任务一：搜索登录模块的所有 API 入口
/agent 任务二：分析数据库连接池的实现
/agent 任务三：生成用户模块的单元测试骨架
```

每个子 Agent 独立执行，主会话等待所有结果后整合。

### 1.6.4 与上下文管理配合

- **大项目初期**：用 `/agent` 探索代码结构，将结果带回主会话作为基础认知
- **长会话中期**：将高消耗的搜索/验证任务剥离到子 Agent
- **切换模块时**：用 `/compact` 压缩主会话，同时用 `/agent` 并行探索目标模块
- **与 [[E-AI与Agent协同开发/02.工作流与方法论/3. Claude Code 上下文管理|上下文管理]]** 中的 `/compact` 配合，实现"主会话专注逻辑 + 子 Agent 负责探索"的分工模式

## 1.7 局限与注意事项

| 局限 | 说明 | 应对方式 |
|------|------|----------|
| **纯文本结果** | 子 Agent 的输出是纯文本，而非结构化 JSON | 在 prompt 中要求固定格式，便于后续解析 |
| **无增量结果流** | 子 Agent 完成后一次性返回完整输出 | 对长任务设置耐心等待 |
| **不继承上下文** | 需在 prompt 中传递背景 | 养成写完整 prompt 的习惯 |
| **并发上限** | 同时运行的子 Agent 数量受限 | 避免一次性派发太多子任务 |
| **成本** | 每个子 Agent 消耗独立的 token 配额 | 仅在确实需要隔离时使用 |

## 1.8 与多 Agent 协作体系的关系

`/agent` 是 Claude Code 中实现多 Agent 协作的具体工具之一，与项目级的多 Agent 协作约定互为补充：

| 维度 | `/agent` | 项目级多 Agent 协作（AGENT.md） |
|------|----------|-------------------------------|
| 范围 | 单个 Claude Code 会话内 | 跨工具、跨会话、跨开发者 |
| 生命周期 | 单次会话，执行完毕即结束 | 项目级定义，持续有效 |
| 协调方式 | 主会话驱动 | CLAUDE.md / AGENT.md 规则约束 |
| 信息共享 | prompt 传递 | 文件系统、共享规则文件 |

详细的多 Agent 协作约定参见 [[E-AI与Agent协同开发/03.工具与配置/2. Claude Code Team 模式协作指南|Team 模式协作指南]]。
