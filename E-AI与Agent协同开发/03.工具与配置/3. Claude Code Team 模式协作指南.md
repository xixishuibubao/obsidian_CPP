# 1. Claude Code Team 模式协作指南

Claude Code 的 **Team 模式**（多 Agent 团队协作）通过 Workflow 编排多个 Agent，实现任务拆分、并行执行与结果聚合。相比单 Agent 的 `/agent` 指令，Team 模式解决的是**多角色、多视角、可编排**的复杂任务。

## 【核心精简提纲】

### 记忆主干

**单 Agent（/agent）** = 一个上下文、一个角色、串行到底；**Team 模式** = N 个 Agent 各司其职、可并行可串行、通过 SendMessage 通信，由 Workflow 脚本编排。

### 启动方式速查

| 模式 | 启动方式 | 适用场景 |
|------|----------|----------|
| `/agent` 单 Agent | Slash command | 一次性的独立子任务（搜索、分析） |
| Team 多 Agent | Workflow 脚本 + `agent()` / `pipeline()` / `parallel()` | 多步骤、多角色、需并行的复杂任务 |

### 核心要点清单

- **Team 不等于人多**——是「角色分离 + 编排逻辑」；每个 Agent 有独立 prompt 和工具集
- **通信机制**：Agent 之间通过 `SendMessage` 工具收发消息，不共享上下文
- **任务分配**：使用 TaskCreate/TaskUpdate/TaskList 做工作追踪
- **工作隔离**：`isolation: "worktree"` 为子 Agent 创建独立 git worktree，互不干扰
- **Workflow 脚本**（`.claude/workflows/`）是 Team 模式的控制层，用 JS 编排多 Agent
- **注意事项**：token 消耗翻倍、调试困难、通信延迟——小任务不值得用 Team

### 关键结论 / 流程

```
简单子任务 → /agent（单 Agent，轻量快速）
多步骤高质量任务 → Team 模式（Workflow 编排 + 多 Agent 并行 + 交叉验证）
编排入口 → .claude/workflows/ 目录下的 Workflow 脚本
Agent 通信 → SendMessage 工具（收/发 + 广播）
工作隔离 → isolation: "worktree"（并行 Agent 独立环境）
```

## 【完整拓展详情】

## 1.1 Team 模式概述

### 1.1.1 什么是 Team 模式

Claude Code 的 Team 模式是指通过 **Workflow 脚本** 编排多个 Agent 组成临时团队，协作完成一个复杂任务。每个 Agent 拥有独立的：

- **系统提示（prompt）**——定义角色和职责
- **可用工具集**——可以限制或扩展
- **上下文窗口**——独立 token 配额，互不干扰
- **工作目录**——可能通过 worktree 隔离获得独立文件系统视图

Team 模式不是"多个 Claude 同时聊天"，而是**有控制层的多 Agent 系统**——Workflow 脚本是控制层，负责定义阶段、派发任务、收集结果。

### 1.1.2 与单 Agent 模式的对比

| 维度 | `/agent` 单 Agent | Team 多 Agent |
|------|-------------------|---------------|
| 上下文 | 共享主会话上下文 | 每个 Agent 独立上下文 |
| 角色 | 单一角色 | 可定义不同角色（架构师/编码/测试） |
| 并行 | 串行执行 | 支持并行执行 |
| 通信 | 和用户对话 | Agent 间通过 SendMessage 通信 |
| 编排 | 无编排层 | Workflow 脚本作为编排层 |
| 隔离度 | 共享工作目录 | 支持 worktree 隔离 |
| 适用 | 搜索/分析/简单子任务 | 多步骤、多角色、需交叉验证的任务 |
| Token 消耗 | 一个上下文 | N 个上下文，消耗翻倍 |

## 1.2 创建团队：Workflow 脚本编排

### 1.2.1 Workflow 脚本基础

Team 模式的入口是放置在 `.claude/workflows/` 目录下的 JavaScript 文件。Workflow 脚本通过 `agent()`、`pipeline()`、`parallel()` 等 API 编排多个 Agent。

```javascript
// .claude/workflows/code-review-team.js
export const meta = {
  name: 'code-review-team',
  description: '多 Agent 团队代码审查：分别从架构、安全、性能三个角度 review',
  phases: [
    { title: '并行 Review', detail: '3 个 Agent 分别从不同角度审查' },
    { title: '汇总', detail: '汇总并交叉验证审查结果' },
  ],
}

// Phase 1: 并行执行三个独立的 Review Agent
phase('并行 Review')
const results = await parallel([
  agent('作为架构师，审查代码整体架构设计', {
    name: 'architect',
    prompt: '你是一位软件架构师，关注模块化、耦合度、扩展性...',
  }),
  agent('作为安全工程师，审查代码安全风险', {
    name: 'security',
    prompt: '你是一位安全工程师，关注注入、XSS、权限校验...',
  }),
  agent('作为性能工程师，审查代码性能瓶颈', {
    name: 'performance',
    prompt: '你是一位性能工程师，关注时间复杂度、内存泄漏、IO...',
  }),
])

// Phase 2: 汇总结果
phase('汇总')
await agent(`汇总以下三个维度的审查结果，合并为一份报告:
  架构: ${results[0]}
  安全: ${results[1]}
  性能: ${results[2]}`, {
  name: 'summarizer',
})
```

### 1.2.2 编排模式

Workflow 支持三种编排模式：

| 模式 | API | 行为 | 适用场景 |
|------|-----|------|----------|
| **串行** | `await agent(...)` | 一个 Agent 完成后下一个才启动 | 有依赖关系的步骤链 |
| **并行** | `await parallel([...])` | 多个 Agent 同时执行 | 无依赖的独立分析 |
| **流水线** | `pipeline(source, ...transforms)` | 数据流式经过多个处理阶段 | 批处理、ETL 类型任务 |

```javascript
// 串行模式：分析 → 生成 → 校验
await agent('步骤1: 分析需求')
await agent('步骤2: 生成代码')
await agent('步骤3: 校验输出')

// 并行模式：多角度同时分析
const [a, b, c] = await parallel([
  agent('角度A'),
  agent('角度B'),
  agent('角度C'),
])

// 流水线模式：数据流处理
const result = await pipeline(
  sourceData,
  transform1,  // 清洗
  transform2,  // 分析
  transform3,  // 格式化输出
)
```

### 1.2.3 Agent 定义参数

调用 `agent()` 时的关键参数：

```javascript
const myAgent = await agent(prompt, {
  name: 'researcher',        // Agent 名称，用于通信寻址
  model: 'sonnet',           // 可选：指定模型（sonnet/opus/haiku）
  prompt: '系统提示...',     // 覆盖默认系统提示
  isolation: 'worktree',    // 可选：使用独立 git worktree 隔离
  tools: ['Read', 'Edit'],  // 可选：限制可用工具
})
```

## 1.3 Agent 通信：SendMessage

### 1.3.1 通信机制

Team 模式中 Agent 之间的通信通过 **SendMessage 工具** 实现。这是 Agent 间唯一的通信通道：

```
┌──────────┐    SendMessage     ┌──────────┐
│ Agent A  │ ──────────────────▶│ Agent B  │
│          │                    │          │
│ 发送消息 │                    │ 接收消息 │
│ 给 B     │                    │ 处理并   │
│          │                    │ 回复     │
└──────────┘ ◀──────────────────└──────────┘
                SendMessage 回复
```

### 1.3.2 基本用法

```javascript
// Agent A 发送消息给 Agent B
await SendMessage({
  to: 'researcher',
  summary: '请求分析结果',
  message: '请分析 src/ 目录下的代码结构，输出模块依赖关系图',
})
```

```javascript
// Agent B 回复消息给 Agent A
await SendMessage({
  to: 'architect',
  summary: '返回分析结果',
  message: '分析完成。模块依赖关系如下: ...',
})
```

### 1.3.3 通信设计要点

| 要点 | 说明 |
|------|------|
| **单向通信** | SendMessage 是异步的——发送方发出消息后继续自己的工作，不阻塞等待回复 |
| **不共享上下文** | Agent 之间不共享对话历史，每次 SendMessage 携带完整上下文 |
| **消息中继** | Agent 需要自己决定何时发送消息、发给谁、是否等待回复 |
| **广播模式** | 发送方可以将同一消息发给多个 Agent（调用多次 SendMessage） |
| **不干扰主会话** | Agent 间的通信不出现在用户的主对话中 |

### 1.3.4 通信作为任务同步

Workflow 层面更可靠的任务同步方式是使用 **TaskCreate/TaskUpdate/TaskList**：

```javascript
// 在 Workflow 脚本中
const taskId = await TaskCreate({
  subject: '分析模块依赖',
  description: '分析 src/ 下各模块的依赖关系',
})

// Agent 完成任务后更新状态
await TaskUpdate({
  taskId: taskId,
  status: 'completed',
})
```

## 1.4 任务分配与并行执行

### 1.4.1 任务追踪体系

Team 模式使用内置的任务工具管理并行工作：

| 工具 | 用途 | 在 Team 中的角色 |
|------|------|------------------|
| `TaskCreate` | 创建任务项 | 分配工作任务 |
| `TaskGet` | 获取任务详情 | 检查前置条件 |
| `TaskUpdate` | 更新任务状态/所有者 | 声明/移交/完成任务 |
| `TaskList` | 列出所有任务 | 工作进度总览 |

### 1.4.2 典型分配流程

```
Workflow 脚本（控制层）
  │
  ├── 1. TaskCreate("分析模块A", owner: "agent-alpha")
  ├── 2. TaskCreate("分析模块B", owner: "agent-beta")
  ├── 3. TaskCreate("汇总结果", blockedBy: ["1", "2"])
  │
  ├── parallel([
  │     agent("分析模块A", { name: "agent-alpha" }),
  │     agent("分析模块B", { name: "agent-beta" }),
  │   ])
  │
  └── 4. agent("汇总结果")   ← 等前两个都完成才执行
```

### 1.4.3 Agent 的自主任务认领

Agent 收到消息后，可以自主查询任务列表并认领：

```javascript
// Agent 启动后
const tasks = await TaskList()
const myTask = tasks.find(t => t.owner === 'researcher' && t.status === 'pending')
if (myTask) {
  await TaskUpdate({ taskId: myTask.id, status: 'in_progress' })
  // ... 执行任务
  await TaskUpdate({ taskId: myTask.id, status: 'completed' })
}
```

## 1.5 工作流隔离：Worktree Isolation

### 1.5.1 为什么需要隔离

在多 Agent 并行执行时，如果所有 Agent 共享同一工作目录，会产生文件冲突：

- Agent A 修改 `src/module/a.cpp`，Agent B 同时修改同一文件
- Agent A 创建临时文件，Agent B 不知情地依赖它
- git 工作区状态不一致，一个 Agent 的改动被另一个覆盖

### 1.5.2 worktree 隔离机制

通过 `isolation: "worktree"` 为子 Agent 创建独立的 git worktree：

```javascript
// 每个子 Agent 获得独立的工作目录
await parallel([
  agent('分析并修改模块A', {
    name: 'agent-alpha',
    isolation: 'worktree',   // ← 自动创建独立 worktree
  }),
  agent('分析并修改模块B', {
    name: 'agent-beta',
    isolation: 'worktree',   // ← 另一个独立 worktree
  }),
])
```

每个 worktree 是：
- 从同一仓库创建出的独立工作副本
- 共享 git 对象库（不重复存储历史）
- 有独立的工作目录和暂存区
- Agent 之间的文件修改完全隔离

### 1.5.3 隔离级别选择

| 隔离级别 | 方式 | 适用场景 |
|----------|------|----------|
| **无隔离** | 共享工作目录 | 纯分析、只读任务 |
| **worktree 隔离** | `isolation: "worktree"` | 涉及文件修改的并行任务 |
| **远程隔离** | `isolation: "remote"` | 需要完全独立环境（安装依赖等） |

### 1.5.4 退出 worktree

Agent 完成任务后，Workflow 脚本可以通过 API 清理 worktree：

```javascript
// Workflow 脚本中清理
await ExitWorktree({ action: 'remove' })
```

## 1.6 团队协作最佳实践

### 1.6.1 角色设计原则

良好的 Team 模式角色设计遵循 **单一职责 + 明确边界**：

```
❌ 不理想的分工
  Agent A: "专家"（什么都做）
  Agent B: "助手"（也什么都做）
  → 角色重叠、边界模糊、结果冗余

✅ 推荐的分工
  Agent A: "架构师"——关注模块划分、接口设计
  Agent B: "编码实现"——按架构师方案实现代码
  Agent C: "测试验证"——编写测试并验证实现
  → 角色清晰、产出互补、可交叉验证
```

### 1.6.2 通信最小化

Agent 之间通信有成本（token + 延迟），应遵循以下原则：

- **能不通信就不通信**——优先通过 Workflow 脚本做数据传递
- **批量传输**——一次发送完整结果，而非多条小消息
- **结构化格式**——使用 JSON 等结构化格式而非自然语言描述
- **只传结果**——不传中间思考过程

### 1.6.3 错误处理策略

```javascript
// 在 Workflow 脚本中做容错
try {
  const result = await parallel([
    agent('分析模块A', { name: 'alpha' }),
    agent('分析模块B', { name: 'beta' }),
  ])
} catch (error) {
  // 单个 Agent 失败不影响其他 Agent
  console.error('部分 Agent 失败:', error)
  // 重新调度失败的子任务
  const retryResult = await agent('重新分析模块A', { name: 'alpha-retry' })
}
```

### 1.6.4 什么时候不该用 Team 模式

| 情况 | 更好的方式 | 原因 |
|------|-----------|------|
| 简单文件搜索 | `/agent` 或直接问 | Team 的编排成本远高于收益 |
| 单步骤任务 | 主会话直接对话 | Workflow 脚本 + 多 Agent 是过度设计 |
| 需要频繁交互确认 | Skill（交互式） | Team 模式是批处理，不适合需要"人点头"的流程 |
| Token 预算紧张 | 单 Agent | N 个 Agent = N 倍 token 消耗 |
| 快速原型 | 直接编码 | Team 的编排时间比编码时间还长 |

### 1.6.5 适用场景推荐

| 场景 | 推荐模式 | 理由 |
|------|----------|------|
| 代码审查（多角度） | Team + parallel | 架构师/安全/性能 3 个 Agent 并行 |
| 复杂重构 | Team + pipeline | 分析 → 重构 → 测试 流水线 |
| 批量文件处理 | Team + parallel + worktree | 每个文件一个 Agent，互不干扰 |
| 知识库批量摄入 | Team + parallel | 多篇笔记同时处理 |
| 交叉验证实验 | Team + parallel | 多个 Agent 独立分析同一问题后投票 |
| 代码生成 + 测试 | Team + pipeline | 先生成再测试，形成闭环 |

## 1.7 完整示例：多 Agent 协作评审流程

```javascript
// .claude/workflows/team-demo.js
export const meta = {
  name: 'team-demo',
  description: '演示 Team 模式：分析 + 审查 + 测试 流水线',
  phases: [
    { title: '需求分析', detail: '架构师分析需求文档' },
    { title: '并行审查', detail: '安全和性能并行审查' },
    { title: '集成测试', detail: '测试 Agent 验证修改' },
  ],
}

// Phase 1: 架构分析
phase('需求分析')
const architecture = await agent(
  '分析需求文档 requirements.md，输出: 1) 模块划分 2) 接口设计 3) 数据流',
  { name: 'architect', isolation: 'worktree' }
)

// Phase 2: 安全和性能并行审查
phase('并行审查')
const [securityReview, perfReview] = await parallel([
  agent(`基于以下架构方案做安全审查: ${architecture}`, {
    name: 'security-engineer',
    isolation: 'worktree',
  }),
  agent(`基于以下架构方案做性能审查: ${architecture}`, {
    name: 'performance-engineer',
    isolation: 'worktree',
  }),
])

// Phase 3: 汇总并生成测试
phase('集成测试')
await agent(`
  汇总以下三份报告生成最终文档:
  架构: ${architecture}
  安全: ${securityReview}
  性能: ${perfReview}

  输出:
  1. 最终架构文档
  2. 待修复问题清单
  3. 测试要点
`, { name: 'summarizer' })
```
