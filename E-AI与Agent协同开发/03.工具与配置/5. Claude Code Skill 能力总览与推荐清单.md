# 1. Claude Code Skill 能力总览与推荐清单

面向 C++/Python 开发者的 Claude Code Skill 使用指南，涵盖全部内置 Skill 与自定义 Skill 的分类说明、优先级推荐、及串联工作流。

## 【核心精简提纲】

### Skill 全景图

- **自定义 Skill x6**（`myskills/`）— 针对本知识库仓库的维护工具链
- **Superpowers x14** — Claude Code 高阶工作流的核心能力（规划/执行/审查/调试）
- **开发工具 x6** — 调试、代码审查、TDD、原型、安全
- **项目管理 x5** — 计划拆解、Issue 管理、深度审查
- **配置工具 x5** — 权限、快捷键、环境初始化

### C++/Python 开发者精选（★ 强烈推荐）

| 类别 | Skill | 场景 |
|------|-------|------|
| ★ 设计先行 | `brainstorming` | 写代码前探索设计 |
| ★ 计划驱动 | `writing-plans` | 多步骤实现前写计划 |
| ★ TDD | `tdd` | C++/Python 红-绿-重构 |
| ★ 并行开发 | `subagent-driven-development` | 同时推进 C++ + Python 任务 |
| ★ 诊断 bug | `diagnose` | 段错误/内存泄漏/异常系统化定位 |
| ★ 代码审查 | `code-review` | 提交前 diff 自动化审查 |
| ★ 安全审查 | `security-review` | C/C++ 缓冲区溢出/Python 注入 |
| ★ 验证先行 | `verification-before-completion` | 完工前跑验证再下结论 |
| ★ Worktree | `using-git-worktrees` | 多分支并行隔离 |
| ★ 深度调研 | `deep-research` | 新技术栈多源调研 |

### 知识库维护链（本仓库专用）

```
收集 → /ingest-note → 清洗格式化入库
提交 → /quick-commit → 自动分析 diff 并 commit
整理 → /squash-commits → 推送前 squash 备份提交
```

## 1.1 背景

本知识库的 Claude Code 环境包含两套 Skill：

1. **内建 Skill**（Claude Code 内置）— 通用开发能力
2. **自定义 Skill**（`myskills/` 目录）— 针对本 Obsidian 知识库的专用工具链

以下逐一说明用途，并面向 C++/Python 开发者给出推荐优先级。

## 1.2 自定义 Skill（myskills/ 目录）

这批 Skill 是为本知识库仓库量身定做的维护工具。

| Skill | 定位 | 用途 |
|-------|------|------|
| `ingest-note` | **核心** | 清洗原始知识文本 → 格式化 Markdown → 自动定位目录 → 更新索引/README → 删除源文件 → 生成 Conventional Commit |
| `quick-commit` | **核心** | 自动 stage 所有改动 → 分析 diff → 检测外部手动修改（如 Obsidian 编辑）→ 生成 commit message → 确认后执行 |
| `squash-commits` | **常用** | 推送前展示本地 commits 列表，自动检测 obsidian-git 备份提交，选择策略后 squash |
| `init-note-vault` | 一次性 | 为 Obsidian/Markdown 知识库初始化笔记规范（标题编号、wikilink、代码块、双语写作等） |
| `init-git-convention` | 一次性 | 注入 Conventional Commits 工作流规范 + 权限预设 |
| `init-win-env` | 一次性 | 在新机器上探测 Git Bash/PowerShell 路径，生成 Shell 规则与权限 |

> 本知识库笔记维护的三板斧：`/ingest-note`（摄入）→ `/quick-commit`（提交）→ `/squash-commits`（推送前整理）。

## 1.3 Superpowers 系列（开发核心）

Claude Code 最高阶的能力集合，通过 `/superpowers:` 前缀调用，覆盖从创意到交付的全流程。

### 1.3.1 规划与设计

| Skill | 场景 | C++/Python 推荐度 |
|-------|------|:---:|
| `brainstorming` | 任何创意/功能开发前：探索用户意图、需求、设计。强制在写代码前先想清楚 | ★★★ |
| `writing-plans` | 拿到需求后：产出结构化的实现计划，识别关键文件和架构决策 | ★★★ |
| `using-superpowers` | 新会话起点：快速查有哪些 Superpowers 可用 | ★★☆ |
| `writing-skills` | 需要自定义 Skill 时：创建/编辑/验证 | ★★☆ |
| `using-git-worktrees` | 多分支并行开发时：创建隔离工作树，互不干扰 | ★★★ |

### 1.3.2 执行与协作

| Skill | 场景 | 推荐度 |
|-------|------|:---:|
| `executing-plans` | 已有写好的计划，在隔离会话中按步骤执行，含审查检查点 | ★★★ |
| `subagent-driven-development` | 多个独立实现任务时：派生子 Agent 并行推进，当前会话继续做其他事 | ★★★ |
| `dispatching-parallel-agents` | 遇到 2 个以上无依赖的任务时并行处理（如同时写 C++ 后端 + Python 脚本） | ★★★ |
| `finishing-a-development-branch` | 实现完成后：提供合并/PR/清理等收尾选项，引导决定 | ★★☆ |

### 1.3.3 代码质量与验证

| Skill | 场景 | 推荐度 |
|-------|------|:---:|
| `requesting-code-review` | 完成主要功能/合并前：要求审查是否满足需求 | ★★★ |
| `receiving-code-review` | 收到审查反馈后：严谨验证再动手，不盲从建议 | ★★★ |
| `verification-before-completion` | 声称完成/修复前：先跑验证命令、确认输出，证据先于断言 | ★★★ |
| `systematic-debugging` | 遇到 Bug/测试失败/异常行为时：复现→最小化→假说→验证→修复→回归，阻止乱试 | ★★★ |
| `test-driven-development` | 实现任何功能前：先写测试，红-绿-重构循环 | ★★★ |

## 1.4 开发专用 Skill

| Skill | 用途 | C++/Python 推荐度 |
|-------|------|:---:|
| `diagnose` | **硬 Bug/性能回归的诊断循环** — 再现→最小化→假说→仪表→修复→回归测试。C++ 段错误/内存泄漏/未定义行为 + Python 异常/性能瓶颈的首选 | ★★★ |
| `code-review` | **审查当前 diff** — 正确性 bug + 可复用性/效率/简化的清理建议。支持 `--comment`（Inline 评论）, `--fix`（直接应用修复） | ★★★ |
| `security-review` | **安全审查待推送变更** — C/C++ 的缓冲区溢出/未初始化内存、Python 注入/反序列化、以及对 AI 生成代码的额外关注 | ★★★ |
| `tdd` | **红-绿-重构** — 比 Superpowers TDD 更专注循环本身，适合纯 TDD 场景 | ★★★ |
| `simplify` | **代码简化** — 从改动的代码中发现复用/简化/效率优化的机会并直接应用修复。不找 bug，纯质量改进 | ★★☆ |
| `prototype` | **快速原型** — 两个分支：可运行终端应用（验证状态/业务逻辑），或多种 UI 变体。适合 C++ 算法验证 / Python 数据分析试探 | ★★☆ |
| `verify` | **运行项目确认改动生效** — 适合在本地跑二进制/测试确认修复有效 | ★★☆ |
| `review` | **审查 Pull Request** — PR 级别的 code review | ★★☆ |
| `loop` | **定时重复执行** — 如每 5 分钟检查 CI 状态、轮询部署进度 | ★☆☆ |

## 1.5 项目管理 Skill

| Skill | 用途 | 场景 |
|-------|------|:---:|
| `to-prd` | 将对话转化为 PRD 并发布到 Issue Tracker | 有明确的 feature 需求需文档化 |
| `to-issues` | 将计划/规格按 tracer-bullet 垂直切片拆分为独立 Issues | 大型功能拆分任务 |
| `triage` | Issue 状态机分类管理（三角色驱动） | 维护开源项目 / 处理 Issue 队列 |
| `grill-me` | 对你的计划/设计穷追猛打，直到达成共识 | 重大决策前压力测试 |
| `grill-with-docs` | 结合项目已有文档 + 术语的审查，边审查边更新文档 | 需要对齐项目领域模型的审查 |
| `write-a-skill` | 创建新的内置风格 Skill（包含渐进式打开和资源打包） | 需要通用 Skill 时 |
| `skill-creator:skill-creator` | 创建/编辑/优化 Skill，含 Eval 评测与方差分析 | 已有 Skill 需迭代优化 |

## 1.7 C++/Python 开发者推荐工作流

### 日常开发循环

```text
/superpowers:brainstorming          # ① 理清需求与设计
  ↓
/superpowers:writing-plans           # ② 产出实现计划
  ↓
/grill-me                            # ③ 压力测试计划（可选，重大功能）
  ↓
/prototype                           # ④ 快速验证核心设计（可选）
  ↓
/superpowers:using-git-worktrees     # ⑤ 创建工作树隔离开发
  ↓
/tdd 或 /superpowers:test-driven-development  # ⑥ 测试驱动实现
  ↓
/superpowers:verification-before-completion    # ⑦ 验证通过再声称完成
  ↓
/code-review                         # ⑧ 审查 diff 质量
  ↓
/security-review                     # ⑨ 安全审查（涉及 I/O/网络时必做）
  ↓
/superpowers:finishing-a-development-branch    # ⑩ 收尾合并/PR
```

### 遇到 Bug 时

```text
/diagnose                            # 系统化诊断循环
  ↓
/superpowers:systematic-debugging    # 备选/补充：复现→最小化→假说→验证
```

### 学习新技术时

```text
/deep-research                       # 多源深度调研 + 引用报告
  ↓
/ingest-note                         # 将学习成果格式化入库（本知识库）
  ↓
/prototype                           # 动手验证理解
```

### 本知识库维护

```text
/ingest-note                         # 清洗原始文本 → 格式化入库
/quick-commit                        # 一键提交（含 AI + 手动改动检测）
/squash-commits                      # 推送前 squash 整理
```

## 1.8 Skill 间关系速查

| 类别 | 进入方式 | 推荐链式组合 |
|------|----------|------------|
| 新功能开发 | `brainstorming` | → `writing-plans` → `using-git-worktrees` → `tdd` → `code-review` |
