# 1. /agent 与 Team 模式对比与效能优化

对比 Claude Code 中 `/agent` 命令（单 Agent）与 Team 模式（多 Agent Workflow）的差异、适用场景，以及如何混合使用提升效能。

## 【核心精简提纲】

### 记忆主干

**`/agent`** = 一次启动 1 个子 Agent，通过主会话通信，适合独立子任务。**Team 模式** = 通过 workflow 编排多个 Agent，支持 pipeline/parallel，适合复杂多步骤流水线。两者是互补关系：`/agent` 做探索与单点执行，Team 做编排与批量推进。

### 选型速查

| 维度 | `/agent` | Team 模式 |
|------|----------|-----------|
| Agent 数量 | 单次启动 1 个 | Workflow 编排多个 |
| 通信方式 | 通过主会话 relay | 通过 SendMessage / pipeline |
| 并行能力 | 无内置并行 | pipeline / parallel 原生支持 |
| 上下文隔离 | 子 Agent 独立上下文 | worktree 隔离可选 |
| 适用任务 | 独立子任务 | 复杂多步骤工作流 |
| 交互性 | 可中途交互确认 | 批处理设计，无交互 |
| 启动门槛 | `/agent 任务描述` 一行命令 | 需设计 workflow / 配置多 Agent |
| 适用人数 | 单人多任务 | 团队 / 复杂流水线 |
| Token 控制 | 子 Agent 用完后销毁 | Budget 管理、checkpoint 恢复 |

### 核心要点清单

- **`/agent` 用于**：独立子任务、调研搜索、单文件修改、辅助问答
- **Team 模式用于**：多步骤流水线、批量文件处理、代码审查+验证、多角度分析
- **混合策略**：`/agent` 探索调研 → Team 批量执行；Workflow 中混用 `/agent` 做子任务
- **效能优化**：控制并行度避免 token 爆炸、worktree 隔离减少冲突、checkpoint 中断恢复
- **决策树**：单一线索不需交互 → `/agent`；多步骤/多角度需编排 → Team

### 关键结论 / 流程

```
任务评估：
  ┌─ 单步/独立/不需交互 → /agent（开箱即用）
  └─ 多步/多角度/需编排 → Team（design workflow → run）
                          └─ 混合：Team 中内嵌 /agent 做子任务探索
```

## 【完整拓展详情】

## 1.1 背景

在 Claude Code 的 Agentic Engineering 体系中，`/agent` 命令和 Team 模式代表了两种不同粒度的任务执行单元：

- **`/agent`**：一次启动一个独立的子 Agent（Sub-agent），拥有独立的上下文窗口，通过主会话（Main Session）进行任务分发和结果回收。适合单个明确目标的子任务。
- **Team 模式**：通过 workflow 编排多个 Agent，支持 `pipeline`（流水线）和 `parallel`（并行）模式，Agent 之间通过 `SendMessage` 直接通信，可选的 worktree 隔离让上下文彻底独立。

详见 [[E-AI与Agent协同开发/01.基础概念/1. 编程范式演进：Spec Coding → Agentic Engineering → Harness Engineering|编程范式演进]] 了解 Agentic Engineering 的宏观演进。

## 1.2 功能对比

### 1.2.1 差异对比表

| 维度 | `/agent` | Team 模式 |
|------|----------|-----------|
| **Agent 数量** | 单次启动 1 个子 Agent | Workflow 编排多个 Agent（可并行） |
| **通信方式** | 通过主会话 relay：主 Agent 分发 → 子 Agent 执行 → 结果回传给主 Agent | Agent 间通过 `SendMessage` 直接通信，或通过 pipeline 数据传递 |
| **并行能力** | 无内置并行。一个 `/agent` 执行完才能启动下一个 | `pipeline()` / `parallel()` 原生支持串行流水线和并行 fan-out |
| **上下文隔离** | 子 Agent 有独立上下文窗口，但与主会话共享项目文件认知 | 可选 worktree 隔离——每个 Agent 拥有独立的文件系统快照 |
| **适用任务** | 单步调研、独立文件修改、辅助问答、小范围重构 | 多步骤流水线、批量文件处理、多角度代码审查 + 验证 |
| **交互性** | 主会话可在子 Agent 执行后检查结果、继续追问 | 批处理设计——workflow 运行中无交互，运行后汇总报告 |
| **启动门槛** | `/agent 任务描述`——一行命令即可启动 | 需要设计 workflow（JavaScript 脚本），配置多 Agent 和通信协议 |
| **Token 控制** | 子 Agent 完成任务后上下文销毁，不占用主会话 token | 通过 budget 管理每阶段 token 上限，支持 checkpoint 中断恢复 |
| **状态持久化** | 无——子 Agent 完成后状态即释放 | 支持 checkpoint——可在失败时从中断点恢复 |
| **可观测性** | 子 Agent 输出在主会话可见 | 提供阶段进度、budget 消耗、每个 Agent 的独立日志 |

### 1.2.2 通信模型对比

```
/agent 通信模型：
  主 Agent ──→ /agent（子 Agent）──→ 结果回传主会话
              (无 peer 通信能力)

Team 模式通信模型：
  Orchestrator
    ├── pipeline: Agent A ──→ Agent B ──→ Agent C   (链式传递)
    ├── parallel:  Agent B₁ ──→ 汇总 ──→ Agent C    (fan-out)
    │              Agent B₂ ──→ 汇总
    │              Agent B₃ ──→ 汇总
    └── mesh:     Agent A ←──→ Agent B ←──→ Agent C  (通过 SendMessage)
```

## 1.3 适用场景分析

### 1.3.1 何时用 `/agent`

`/agent` 适合 **目标明确、边界清晰、只需一次执行** 的任务：

| 场景 | 示例 | 说明 |
|------|------|------|
| **单步调研** | `/agent 调研 Rust 的 Pin<Box<dyn Future>> 用法` | 子 Agent 独立搜索、整理，不污染主上下文 |
| **独立文件修改** | `/agent 将 src/utils.rs 中的错误处理从 panic 改为 Result` | 文件级改动，子 Agent 专注这一件事 |
| **辅助问答** | `/agent 解释这段 CMakeLists.txt 中 target_link_libraries 的 PUBLIC/PRIVATE 区别` | 旁路查询，不占用主会话上下文 |
| **小范围代码生成** | `/agent 在 src/下生成一个通用的 LRU Cache 实现` | 单输出的生成任务 |
| **知识库笔记查询** | `/agent 在 E-AI 目录下找关于 Vibecoding 的笔记并总结要点` | 搜索 + 摘要，子 Agent 完成后即销毁 |

使用 `/agent` 的好处是不会在主会话中留下大量中间步骤的 token 消耗。完成后结果以消息形式返回，主会话可选择是否深入讨论。

### 1.3.2 何时用 Team 模式

Team 模式适合 **多步骤、多角度、需要编排** 的任务：

| 场景 | 示例 | 说明 |
|------|------|------|
| **多步骤流水线** | 代码分析 → 生成测试 → 运行验证 → 输出报告 | pipeline 串联，每步专注一个职责 |
| **批量文件处理** | 对 raw/ 下 20 个文件并行格式化、清洗、写入知识库 | parallel 模式，每个 Agent 处理一个文件 |
| **多角度代码审查** | 同时从正确性、性能、安全三个角度 review | parallel fan-out，汇总投票 |
| **交叉验证** | 3 个 Agent 各自独立分析同一个 Bug，对比结论 | 消除单 Agent 偏见 |
| **代码审查+验证** | review + 修改建议 + 运行测试验证修改 | 自动化的修改-验证闭环 |
| **批量重构** | 重命名符号 → 更新引用 → 检查编译 → 格式化代码 | 多步骤连锁变更 |

### 1.3.3 决策路径

```
? 任务是否包含多个独立步骤或可并行的维度？
  ├── 否 → ? 是否需要主会话继续追问？
  │       ├── 是 → 直接在对话中完成（或考虑 Skill）
  │       └── 否 → /agent 启动一次子 Agent
  └── 是 → ? 需要人在中间确认/选择吗？
           ├── 是 → Skill（交互式）或拆分后手动触发
           └── 否 → Team 模式（workflow 编排）
```

## 1.4 混合使用策略

### 1.4.1 策略一：`/agent` 探索调研 → Team 批量执行

先用 `/agent` 做快速探索和方案验证，确定方向后再用 Team 模式批量落地。

```plaintext
Phase 1: /agent 调研（单次探索）
  /agent 调研现有知识库中所有关于 V4L2 的笔记，列出缺失的主题

Phase 2: Team 批量执行（基于调研结果）
  pipeline:
    Agent A: 根据调研结果生成笔记大纲
    Agent B: 并行处理各章节内容
    Agent C: 格式化和 wikilink 检查
    Agent D: 更新索引并 commit
```

### 1.4.2 策略二：在 Workflow 中混用 `/agent`

Team Workflow 中的某个步骤如果需要独立探索，可以启动一个 `/agent` 子任务：

```javascript
// 伪代码：Workflow 中混用 /agent
phase('分析代码结构')
const analysis = await agent('分析 src/trade 模块的架构')

phase('用 /agent 探测潜在问题')
const issues = await agent(`基于上述架构分析，用 /agent 模式分别检查:
  1. /agent 检查并发安全性
  2. /agent 检查内存泄漏风险
  3. /agent 检查错误处理完整性
  汇总结果`)

phase('生成修复方案')
await agent(`基于 issues 生成修复方案，按优先级排序`)
```

### 1.4.3 策略三：`/agent` 做快速辅助，Team 做主干流程

日常开发中，小问题随手用 `/agent` 解决，不打断 Team 的主流程：

```plaintext
# 主流程：Team 模式执行 CI 检查
pipeline:
  Agent A: 运行 linter
  Agent B: 运行单元测试
  Agent C: 检查测试覆盖率
  
# 并行辅助：/agent 顺手修复 lint 告警
/agent 修复 Agent A 报告中编号 W-003 和 W-007 的 lint 告警
```

### 1.4.4 策略四：Team 执行 → `/agent` 验证 → Team 继续

```plaintext
pipeline:
  Agent A: 批量修改 10 个文件的接口签名（Team 执行）
  
  # 人工/agent 验证点
  /agent 检查修改后的接口签名是否与调用方一致
  
  Agent B: 更新文档和测试（Team 继续）
```

## 1.5 效能优化建议

### 1.5.1 Token 控制

| 问题 | 优化方式 |
|------|----------|
| Team 模式 token 消耗快 | 为每个阶段设置 `budget` 上限 |
| 子 Agent 上下文膨胀 | 用 `.claudeignore` 前置过滤，避免加载无关文件 |
| 大量并行导致 token 峰值高 | 控制 `parallel` 的并发数（建议 3~5 个为上限） |
| 任务中断后重头开始 | 使用 workflow 的 `checkpoint` 机制，从中断点恢复 |

### 1.5.2 并行度调整

- **CPU 密集型**（编译、静态分析）：并行度 = min(文件数, 核心数 - 1)
- **IO 密集型**（网络请求、文件读写）：可适当提高并行度（8~12）
- **LLM 密集型**（代码生成、review）：建议并行度 3~5，避免 API 限流

```javascript
// 控制并行度的示例
const BATCH_SIZE = 4

phase('批量 review')
for (let i = 0; i < files.length; i += BATCH_SIZE) {
  const batch = files.slice(i, i + BATCH_SIZE)
  await parallel(
    ...batch.map(f => agent(`review ${f}`))
  )
}
```

### 1.5.3 Worktree 隔离

在 Team 模式中使用 worktree 隔离可以减少 Agent 间的文件冲突：

```plaintext
# 每个 Agent 在独立 worktree 中工作
Agent A: worktree 1 → 修改模块 A
Agent B: worktree 2 → 修改模块 B
Agent C: worktree 3 → 修改模块 C
汇总 Agent: 合并三个 worktree 的改动
```

**适用场景**：
- 多个 Agent 同时修改同一仓库
- 需要对比不同方案（每个 worktree 一个方案）
- 需要保持主仓库干净的读操作（review、分析）

### 1.5.4 上下文隔离原则

| 层级 | 隔离手段 | 适用 |
|------|----------|------|
| **项目级** | `.claudeignore` 前置过滤 | 排除 assets/log/ 等无关目录 |
| **任务级** | `/agent` 独立上下文 | 子任务不污染主会话 |
| **Agent 级** | Team 模式独立上下文窗口 | 每个 Agent 只看自己需要的文件 |
| **文件系统级** | worktree 隔离 | 彻底独立的文件快照 |
