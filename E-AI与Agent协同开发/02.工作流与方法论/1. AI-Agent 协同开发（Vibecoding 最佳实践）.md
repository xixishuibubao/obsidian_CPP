> **Vibecoding**（共鸣编码）：人定 What & Why，Agent 做 How & Explore。五阶段结构化交付，目标 L3/L4 工程型协作。

# 1. 概述：从灵感到交付的 Agentic 工作流

## 【核心精简提纲】

### 记忆主干

```plain
想法 → 压力测试 → PRD → 细节补充 → 设计定调 → 分步实现（commit/测试/重构）
```

人负责方向与判断；Agent 负责探索、生成、迭代。核心不是跑得更快，而是**不跑偏地更快**。

### 核心要点清单

- **阶段 1 压力测试**：adversarial 质疑可行性、风险、MVP 范围（garbage in → garbage out 放大）
- **阶段 2 PRD**：Plan mode 生成 → 人逐段纠正 Agent 过度推断
- **阶段 3 细节**：grill-me 式追问边界；决策记 ADR
- **阶段 4 定调**：AGENT.md / 风格样本 / vibe review
- **阶段 5 交付**：小步 commit + 测试金字塔 + debug 循环 + 中期重构
- **成熟度**：L1 冲动 → L4 人机分工明确

### 学习路径

1. 读 [[E-AI与Agent协同开发/01.基础概念/1. 编程范式演进：Spec Coding → Agentic Engineering → Harness Engineering|编程范式]] 定位 Agentic 层
2. 按五阶段走完一个小 feature
3. 配合 [[E-AI与Agent协同开发/02.工作流与方法论/2. AI如何辅助个体成为超级程序员|模块化 PRD 路线]]
4. 版本管理见 [[F-版本管理/2. log规范|Commit 规范]] + [[F-版本管理/1. git|Git 基础]]

### 关键结论 / 流程

| 阶段 | 退出标准（摘要） |
|------|------------------|
| 压力测试 | 3 轮 adversarial + 2 个风险预案 + MVP 清晰 |
| PRD | 有 No-go zone + 可度量验收标准 |
| 细节 | 接口格式、错误码、边界条件已定 |
| 定调 | AGENT.md + 风格样本就位 |
| 交付 | 每 step 有 commit + 测试 |

## 【完整拓展详情】
Vibecoding 不是"有了想法就开写"，而是**结构化的人机协作**。核心契约是：**人负责 What & Why，Agent 负责 How & Explore**。

全流程分为五个阶段，形成一个闭环：

```plain
想法 → [压力测试] → [PRD 生成] → [细节补充] → [定调] → [迭代实现]
                                                          ↕
                                                  版本管理 · 测试 · 重构
```

| 阶段 | 核心目标 | 典型工具 / Skill |
|------|----------|-----------------|
| 1.1 压力测试 | 验证想法的合理性与逻辑严密性 | `startup`、`/plan`、adversarial agent |
| 1.2 PRD 对齐 | 生成产品需求文档并交互修正 | Claude Code Plan mode、`/spec` |
| 1.3 细节补充 | 补全边缘情况、技术选型、约束 | `grill-me`、Q&A agent |
| 1.4 设计定调 | 明确项目风格、架构风味 | 风格指南、architectural decision records |
| 1.5 迭代交付 | 分步实现 + 测试 + 重构 | Git、TDD、CI、refactoring agent |

# 2. 阶段一：压力测试 — 先质疑，再投入

## 2.1 为什么需要压力测试

AI-Agent 容易产生 **garbage in, garbage out** 的放大效应：一个模糊但有漏洞的想法，经过 Agent 自动补全后会变成看似合理实则错误的实现。压力测试的目的就是在投入代码之前，用最小的成本把逻辑漏洞暴露出来。

## 2.2 实操方法

### 2.2.1 使用 Startup Skill 做快速验证

```bash
# 假设你已配置了 startup skill
/startup "我想做一个基于 FUSE 的版本化文件系统，允许按时间戳回溯文件状态"

# Agent 会产出：
# - 可行性分析（技术上是否可做）
# - 风险点列举（性能瓶颈？POSIX 兼容性？）
# - 已有替代方案对比（git annex? Nix?）
# - 推荐的技术栈选型
```

Startup 类 skill 通常内置了 **adversarial review**（对抗性审视），自动从多个角度攻击想法的薄弱环节：

- **技术可行性**：所选方案是否成熟，依赖是否可获取
- **逻辑一致性**：目标之间是否存在矛盾（如"实时同步"和"零带宽占用"）
- **范围合理性**：MVP 是否最小可行，是否会陷入 scope creep
- **已有生态**：社区有没有现成方案可以站在肩膀上

### 2.2.2 手动压力测试框架

如果当前环境没有合适的 skill，可以用以下 prompt 框架手动做压力测试：

```plain
请你扮演一个 adversarial reviewer，从以下角度质疑我的想法：

【想法描述】：<一句话描述>

请分析：
1. 这个想法最大的三个风险是什么？
2. 在什么情况下这个想法会失败？
3. 如果资源减半，还能做出有价值的东西吗？
4. 已有生态系统中有没有替代方案？为什么不直接用？
5. 目标的某两个子目标之间是否存在矛盾？
```

### 2.2.3 压力测试的退出标准

- [ ] 想法的核心逻辑经得起 3 轮 adversarial 追问
- [ ] 识别出至少 2 个关键风险并有应对预案
- [ ] 确认 MVP 范围清晰，不存在"先做出来再看"的模糊地带
- [ ] 和已有方案相比有明确的差异化价值

> 通过压力测试并不意味着想法一定可行，而是确保在进入 PRD 阶段前，你已经**知道自己不知道什么**。

# 3. 阶段二：PRD 生成与对齐

## 3.1 从想法到文档

通过压力测试后，进入需求工程阶段。核心目标是生成一份结构化的 **PRD（Product Requirements Document）**，作为后续所有工作的唯一真相源（single source of truth）。

### 3.1.1 使用 Plan Mode 生成 PRD

```plain
/plan "基于以下想法生成 PRD：<经过压力测试的最终描述>"
```

Claude Code 的 Plan mode 会产出包含以下结构的文档：

```plain
# PRD: <项目名称>

## 1. 背景与目标
- 解决什么问题
- 目标用户
- 成功指标

## 2. 功能需求
- MVP 功能列表（Must Have）
- 迭代二功能（Should Have）
- 未来功能（Nice to Have）

## 3. 非功能需求
- 性能指标
- 安全要求
- 兼容性要求

## 4. 技术选型
- 推荐方案及理由
- 被排除的方案及原因

## 5. 约束与假设
- 已知约束
- 外部依赖

## 6. 里程碑
- 阶段划分与交付物
```

### 3.1.2 交互修正：对齐意图

Agent 生成的 PRD 大概率会**偏离原始意图** — 这是正常现象，因为 Agent 会自动补全你未明确表达的部分。关键步骤是逐段审查并交互修正：

```plain
# 示例修正交互

你：PRD 第 2.3 节提到"支持实时同步"，但我们的 MVP 只需要手动触发同步。
    请改为"手动触发 + 定时任务"。

Agent：已修正。同时我注意到第 3.2 节的性能指标中
       "写入延迟 < 50ms" 在手动模式下不再适用，是否需要同步调整？

你：好问题。改为"定时同步延迟 < 5s"。
```

这轮对话的价值在于：**人纠正 Agent 的过度推断，Agent 补充人未考虑到的关联影响**。

### 3.1.3 PRD 对齐检查清单

- [ ] 每个功能条目是否都对应到一个具体的用户故事
- [ ] "不做什么"是否明确列出（No-go zone）
- [ ] 验收标准是否可度量（而非"用户体验好"这类模糊表述）
- [ ] 非功能需求是否与 MVP 规模匹配（不要 MVP 阶段就要求 99.999% 可用性）
- [ ] 每个参与者（包括未来的你）看过后都能理解项目范围

# 4. 阶段三：细节补充

## 4.1 为什么需要专门的细节补充阶段

PRD 定义了"做什么"，但 agent 级别的实现需要更细粒度的上下文：输入输出格式、错误处理策略、日志规范、数据模型字段设计等。这些细节在 PRD 中逐一罗列会使其臃肿，但缺失又会导致 Agent 在实现时自由发挥，造成前后不一致。

## 4.2 使用 Grill-Me 类 Skill 做地毯式追问

```plain
/grill-me "针对这个 FUSE 版本化文件系统项目，请深入挖掘
           我可能忽略的细节问题"
```

Grill 类工具会从实现者的角度提出尖锐问题：

```plain
Q1: 文件 overwrite 时，是创建新版本还是覆盖当前版本？
    → 影响：决定了 version model 的设计

Q2: 目录重命名时，子文件的历史版本如何处理？
    → 这是 FUSE 实现中最常见的坑

Q3: 多进程同时写入一个文件时的并发模型是什么？
    → last-write-wins？还是 transaction？

Q4: 元数据（权限、owner）是否版本化？
    → 大部分文件系统版本工具只版本化内容，忽略元数据

Q5: 磁盘空间满了之后的行为是什么？
    → 阻塞写入？丢弃最旧版本？报警？
```

## 4.3 细节文档化

每个细节决策应当记录为 **ADR（Architecture Decision Record）** 或直接更新到 PRD 的附录：

```markdown
### ADR-001: 版本模型采用 copy-on-write

**上下文**：文件被 overwrite 时需要保留上一版本
**决策**：采用 copy-on-write + 增量存储
**后果**：写入性能降低约 30%，但存储效率提高
**替代方案**：全量快照（存储膨胀）、diff-based（实现复杂）
```

> ADR 的格式参考 [[5. UML语法|架构决策记录]] 的轻量化方案。

## 4.4 细节补充的退出标准

- [ ] 所有外部接口的输入输出格式已定义
- [ ] 错误码和异常处理策略已确定
- [ ] 数据模型的核心字段已定稿
- [ ] 边界条件（空文件、超大文件、并发写入、磁盘满）有处理方案
- [ ] 与 PRD 无矛盾之处

# 5. 阶段四：设计定调

## 5.1 项目的"风味"

Design Vibe（设计基调 / 风味）是项目中常被忽略但至关重要的维度。它决定了代码库的气质 — 是偏工程化的严谨，还是偏原型化的快速迭代。根据项目类别，基调可以分为几个典型方向：

| 项目类别 | 推荐基调 | 特征 |
|----------|----------|------|
| CLI 工具 / 库 | **严谨工程派** | 完整的错误处理、单元测试 > 90% 覆盖率、严格类型 |
| Web 原型 / 个人工具 | **快速迭代派** | 功能性优先、测试覆盖核心路径、适度技术债可接受 |
| 基础设施 / 中间件 | **防御性编程派** | 每个边界都做防御检查、panic-safe、极致的错误信息 |
| 学习项目 / Demo | **教学清晰派** | 注释丰富、可读性优先、不追求性能极致 |
| 团队协作项目 | **约定驱动派** | ESLint + Prettier + commitlint + CI 门禁齐全 |

## 5.2 定调的具体执行

确定基调后，需要通过以下手段确保 Agent 一致地遵循：

### 5.2.1 CLIENT.md / AGENT.md

在项目根目录创建对 Agent 的指引文件（类似本 vault 的 [[CLAUDE|CLAUDE.md 全局规则]]）：

```markdown
# AGENT.md — 本项目协作规则

## 设计基调
- **防御性编程**：每个外部输入都必须验证，不信任任何参数
- **测试优先**：所有 PR 必须附带测试，覆盖率不降低
- **文档同步**：公共接口必须有 doc comment，和代码一起更新

## 代码风格
- 使用项目 .clang-format，不讨论格式
- 错误处理使用 Result<T, E> 模式，不抛异常

## 沟通约定
- 建议实现方案时，优先考虑内存安全和性能
- 有歧义时在代码中写 TODO 并询问，不要擅自做决定
```

> 关于 AGENT.md 的详细用法参考 [[01-repo-structure|仓库结构说明]] 中的配置文件 section。

### 5.2.2 生成风格样本

给 Agent 提供一段该项目风格的**样本代码**，比写在文档里有效得多：

```plain
请参考以下风格样本实现新功能 —— 注意错误处理模式、注释风格、
命名约定要与此一致：

<提供一段符合基调的参考代码>
```

### 5.2.3 基调一致性检查

在代码审查环节，增加一个专门的审查维度：

```plain
/review-vibe "检查本次变更是否与项目的防御性编程基调一致"
```

# 6. 阶段五：迭代实现与工程化

## 6.1 分步推进策略

这是开始写代码的阶段。核心原则：**小步快跑 + 频繁提交 + Agent 辅助迭代**。

### 6.1.1 理想的分步粒度

```plain
Step 1: 项目脚手架 + 核心数据结构    →    commit + test
Step 2: 最小可行路径（happy path）   →    commit + test
Step 3: 错误处理 + 边界条件         →    commit + test
Step 4: 性能优化 / 功能扩展          →    commit + test
Step 5: 集成测试 + 端到端验证       →    commit + test
```

每个 step 的代码主要由 Agent 生成，但人负责：

1. **描述需求**：用自然语言告诉 Agent 这一步要做什么
2. **审查产出**：读 diff，关注"做了没"和"做得对不对"
3. **修正方向**：如果 Agent 走偏了，及时纠正
4. **提交版本**：每个步完成后创建本地 commit

## 6.2 版本管理：本地高频提交 + 远端精简推送

遵循本 vault 的 Git 工作流（本地高频 commit，推远端前 squash）：

```bash
# 1. 局部提交（每个 step 完成后立即执行）
git add -A && git commit -m "feat(version-fs): 实现 copy-on-write 写入路径"

# 2. 累积几个 step 后，在推远端前整理
git rebase -i HEAD~N
# 或
git reset --soft HEAD~N && git commit -m "feat(version-fs): 实现核心写入功能"

# 3. 推送到远端
git push origin main
```

### 6.2.1 与 Agent 协作的提交策略

```plain
你：请实现 Step 2 — 最小可行的文件读取路径。
Agent：已实现，diff 如下 <...>

你：看起来正确，提交吧。
Agent：git add + git commit
```

> 可以将提交任务委托给 `/quick-commit` skill。详见 [[F-版本管理/1. git|Git 基础]]。

## 6.3 测试：从单元到集成的测试金字塔

### 6.3.1 单元测试（每个 step）

在每个 step 对应的 agent 请求中，要求同步生成测试：

```plain
请实现 FUSE read 路径，并同时生成对应的单元测试 ——
覆盖：正常读取、读取空文件、读取已删除文件、offset+size 边界
```

Agent 产出的测试应当：

- 和实现代码同时提交（不要事后补）
- 覆盖 happy path + 边界条件 + 预期的错误路径
- 不依赖外部环境（mock 文件系统、mock 网络）

### 6.3.2 回归测试（跨 step）

当完成 Step 3 后，需要验证 Step 1 + Step 2 的功能没有被破坏：

```bash
# 运行全部现有测试
cmake --build build && ctest --output-on-failure

# 或
go test ./... -count=1
```

可以将回归测试写入 CI（GitHub Actions / GitLab CI）：

```yaml
# .github/workflows/test.yml
test:
  steps:
    - run: go test ./... -count=1 -race
    - run: go vet ./...
```

### 6.3.3 测试风格

参考项目测试约定，在 AGENT.md 中约定测试风格：

```markdown
## 测试约定
- 使用 table-driven test（Go）或 parameterized test（C++）
- 测试函数名统一：Test_功能_场景_期望结果
- mock 接口放在 testutil/ 目录下
```

## 6.4 Debug & Bug Fix

Agent 在实现过程中会产生 bug，这是常态。关键在于建立高效的 debug 循环：

```plain
你：Step 2 的读取测试失败了，错误信息是 "permission denied"
    请帮我 debug，在 FUSE 调试模式下运行
    fusermount -u mountpoint && ./build/version-fs -d mountpoint

Agent：[运行、分析日志、定位到错误在权限检查逻辑中]
       发现 getattr 返回的 st_mode 没有设置读权限位。
       修正方案：在 version 数据初始化时设置 0644。
```

Debug 循环的核心模式：

```plain
复现错误 → Agent 分析日志 → 定位根因 → 修复 + 新增测试验证修复 → 回归测试
```

## 6.5 架构重构

项目进行到中期，随着功能增多，初期设计的抽象层会出现 leaky abstraction。这是健康的信号，说明你对问题的理解加深了。

### 6.5.1 重构触发信号

- 添加一个新功能需要修改 5+ 个文件
- 测试越来越难写（需要大量的 mock 设置）
- Agent 生成的代码开始出现重复的模式
- 代码审查时需要反复解释同一段逻辑

### 6.5.2 重构步骤

```plain
1. 识别重构目标：哪个模块耦合过高？
2. 写重构方案：当前 → 目标架构对比图
3. 分步执行：每次只动一个抽象层
4. 保持测试通过：每步后运行完整回归测试
5. 用 Agent 辅助：让 Agent 负责机械性的代码移动和 import 修正
```

> 任何时候先 `git stash` 或另开分支保护现有工作，再开始重构。

# 7. 总结：Vibecoding 成熟度模型

| 等级 | 特征 | 典型状态 |
|------|------|----------|
| L1 — 冲动型 | 有想法就写，让 Agent 自由发挥 | 代码能跑但不可维护，无测试，无文档 |
| L2 — 结构型 | 先定范围再写，有基础版本管理 | 有 PRD 雏形，本地 commit，部分测试 |
| L3 — 工程型 | 全流程结构化，测试驱动，定期重构 | ADR 完整，CI 跑测试，代码风格一致 |
| L4 — 协作型 | 人机分工明确，Agent 负责执行与探索，人负责方向与决策 | 高效反馈循环，自动化的质量门禁 |

本文描述的是从 L2 向 L3 / L4 过渡的方法论。**Vibecoding 的核心不是跑得更快，而是在不跑偏的前提下跑得更快。**
