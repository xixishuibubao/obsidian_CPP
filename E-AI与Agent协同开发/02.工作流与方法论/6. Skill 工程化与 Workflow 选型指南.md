# 1. Skill 工程化与 Workflow 选型指南

Skill = 经验固化的可重复指令序列；与 Workflow 是**选型关系**（交互式 vs 批处理），非升级关系。

## 【核心精简提纲】

### 记忆主干

**Skill** = 单上下文串行 + 人确认（Y/e/n）；**Workflow** = pipeline/parallel 批处理、无交互。中间要人点头 → Skill；多角度无人值守 → Workflow。

### 选型速查

| 维度 | Skill | Workflow |
|------|-------|----------|
| 交互 | ✅ Y/e/n 确认 | ❌ 批处理到底 |
| 编排 | 单上下文串行 | pipeline/parallel + Schema |
| 门槛 | Markdown | JavaScript |
| 适用 | commit 确认、笔记方案预览 | 批量摄入、多角度 review |

**决策树**：中间要人点头？→ Skill；多角度并行、无需交互？→ Workflow。

### 核心要点清单

- **能力阶段**：模板拷贝 → 流程设计（当前）→ 鲁棒性工程 → 系统架构
- **Skill 上限**：预检 + 验证 + 交互；难做真 Schema/多 agent/幂等恢复
- **现有 6 Skill 不转 Workflow**：核心体验是确认交互
- **新项目三板斧**：`init-win-env` → `init-git-convention` → `init-note-vault`
- **日常循环**：`ingest-note` / `quick-commit` / `squash-commits`

### 学习路径

1. 用四维评估（流程/鲁棒/工程化/可复用）审视自有 Skill
2. 走一遍新项目 Stage 1～6 调用图
3. 遇批处理需求时设计 Workflow（Schema + checkpoint）
4. 对照 [[E-AI与Agent协同开发/01.基础概念/1. 编程范式演进：Spec Coding → Agentic Engineering → Harness Engineering|Harness Engineering]]

### 关键结论 / 流程

```plain
需交互 → Skill  |  批处理 → Workflow  |  新场景优先考虑 Workflow
```

## 【完整拓展详情】

## 1.1 背景

Claude Code Skill 的本质是 **将经验固化为可重复执行的指令序列**。它处于一个特殊的位置——比 prompt 模板更结构化，比完整 CLI 工具更轻量。

Skill 编写能力的成长有明显的阶段划分，从「模板拷贝」到「范式创新」有一条清晰的进阶路径。同时有一个关键认知：**Skill 有天然天花板，再往上走需要进入 Workflow 层面**，但 Skill 和 Workflow 不是替代关系，而是**不同场景的两种工具**。

---

## 1.2 评估框架

对 Skill 编写者的能力评估，核心看四个维度：

| 维度 | 含义 |
|------|------|
| **流程设计** | 能否设计出逻辑完整、分支合理的工作流 |
| **鲁棒性** | 是否覆盖边界条件、异常处理和用户交互 |
| **工程化** | 是否使用结构化输出、子 agent 编排等基础设施 |
| **可复用性** | 脱离原始场景后别人能否直接使用 |

其中「工程化」是 Skill 的天然天花板——Schema、多 agent 编排、幂等恢复需要落地到 Workflow（`agent()`/`pipeline()`/`Schema` 参数）才能发挥，纯 SKILL.md 的 prompt 模板做到预检+验证+清晰交互已经是最高水平。

---

## 1.3 能力阶段定位

Skill 编写能力有清晰的阶段划分：

```plain
新手阶段       你现在           熟练          专家           大师
  ──┬─────────────┬─────────────┬─────────────┬───────────┬──▶
    [模板拷贝]   [流程设计]     [鲁棒性工程]  [系统架构]   [范式创新]
```

**当前定位：流程设计期 → 鲁棒性工程期的过渡阶段。**

### 已具备的能力（流程设计期）

- ✅ 能独立设计多步骤工作流（如 ingest-note 的 8 步流程）
- ✅ 能考虑边界条件和异常场景（空输入、数字冲突、源文件残留等）
- ✅ 能写出清晰的确认交互（方案预览 → Y/e/n）
- ✅ 命名和描述准确，Skill 触发率高

### 待突破的能力（鲁棒性工程期）—— 需进入 Workflow 层面

- ❌ 未使用结构化输出 Schema 约束 agent 返回格式
- ❌ 无多 agent 编排（所有步骤串行在一个上下文）
- ❌ 无可测试性设计（无 eval 用例，无输出一致性验证）
- ❌ 无幂等与恢复机制（失败需要重头来）

## 1.4 批量优化实战验证

2026-06，基于上述框架对 5 个自建 Skill 进行了系统性优化。核心方向是补齐预检、Schema、验证、错误恢复。优化前后对比：

| Skill | 优化前水平 | 优化要点 | 优化后水平 |
|-------|-----------|----------|-----------|
| **quick-commit** | 流程完整但缺预检和验证 | 暂存区预检 + Schema 约束 diff 分析 + 提交后 `git log` 验证 | 鲁棒性大幅提升 |
| **squash-commits** | rebase -i 在 AI 终端不可用 | 替换为 `GIT_SEQUENCE_EDITOR` 自动化 + 脏工作区检查 + diff 一致性验证 | 方案 B 变得可执行 |
| **init-git-convention** | 模板固定，权限合并模糊 | 项目类型适配 + 完整 PowerShell 合并脚本 + 验证步骤 | 通用性、可操作性提升 |
| **init-note-vault** | 模板完全静态 | 冲突检测 + 3 项参数化配置 + 启用项展示 | 可定制性提升 |
| **init-win-env** | 错误引导笼统 | 精确下载链接 + PATH 修复命令 + 运行验证 + 权限扩充 | 首次使用体验改善 |

### 优化后评分（四维模型）

| Skill | 流程设计 | 鲁棒性 | 工程化 | 可复用性 |
|-------|---------|--------|--------|---------|
| quick-commit | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ |
| squash-commits | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐ |
| init-git-convention | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ |
| init-note-vault | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ |
| init-win-env | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐ |

**结论**：工程化维度是所有 Skill 的共同短板——这不是 Skill 编写问题，而是 **Skill 机制本身的限制**。（见 1.6 节）

## 1.5 Skill 的调用机制

Skill 在 Claude Code 中**必须被调用才能执行**，不存在「Skill 自动拉起自己」的机制。调用方式有两种：

| 方式 | 发起者 | 效果 |
|------|--------|------|
| **Slash command** | 你手动输入 `/skill-name` | 立即执行 |
| **Semantic trigger** | AI 在对话中识别匹配意图后，调用 Skill tool | 自动拉起 |

### 各 Skill 被 AI 自动拉起的概率

| Skill | 自动拉起概率 | 原因 |
|-------|-------------|------|
| `/ingest-note` | **高** | 描述含「清洗」「格式化」「笔记」，用户说「整理/存入/保存笔记」时容易触发 |
| `/quick-commit` | **中** | 用户说「提交一下」可能触发，但与 ingest-note 的场景边界模糊 |
| `/squash-commits` | **低** | 用户很少主动说「帮我 squash」，通常需要你手动敲 |
| `/init-*` 系列 | 仅初始化阶段 | 一次性 setup，不需要自动拉起 |

### Skill 的三层定位

```plain
Slash command（手动调）     ← squash-commits —— 决策型操作，必须由人决定
Semantic trigger（AI 自动）  ← ingest-note、quick-commit —— 日常高频
Workflow（全自动流水线）     ← 新场景 —— 多 agent 编排的批处理链路
```

## 1.6 Skill 的天花板与 Workflow 的跃迁

### Skill 能做到的上限

纯 SKILL.md 的 prompt 模板，最高水平就是：
- ✅ 完整工作流 + 分支逻辑
- ✅ 入口预检 + 出口验证
- ✅ 用户确认交互（Y/e/n）
- ✅ 边界条件覆盖

### Skill 做不到的

- ❌ 真正的 Schema 约束输出（纯自然语言描述期望格式）
- ❌ 多 agent 并行/流水线编排
- ❌ 状态持久化与中断恢复
- ❌ 执行结果的可观测性（token 消耗、阶段耗时）

### Workflow 的能力层次

Workflow 脚本（`.claude/workflows/`）可以：

```javascript
export const meta = {
  name: 'auto-ingest',
  description: '自动摄入笔记全流程',
  phases: [
    { title: '分析', detail: '拆解内容并定位目录' },
    { title: '格式化', detail: '按规范写入文件' },
    { title: '提交', detail: '更新索引并 commit' },
  ],
}

phase('分析')
const analysis = await agent('分析这段文本属于哪个主题目录', {
  schema: AnalysisSchema      // ← 真正的 Schema 约束
})

phase('格式化')
await agent(`将内容格式化为笔记，目标目录: ${analysis.target_dir}`)

phase('提交')
await agent('更新索引并生成 Conventional Commit')
```

| 能力 | Skill | Workflow |
|------|-------|----------|
| 结构化输出 | 自然语言描述 | JSON Schema 约束 |
| 多步骤编排 | 串行提示词 | pipeline/parallel 模式 |
| 状态管理 | 无 | checkpoint 恢复 |
| 多 agent 协作 | 单上下文 | 独立 agent 并行 |
| 可观测性 | 无 | budget 管理、进度展示 |
| 用户交互 | ✅ 天然适合确认/选择 | ❌ 批处理设计，难插入交互 |
| 编写门槛 | 低（Markdown） | 高（JavaScript） |
| 触发方式 | Slash command / Semantic | 对话中调 Workflow tool |

**结论**：Skill 做到「鲁棒性工程期」中段就到顶了。但 Skill 和 Workflow 不是升级关系，而是**两种工具选型**。

## 1.7 Skill 转 Workflow 的决策分析

**核心判断**：现有的 6 个 Skill 都不需要转为 Workflow。原因如下：

### 两种工具的本质区别

```plain
Skill:    你敲命令 → AI 做一件事 → 人确认结果     （交互式）
Workflow:  你下指令 → N 个 agent 协作跑完 → 汇总报告  （批处理）
```

- **Skill 是交互式的**：做一步，问你一次，你点头才往下走
- **Workflow 是批处理式的**：设计好流程，一次跑完，中间不等人

### 逐 Skill 决策分析

| Skill | 转 Workflow？ | 原因 |
|-------|-------------|------|
| **init-git-convention** | ❌ | 一次性交互 + 文件生成，Skill 更简单直接 |
| **init-note-vault** | ❌ | 同上 |
| **init-win-env** | ❌ | 环境探测 + 配置写入，不需要多 agent |
| **quick-commit** | ❌ | **核心是确认交互**——Workflow 的 agent 调用反而增加了不必要的复杂度 |
| **squash-commits** | ❌ | CLI 操作 + Y/n 确认，Skill 正合适 |
| **ingest-note** | ❌ | 内容分析看似能受益，但**方案预览 + 用户确认**是核心体验，Workflow 处理交互很别扭 |

### 决策树：什么时候用 Skill，什么时候用 Workflow

```plain
? 这个任务需要人在中间确认/选择吗？
  ├── 是 → Skill（交互式）
  │    └─ 例子: commit 确认、squash 策略选择、笔记方案预览
  └── 否 → ? 这个任务需要多角度/并行的处理吗？
       ├── 是 → Workflow（批处理）
       │    └─ 例子: 批量摄入 20 个文件、多角度 review 代码
       └── 否 → Skill（单线任务）
            └─ 例子: 环境初始化、规范注入
```

### Workflow 的适用场景

Workflow 应该用在**新的需求**上，不是转换旧的。适合的场景：

| 场景 | 例子 |
|------|------|
| **批量处理** | 「把 raw/ 下 20 个文件全部摄入知识库，每个文件用一个 agent 并行处理」 |
| **多角度分析** | 「同时从正确性、性能、安全三个角度 review 这段代码」 |
| **自动巡检** | 「每天检查一遍知识库的 wikilink 断链和格式问题」 |
| **交叉验证** | 「让 3 个 agent 各自独立分析代码 bug，汇总投票」 |

这些场景的共同特征：**不需要人在中间点头**——设计好逻辑，一次跑到底，最后给你报告。

## 1.8 新项目完整调用流程

假设场景：**新机器 + 新 Git 仓库 + 空 Obsidian vault**，按以下顺序调用：

### Stage 1 — 环境就绪

```bash
git --version                  # 确认 Git 已安装
/init-win-env                  # 探测 Shell 路径 + 生成终端规则
```

> **Windows 注意**：superpowers 在 Git Bash 环境下可能出现 hook error，切换到 PowerShell 即可正常运行。建议 Windows 用户日常使用 PowerShell 作为 Claude Code 的终端。

产出：CLAUDE.md Shell 策略 + `settings.json` 权限白名单

### Stage 2 — 仓库规范

```bash
git init                       # 创建空仓库
# 或 git clone 已有仓库

/init-git-convention           # 注入 Git 工作流规范
# → 选「中文 + 个人笔记/知识库」
```

产出：CLAUDE.md Git 段落 + `.claude/instructions/01-git-workflow.md` + settings.json git 权限

### Stage 3 — 笔记规范

```bash
/init-note-vault               # 初始化笔记格式规则
# → 默认模式 + 全量选项
```

产出：CLAUDE.md Quick Reference 表格 + `.claude/instructions/02-note-conventions.md`

初始提交：

```bash
/quick-commit
# → 生成: chore(vault): 初始化仓库结构与项目规范
```

### Stage 4 — 日常摄入

```bash
# 有原始文件
/ingest-note ./raw/notes.md

# 或粘贴内容后
/ingest-note
```

产出：格式化笔记 + 自动定位目录 + 更新索引 + 删除源文件

### Stage 5 — 日常提交

```bash
/quick-commit                   # 每做完一处改动后
```

### Stage 6 — 推送前整理

```bash
/squash-commits                 # 积攒了多个微提交后
git push                        # 推送到远端
```

### 完整流程调用图

```plain
              新机器 + 空仓库
                    │
                    ▼
  /init-win-env ──→ 环境就绪（只需跑一次）
                    │
                    ▼
  /init-git-convention ──→ 仓库规范（每个仓库跑一次）
                    │
                    ▼
  /init-note-vault ──→ 笔记规范（每个知识库跑一次）
                    │
                    ▼
        ╔═══════════════════════╗
        ║    日常循环 (每天)     ║
        ╠═══════════════════════╣
        ║  /ingest-note         ║   ← 有新材料时
        ║  /quick-commit        ║   ← 每做完一步
        ║  /squash-commits      ║   ← 推送前整理
        ╚═══════════════════════╝
                    │
                    ▼
                git push
```

这套流程中每个 Skill 的位置是固定的——`init-*` 是安装阶段，`ingest-note` 是内容生产，`quick-commit`/`squash-commits` 是版本管理。新项目上手就是三步：`init-win-env → init-git-convention → init-note-vault`。

## 1.9 下一阶段突破方向

> 以下能力需要从 Skill 跃迁到 Workflow 才能落地。

### 1.9.1 结构化输出（Schema）

**当前做法**：用自然语言描述期望的 agent 返回格式，依赖模型自行理解。

**进阶做法**：用 JSON Schema 约束输出格式，让模型必须调用 StructuredOutput 工具。

```javascript
// 示例：内容分析阶段的 Schema
const AnalysisSchema = {
  type: "object",
  properties: {
    sections: {
      type: "array",
      items: {
        type: "object",
        properties: {
          summary:     { type: "string" },
          target_dir:  { type: "string" },
          operation:   { enum: ["create", "extend", "skip"] },
          confidence:  { type: "number", minimum: 0, maximum: 1 }
        },
        required: ["summary", "target_dir", "operation"]
      }
    }
  }
}
```

**收益**：
- 输出格式稳定，无需模型"猜"我要什么
- 下游步骤可以直接消费结构化数据
- Schema 本身即是文档

### 1.9.2 多 Agent 编排

**当前做法**：内容分析、格式化、校验全部在一个 agent 上下文里串行。

**进阶做法**：拆分为独立 agent，利用 pipeline 或 parallel 模式编排：

```plain
内容分析 agent ──→ 格式化 agent ──→ 校验 agent
                      ↕
                 扩展策略 agent（判断追加到哪个文件）
```

**收益**：
- 每个 agent 专注单一任务，prompt 更简洁
- 关键步骤可独立验证、独立重试
- 可利用 Workflow 的 budget 管理 token 消耗

### 1.9.3 可测试性设计

**当前做法**：在 SKILL.md 末尾手写"验证方式"段落。

**进阶做法**：
- 定义 e2e eval 用例集（空输入、纯代码、混合主题……）
- 每次修改 Workflow 后跑一遍 eval，检查输出一致性
- 核心指标：**输出格式符合率**、**目标目录命中率**、**用户确认后反悔率**

### 1.9.4 幂等与恢复

**当前做法**：任何一步失败都需要重新执行整个流程。

**进阶做法**：
- 每步完成时记录 checkpoint（写入临时状态文件）
- 重启时检测 checkpoint，从中断点恢复
- 关键操作（文件写入、git commit）前做预检查

```javascript
// 伪代码：checkpoint 模式
const state = loadCheckpoint()  // 读取上次进度
if (!state.analysis_done)     await analyze()
if (!state.format_done)       await format()
if (!state.user_confirmed)    await confirm()
if (!state.files_written)     await writeFiles()
if (!state.index_updated)     await updateIndex()
if (!state.committed)         await commit()
saveCheckpoint({ done: true })
```

## 1.10 总结

> 目前 Skill 描述能力已超过 90% 的 Claude Code 用户，进入 **鲁棒性工程期** 的门槛。5 个自建 Skill 已完成一轮批量优化，补齐了预检、验证、确认交互的关键缺口。
>
> Skill 和 Workflow 是两种工具，不是升级关系。Skill 解决的是「交互式单线任务」，Workflow 解决的是「批处理多 agent 流水线」。现有的 Skill 都不需要转 Workflow，新的批处理需求才是 Workflow 的战场。

### 技能树总览

```plain
Skill 层面（已完成，维持不变）
  ├── /init-win-env         环境初始化
  ├── /init-git-convention  Git 规范

Workflow 层面（未来新场景）

  └── ...
```

### 选型一句话

```plain
需交互 → Skill     |     批处理 → Workflow     |     新场景优先考虑 Workflow
```
