# 1. AI 辅助个体成为超级程序员的工作流路线

AI 上限由人的**结构设计 × 问题拆解**决定；三步飞轮：模块化 PRD → 接口优先框架 → 选择性审计 + 规则沉淀。

## 【核心精简提纲】

### 记忆主干

| 角色 | 负责 | 不负责 |
|------|------|--------|
| 开发者 | 结构设计、拆解、方向、质量审计 | 逐行编码、重复劳动 |
| AI | 生成、测试、探索、模式匹配 | 架构决策、价值判断 |

### 核心要点清单

- **模块化 PRD**：拒绝大而全；每模块独立可交付、边界清晰、接口级依赖
- **AI 追问**：暴露盲区、压缩决策成本、产出即 ADR
- **千行原则**：单模块 ~1000 行 = AI 可完整理解 + 人可 review 的规模
- **接口优先**：只定义接口与数据结构；依赖接口不依赖实现
- **选择性审计**：查设计一致性、抽象泄漏、规则遵守；不逐行审
- **规则闭环**：偏差 → 修正 → 写入 CLAUDE.md/AGENT.md → 下次自动避让

### 学习路径

1. 把一个需求拆成 3～5 个模块 PRD
2. 只写头文件/接口 + 让 AI 生成测试
3. 审计后沉淀一条规则到项目 harness
4. 与 [[E-AI与Agent协同开发/02.工作流与方法论/1. AI-Agent 协同开发（Vibecoding 最佳实践）|Vibecoding]] 压力测试/细节阶段对齐

### 关键结论 / 流程

```plain
模块化 PRD → 框架代码 → AI 实现 → 审计 → 规则沉淀
                ↑___________________________|
```

## 【完整拓展详情】

## 1.1 核心理念：人机协作的契约

在 AI 辅助编程中，人的角色从"写代码的人"转变为"设计系统的人"。这种转变的底层契约是：

| 角色 | 负责 | 不负责 |
|------|------|--------|
| **开发者** | 结构设计、问题拆解、方向判断、质量审计 | 逐行编码、重复劳动、细节记忆 |
| **AI** | 代码生成、测试编写、方案探索、模式匹配 | 架构决策、需求定义、价值判断 |

这套路线包含三个递进步骤，最终形成开发者能力的成长飞轮：

```plain
模块化 PRD → 接口优先的框架代码 → 选择性介入审计
     ↑                                  |
     └──────── 能力反哺 ←──────────────┘
```

---

## 1.2 步骤一：模块化 PRD

### 1.2.1 为什么拒绝全局需求文档

传统开发中，需求文档往往追求"大而全"——试图在编码开始前穷举所有功能。在 AI 协作语境下，这种方式有两个致命问题：

1. **一次性信息过载**：AI 的上下文窗口有限，长文档导致关键信息被稀释
2. **过早细化**：未经验证的功能细节浪费了 AI 和人的精力

> **原则**：不要写大而全的全局需求文档，而是专注于一块块模块化的文档。

### 1.2.2 模块级文档的颗粒度

每个 PRD 模块应该聚焦于**最核心的功能需求**，并满足以下特征：

- **独立可交付**：每个模块对应一个可独立实现的子功能
- **边界清晰**：明确输入、输出、依赖、约束
- **精细颗粒度**：精确到具体的接口行为，而非模糊的业务描述
- **优先级排序**：核心路径优先，边缘功能延后

一个好的模块 PRD 通常覆盖：

```plain
模块名称：
功能描述（一句话）：
输入输出规格：
依赖模块（接口级，非实现级）：
边界条件：
验收标准：
```

### 1.2.3 借助 AI 追问来暴露盲区

这一步的关键不是"让 AI 帮你完善文档"，而是**让 AI 提出问题**，帮助开发者发现自己未考虑到的维度：

```plain
开发者：我的项目需要一个文件版本管理模块。
AI:      好。几个问题帮助你细化：
         1. 版本存储策略是什么？
            copy-on-write? diff-based? 全量快照？
         2. 允许 overwrite 还是 append-only？
         3. 元数据（权限、owner）是否版本化？
         4. 磁盘空间满时的行为是什么？
```

这种交互的价值在于：

- **暴露 thinking blind spots**：AI 从实现角度看问题，天然覆盖开发者容易忽略的边界条件
- **压缩决策成本**：在编码前解决设计歧义，避免实现阶段的返工
- **沉淀设计决策**：每个 Q&A 的结果都是 ADR，无需额外文档工作

> 这一步与 [[1. AI-Agent 协同开发（Vibecoding 最佳实践）|Vibecoding 工作流]] 中的"压力测试 + 细节补充"阶段相对应。

## 1.3 步骤二：接口优先的框架代码

> 核心思路：只定义接口和数据结构，不包含具体实现和逻辑。**分层次设计**。

### 1.3.1 千行代码原则

将项目通过接口定义拆解成**较少行数代码可以实现的小模块**，推荐以 **1000 行**为分界线。

为什么是 1000 行？

| 考量维度 | 说明 |
|----------|------|
| **AI 上下文适配** | 当前 LLM 的上下文窗口虽大，但高精度处理的区域集中在开头和结尾；1000 行代码恰好是一个 AI 能完整理解并保持一致性的规模 |
| **审查认知负载** | 人类的 code review 注意力曲线在 500-1000 行后急剧下降；模块太小则接口碎片化，太大则审查失真 |
| **单一职责** | 1000 行是实现一个完整抽象层（一个模块、一个 service、一个 provider）的典型规模 |
| **可测试性** | 千行模块的测试集通常不会超过 200-300 个用例，CI 可在秒级完成 |

> 1000 行是参考值，不是铁律。核心判断标准是：**这个模块能不能用一句话说清楚它做什么**。

### 1.3.2 每个模块只关注自己的功能

模块设计的两条基本约束：

1. **模块可以依赖其他模块的接口，但不可依赖其具体实现方式**
   - 通过接口依赖实现编译期解耦
   - 允许 mock 接口进行单元测试，不需要加载整个运行时环境

2. **尽量把可自动测试的逻辑剥离**
   - 纯业务逻辑 -> 参数化测试
   - 数据转换 -> 输入输出对测试
   - 状态机 -> 状态转移表测试
   - 将"非纯粹"的部分（IO、网络、外部调用）推到模块边界

> 这与 [[1. 编程范式演进：Spec Coding → Agentic Engineering → Harness Engineering|编程范式演进]] 中的 Spec Coding 一脉相承——精确的接口规格是 AI 高质量产出的基础。

### 1.3.3 让 AI 生成测试代码验证逻辑正确性

在框架代码定义完成后，将测试代码生成交给 AI：

```plain
开发者：以下是 Parser 模块的接口定义和数据结构。
        请生成覆盖以下场景的单元测试：
        - Happy path：标准输入
        - 边界条件：空输入、超长输入、特殊字符
        - 错误路径：格式错误、缺少必要字段
```

这样做的好处：

- **AI 的"模式匹配"能力**在此场景下发挥最佳——测试模式高度重复
- **开发者只需确认覆盖度**，不需要写重复的测试代码
- 测试同时也是**规格的 live documentation**

## 1.4 步骤三：选择性介入审计

### 1.4.1 审计的关注点

开发者应当介入代码审计，但**不必关注每一行细节**。审计焦点应放在三层：

| 审计层级 | 关注点 | 检查频率 |
|----------|--------|----------|
| **设计一致性** | AI 是否破坏了预定义的接口设计？ | 每个模块完成时 |
| **抽象泄漏** | 本应隐藏的实现细节是否暴露到了接口中？ | 每个模块完成时 |
| **规则遵守** | 是否符合已沉淀的编码规则？ | 自动化门禁 |
| **逐行细节** | ❌ 不需要——交给 AI 和测试 | 不检查 |

### 1.4.2 规则反馈闭环

当发现 AI 破坏设计时：

1. **及时修正**：指出问题，让 AI 重新生成
2. **把约束固化到规则文件**：将本次发现的问题转化为可复用的规则（写入 CLAUDE.md / AGENT.md）

这个闭环是整条工作流中**最关键的能力积累机制**：

```plain
发现偏差 → 修正实现 → 沉淀规则 → 下次自动避让
    ↑                          |
    └──────────────────────────┘
```

每一次审计+规则沉淀，都在提升 AI 产出的质量基线。长期来看，规则文件越丰富，需要人为介入的频率越低。

> 关于 AGENT.md 的详细实践，参考 [[1. AI-Agent 协同开发（Vibecoding 最佳实践）|Vibecoding 最佳实践]] 中的"设计定调"章节。

## 1.5 对开发者的启迪——强制思考

这条工作流最被低估的价值是：**它强制开发者思考**。不是思考每一行代码怎么写，而是思考更高维度的问题。

### 1.5.1 代码结构设计能力

- 如何划分模块边界？
- 接口的稳定性和演进性如何平衡？
- 什么逻辑应该放在模块内部，什么应该暴露？
- 模块间如何通信？同步还是异步？

这些问题在传统编码中被实现细节淹没，但在"只写接口"的工作流中被放大到无法回避。**结构设计能力**成为决定系统质量的第一要素。

### 1.5.2 拆解问题的能力

一个模糊的需求 → 拆解为多个独立模块 → 每个模块定义清晰的接口 → 交给 AI 逐个击破。

这种拆解能力包含：

| 子能力 | 表现 |
|--------|------|
| **水平拆解** | 按功能切分（Parser / Storage / API / UI） |
| **垂直拆解** | 按抽象层次切分（接口层 / 业务层 / 基础设施层） |
| **优先级排序** | 识别核心路径 vs 边缘功能，MVP 先行 |
| **依赖编排** | 确定模块间的依赖关系和实现顺序 |

> 拆解问题这件事，**AI 无法代劳**。一个好的问题拆解能让 AI 如虎添翼，一个糟糕的拆解则会让 AI 陷入无限修复循环。

### 1.5.3 AI 能力的上限由什么决定

```plain
传统开发的效率瓶颈：编码速度 × 调试效率
AI 协作的效率瓶颈：结构设计 × 拆解质量
```

一个简单的对比：

| 开发者能力 | 传统模式下 | AI 协作模式下 |
|-----------|-----------|--------------|
| 结构设计强 + 拆解强 | ⭐⭐⭐ 优秀工程师 | ⭐⭐⭐⭐⭐ 超级程序员 |
| 结构设计强 + 拆解弱 | ⭐⭐⭐ 优秀工程师 | ⭐⭐⭐ 被 AI 放大但有限 |
| 结构设计弱 + 拆解强 | ⭐⭐ 普通 | ⭐⭐⭐ 尚可，但系统质量受限 |
| 结构设计弱 + 拆解弱 | ⭐ 挣扎 | ⭐ AI 也无法拯救 |

> **结论**：AI 编程时代，开发者的结构设计和问题拆解能力，直接决定了 AI 工具的上限。

## 1.6 补充：工作流全景与能力飞轮

将三个步骤放在一起，可以看到一个完整的能力成长循环：

```plain
┌─────────────────────────────────────────────────────────┐
│                   第一轮：从零到一                        │
│                                                         │
│  模块化 PRD ──→ 框架代码 ──→ AI 实现 ──→ 审计 ──→ 规则沉淀 │
│                    ↓                      ↑             │
│                Developer 理解了模块边界 ────┘             │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                   第二轮：迭代演进                        │
│                                                         │
│  细化 PRD ──→ 调整接口 ──→ AI 生成 ──→ 审计更少 ──→ 规则更丰富 │
│                                                    ↓      │
│                                          AI 产出质量持续提升 │
└─────────────────────────────────────────────────────────┘
```

这个循环的核心意义：

- **短期看**：每一步产出在提升产品质量
- **中期看**：规则文件积累减少了重复审计的成本
- **长期看**：开发者自身的结构设计和拆解能力在实战中持续提升

**AI 不会让开发者变得"懒"——它淘汰的是拒绝思考的人，放大的是善于思考的人。**
