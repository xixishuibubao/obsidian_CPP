# 1. 遗留代码系统分析与重构

超大/混乱 C/C++ 工程：**不一次性加载全量** → 索引 + 检索 + 分层切片 → 局部精读 + LLM 辅助。

## 【核心精简提纲】

### 记忆主干

五阶段：**全局摸底** → **符号索引**（cscope/ctags）→ **分层切片**（入口→调度→业务→公共）→ **四大脉络**（执行/数据/耦合/外部依赖）→ **输出重构清单**。

### 核心要点清单

- **工具三件套**：rg（检索）+ cscope（调用链）+ cloc/tree（体量）
- **入口优先**：从 `main`/初始化出发，逐层展开，不逐行硬读
- **切片单位**：函数调用链，非文件目录
- **降噪**：黑白名单忽略 printf/日志；超长函数按分支切
- **LLM 用法**：每次只喂一个函数/切片；禁止整库粘贴
- **重构模式**：绞杀者（Wrap→新架构→逐步替换）

### 学习路径

1. `tree` + `cloc` + `rg main` 建立第一印象
2. `cscope -bq` 练「查定义→查调用方→查被调方」
3. 画主循环时序骨架（Markdown 缩进即可）
4. 结合 [[E-AI与Agent协同开发/02.工作流与方法论/3. Claude Code 上下文管理|上下文管理]] 控制喂给 Agent 的粒度

### 关键结论 / 流程

```plain
索引建立 → 入口→主循环→业务模块 → 数据流表 → 问题清单 → 重构优先级
```

## 【完整拓展详情】

## 1.1 核心理念

遗留代码（Legacy Code）系统分析的核心原则：**不依赖一次性加载全量代码，先宏观拓扑，再分层切片，最后局部精读**。

混乱代码的痛点在于无模块边界、调用杂乱、命名随意、注释缺失。优先抓**文件依赖、调用链、入口、数据流转**四大主线，而非逐行读逻辑。

| 阶段 | 目标 | 核心工具 |
|------|------|----------|
| 全局摸底 | 目录拓扑、入口定位、体量评估 | tree, cloc, rg, find |
| 符号索引 | 建立函数/变量/宏索引，点对点查询 | cscope, ctags, cflow |
| 分层切片 | 按调用层级逐块分析，绕开上下文限制 | rg, sed, cscope |
| 脉络梳理 | 执行流程、数据流转、模块耦合 | cscope, CodeViz, Graphviz |
| 重构规划 | 标记坏味道，制定重构优先级 | 人工 + LLM 辅助 |

## 1.3 第一步：全局摸底

目标：搞清楚目录分布、文件体量、入口文件、核心后缀，建立第一印象，不读代码细节。

### 1.3.1 目录与文件结构扫描

```bash
# 展示目录层级，过滤编译产物和临时文件
tree -L 3 -I 'build|log|bin|tmp' > dir_tree.txt

# 统计源码文件总数
find . -name "*.c" -o -name "*.cpp" -o -name "*.h" | wc -l

# 按目录统计代码量，定位大头目录
find . -type f \( -name "*.c" -o -name "*.cpp" -o -name "*.h" \) \
  | awk -F/ '{print $1"/"$2}' | sort | uniq -c | sort -rn

# 统计全工程代码、注释、空行
cloc --include-ext=c,cpp,h .
```

手动标记三类目录：
- **入口目录**：主程序、main、启动脚本所在
- **公共目录**：通用工具、基础库、全局结构体
- **业务目录**：功能代码、外设 / 业务逻辑

重点关注：
- 无注释、注释率极低的目录/文件（逻辑最混乱）
- 单文件行数极高的文件（大概率职责混杂）

### 1.3.2 入口与核心符号定位

混乱工程优先找入口，顺着入口牵出主流程：

```bash
# 找所有 main 函数（C/C++ 标准入口）
rg -n "int main" --glob '*.{c,cpp}' .

# 嵌入式/后台无 main 场景：找初始化函数、回调入口
rg -n "init|Init|CreateTask|Start|Run" --glob '*.{c,cpp}' .

# 查找全局结构体
rg "struct " --glob '*.h' -n

# 查找高频函数（被大量调用的 = 公共核心函数）
rg -c "InitSystem" .
```

原则：所有脉络从「入口函数」出发，逆向/正向追踪，不用一次性加载全代码。

### 1.3.3 依赖关系提取

```bash
# 检索头文件引用，看文件之间耦合
rg "^#include \"" --glob '*.{c,h,cpp}' .
```

标记循环依赖、跨模块乱引用——这是架构混乱的重灾区。

## 1.4 第二步：建立符号索引

用 cscope + ctags 建立全局符号库，实现点到点跳转查询，彻底不用加载全量上下文。

### 1.4.1 Cscope 索引

```bash
# 1. 生成源码清单，排除编译产物、日志、临时文件
find . -name "*.c" -o -name "*.cpp" -o -name "*.h" > cscope.files

# 2. 构建索引（毫秒/秒级完成）
cscope -bq

# 3. 交互式查询
cscope
```

cscope 交互界面核心功能：
- **查函数定义**：输入函数名 → 找到函数所在文件和行号
- **查函数被谁调用**：向上追溯调用链
- **查函数调用了谁**：向下梳理执行分支
- **查全局变量/结构体/宏**：定位数据结构，理清数据流转

实操路线：`main()` → 查看它调用的所有子函数 → 逐个子函数再查其调用链 → 逐层展开调用树。

### 1.4.2 Ctags

```bash
ctags -R --c-kinds=fp .
```

在 Vim / VS Code 中可一键跳转到函数和变量定义。

### 1.4.3 Cflow 调用树

```bash
# 单个文件调用树
cflow main.c

# 批量导出所有核心文件调用树到文本
find . -name "*.c" -exec cflow {} \; > call_tree.txt
```

纯文本输出，拆分保存，分段阅读，不受上下文窗口限制。

## 1.5 第三步：分层切片梳理

不按文件分片，而按「调用层级」切片。梳理顺序：**入口层 → 主调度层 → 业务模块层 → 底层公共层**。

### 1.5.1 启动入口层

只看 `main` 函数、启动脚本、初始化入口，记录三件事：
1. **启动顺序**：先初始化什么、再做什么
2. **一级调用函数**：只记函数名，不深入内部
3. **全局资源初始化**：内存、硬件、网络、全局变量

> 这一层代码量极小，任意 LLM 都能完整处理。

### 1.5.2 主调度层

从 main 往下，找到**死循环/事件循环/任务调度**（绝大多数后台/嵌入式工程的心脏）。

特征：`while(1)`、`for(;;)`、任务队列、事件分发。

切片分析：
- 循环里依次执行哪些业务模块
- 模块之间的执行顺序、触发条件
- 模块之间如何传数据（全局变量/函数参数）

**作用**：直接拿到程序的运行时序骨架。

### 1.5.3 业务模块层

根据主循环调用的函数/分支，**按功能切分独立模块**（不是按目录）。

示例：主循环依次调用 `采集数据` → `数据处理` → `协议封装` → `日志输出`，拆成 4 个独立切片逐个分析。

### 1.5.4 底层公共层

- 公共工具：字符串、加解密、队列、读写接口（多个模块共用，最后统一梳理）
- 边缘代码：极少被调用、废弃逻辑、分支代码（最后处理，甚至暂时忽略）

### 分层原则

1. **先抓主干，丢弃分支**：大量 `if/else`、废弃注释、死代码、调试代码暂时跳过
2. **以函数调用链为单位切片，而非文件**：一个文件可能混多个功能
3. **不一次性深挖**：一层梳理完，再进入下一层子函数

## 1.6 第四步：梳理四大核心脉络

### 1.6.1 执行流程脉络

基于 main + cscope 调用链，记录三层结构：

| 阶段 | 内容 | 说明 |
|------|------|------|
| **启动阶段** | 初始化硬件、内存、配置、全局变量、模块注册 | 建立运行环境 |
| **主循环/主逻辑** | while 死循环、消息轮询、任务调度 | 业务核心 |
| **退出/销毁阶段** | 资源释放、收尾逻辑 | 清理 |

技巧：只记录主干流程，暂时屏蔽分支、异常、日志函数。

### 1.6.2 数据脉络

混乱工程几乎都依赖大量全局变量，这是数据流转核心：

- 用 rg/cscope 检索 `global`、全局结构体、全局数组、静态变量
- 记录：哪个函数赋值、哪个函数读取、跨哪些文件传递
- 梳理头文件依赖，标记循环依赖、跨模块乱引用

用表格记录数据流：

| 数据名称 | 生产者 | 处理函数 | 消费者 |
|----------|--------|----------|--------|
| （全局变量/结构体） | 初始化/接收 | 中间修改函数 | 存储/发送/展示 |

### 1.6.3 模块耦合脉络

按物理目录 + 调用关系，手动划分临时逻辑模块：

| 模块类型 | 说明 |
|----------|------|
| **工具类** | 通用函数（字符串、内存、计算、日志） |
| **业务类** | 核心业务逻辑 |
| **底层驱动/接口** | 硬件、网络、IO |
| **废弃代码** | 注释掉、从未被调用的函数/文件 |

判断耦合：
- A 文件大量调用 B 文件函数 → 强耦合
- 跨多层目录互相引用 → 架构问题

### 1.6.4 外部依赖脉络

检索 `#include <xxx.h>`、系统调用、第三方函数，分清：**自研代码 / 系统库 / 三方组件**，后续重构做边界隔离。

## 1.7 第五步：应对混乱代码的特殊策略

### 1.7.1 函数切片阅读

超长函数（上千行）逐段拆分分析：

1. 用检索定位 `if/else`、`switch`、`for/while`、`goto` 按代码块切割
2. 每一个分支单独记录逻辑，不强行串联全函数
3. 标记 goto、多层嵌套、重复代码（重构重点）

### 1.7.2 黑白名单降噪

混乱代码充斥大量日志、打印、空判断、冗余代码：

- **白名单**：只关注函数调用、变量读写、分支判断、接口调用
- **黑名单**：暂时忽略 printf、日志函数、空行、无效注释、调试代码

```bash
# 过滤掉打印语句，只看核心逻辑
rg -v "printf|LOG|print" --glob '*.c' main.c
```

### 1.7.3 分批次迭代

超大工程不要一次性梳理完，按优先级分批：

| 批次 | 内容 |
|------|------|
| 第一批 | 入口 + 主流程（必须吃透） |
| 第二批 | 核心业务模块 + 高频调用函数 |
| 第三批 | 工具函数、辅助逻辑 |
| 最后一批 | 冷门分支、废弃代码、遗留代码 |

### 1.7.4 识别历史补丁代码

混乱工程典型特征：在原函数里不断追加 `if (flag)` 临时逻辑、补丁代码。

识别特征：
- 同一段逻辑多套分支、临时标记位、注释掉的旧代码
- 后加的代码和原有风格完全不一致

单独标记补丁逻辑，重构时优先剥离。

### 1.7.5 轻量可视化

三种极简方式记录架构：
- **文本思维导图**：Markdown 层级缩进写调用关系
- **简易流程图**（DrawIO/ProcessOn）：只画「函数块 + 调用箭头」
- **依赖矩阵表**：表格记录「文件 A → 依赖文件 B/C」

## 1.8 结合 LLM 的使用技巧

适配超大代码 + 低上下文窗口的限制。

### 1.8.1 绝对不要做的事

- 把整个工程或整个大文件一次性粘贴进对话
- 试图让 AI 一次性理解全架构

### 1.8.2 逐切片迭代

1. **喂入口代码**：粘贴 main 函数 → 让 AI 总结启动流程、一级调用函数
2. **按一级函数逐个提取代码**：AI 指出调用了 `SystemInit()`，单独提取该函数代码分析
3. **逐层递归**：子函数分析完，再提取下级调用，单链路单次分析
4. **汇总全局脉络**：手动拼接调用链、模块依赖、数据流向

### 1.8.3 超长单文件拆分

```bash
# 列出所有函数行号
grep -n "^[a-zA-Z]" big.cpp

# 按函数边界截取代码片段
sed -n '100,200p' big.cpp > part1.cpp
```

保证每一段都是完整函数，再送入 AI。

### 1.8.4 提示词优化

```plaintext
这段代码架构混乱、命名不规范、注释缺失，请只分析：
1. 函数核心功能
2. 调用了哪些其他函数
3. 输入输出/依赖的全局变量
4. 关键执行分支
不用优化代码，不用补逻辑，只梳理执行脉络。
```

## 1.9 相关方法论

以下 5 套工作流与上述步骤高度匹配，可根据场景选用。

### 1.9.1 Legacy Code 四步流

出自《Working Effectively with Legacy Code》行业标准：

1. **摸底（Inventory）**：目录、文件、代码行数、编译产物、依赖
2. **抓入口与主链路（Find Entry Points）**：main、初始化、任务入口、回调
3. **抽取关键符号与依赖（Extract Signatures）**：函数、全局变量、头文件依赖
4. **画脉络、分模块、标坏味道（Map & Partition）**：调用树、数据流向、标记超长函数/全局变量泛滥

特点：不一次性读全代码，只建索引 + 检索 + 跳转，完美适配超 1M 上下文场景。

### 1.9.2 Superpowers 7 阶段逆向流

AI 编程 Agent 常用方案：

| 阶段 | 目标 |
|------|------|
| Scan | 全局扫描，输出工程概览报告 |
| Index | 建符号索引（函数、类、全局变量、宏） |
| Find Critical Path | 找关键路径（入口、高频调用、核心业务流程） |
| Decompose | 逐层拆解，主流程 → 子模块 → 关键函数 |
| Map Dependencies | 依赖图谱（文件、调用、数据、循环依赖） |
| Assess Health | 坏味道扫描（超长函数、重复代码、全局变量） |
| Plan Refactor | 重构优先级（先解耦、再删冗余、再模块化） |

### 1.9.3 cscope + ctags + rg 三神器流

行业里称「无 IDE 超大 C/C++ 工程梳理标准流」：

1. 生成文件列表：`find . -name *.c ... > cscope.files`
2. 建索引：`cscope -bq`、`ctags -R`
3. 入口检索：`rg main` 定位启动点
4. 逐层查调用链：cscope 查函数定义 → 查被调用 → 查调用谁
5. 模块划分：按目录 + 调用关系临时分组
6. 问题标记：循环依赖、超长函数、废弃代码

特点：轻量、无 GUI、不受上下文限制，10w 行到千万行都能处理。

### 1.9.4 绞杀者模式

针对架构完全混乱、无法整体重写、业务不能停的场景：

1. **梳理（Map）**：把现有系统脉络画清楚
2. **分区（Partition）**：按功能/边界切成新旧可隔离块
3. **封装（Wrap）**：给旧代码加薄接口层，内部不动
4. **新功能走新架构**：新代码用新结构、新模块
5. **逐步替换旧代码**：小步快跑，不影响业务
6. **最终淘汰旧层**

特点：不要求一次看懂全系统，先隔离再消化。

### 1.9.5 分段精读 + 拼接流

适配大模型窗口限制的团队实践：

1. **全局索引**：先做 cscope/ctags，AI 不读全代码
2. **切块提问**：每次只丢 1 个文件或 1 个函数给 AI，问功能、输入输出、依赖、副作用
3. **AI 输出摘要**：每个块生成一段精简描述
4. **人工拼接全局脉络**：把各块摘要拼成完整调用链/数据流
5. **迭代补全**：缺哪块再让 AI 精读哪块

特点：绕开窗口限制，人工控全局，AI 做局部精读。

## 1.10 最终输出

梳理完毕后整理成文档，作为重构依据：

| 输出项 | 内容 |
|--------|------|
| **工程基础信息** | 目录结构、文件数量、代码行数、编程语言 |
| **程序入口清单** | 所有 main/启动函数、启动流程简述 |
| **全局调用主干** | 主流程调用树（分层） |
| **逻辑模块划分** | 每个模块职责、包含文件、对外接口 |
| **数据流转** | 核心全局变量/结构体、读写链路 |
| **问题清单** | 循环依赖、超长函数、重复代码、goto 滥用、废弃代码、强耦合点 |
| **重构优先级** | 先解耦、再删冗余、再规范模块化 |

## 1.11 核心原则总结

1. 放弃一次性加载全代码，用索引 + 检索 + 调用链点对点分析
2. 从入口切入，逐层递归，由粗到细，先骨架后细节
3. 把连续调用的一组函数归为一个「逻辑单元」，忽略原有烂架构
4. 先划分逻辑单元，再管规范，先「看懂跑通」再谈架构优化
5. 碎片化阅读 + 分批整合，用工具降噪，屏蔽无效代码干扰

这套方案不依赖 AI 上下文窗口，纯本地工具 + 人工分层梳理，可支撑任意体量的传统 C/C++ 混乱工程。
