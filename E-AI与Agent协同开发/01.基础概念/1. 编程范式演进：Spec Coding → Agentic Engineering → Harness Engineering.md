> AI 辅助编程三级演进：Spec Coding（规格）→ Agentic Engineering（协作）→ Harness Engineering（基础设施）。叠加而非替代。

# 1. 编程范式演进：Spec Coding → Agentic Engineering → Harness Engineering

## 【核心精简提纲】

### 记忆主干

Spell Casting（咒语，已淘汰）之后的三层工程化范式：**写清规格** → **与 Agent 迭代协作** → **构建 Harness 约束与工具链**。选型看项目复杂度 × 团队规模 × 可靠性，非追求「更先进」。

### 核心要点清单

- **Spec Coding**：精确规格 + 审查；适合 API、纯函数、数据转换
- **Agentic Engineering**：目标 + 迭代；适合新项目、功能迭代、重构（= [[E-AI与Agent协同开发/02.工作流与方法论/1. AI-Agent 协同开发（Vibecoding 最佳实践）|Vibecoding]]）
- **Harness Engineering**：权限、上下文、记忆、MCP、质量门禁；团队规模化必备
- **叠加关系**：内层能力不消失——Spec 内化进 Agent 输入，Harness 固化 Agent 实践
- **横切能力**：AI 心智模型、批判性思维、反馈闭环设计

### 学习路径

1. 从 Spec Coding 练「写清接口与边界」
2. 进入 Vibecoding 五阶段工作流
3. 用 CLAUDE.md / MCP 搭建轻量 Harness
4. 对照 [[E-AI与Agent协同开发/02.工作流与方法论/5. TDD-BDD-SDD-DDD：AI时代开发方法论选择|SDD/BDD/TDD 选型]]

### 关键结论 / 流程

```plain
任务清晰可规格化？ → Spec Coding
需探索与迭代？     → Agentic（+ Spec 作契约）
团队要统一 Agent 行为？ → Harness
```

## 【完整拓展详情】
## 1.1 为什么要区分这些范式

随着 Claude、GPT 等大语言模型的能力快速提升，开发者与 AI 的协作方式也在快速演变。从最初的"AI 补全单行代码"，到今天已经形成了三种相对独立的编程范式：

```plain
Spell Casting          → 以 prompt 为中心的"施法"模式
    ↓
Spec Coding            → 以规格说明为中心的"描述"模式
    ↓
Agentic Engineering    → 以协作为中心的"代理"模式
    ↓
Harness Engineering    → 以基础设施为中心的"赋能"模式
```

> 其中最原始的是 **Spell Casting**（咒语编程）— 把 AI 当作 oracle，期望一个精心构造的 prompt 就能产出完美结果。这一模式因不稳定、不可控、不可复用，已逐渐被后续范式取代。本文从 **Spec Coding** 开始讨论真正工程化的范式。

这四种模式并非替换关系，而是**叠加演进**：每次迭代都在保留前一层能力的基础上，解决了前一层无法触及的问题。

| 范式 | 核心问题 | 协作模型 | 开发者角色 |
|------|----------|----------|-----------|
| Spell Casting | "如何写 prompt 让 AI 给出正确答案？" | 一问一答 | prompt 工程师 |
| Spec Coding | "如何让 AI 稳定产出符合预期的代码？" | 规格→生成→审查 | 规格编写者 + 审查者 |
| Agentic Engineering | "如何让 AI 主动探索并迭代解决方案？" | 需求→迭代→交付 | 方向制定者 + 协作者 |
| Harness Engineering | "如何构建让 AI 高效工作的基础设施？" | 框架→Agent→反馈 | 基础设施构建者 |

## 1.2 Spec Coding — 规格驱动编程

### 1.2.1 定义

Spec Coding（规格驱动编程）是指开发者编写详细的规格说明（Specification），由 AI 根据规格生成实现代码的协作模式。核心假设是：**输入越精确，输出越可靠**。

```plain
开发者                  AI
  │                     │
  │── 规格说明 ────────→│
  │  (接口定义 + 行为    │
  │   描述 + 边界条件)   │
  │                     │── 生成实现代码
  │←──── 代码/PR ──────│
  │                     │
  │── 审查反馈 ────────→│── 修正
```

### 1.2.2 核心技能

- **规格编写能力**：能用自然语言精确描述接口语义、行为约束、错误处理策略
- **边界感知能力**：能在规格中覆盖正常路径之外的边界条件与异常场景
- **Review 能力**：能快速判断 AI 生成的实现是否忠实于规格
- **例化思维**：善于用输入-输出对（example-driven）而非抽象描述来定义行为

### 1.2.3 适用场景

| 场景 | 适合度 | 原因 |
|------|--------|------|
| API / SDK 开发 | ⭐⭐⭐⭐⭐ | 接口边界清晰，输入输出可精确定义 |
| 数据转换逻辑 | ⭐⭐⭐⭐⭐ | 纯函数，无副作用，规格即测试 |
| 算法实现 | ⭐⭐⭐⭐ | 需配合测试用例，避免歧义 |
| UI 组件开发 | ⭐⭐⭐ | 视觉规格难以用纯文本描述 |
| 系统集成 | ⭐⭐ | 涉及多组件交互，规格难以覆盖所有状态 |

### 1.2.4 优点

- **可预测性强**：输入-输出关系明确，AI 产出方差小
- **可审查性高**：规格即文档，review 时能快速判断正确性
- **可测试性好**：规格可直接转化为测试用例
- **低认知负荷**：不需要理解 AI 内部机制，只需关注规格本身

### 1.2.5 缺点

- **规格编写成本高**：详细规格可能比代码本身还长
- **覆盖盲区**：无法预见的交互行为无法写在规格中
- **缺乏探索性**：AI 不会主动提示更好的设计方案
- **适用面有限**：对探索性、创新性任务不友好

## 1.3 Agentic Engineering — 代理协作工程

### 1.3.1 定义

Agentic Engineering（代理协作工程）是指开发者将 AI 视为具备自主探索能力的协作伙伴，通过目标描述 + 迭代交互来驱动开发的模式。核心假设是：**AI 能自主探索解决方案空间，人负责方向判断**。

```plain
开发者                  AI Agent
  │                     │
  │── 目标/需求 ───────→│
  │  (What & Why)       │
  │                     │── 方案探索
  │                     │── 实现迭代
  │←── 进展/问题 ──────│
  │                     │
  │── 方向修正 ────────→│── 继续迭代
  │                     │
  │←──── 交付 ─────────│
```

这正是 [[1. AI-Agent 协同开发（Vibecoding 最佳实践）|Vibecoding]] 所描述的工作模式 — 开发者提供方向与判断，Agent 负责探索、生成与迭代。

### 1.3.2 核心技能

- **目标分解能力**：能将模糊需求拆解为 Agent 可执行的里程碑
- **交互式审校能力**：在 Agent 迭代过程中及时识别偏差并提供修正
- **上下文管理能力**：知道何时开启新对话、何时复用已有上下文
- **Prompt 工程深度**：不仅是写 prompt，而是设计多轮交互策略
- **结果评估能力**：能判断 Agent 的探索成果是否达到质量标准
- **工具调用设计**：知道为 Agent 配备哪些工具（搜索、执行、测试）能最大化其效能

### 1.3.3 适用场景

| 场景 | 适合度 | 原因 |
|------|--------|------|
| 全新项目启动 | ⭐⭐⭐⭐⭐ | Agent 可快速搭建骨架，探索技术选型 |
| 功能迭代开发 | ⭐⭐⭐⭐ | 在已有架构上，Agent 能自主完成大部分实现 |
| Bug 修复 | ⭐⭐⭐⭐ | Agent 可自主定位→修复→验证 |
| 代码重构 | ⭐⭐⭐⭐ | Agent 能处理大批量机械性修改 |
| 技术选型调研 | ⭐⭐⭐ | 需人做最终决策，Agent 提供信息 |
| 性能优化 | ⭐⭐ | 需要深度的系统理解，Agent 容易浅尝辄止 |
| 安全审计 | ⭐⭐ | 高风险场景仍需人工主导 |

### 1.3.4 优点

- **生产力大幅提升**：Agent 能并行探索多个方向
- **解放创造力**：开发者从机械编码中解脱，聚焦于设计和决策
- **迭代速度快**：人机协同的反馈循环远快于传统开发
- **适合探索性工作**：Agent 能尝试多种方案，人从中选择最佳路径

### 1.3.5 缺点

- **输出方差大**：Agent 的自主性带来不确定性
- **上下文窗口约束**：长对话后 Agent 容易"遗忘"早期约定
- **调试复杂化**：错误可能来自 Agent 的自主决策路径
- **需要新的管理技能**：管理 Agent 与管理人类开发者不同
- **安全风险**：Agent 的自主操作可能引入未预期的副作用

## 1.4 Harness Engineering — 框架工程

### 1.4.1 定义

Harness Engineering（框架工程）是指构建基础设施、框架和工具链，使 AI-Agent 能在受控环境中高效、可靠、安全地完成任务的开发模式。核心假设是：**Agent 的能力上限由它所在的框架决定**。

```plain
                      Harness（框架层）
  ┌─────────────────────────────────────────────┐
  │  权限控制      上下文管理      记忆持久化      │
  │  ┌──────┐    ┌──────────┐   ┌──────────┐   │
  │  │ allow│    │ context  │   │ memory   │   │
  │  │ deny │    │ window   │   │ store    │   │
  │  │ list │    │ strategy │   │          │   │
  │  └──────┘    └──────────┘   └──────────┘   │
  │                                              │
  │  工具注册     反馈循环      质量门禁         │
  │  ┌──────┐    ┌──────────┐   ┌──────────┐   │
  │  │ MCP  │    │ test     │   │ lint     │   │
  │  │ tools│    │ verify   │   │ review   │   │
  │  └──────┘    └──────────┘   └──────────┘   │
  └─────────────────────────────────────────────┘
           ↕                          ↕
    ┌──────────┐              ┌──────────┐
    │ Developer│←──────────→│  AI Agent │
    └──────────┘              └──────────┘
```

Harness 的核心意义在于：**将 Agentic Engineering 中的经验性实践固化为可重复的基础设施**。每一次"这个 prompt 有效"、"这个流程靠谱"、"这个约束必要"的经验，都应被编码进 harness 中，而不是留在开发者的个人笔记里。

### 1.4.2 Harness 的构成要素

一个成熟的 harness 通常包含以下组件：

| 组件 | 作用 | 示例 |
|------|------|------|
| **权限沙箱** | 约束 Agent 的操作范围 | 文件系统白名单、网络访问控制、命令执行黑名单 |
| **上下文路由** | 管理 Agent 的信息来源 | 自动加载相关文件、检索记忆、注入项目规则 |
| **反馈回路** | 提供闭环验证 | 自动运行测试、lint 检查、diff review |
| **护栏系统** | 防止 Agent 越界 | 操作确认机制、回滚能力、紧急终止 |
| **工具注册表** | 定义 Agent 能使用的工具 | MCP servers、自定义 scripts、API 网关 |
| **记忆系统** | 跨会话的知识持久化 | 项目记忆、用户偏好、决策记录 |
| **编排引擎** | 多 Agent 协作调度 | 任务分解、结果聚合、冲突解决 |

### 1.4.3 核心技能

- **系统设计能力**：能设计 Agent 运行环境的整体架构
- **安全工程思维**：理解最小权限原则、沙箱隔离、操作审计
- **工具链构建能力**：能开发/集成让 Agent 效能倍增的工具
- **编排与调度能力**：设计多 Agent 的协作流程与通信协议
- **可靠性工程**：确保 Agent 产出的稳定性与可复现性
- **UX 设计（面向开发者）**：设计开发者与 Harness 的交互体验
- **元编程能力**：编写生成代码的代码、配置 Agent 行为的配置

### 1.4.4 适用场景

| 场景 | 适合度 | 原因 |
|------|--------|------|
| 团队级 AI 协作平台 | ⭐⭐⭐⭐⭐ | 需要统一的规则、权限、工具链 |
| CI/CD 集成 | ⭐⭐⭐⭐⭐ | Agent 可在 harness 中安全地自动执行 |
| 大规模代码库维护 | ⭐⭐⭐⭐ | 需要精细的上下文管理和操作约束 |
| 合规性敏感项目 | ⭐⭐⭐⭐ | 需要在严格审计下使用 Agent |
| 个人效率提升 | ⭐⭐⭐ | 简单的 harness（MCP + memory）就够用 |
| 多 Agent 协作系统 | ⭐⭐⭐⭐⭐ | 无 harness 几乎无法管理 |

### 1.4.5 优点

- **可靠性与安全性**：Agent 在约束框架内操作，风险可控
- **可复现性**：相同 harness 下 Agent 行为一致性好
- **团队标准化**：所有成员共享相同的 Agent 协作基础设施
- **知识沉淀**：项目经验编码为 harness 配置，不随人员流失
- **渐进自动化**：可以逐步增加 harness 的自动化程度

### 1.4.6 缺点

- **构建成本高**：成熟的 harness 需要大量前期投入
- **灵活性损失**：框架约束可能限制 Agent 的创造性
- **维护负担**：harness 本身需要持续演进
- **学习曲线陡**：团队成员需要学习 harness 的使用方式
- **过度工程风险**：容易在 harness 上过度投资，而实际需求很简单

## 1.5 演进关系：三层范式的有机衔接

这三种范式不是非此即彼的选择，而是一个**层层叠加**的演进过程：

```plain
                    ┌──────────────────────────┐
                    │     Harness Engineering   │
                    │  (框架层：基础设施与工具链)   │
                    │                           │
                    │  ┌────────────────────┐   │
                    │  │  Agentic Engineer. │   │
                    │  │ (协作层：人机协同)    │   │
                    │  │                     │   │
                    │  │  ┌──────────────┐  │   │
                    │  │  │ Spec Coding  │  │   │
                    │  │  │ (规格层：精准  │  │   │
                    │  │  │  描述)       │  │   │
                    │  │  └──────────────┘  │   │
                    │  └────────────────────┘   │
                    └──────────────────────────┘
```

### 1.5.1 Spec Coding → Agentic Engineering

在 Agentic Engineering 中，Spec Coding 的能力并未消失，而是**内化**为 Agent 的输入技能：

- **Spec 不再是人写给 AI 的指令**，而是人写给 Agent 的**契约**
- Agent 会主动帮你补全规格中未覆盖的边界
- 但**高质量的规格仍然是高质量产出的基础**
- 区别在于：Agentic Engineering 允许规格在迭代中逐步细化，而非一次性写死

### 1.5.2 Agentic Engineering → Harness Engineering

当 Agentic Engineering 在团队中推广时，Harness 成为必需品：

- **单打独斗时**：你可以在对话中自己管理上下文和规则
- **团队协作时**：必须通过 harness 统一 Agent 的行为标准
- **简单项目时**：一个 MCP server 就够用
- **复杂系统时**：需要权限管理、多 Agent 编排、审计日志
- Harness 不是替代 Agentic 工作流，而是**使其可规模化**

### 1.5.3 三层的协作关系

| 层级 | 关注点 | 决策者 | 变更频率 |
|------|--------|--------|----------|
| Spec Coding | 具体功能的输入输出行为 | 开发者 | 每个任务 |
| Agentic Engineering | 与 Agent 的交互策略与流程 | 开发者 + Agent | 每个项目 |
| Harness Engineering | Agent 运行的基础设施 | 架构师 / 平台团队 | 按需演进 |

```plain
                      ┌──────────────────────┐

        │                                         │

        │   Spec Coding   Agentic     Harness    │
        │                   Engineering Engineer.│
        └─────────────────────────────────────────┘
```

### 1.6.2 横跨所有范式的核心能力

| 能力 | 为什么重要 | 如何培养 |
|------|-----------|---------|
| **AI 心智模型** | 准确预测 AI 的行为与局限性 | 大量实践 + 阅读 AI 原理（Transformer、RLHF 基础） |
| **批判性思维** | 不盲目信任 AI 输出，保持质疑 | 始终验证结果，分析 AI 的错误模式 |
| **快速原型能力** | 用最小成本验证 AI 是否能完成任务 | 先写 prompt 测试，再投入完整实现 |
| **反馈设计** | 设计高效的闭环验证机制 | 在每个环节嵌入自动化检查 |

### 1.6.3 从 Spec Coding 到 Harness 的成长路径

```plain
初级阶段（Spec Coding 为主）
├── 能写清晰的规格说明
├── 能通过 prompt 控制 AI 的输出质量
└── 能有效 review AI 生成的代码

中级阶段（Agentic Engineering 为主）
├── 能设计多轮交互策略
├── 能为 Agent 配备合适的工具
├── 能管理 Agent 的上下文和记忆
└── 能处理 Agent 的"幻觉"与偏差

高级阶段（Harness Engineering 为主）
├── 能设计 Agent 运行的基础设施
├── 能构建多 Agent 协作系统
├── 能设计安全沙箱与权限模型
├── 能将团队经验编码为可复用的配置
└── 能度量并持续优化 Agent 的产出质量
```

```plain
Harness Engineering    "搭什么框架"           基础设施构建者
```

- **Spell Casting** 已被证明不可靠——同样的 prompt 在不同上下文产生不同结果
- **Spec Coding** 是进入工程化的第一步，适合接口清晰、行为可定义的任务
- **Agentic Engineering** 是当前的主流范式，实现生产力质的飞跃
- **Harness Engineering** 是规模化落地的关键，将经验固化为基础设施
- 三种范式是**叠加演进的**，高级开发者应同时具备各层级的能力
- 范式选择取决于**项目复杂度 × 团队规模 × 可靠性要求**，而非追求"更先进"的范式
