# 1. Git 基础

Git 分布式版本控制：本地 `commit` 留痕，远程 `push/pull` 协作；分支隔离开发，`merge/rebase` 整合历史。

## 【核心精简提纲】

### 记忆主干

工作区 → `git add` 暂存区 → `git commit` 本地仓库 → `git push` 远程。日常三板斧：`status` 看状态、`diff` 看差异、`log --oneline --graph` 看历史。切分支前未完工用 `stash` 暂存。

### 核心要点清单

- **三区**：working directory / staging area / repository（HEAD 指向当前 commit）
- **远程**：`clone` 一步到位；已有目录 `init` + `remote add origin` + `push -u`
- **分支**：`switch -c` 创建并切换；`merge` 合并；`branch -d` 删已合并分支
- **协作**：feature 分支开发 → `rebase`/`merge` → PR → Squash merge
- **回退**：`reset --soft` 保留改动只移指针；`--hard` 丢弃改动；`reflog` 救误操作
- **变基**：`rebase -i` 整理提交；`hash` 指要保留/修改的 commit，非其父提交
- **储藏**：`stash push -m` 暂存 WIP；`pop` 恢复；与 `status`（只读查询）互补

### 学习路径

1. 配 SSH + `user.name/email` → `clone` 或 `init` 建仓库
2. 练 `add/commit/log/diff/branch/switch`
3. 学 `pull/push/merge` 与 feature 分支协作流
4. 进阶：`rebase -i`、`reset`、`stash`、`worktree`、浅克隆 `--depth=1`

### 关键结论 / 流程

```
开发 → add → commit（本地高频）
     → fetch/rebase → push（同步远端）
切分支前：status → stash → switch → 修完 → pop
```

#### 2.2.1 git首次安装

```Bash

# 配置完成后可以开始使用git

#### 2.2.2 git本地仓库

```Bash
git init # 在本地初始化一个空的仓库
git add a.x # 上传a.x 也可以 git add . 上传当前目录所有内容 此时只是上传文件到暂存区
git status # 查看当前暂存区状态
git commit -m "输入你的提交内容注释"  # 将暂存区文件传到仓库
git log # 查看上传记录 
git log --oneline # 查看简单的记录,log指令下使用q退出。
git log --graph --oneline # 查看分支图（合并/分支拓扑一目了然）
git log --graph --oneline --all # 查看所有分支的分支图
git diff #比较本地和仓库的差别，也可以比较不同版本的差异

# HEAD 意思是比较HEAD也就是最新版本和上一个版本，~后可以加数字表示从HEAD
# 回退几个版本
git diff HEAD~ 
# 在log可以看到每次提交前面有个版本号 ，这里可以使用版本号查询差异
git diff version_x version_y  
# 从本地和仓库删除名为filename的文件，
git rm filename
# 若是只从仓库删除 加上--cached修饰
git rm --cached filename
```

#### 2.2.3 git远程仓库

- ssh配置：
    1. `cd ~/.ssh`
    2. `ssh-keygen -t rsa -b 4096 -C "你的邮箱"`,生成公钥。
    3. `Enter`后输入文件名，然后输入私钥密码，默认不输入直接`Enter`。
    4. `cat 秘钥.pub`,复制内容并绑定到仓库网站的设置里。
    5. 在`.ssh`文件夹下创建一个`config`文件(无后缀文件),填入配置。如下：
       - ```bash
                #github
		        Host github.com
			        Hostname ssh.github.com
			        Port 22 # 443也可以
			        PreferredAuthentications publickey  
			        IdentitiesOnly yes  
			        IdentityFile ~/.ssh/id_rsa # 根据你的私钥路径修改  
			        TCPKeepAlive yes  
			        ServerAliveInterval 60
         ```

#### 2.2.4 关联本地和远程

1. 本地仓库关联单个仓库:
	1. 无本地仓库直接关联远程仓库：`git clone`,无需`git init`。
	2. 有本地仓库再关联远程仓库：
		1. `git init`，初始化本地仓库。
		2. `git remote add origin [SSH链接1]`,一个本地仓库可以关联多个远程仓库。
		3. `git push -u origin main`,首次推送要关联上游具体分支。
	- 注：`remote` 表示远程仓库，`origin` 是远程仓库的默认别名（可自定义）。

2. 本地仓库关联第二个仓库：
	-  再次`git remote add origin [SSH链接2]`,关联第二个仓库`push`，但不关联`pull`操作。这个操作会覆盖原push，需要再执行关联SSH2后，再次执行`git remote add origin [SSH链接1]`,
	-  `git remote add newname [SSH链接2]`,关联第二个仓库完整，操作时要选择哪个仓库。
	-  `git remote show origin`，显示当前仓库的推拉上游关系。
3. 有时可能会提示无法关联同一个仓库：
	- 使用`git remote set-url --add --push origin [SSH链接2]`,可以追加`push`地址。
#### 2.2.5 推拉和分支

1. 分支名称统一（可选操作）：`git branch -M main` ，将本地分支名称修改为`main`。
	- 目的是与远程仓库主分支同名，避免后续操作混淆。
	- 若不指定具体分支，`git pull`/`git push` 默认操作对象为 `main` 分支；
	- 若推送的分支在远程不存在，会自动创建该分支。

2. 推拉操作：
	- `git pull origin main`,拉取远程`main`分支的最新代码。
    - `git push origin x1`：将本地分支 x1 推送到远程仓库。

    - `git branch`,查询本地所有分支，显示数量和名称，当前分支前有 `*`标记。
    - `git branch xx`,新建本地`xx`分支。
    - `git switch dev`,切换到本地`dev`分支。
    
3. 分支合并删除：
	1. `git merge xxx`,合并`xxx`分支到当前本地分支。
	2. `git branch -d xxx`,删除已合并的分支 `xxx`。
		- 仅能删除已合并的分支。
	3. `git branch -D xxx`。
		- 强制删除未合并的分支，可能丢失修改。
4. 分支切换:不同分支的文件状态是独立的。
	1. 当不同分支间存在差异，如`main`分支删除了`1.cpp`，但`dev`分支未删除。
		- 切换分支后,工作目录会自动更新为对应分支的文件状态。
			- `git switch dev` ,`1.cpp`仍存在。
			- `ls`查看文件列表，通过`git status`查看分支状态。

## 1.3 协作流程

## 3.1 merge/pr

```Bash
# 1. 克隆并创建分支
git clone xxx.git
git switch -c feature/new-feature #feature/new-feature 是一个完整的分支名

# 2. 开发并多次提交
git add . && git commit -m "feat: 实现用户登录核心功能" 
git add . && git commit -m "fix: 修复表单验证失败的边界 case" 
git add . && git commit -m "perf: 优化数据库查询效率"

# 3. 同步远程 main 并压缩提交
git fetch origin
git rebase -i origin/main  # 压缩提交

# 4. 解决冲突（如有）
git rebase --continue

# 5.(二选一)
# 5.1 推送到远程
git push origin feature/new-feature
# 5.2 直接本地merge
git switch main
git merge feature/new-feature
git push origin main

# 6. 创建 PR 并选择 "Squash and Merge"（可选）
# 7. 删除分支
git branch -d feature/new-feature
git push origin --delete feature/new-feature
```

## 3.2 worktree

```Bash
# 不同于clone多个文件夹，git worktree共享同一个.git文件，
# 即分支共享，减少了文件大小

git clone xxx.git
git worktree add ../mydev origin dev #创建dev分支的本地分支在../mydev

# 所有的git branch 操作，c d 增删，都可以在这两个文件夹内观测到
# 正常的提交是一样的，log和暂存区独立
cd mydev # 在独立的文件夹开发，独立暂存区，但branch共享
cd main # 在独立的文件夹开发，独立暂存区，但branch共享

# 用后删除
# 方法1：使用 git worktree remove（Git 2.15+）
git worktree remove ../mydev  # 自动清理目录并更新引用,不可以在mydev内使用

# 方法2：手动删除（你的方式）
cd .. && rm -rf mydev         # 删除工作树目录
cd main-repo && git worktree prune  # 清理失效的工作树引用
```

## 1.4 `git reset`
1. `git reset`会移动当前的`git`指针。
	- `git reset --hard HEAD~5`,会完全重置本地状态到之前的版本，适用于错误恢复。
	- `git reset --soft HEAD~5`,只移动指针，但内容没变，中间提交的冲突内容被视为一次修改，不需要处理冲突，等同直接在旧版本上修改到`HEAD`。
2. 由于本地指针早于远程，需要配个`git push --force-with-lease`进行安全的强制推送。
3. `git reset --hard HEAD~5`到了错误的版本，可以`git reflog`查看`git`的指针记录，再`git reset --hard hash值`回到正确的原版本。
## 1.5 `git rebase`
1. `git rebase -i hash`,先变基到`hash`值对应的版本，然后重新修改提交记录，中途需要处理冲突。
- `git rebase`和`git reset`的`hash`版本应当是你要保留或修改`commit`的版本，而不是他的上一个版本。

## 1.6 浅克隆加速
- `git clone --depth=1 --branch main <远程仓库地址>`， 只拉取主线最新 1 次提交（--depth=1 表示仅获取最新1层历史） ，大项目可以加速拉取，但不保有提交历史，工具类常用。

## 1.7 `git stash` — 临时储藏

## 7.1 用途

当你正在开发一个功能，需要临时切换到其他分支处理紧急事务，但当前修改还没完成、不适合提交时，`stash` 可以将当前工作区（`working directory` + `staging area`）的改动**临时储藏**起来，让工作区恢复干净。

```
stash 前：                        stash 后：
工作区:  修改了 a.cpp, b.cpp      工作区:  干净（回到 HEAD 状态）
暂存区:  有 staged 的文件          暂存区:  空

                 │
                 ▼
           ┌─────────────┐
           │ stash 栈    │
           ├─────────────┤
           │ stash@{0}   │  ← a.cpp 和 b.cpp 的修改存到这里
           └─────────────┘
```

## 7.2 基本操作

```bash
# 储藏当前所有修改（工作区 + 暂存区）
git stash

# 储藏时添加描述信息（推荐，便于识别）
git stash push -m "feat: 用户登录验证逻辑（未完成）"

# 查看储藏列表
git stash list

# 恢复到最近一次 stash 并删除该 stash
git stash pop

# 恢复但不删除 stash
git stash apply

# 恢复指定的 stash（不一定是最近的那个）
git stash apply stash@{1}
git stash pop stash@{1}

# 删除指定的 stash
git stash drop stash@{1}

# 清空所有 stash
git stash clear
```

### 场景示例

```bash
# 1. 在 dev 分支上开发到一半
git add main.cpp
git status
#  Changes to be committed:  main.cpp

# 2. 突然要修一个紧急 bug
git stash push -m "WIP: 重构请求处理逻辑"
# Saved working directory and index state WIP ...

# 3. 工作区已干净，可以切分支修 bug
git switch hotfix
# ... 修 bug、提交 ...

# 4. 切回来继续开发
git switch dev
git stash pop
# 之前的修改全部恢复
```

## 7.3 进阶用法

### 储藏未跟踪文件

默认 `git stash` 只储藏被 Git 跟踪（tracked）的文件。要连带新文件一起储藏：

```bash
git stash push -u                   # -u = --include-untracked
git stash push --include-untracked  # 同上，完整写法
```

### 储藏时保留暂存区状态

默认 `stash` 把工作区和暂存区的改动全部存起来，恢复时都回到工作区。要保留暂存区状态：

```bash
git stash push --staged    # 只储藏暂存区的改动，工作区改动不动
```

### 从 stash 创建分支

如果恢复 stash 时遇到冲突，可以基于 stash 创建分支：

```bash
git stash branch new-branch stash@{0}
# 创建一个新分支，应用 stash 的修改，然后删除 stash
```

## 7.4 常用选项汇总

| 选项 | 说明 |
|------|------|
| `push -m "msg"` | 带描述信息的储藏 |
| `-u` / `--include-untracked` | 连带未跟踪文件一起储藏 |
| `--staged` | 只储藏暂存区 |
| `list` | 查看储藏列表 |
| `pop` | 恢复并删除最上面的 stash |
| `apply [stash@{N}]` | 恢复但不删除 |
| `drop [stash@{N}]` | 删除指定 stash |
| `clear` | 清空所有 stash |
| `branch <name> [stash@{N}]` | 从 stash 创建新分支 |

## 1.8 `git status` vs `git stash` 的区别

二者**完全不同**，不存在"选哪个"的问题——它们是互补的，服务于不同的步骤。

| 维度 | `git status` | `git stash` |
|------|-------------|-------------|
| **本质** | **查询命令**（只读） | **操作命令**（修改状态） |
| **作用** | 查看当前工作区与暂存区的状态 | 将当前修改临时保存并清空工作区 |
| **影响工作区** | ❌ 不改变任何文件 | ✅ 暂存修改，恢复工作区为 HEAD |
| **频率** | 日常高频（随时查看状态） | 低频（切换分支前） |
| **输出** | 哪些文件修改了、删除了、新增了 | 无直接输出，存到 stash 栈 |
| **是否可恢复** | — | ✅ 通过 `pop`/`apply` 恢复 |
| **常见搭配** | `git add` 前查看改了什么 | `git switch` 前保存进度 |

### 典型组合流程

```bash
# 1. status → 看清当前改了什么
git status
# Changes not staged for commit:
#   modified:   src/server.cpp
#   modified:   src/config.h

# 2. stash → 临时存起来，腾出干净工作区
git stash push -m "WIP: 配置解析重构"
git status
# nothing to commit, working tree clean  ← 干净了

# 3. 切分支、修 bug ...
git switch hotfix
# ...
git switch dev

# 4. pop → 恢复之前的修改
git stash pop
git status
# modified: src/server.cpp
# modified: src/config.h              ← 回来了
```

### 一句话总结

> **`git status` 是问"我改了什么？"，`git stash` 是说"先帮我存起来"。**
