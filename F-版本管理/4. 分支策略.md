# 1. 分支策略概述

团队约定的分支创建、命名、合并与发布模型；选型取决于发布频率、团队规模与 CI/CD 成熟度。

## 【核心精简提纲】

### 记忆主干

固定版本周期 → **Git Flow**；SaaS 强 CI、日多次部署 → **GitHub Flow** / **Trunk-Based**；多环境逐级晋级 → **GitLab Flow**。

### 策略速查

| 策略 | 常驻分支 | 复杂度 | 典型场景 |
|------|----------|--------|----------|
| **Git Flow** | main + develop | 高 | 固定版本号、需 release/hotfix 隔离 |
| **GitHub Flow** | 仅 main | 低 | SaaS、CI/CD 成熟、持续部署 |
| **GitLab Flow** | main + 环境分支 | 中 | staging→production 逐级晋级 |
| **Trunk-Based** | main（极短命 feature） | 低 | 大厂、feature flag、强自动化 |

### 核心要点清单

- **Git Flow**：`feature/*`→develop，`release/*`→main+tag，`hotfix/*` 从 main 双向合并
- **GitHub Flow**：main 永远可部署；feature 分支 + PR 审查 + Squash merge
- **合并策略**：Squash（main 线性）/ merge commit（保留分支图）/ rebase merge
- **本 vault**：本地高频 commit，推远端前 squash（见 [[F-版本管理/2. log规范|Commit 规范]]）

### 学习路径

1. 个人项目：GitHub Flow（main + feature + PR）
2. 有版本号产品：理解 Git Flow 的 release/hotfix
3. 多环境：GitLab Flow 环境分支
4. 对照 [[F-版本管理/1. git|Git 基础]] 练 `switch`/`merge`/`rebase`

### 关键结论 / 流程

```plain
固定版本周期？ → Git Flow
每天多次部署 + 强 CI？ → GitHub Flow / Trunk-Based
多环境验证？ → GitLab Flow
```

## 【完整拓展详情】

主流策略对比：

| 策略 | 复杂度 | 发布频率 | 适用团队 |
|------|--------|----------|----------|
| **Git Flow** | 高 | 低频（版本发布） | 有固定 release 周期的产品 |
| **GitHub Flow** | 低 | 高频（持续部署） | CI/CD 成熟、快速迭代的团队 |
| **GitLab Flow** | 中 | 依环境而定 | 多环境部署（staging/production） |
| **Trunk-Based** | 低 | 极高（每日多次） | 高度自动化、小团队 |

## 2. Git Flow

Git Flow 由 Vincent Driessen 于 2010 年提出，是一套**严格的、面向版本发布**的分支模型。核心思路是维护**两条常驻分支**，配合**三类临时分支**完成开发、发布和热修复。

## 2.1 常驻分支

```text
main        ───●─────────────────────●────────  (稳定发布版)
                \                   /
develop         ─●────●────●────●───             (集成开发)
```

| 分支 | 角色 | 说明 |
|------|------|------|
| **`main`** | 生产就绪 | 只接受来自 `release` 和 `hotfix` 的合并，每个 merge 对应一个发布的版本 |
| **`develop`** | 集成分支 | 日常开发的基准分支，所有 `feature` 分支从此分出、合并回来 |

## 2.2 临时分支

### 2.2.1 Feature 分支 — `feature/*`

从 `develop` 分出，开发完成后合并回 `develop`：

```bash
# 创建并切换到 feature 分支
git switch -c feature/user-login develop

# ... 多次提交 ...
git add . && git commit -m "feat: 实现用户登录"

# 开发完成，合并回 develop
git switch develop
git merge --no-ff feature/user-login  # --no-ff 保留分支历史
git branch -d feature/user-login
```

**命名约定：** `feature/<简要功能名>`，如 `feature/oidc-auth`、`feature/dark-mode`。

### 2.2.2 Release 分支 — `release/*`

当 `develop` 积累到可发布状态时，创建 release 分支，进入**冻结期**（只修 Bug，不增功能）：

```bash
# 从 develop 创建 release 分支
git switch -c release/1.2.0 develop

# 在 release 分支上修 Bug、更新版本号
git commit -m "fix: 修复 release 测试发现的空指针"
git commit -m "chore: bump version to 1.2.0"

# 合并到 main（打 tag）
git switch main
git merge --no-ff release/1.2.0
git tag -a v1.2.0 -m "Release v1.2.0"

# 合并回 develop（确保修复同步到开发线）
git switch develop
git merge --no-ff release/1.2.0

# 删除 release 分支
git branch -d release/1.2.0
```

**命名约定：** `release/<版本号>`，如 `release/1.2.0`、`release/2.0.0-rc1`。

### 2.2.3 Hotfix 分支 — `hotfix/*`

生产环境出现紧急 Bug，从 `main` 分叉，修复后同时合并回 `main` 和 `develop`：

```bash
# 从当前生产版本创建 hotfix
git switch -c hotfix/1.2.1 main

# 修复并提交
git commit -m "fix: 修复生产环境 OOM 异常"

# 合并回 main + 打 tag
git switch main
git merge --no-ff hotfix/1.2.1
git tag -a v1.2.1 -m "Hotfix v1.2.1"

# 同步回 develop
git switch develop
git merge --no-ff hotfix/1.2.1

git branch -d hotfix/1.2.1
```

**命名约定：** `hotfix/<版本号>` 或 `hotfix/<缺陷简述>`，如 `hotfix/1.2.1`、`hotfix/null-pointer-login`。

## 2.3 完整生命周期图

```text
        feature/*                develop              main
         │                        │                    │
         ├── feature/login ─────► │                    │
         │                        ├── release/2.0 ──► ● v2.0 (tag)
         │                        │                    │
         │                        │                    ├── hotfix/2.0.1 ──► ● v2.0.1
         │                        │◄───────────────────┘
         │                        │                    │
```

## 2.4 Git Flow 的优缺点

| 优点 | 缺点 |
|------|------|
| 严格规范，适合大型团队 | 流程复杂，学习成本高 |
| 版本发布可追溯（tag + release） | 分支生命周期长，合并频繁 |
| hotfix 与正常开发隔离 | `develop` vs `main` 常不同步 |
| 适合传统软件（有版本号） | 对持续部署不太友好 |

## 3. GitHub Flow

GitHub Flow 由 GitHub 团队推广，核心理念是**简化**：只有一条常驻分支 `main`，所有开发在 feature 分支上完成，通过 Pull Request 审查后合并。

## 3.1 核心规则

1. **`main` 永远是可部署的** — 任何时刻 `main` 的代码都是生产就绪的
2. **从 `main` 开新分支** — 每次开发一个新功能或修 Bug
3. **频繁推送并开 PR** — 即使 WIP（Work In Progress）也鼓励推送，便于协作
4. **PR 审查通过后合并回 `main`** — 代码审查是质量门禁
5. **合并后立即部署** — CI/CD 自动触发部署

## 3.2 标准工作流

```bash
# 1. 从 main 创建 feature 分支
git switch -c feat/add-search main

# 2. 日常开发，频繁提交
git commit -m "feat: 添加搜索页面骨架"
git commit -m "feat: 实现搜索 API 调用"
git commit -m "fix: 处理搜索输入防抖"

# 3. 推送到远程，创建 PR
git push origin feat/add-search
# → 在 GitHub 上创建 Pull Request

# 4. PR 审查 → CI 通过 → 合并（Squash and Merge）
# → 自动部署到生产环境

# 5. 删除远端分支
git push origin --delete feat/add-search
```

## 3.3 PR 合并策略

| 策略 | 效果 | 适用场景 |
|------|------|----------|
| **Create a merge commit** | `--no-ff` 保留完整分支历史 | feature 分支上有多人协作 |
| **Squash and merge** | 所有 commits 压缩成一个 | 单人开发，保持 main 线性历史 |
| **Rebase and merge** | 逐个 rebase 提交到 main 顶端 | 追求完美线性历史 |

**推荐：** 默认用 **Squash and merge**。优点是 `main` 历史简洁，每个功能对应一个 commit；缺点是丢失了 feature 分支的内部提交细节，但 PR 页面中仍然可查。

## 3.4 GitHub Flow 示例（完整流程）

```bash
# === 场景：添加用户头像上传功能 ===

# 1. 从最新的 main 创建分支
git checkout main
git pull origin main
git switch -c feat/avatar-upload

# 2. 开发迭代
echo "avatar upload logic" > avatar.go
git add avatar.go && git commit -m "feat: 添加头像上传核心逻辑"
echo "tests" > avatar_test.go
git add avatar_test.go && git commit -m "test: 添加头像上传测试用例"
git push origin feat/avatar-upload

# 3. 在 GitHub 上创建 PR → 同事 review → CI 通过 → Squash and Merge
# 4. 本地同步
git switch main
git pull origin main

# 5. 删除本地分支
git branch -d feat/avatar-upload
```

## 3.5 GitHub Flow 的优缺点

| 优点 | 缺点 |
|------|------|
| 极简，学习成本低 | 没有 develop 分支，`main` 稳定性依赖 CI 和审查 |
| 天然适配 CD（Continuous Deployment） | 难以同时管理多个版本（如 LTS 与最新版） |
| PR 即代码审查入口 | 不适合严格版本号管理的传统软件 |
| CI 集成度高 | 紧急 hotfix 和生产发布重叠时可能混乱 |

## 4. Git Flow vs GitHub Flow 对比

| 维度 | Git Flow | GitHub Flow |
|------|----------|-------------|
| **常驻分支** | `main` + `develop` | 只有 `main` |
| **临时分支** | `feature/*`、`release/*`、`hotfix/*` | 只有 feature 分支（对应 PR） |
| **发布方式** | release 分支 → tag → 部署 | 合并到 `main` → 自动部署 |
| **hotfix 处理** | 从 `main` 分叉，双向合并 | 同 feature 分支流程（PR + CI） |
| **代码审查** | 可选（多在 merge 前审查） | 强制（PR 审查是流程核心） |
| **CI/CD 依赖** | 低（流程本身保证质量） | 高（CI 是质量唯一门禁） |
| **历史复杂度** | 分支图较复杂（`--no-ff` 有 merge bubble） | 线性历史（Squash merge） |
| **适用场景** | 版本软件、嵌入式、移动应用 | Web 服务、SaaS、持续部署 |

## 4.1 如何选择

```plain
你的团队符合以下条件吗？
├── 有固定版本号/发布周期（如 v1.0 → v2.0） → 考虑 Git Flow
├── 需要同时维护多个线上版本（LTS + 最新） → 考虑 Git Flow
├── 每天多次部署到生产环境                 → GitHub Flow
├── 团队人数 < 10，产品是 SaaS/Web 服务     → GitHub Flow
└── 已有成熟 CI/CD 流水线                  → GitHub Flow 或 Trunk-Based
```

## 5. 补充：GitLab Flow & Trunk-Based Development

## 5.1 GitLab Flow

GitLab 提出的折中方案，在 `main` 之外引入**环境分支**（如 `staging`、`pre-production`、`production`），代码逐级向上合并：

```text
feature  →  main  →  staging  →  production
```

适用于需要多环境验证但不能做到「合并即上线」的场景。

## 5.2 Trunk-Based Development（主干开发）

- 所有开发者直接在 `main`（trunk）上提交或创建**极短命的 feature 分支**（存活 < 1 天）
- 通过 **feature flag** 控制未完成功能的可见性
- 要求：高度自动化测试、小步提交、快速 Code Review
- 适合：Google、Netflix 等超大规模工程团队
