# 1. 跨平台动态库加载机制

Linux 与 Windows 在动态库加载机制上的核心差异：**搜索策略、路径嵌入方式、符号解析模型**。

## 【核心精简提纲】

### 记忆主干

- **Linux**：`ld-linux.so` 按优先级搜索 rpath / `LD_LIBRARY_PATH` / ld.so.cache / 默认路径；`$ORIGIN` 解耦路径与部署目录；`dlopen` 按需加载
- **Windows**：进程目录 → 系统目录 → Windows 目录 → `PATH`；`LoadLibrary` 可用绝对/相对路径；Known DLLs 机制劫持系统库搜索
- **`$ORIGIN`**：运行时展开为 `.so`/可执行文件所在目录，`-Wl,-rpath,'$ORIGIN:$ORIGIN/lib'` 实现 relocatable 部署
- **Windows 无 rpath 等价物**：依赖 `.exe` 所在目录、`SetDllDirectory`、manifest 或 `PATH`；部署捆绑 DLL 时优先放在 exe 同目录

### 核心要点清单

- **Linux 搜索顺序**（GNU ld）：`DT_RPATH` → `LD_LIBRARY_PATH` → `DT_RUNPATH` → `/etc/ld.so.cache` → `/lib` → `/usr/lib`
- **Windows 搜索顺序**（Safe DLL Search Mode）：已知 DLL → 进程目录 → 系统目录（`System32`）→ 16 位系统目录 → Windows 目录 → 当前目录 → `PATH`
- **`$ORIGIN`** 是 `ld-linux.so` 的内置变量，不是 shell 变量 — 编译时写入 ELF，运行时展开
- `DT_RPATH` vs `DT_RUNPATH`：前者优先级高于 `LD_LIBRARY_PATH`，后者低于；现代 Linux 推荐 `DT_RUNPATH`（`--enable-new-dtags`）
- **Windows 已知 DLL**（Known DLLs）：注册表 `HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\KnownDLLs` 中的系统库优先从 `%windir%\System32` 加载，即使 `PATH` 或进程目录有同名文件
- **DLL 重定向**（`.local` 文件 / manifest）：Windows 可通过 manifest 或 `app.exe.local` 文件强制优先使用私有 DLL

### 关键结论 / 流程

```plain
Linux relocatable 部署：-Wl,-rpath,'$ORIGIN:$ORIGIN/lib' →
  运行时 $ORIGIN 展开为可执行文件目录 → 优先在该目录及 lib/ 子目录查找 .so
Windows 对应做法：将 DLL 放在 .exe 同级目录 →
  系统 Safe DLL Search Mode 默认优先搜索进程目录
```

## 1.1 Linux 动态库加载机制

相关笔记：[[C-Linux生态/01.Linux环境/3. rpath与链接路径|rpath 与链接路径]]、[[C-Linux生态/01.Linux环境/4. 加载与劫持|加载与劫持]]

### 1.1.1 加载器

Linux 由 `ld-linux.so`（`/lib64/ld-linux-x86-64.so.2`）在程序启动时负责解析所有 `DT_NEEDED` 条目，链接共享库。

```bash
# 查看程序使用的动态链接器
readelf -l your_app | grep interpreter

# 输出示例
[Requesting program interpreter: /lib64/ld-linux-x86-64.so.2]
```

### 1.1.2 搜索优先级（GNU ld）

| 优先级 | 机制 | 说明 |
|--------|------|------|
| **1（最高）** | `DT_RPATH` | ELF 硬编码（旧式，已弃用） |
| **2** | `LD_LIBRARY_PATH` | 环境变量，进程级 |
| **3** | `DT_RUNPATH` | ELF 硬编码（新式，推荐） |
| **4** | `/etc/ld.so.cache` | ldconfig 缓存的系统库 |
| **5（最低）** | `/lib`、`/usr/lib` | 默认系统库目录 |

> `DT_RPATH` 优先于 `LD_LIBRARY_PATH`，而 `DT_RUNPATH` 低于 `LD_LIBRARY_PATH`。GCC 默认写 `DT_RPATH`；加 `-Wl,--enable-new-dtags` 后写 `DT_RUNPATH`。

### 1.1.3 隐式加载与显式加载

- **隐式加载**：编译时 `-lfoo` 写入 `DT_NEEDED`，程序启动时由 `ld-linux.so` 自动加载
- **显式加载**：`dlopen("libfoo.so", flags)` 运行时手动加载，不由 `ld-linux.so` 管理

> `dlopen` 不搜 rpath / runpath（见 [[C-Linux生态/01.Linux环境/4. 加载与劫持|加载与劫持]]），仅搜 `LD_LIBRARY_PATH` + ld.so.cache + 默认目录。

### 1.1.4 符号劫持

- `LD_PRELOAD`：在所有库之前加载，同名符号覆盖。多个库用 `:` 分隔，左优先
- `LD_LIBRARY_PATH` 劫持：通过改变搜索目录次序，使系统库被替代
- setuid/setgid 程序忽略上述环境变量

## 1.2 Windows DLL 加载机制

### 1.2.1 DLL 类型

| 类型 | 说明 |
|------|------|
| **隐式链接（Load-Time）** | 编译时链接 `.lib`（导入库），PE 头写 `IMAGE_DIRECTORY_ENTRY_IMPORT`，进程启动由 `LdrLoadDll` 自动加载 |
| **显式链接（Run-Time）** | `LoadLibrary("foo.dll")` / `LoadLibraryEx` 运行时手动加载 |
| **延迟加载（Delay-Load）** | `/DELAYLOAD` 链接器选项，首次调用函数时才通过 `__delayLoadHelper` 加载 |

### 1.2.2 加载 API

```cpp
#include <windows.h>

// 显式加载（标准）
HMODULE hMod = LoadLibrary(TEXT("foo.dll"));
// 或指定更多搜索选项
HMODULE hMod = LoadLibraryEx(TEXT("foo.dll"), NULL, LOAD_WITH_ALTERED_SEARCH_PATH);

FARPROC func = GetProcAddress(hMod, "function_name");
FreeLibrary(hMod);
```

Windows 无 `dlerror()` 等效函数；获取错误用 `GetLastError()` + `FormatMessage`。

### 1.2.3 搜索顺序（Safe DLL Search Mode）

Vista+ 默认启用 Safe DLL Search Mode，搜索顺序如下：

| 优先级 | 路径 | 说明 |
|--------|------|------|
| **1** | Known DLLs | 注册表锁定的系统 DLL（kernel32.dll、user32.dll 等） |
| **2** | 进程 .exe 所在目录 | DLL 放在此目录优先级最高（等同 Linux `$ORIGIN`） |
| **3** | 系统目录 `%SystemRoot%\System32` | 64 位系统 DLL |
| **4** | 32 位兼容目录 `%SystemRoot%\SysWOW64` | 32 位兼容 DLL |
| **5** | Windows 目录 `%SystemRoot%` | |
| **6** | 进程当前工作目录（CWD） | 默认开启 Safe Search 后，此位置优先级下降 |
| **7** | `PATH` 环境变量 | 最后搜索 |

> **Known DLLs** 是 Windows 特有的安全机制：注册表 `/HKLM/.../Session Manager/KnownDLLs/` 中列出的系统 DLL 永远从 `%windir%\System32` 加载，即使 `PATH` 或进程目录中有同名的恶意 DLL。

#### 禁用 Safe Search（旧模式兼容）

```cpp
// 等效于 XP/2003 旧顺序（CWD 优先级大幅提升）
SetDllDirectory(TEXT(""));   // 传入空字符串移除 Safe Search
```

或通过注册表 `HKEY_LOCAL_MACHINE\System\CurrentControlSet\Control\Session Manager\SafeDllSearchMode` 设为 `0`。

### 1.2.4 DLL 搜索自定义

```cpp
// 在进程目录后、系统目录前插入自定义搜索路径
AddDllDirectory(TEXT("C:\\app\\plugins"));

// 设定默认搜索策略（需配合 AddDllDirectory）
SetDefaultDllDirectories(LOAD_LIBRARY_SEARCH_DEFAULT_DIRS);
```

> **`SetDllDirectory(path)`**：
> - 传入有效路径：该路径插入到系统目录前搜索
> - 传入 `""`（空串）：移除 Safe Search 保护，回退到旧顺序（CWD 优先）
> - 传入 `NULL`：恢复默认 Safe Search 行为

### 1.2.5 DLL 重定向与 Manifests

Windows 提供多种机制让应用程序使用私有 DLL，避免与系统或其他应用冲突：

| 机制 | 描述 |
|------|------|
| **`.local` 文件** | 在 `.exe` 同目录创建 `app.exe.local` 空文件，系统优先从该目录加载 DLL（Vista+ 部分失效） |
| **Side-by-Side (SxS) Assembly** | 通过 manifest 指定 DLL 版本，在 `%windir%\WinSxS` 管理不同版本共存 |
| **Manifest 嵌入** | 在 `.exe` 中嵌入 manifest，`<dependentAssembly>` 指定精确版本 |
| **私有 Assembly** | manifest 指定 `\<dependentAssembly\>\<assemblyIdentity\>` + DLL 放在 exe 目录 |

## 1.3 `-Wl,-rpath,'$ORIGIN:$ORIGIN/lib'` 详解

### 1.3.1 `$ORIGIN` 是什么

`$ORIGIN`（也可写作 `$ORIGIN` 或 `${ORIGIN}`）是 **GNU ld-linux.so 的内置变量**，在运行时被动态链接器展开为**可执行文件所在目录**的绝对路径。

- 不是 shell 变量 — 编译时 `$` 字面写入 ELF，运行时由 `ld-linux.so` 替换
- 在 `dlopen` 和隐式加载中均有效
- 对 `.so` 自身也有意义：当 `.so` 依赖另一 `.so` 时，`$ORIGIN` 展开为依赖者（`.so`）所在目录

### 1.3.2 语法与组合

```bash
# 基本用法：搜索可执行文件同目录
gcc main.c -o app -Wl,-rpath,'$ORIGIN'

# 多路径：先搜同目录，再搜同目录下的 lib/
gcc main.c -o app -Wl,-rpath,'$ORIGIN:$ORIGIN/lib'

# 混合固定路径与 $ORIGIN
gcc main.c -o app -Wl,-rpath,'$ORIGIN/lib:/usr/local/lib'

# 对 .so 自身也有效
gcc -shared -fPIC -o libfoo.so foo.c -Wl,-rpath,'$ORIGIN/plugins'
```

> **路径分隔符**：Linux 使用 `:`，与 `PATH` 风格一致。
>
> **单引号的作用**：阻止 shell 展开 `$ORIGIN`，确保 `$ORIGIN` 字面写入 ELF 的 `.dynamic` 段。若误用双引号，shell 会将 `$ORIGIN` 展开为空字符串。

### 1.3.3 两种 DTAG 选项对比

```bash
# DT_RPATH（旧式，优先级高于 LD_LIBRARY_PATH）
gcc -Wl,-rpath,'$ORIGIN/lib' ...

# DT_RUNPATH（新式，推荐）
gcc -Wl,-rpath,'$ORIGIN/lib' -Wl,--enable-new-dtags ...

# 同时写入两者（GCC 默认行为 + --enable-new-dtags）
```

**选择建议**：

| 场景 | 推荐 | 原因 |
|------|------|------|
| 自包含部署 | DT_RUNPATH | 允许用户用 `LD_LIBRARY_PATH` 覆盖（debug / override） |
| 安全敏感程序 | DT_RUNPATH | 不优先于环境变量，setuid 时环境变量被忽略 |
| 绝对控制 | DT_RPATH | 不允许被环境变量绕过（极少数场景） |
| CMake 默认 | DT_RUNPATH | CMake 3.x 通过 `CMAKE_BUILD_RPATH` 默认启用 `--enable-new-dtags` |

### 1.3.4 验证

```bash
# 查看 rpath
readelf -d app | grep -E 'RPATH|RUNPATH'

# 输出示例（DT_RUNPATH）
0x000000000000001d (RUNPATH) Library runpath: [$ORIGIN:$ORIGIN/lib]

# 调试加载过程
LD_DEBUG=libs ./app
```

### 1.3.5 CMake 等效配置

```cmake
# 等效于 -Wl,-rpath,'$ORIGIN:$ORIGIN/lib'
set_target_properties(myapp PROPERTIES
    BUILD_RPATH "\$ORIGIN:\$ORIGIN/lib"
    INSTALL_RPATH "\$ORIGIN:\$ORIGIN/lib"
    BUILD_WITH_INSTALL_RPATH TRUE    # 构建时即使用 install RPATH
    INSTALL_RPATH_USE_LINK_PATH TRUE # 将链接器搜索路径加入 RPATH
)
```

> **CMake 转义`$`**：CMake 中 `$ORIGIN` 需写为 `\$ORIGIN`，否则 CMake 会将其解释为变量引用。

### 1.3.6 `$ORIGIN` 与 dlopen 的配合

`dlopen` **不搜索 rpath / runpath**（见 [[C-Linux生态/01.Linux环境/3. rpath与链接路径|rpath 与链接路径]] 1.5 节）。

若希望 `dlopen` 使用相对路径部署：

```cpp
// 方案 A：传绝对路径（推荐）
char exe_dir[1024] = {0};
// 从 /proc/self/exe 或 argv[0] 推导 exe 目录
sprintf(full_path, "%s/plugins/libplugin.so", exe_dir);
void* h = dlopen(full_path, RTLD_LAZY);

// 方案 B：加载前临时设置 LD_LIBRARY_PATH
setenv("LD_LIBRARY_PATH", "/opt/app/plugins", 1);
void* h = dlopen("libplugin.so", RTLD_LAZY);
```

## 1.4 搜索顺序对比一览

| 优先级 | Linux | Windows |
|--------|-------|---------|
| **1** | `DT_RPATH` | Known DLLs（注册表锁定）|
| **2** | `LD_LIBRARY_PATH` | 进程 `.exe` 目录 |
| **3** | `DT_RUNPATH` | `%SystemRoot%\System32` |
| **4** | `/etc/ld.so.cache` | `%SystemRoot%\SysWOW64`（32 位兼容）|
| **5** | `/lib`, `/usr/lib` | `%SystemRoot%` |
| **6** | — | 当前工作目录（CWD）|
| **7** | — | `PATH` 环境变量 |

> **Windows 注**：此表为 Vista+ Safe DLL Search Mode 默认顺序。XP/2003 及 Safe Search 关闭时，CWD 的优先级提升到系统目录之前。

### 1.4.1 关键差异矩阵

| 维度 | Linux | Windows |
|------|-------|---------|
| **运行时路径嵌入** | `-Wl,-rpath` 写入 ELF | 无等价物，依赖 .exe 目录 / `PATH` / manifest |
| **相对路径变量** | `$ORIGIN`（ld-linux.so 展开） | 无内置变量，但 `MAX_PATH` + exe 目录等价 |
| **环境变量覆盖** | `LD_LIBRARY_PATH` 强力覆盖 | `PATH` 影响小（搜索顺序末尾） |
| **符号劫持** | `LD_PRELOAD` 易用，显式控制 | 无直接等价；Known DLLs 阻止系统库劫持 |
| **`dlopen`/`LoadLibrary`** | 不搜 rpath | 遵循 Safe Search 顺序 |
| **系统库缓存** | `/etc/ld.so.cache`（`ldconfig`） | 无缓存机制，按顺序搜目录 |
| **多版本共存** | SONAME 软链 + ld.so.cache | SxS / WinSxS / manifests |
| **手动加载错误信息** | `dlerror()` 返回字符串 | `GetLastError()` + `FormatMessage` |

### 1.4.2 部署对标

| Linux 部署策略 | Windows 等效做法 |
|---------------|-----------------|
| `$ORIGIN/lib` 放 `.so` | DLL 放在 `.exe` 同目录 |
| `$ORIGIN/plugins` 放插件 `.so` | exe 目录下建 `plugins\` + `LoadLibrary` 传绝对路径 |
| `rpath` + `DT_RUNPATH` | manifest + `.local` 文件 |
| `LD_LIBRARY_PATH` | 临时改 `PATH`（不推荐）或 `SetDllDirectory` |
| `ldconfig` 注册系统库 | Installer 将 DLL 放入 `System32`（需要管理员权限）|

## 1.5 常见踩坑

### Linux 侧

| 现象 | 原因 | 处理 |
|------|------|------|
| `ldd` 显示 `not found` 但 `.so` 存在 | rpath 路径错或 `ldconfig` 未更新 | `readelf -d` 查 rpath；`LD_DEBUG=libs` 追踪 |
| `$ORIGIN` 未生效 | 双引号 `"$ORIGIN"` 被 shell 展开成空字符串 | 改用单引号 `'$ORIGIN'` |
| `dlopen` 找不到插件 `.so` | `dlopen` 不搜 rpath | 传绝对路径或设 `LD_LIBRARY_PATH` |
| 交叉编译 rpath 写入主机路径 | 误用 `-rpath=/usr/lib` | 使用 `$ORIGIN` 绕过路径绑定 |

### Windows 侧

| 现象 | 原因 | 处理 |
|------|------|------|
| DLL 加载失败（错误 126 / 127） | 缺依赖 DLL、搜索路径不含目标、架构不匹配 | `Dependency Walker` 或 `dumpbin /dependents` 分析 |
| `LoadLibrary` 返回 NULL | DLL 路径错、未满足依赖、x64/x86 冲突 | `GetLastError()` + `FormatMessage`；检查位数 |
| Known DLLs 挡掉私有同名库 | 系统库名被注册表锁定 | 私有 DLL 不要用系统库名（如 `kernel32.dll`）|
| 同一目录有同名 `.dll` 不同版本 | DLL Hell | 使用 manifest + SxS 或重新命名 |
| `PATH` 中已删路径的旧 DLL 被加载 | `PATH` 环境变量残留 | `where foo.dll` 查实际来源 |

### 跨平台

```cpp
// 跨平台路径辅助函数模式
#if defined(_WIN32)
    HMODULE hMod = LoadLibrary(TEXT("plugins\\foo.dll"));
    if (!hMod) {
        DWORD err = GetLastError();
        // FormatMessage 获取详情
    }
#else
    void* hMod = dlopen("plugins/libfoo.so", RTLD_LAZY);
    if (!hMod) {
        fprintf(stderr, "dlopen: %s\n", dlerror());
    }
#endif
```

## 1.6 选型指南

| 部署场景 | Linux 推荐 | Windows 推荐 |
|---------|-----------|-------------|
| **自包含应用** | `-Wl,-rpath,'$ORIGIN:$ORIGIN/lib'` + DT_RUNPATH | DLL 全放 `.exe` 同目录 |
| **系统级库** | `ldconfig` + `/etc/ld.so.conf.d/` | 安装到 `System32`（管理员） |
| **插件系统** | `dlopen` + 传绝对路径（`/proc/self/exe` 推导） | `LoadLibrary` + exe 目录拼接 |
| **调试/开发** | `LD_LIBRARY_PATH` | `SetDllDirectory` 或 `PATH` 前置 |
| **第三方二进制修复** | `patchelf --set-rpath` | 无直接工具；`SetDllDirectory` 或修 `PATH` |
| **安全敏感（防劫持）** | `DT_RPATH`（不可覆盖），避免 setuid | Known DLLs 保护系统库；移除 `.local` 文件 |
