# 1. Claude Code Skill 设计方法论

## 1.1 背景

Claude Code 的 **Skill**（技能）是可复用的 prompt 模板，通过 `/skill-name` 触发。Skill 用于封装**高频、多步骤、有决策分支**的任务流程，使 Agent 的行为可重复、可移植。

本笔记记录从约束性配置（CLAUDE.md）中提取 Skill 的分析方法论，作为后续设计的参考框架。

## 1.2 核心理念

在设计 Skill 之前，必须区分两类不同性质的配置：

### 1.2.1 声明式约束（Declarative Rules）

特征：
- 描述「怎么做」，而非「做什么」
- 没有固定的步骤序列
- 是 Agent 的行为准则，而非用户的主动行为

**不适合 Skill 化**，应保留在 CLAUDE.md 中。

示例：
```
Prefer Git Bash when available.
Commit after each change.
Root CLAUDE.md should stay under 120 lines.
```

### 1.2.2 可执行任务（Executable Tasks）

特征：
- 有明确的输入和输出
- 有可重复的步骤序列
- 包含决策分支点
- 用户高频执行

**适合 Skill 化**，应封装为 `.claude/skills/*.md`。

示例：
```
分析 git diff → 生成 commit message → 执行 commit
查看本地 commits → 检测备份提交 → 选择 squash 策略 → 执行
```

### 1.2.3 核心决策树

```
某条规则
  ├─ 是声明式约束？ → 留在 CLAUDE.md
  │
  └─ 是可执行任务？
       ├─ 步骤数 = 1？       → 留在 CLAUDE.md（太简单）
       ├─ 使用频率 = 低？    → 留在 CLAUDE.md（不值得 Skill 化）
       └─ 步骤数 > 1 且 频率 ≥ 中？
            ├─ 有跨项目通用性？ → 全局 Skill（~/.claude/skills/）
            └─ 项目特有？       → 项目 Skill（repo/.claude/skills/）
```

## 1.3 设计流程

### 1.3.1 约束盘点

遍历所有 CLAUDE.md 及子指令文件，提取每条原子规则。

以本机环境为例：

| 来源 | 原子规则 | 分类 |
|------|----------|------|
| 全局 CLAUDE.md | Prefer Git Bash | 声明式 ✅ |
| 全局 CLAUDE.md | Squash before push | 声明式 ✅ |
| 全局 CLAUDE.md | Conventional Commits | 格式规范，供 Skill 引用 |
| 项目 04-shell.md | Git Bash 路径 | 运行时探测，不硬编码 |
| 项目 03-git.md | Squash 方法 A/B | 子指令引用，供 Skill 使用 |

### 1.3.2 频次与复杂度评估

对每条潜在的可执行任务打分：

| 维度 | 评分标准 |
|------|----------|
| **频次** | 每次改动 ★★★★★ / 每天 ★★★★ / 每周 ★★★ / 每月 ★★ / 偶尔 ★ |
| **步骤数** | >5 步 ★★★ / 3-5 步 ★★ / 1-2 步 ★ |
| **决策分支** | >3 个分支 ★★★ / 1-2 个 ★★ / 无 ★ |
| **风险** | 破坏性操作需确认 ↑ |

**推荐线**：频次 ≥ ★★★ 且 步骤数 ≥ ★★

### 1.3.3 划界

明确 Skill 的边界 —— 什么不做：

```
/quick-commit 的边界
  ├─ 做：git add -A、分析 diff、生成 message、commit
  ├─ 不做：git push（留给用户）
  └─ 不做：精细 stage 控制（用户手动 git add）
```

边界划分原则：
- **单一职责**：一个 Skill 只做一件事
- **保留用户决策点**：破坏性操作、对外交互，留确认环节
- **互补不重叠**：Skill 之间职责清晰，`/quick-commit` 只管提交，`/squash-commits` 只管整理

### 1.3.4 可移植性设计

Skill 文件中的路径处理策略：

| 路径类型 | 处理策略 | 示例 |
|----------|----------|------|
| **绝对路径** | 运行时探测，不硬编码 | Git Bash 路径、用户目录 |
| **相对路径** | 保留为模板 | `.claude/instructions/`、`picture/` |
| **环境变量** | 引用占位符，运行时替换 | `{detected_bash_path}`、`{user}` |

跨项目适用性决定 Skill 存放位置：

```
~/.claude/skills/          → 全局：任何项目通用
repo/.claude/skills/       → 项目：仅该仓库使用
```

### 1.3.5 模块拆分

大型 Skill 应支持模块化选择，而非捆绑所有功能：

```
/init-note-vault
  ├─ [默认模式] 完整规则集（标题编号、wikilink、无 frontmatter...）
  └─ [精简模式] 最小规则集，提示用户可扩展
```

模块划分原则：
- **Core vs Template**：Core 默认选中，Template 按需选
- **正交性**：模块间不重叠，可独立启用
- **可演进**：可随时加新模块而不破坏旧行为

## 1.4 实例：从全局规则到 5 个 Skill

以本机 `C:\Users\AS\.claude\CLAUDE.md` + 项目指令文件为例，完整推导过程：

```
43 条原子约束
  ├─ 声明式（留在 CLAUDE.md）→ Shell 终端策略、UTF-8 补丁、120 行限制
  ├─ 项目特定（不 Skill 化） → 具体科目列表、具体图片清单
  └─ 可执行任务
       ├─ 频次低 → 作为 init 系列 Skill 的模块
       │    ├─ /init-win-env        (Shell 环境配置)
       │    ├─ /init-git-convention  (Git 工作流规范)
       │    └─ /init-note-vault      (笔记知识库规范)
       │
       └─ 频次高 → 独立动作 Skill
            ├─ /quick-commit    (日常高频：提交改动)
            └─ /squash-commits  (次高频：推送前整理)
```

### 1.4.1 Skill 职责矩阵

| Skill | 类型 | 频次 | 步骤 | 输入 | 输出 |
|-------|------|------|------|------|------|
| `/init-win-env` | 初始化 | ★★ | 4 | 本机环境 | CLAUDE.md + settings.json |
| `/init-git-convention` | 初始化 | ★★ | 5 | 语言偏好 | CLAUDE.md + 子指令 + settings.json |
| `/init-note-vault` | 初始化 | ★★ | 4 | 模式选择 | CLAUDE.md + 子指令 |
| `/quick-commit` | 动作 | ★★★★★ | 5 | git diff | commit |
| `/squash-commits` | 动作 | ★★★★ | 5 | commits 列表 | 干净历史 |

## 1.5 设计原则总结

```
1. 区分声明式与可执行     → 声明式留 CLAUDE.md，可执行封 Skill
2. 频次 + 步骤数决定优先级  → 高频多步骤优先做
3. 单一职责 + 明确边界    → 一个 Skill 只做一件事
4. 可移植性             → 绝对路径运行时探测，相对路径保留模板
5. 全局 vs 项目          → 通用放 ~/.claude/skills/，特有放 repo
6. 模块化拆分            → 大型 Skill 支持按需选择模块
7. 保留用户确认环节       → 破坏性操作必须弹确认
```
