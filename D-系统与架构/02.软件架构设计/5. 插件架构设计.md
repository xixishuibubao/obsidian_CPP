# 1. 插件架构设计

## 【核心精简提纲】

### 记忆主干

插件机制 = 运行时动态加载 `.so`（Linux）/ `.dll`（Windows），在不重启程序、不重新编译主程序的前提下扩展功能。主程序是壳，插件是功能模块。

### 核心要点清单

- 一个 .so 可注册多个算法
- 热切换只需改字符串
- `extern "C"` 必须加：C++ 名字粉碎后 `dlsym` 找不到符号

## 1.1 插件架构的应用场景

| 场景 | 说明 |
|------|------|
| **热切换算法** | 运行时切换高斯模糊 / Canny / NPU 推理，不重启 |
| **模块化部署** | 算法都做成独立 `.so`，客户要什么功能就放什么 `.so` |
| **远程热更新** | 覆盖 `.so` 文件即可更新算法，不用更新固件 |
| **解耦架构** | 主程序只管调度，插件只管实现，大型项目不混乱 |

RK3588 视觉项目典型结构：

```plain
main_app (主程序)
    ↓
plugins/
    libdetector.so    (NPU目标检测)
    libpreprocess.so  (OpenCL预处理)
    libencoder.so     (视频编码)
    libalgo_custom.so (自定义NEON算法)
```

## 1.3 插件接口设计方式

核心原则：主程序通过**固定的接口名**查找插件符号，接口不变，实现可替换。

### 1.3.1 C 风格：同名函数（最简单）

所有 `.so` 导出**同名的 C 函数**，主程序按固定名字查找：

```cpp
// plugin.h — 插件接口约定
extern "C" void process_frame(unsigned char* data, int w, int h);
extern "C" const char* plugin_name();
```

```cpp
// gray_plugin.cpp — 插件实现
#include <cstdio>
extern "C" void process_frame(unsigned char* data, int w, int h) {
    printf("灰度化处理中...\n");
}
extern "C" const char* plugin_name() { return "灰度化插件"; }
```

编译插件：
```bash
aarch64-linux-gnu-g++ -fPIC -shared gray_plugin.cpp -o libgray_plugin.so
```

主程序加载：
```cpp
void* handle = dlopen("./libgray_plugin.so", RTLD_LAZY);
auto name   = (const char*(*)())dlsym(handle, "plugin_name");
auto process = (void(*)(unsigned char*,int,int))dlsym(handle, "process_frame");
printf("插件名：%s\n", name());
process(data, 1920, 1080);
dlclose(handle);
```

### 1.3.2 C++ 接口风格：纯虚类 + 工厂函数（工业标准）

```cpp
// plugin_interface.h
#pragma once
class IPlugin {
public:
    virtual ~IPlugin() = default;
    virtual const char* name() = 0;
    virtual bool process(unsigned char* rgb, int w, int h) = 0;
};

// 所有 SO 必须导出这个工厂函数
extern "C" IPlugin* create_plugin();
```

```cpp
// test_plugin.cpp
#include "plugin_interface.h"

class TestPlugin : public IPlugin {
public:
    const char* name() override { return "Test插件"; }
    bool process(unsigned char* rgb, int w, int h) override {
        // 算法：灰度、滤波、NEON、OpenCL、NPU...
        return true;
    }
};

extern "C" IPlugin* create_plugin() { return new TestPlugin(); }
```

主程序加载：
```cpp
void* handle = dlopen("./libtest_plugin.so", RTLD_NOW);
auto create = (IPlugin*(*)())dlsym(handle, "create_plugin");
IPlugin* plugin = create();
plugin->process(rgb_buf, 1920, 1080);
delete plugin;     // 先删对象
dlclose(handle);   // 再卸载 so
```

### 1.3.3 函数指针表风格（高性能）

SO 导出一个**结构体指针**，内含所有函数指针。主程序查找一次即可永久调用：

```cpp
// plugin_api.h
struct PluginAPI {
    void (*init)();
    void (*process)(unsigned char*, int, int);
    void (*release)();
};

extern "C" PluginAPI* get_plugin_api();
```

适合高帧率图像处理、AI 推理前后处理等对延迟敏感的场景。

## 1.4 接口设计方式对比

| 风格 | 优点 | 缺点 | 推荐场景 |
|------|------|------|----------|
| **C 同名函数** | 最简单，无需基类，跨语言友好 | 每个 .so 只能导出固定几个函数 | 轻量级插件、面试常考 |
| **C++ 纯虚接口** | 面向对象、易于扩展、一个 SO 可支持多个接口 | 需注意 C++ ABI 兼容、`extern "C"` 导出工厂 | **工业标准，首推** |
| **函数指针表** | 性能最高，查找一次永久使用 | 写法较繁琐 | 高帧率图像、低延迟场景 |

## 1.5 热插拔的 5 种工业方案

### 方案 1：同名函数 + dlopen（最简单）

所有 .so 导出同名 C 函数，主程序 `dlsym` 查找。换 .so = 换功能。

### 方案 2：C++ 接口 + 工厂自动注册（工业标准）

```cpp
// 每个 .so 启动时自动注册到全局工厂
// 主程序通过字符串获取插件
auto algo = Factory::create("face_detect");
algo->process();
```

- 不用硬编码函数名
- 一个 .so 可注册多个算法
- 热切换只需改字符串

### 方案 3：函数指针表 + 结构体导出（高性能）

SO 导出一个 `PluginAPI` 结构体，内含所有函数指针。

### 方案 4：多进程热重启（最稳定）

主程序（守护进程）永远不重启，功能逻辑放子进程。热插拔 = 杀掉旧子进程 → 启动新子进程。适合车载、机器人、安防等对稳定性要求高的场景。

### 方案 5：脚本热加载（最灵活）

C++ 主程序不变，逻辑写在 Lua/Python 脚本中，替换脚本文件即完成热更新，无需编译 .so。

### 方案对比

| 方案 | 难度 | 性能 | 稳定性 | 适用场景 |
|------|------|------|--------|----------|
| 同名函数 + dlopen | ⭐ | ⭐⭐⭐ | ⭐⭐ | 轻量级插件 |
| **C++ 接口 + 工厂** | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | **客户端标准架构，首推** |
| 函数指针表 | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | 高性能视觉算法 |
| 多进程热重启 | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐⭐ | 生产环境终极方案 |
| 脚本热加载 | ⭐ | ⭐⭐ | ⭐⭐ | 快速迭代调试 |

## 1.6 dlopen/dlclose 生命周期管理

**黄金法则：`dlopen` 在加载插件时打开，`dlclose` 只能在删除对象后调用，对象存活期间绝不能 dlclose。**

```cpp
// ❌ 错误：创建对象后立刻 dlclose
void* handle = dlopen("./libplugin.so", RTLD_NOW);
IPlugin* p = create_plugin();
dlclose(handle);           // .so 被卸载
p->process();              // 段错误！虚函数表已被移除

// ✅ 正确
void* handle = dlopen("./libplugin.so", RTLD_NOW);
IPlugin* p = create_plugin();
p->process();              // 使用期间 .so 必须常驻
delete p;                  // 先销毁对象
dlclose(handle);           // 再卸载 .so
```

**工厂模式中必须维护句柄表**：

```cpp
class PluginFactory {
    std::unordered_map<std::string, void*> m_handles;  // 保存所有 .so handle
    std::unordered_map<std::string, IPlugin*> m_plugins;

public:
    IPlugin* load(const std::string& name, const std::string& path) {
        void* handle = dlopen(path.c_str(), RTLD_NOW);
        m_handles[name] = handle;
        auto create = (IPlugin*(*)())dlsym(handle, "create_plugin");
        auto plugin = create();
        m_plugins[name] = plugin;
        return plugin;
    }

    void unload(const std::string& name) {
        delete m_plugins[name];      // 先删对象
        dlclose(m_handles[name]);     // 再卸 .so
    }
};
```

## 1.7 嵌入式项目实践要点

1. **`extern "C"` 必须加**：C++ 名字粉碎后 `dlsym` 找不到符号
2. **`-fPIC -shared`**：编译插件时必加，生成位置无关的动态库
3. **`-ldl`**：主程序编译时必须链接
4. **`$ORIGIN` 配合 rpath**：插件路径用相对路径时用 `-Wl,-rpath,'$ORIGIN/plugins'`，参见 [[C-Linux生态/01.Linux环境/3. rpath与链接路径|运行时链接路径]] 和 [[B-构建与脚本/01.构建工具/1. g++|rpath 编译选项详解]]
5. **C 风格函数接口比 C++ 虚函数更安全**：C++ 虚函数的 ABI 在不同编译器版本间可能不兼容，跨编译器分发插件建议用 C 风格
