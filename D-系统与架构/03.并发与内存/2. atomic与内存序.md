# 1. atomic 与内存序

C++ `std::atomic` 提供无锁的原子操作，配合**内存序（memory_order）**控制多线程间的可见性和执行顺序。仅靠 CPU 缓存一致性协议（MESI）不足以保证多线程程序的正确性。

## 1.1 为什么缓存一致性不够

MESI 协议确保不同 CPU 缓存中同一地址的数据副本一致，但仍存在以下问题：

### 1.1.1 操作非原子性

对变量的修改（如 `i++`）在汇编层面分解为"读-改-写"三步，非原子操作：

```asm
mov eax, [i]    ; 读
add eax, 1      ; 改
mov [i], eax    ; 写
```

两个线程并发 `i++` 可能交错：线程A 读 `i=0` → 线程B 读 `i=0` → 线程A 写 `i=1` → 线程B 写 `i=1`，结果 `i=1` 而非 2。

### 1.1.2 指令重排序

编译器或 CPU 为优化性能可能重排序指令（保证单线程语义不变），多线程场景下导致意外结果：

```cpp
// 线程A
x = 1;          // 写数据
ready = true;   // 写标记（可能被重排序到 x=1 之前）

// 线程B
while (!ready); // 等待标记
cout << x;      // 可能读到 0（因 x=1 被重排序到 ready=true 之后）
```

### 1.1.3 内存可见性延迟

缓存的写传播需时间，一个线程的修改可能对其他线程暂时不可见。

### 1.1.4 伪共享（False Sharing）

缓存以**缓存行**为单位（通常 64 字节）。两个无关变量位于同一缓存行时，不同线程各自修改不同变量仍会导致缓存行失效，互相影响性能。

**解决方案**：使用 `std::atomic` 配合适当内存序，保证原子性、禁止重排序、控制可见性。

## 1.2 内存序概述

C++ 标准定义 6 种内存序，按约束强度分为三类：

| 类别 | 内存序 | 约束强度 | 典型场景 |
|------|--------|----------|----------|
| 弱序 | `relaxed` | 仅保证原子性，无顺序约束 | 计数器（统计访问次数） |
| 中序 | `acquire` / `release` | 建立线程间因果关系 | 生产者-消费者模型（90% 场景） |
| 中序 | `consume` | 仅约束依赖原子结果的代码（C++17 不建议使用） | 指针依赖（极少使用） |
| 强序 | `acq_rel` | acquire + release 组合 | 读-改-写操作需要双向同步 |
| 强序 | `seq_cst` | 全局总序，禁止任何跨操作重排序 | 省事默认、简单逻辑 |

内存序通过禁止特定类型的**重排序**和强制**缓存同步**来约束执行顺序。

## 1.3 内存序详解

### 1.3.1 `memory_order_relaxed`（松散）

**仅保证原子操作本身的原子性**，不提供任何内存屏障或执行顺序保证。编译器和 CPU 可自由重排序其前后指令。

```cpp
std::atomic<int> x(0), y(0);

// 线程A
x.store(1, std::memory_order_relaxed);   // A
y.store(1, std::memory_order_relaxed);   // B（可能被重排序到 A 之前）

// 线程B：即使看到 y=1，x 仍可能为 0
while (y.load(std::memory_order_relaxed) != 1);
int val = x.load(std::memory_order_relaxed);  // 可能为 0
```

**适用**：纯计数器，无需用它同步其他数据的场景：
```cpp
std::atomic<int> g_count{0};
g_count.fetch_add(1, std::memory_order_relaxed);  // 仅需原子性
```

### 1.3.2 `memory_order_acquire` / `memory_order_release`

最常用的多线程同步方式。通过"释放-获取对"（release-acquire pair）建立线程间的 happens-before 关系。

| 操作 | 约束 |
|------|------|
| **Release**（如 `store(release)`） | 当前线程中，release **之前**的所有读写，**不会被重排序到 release 之后** |
| **Acquire**（如 `load(acquire)`） | 当前线程中，acquire **之后**的所有读写，**不会被重排序到 acquire 之前** |

```cpp
std::atomic<bool> ready{false};
int data = 0;

// 线程A（生产者）
data = 42;
ready.store(true, std::memory_order_release);  // release：data=42 不会排到后面

// 线程B（消费者）
if (ready.load(std::memory_order_acquire)) {   // acquire：之后的读不会排到前面
    assert(data == 42);                        // 一定成立
}
```

**适用场景**：90% 的多线程同步——生产者-消费者、标志位同步、发布数据。

### 1.3.3 `memory_order_seq_cst`（顺序一致）

最强保证。等于在原子操作前后插入**全屏障**，禁止任何跨操作重排序，且所有线程看到的原子操作顺序完全一致。

```cpp
std::atomic<int> x(0), y(0);

// 线程A
x.store(1, std::memory_order_seq_cst);   // A
int a = y.load(std::memory_order_seq_cst); // B

// 线程B
y.store(1, std::memory_order_seq_cst);   // C
int b = x.load(std::memory_order_seq_cst); // D
```

所有线程观察到的操作顺序全局统一（如 A→B→C→D 或 C→D→A→B 等），绝不会出现 `a=0 && b=0`。

**适用**：简单代码不想思考内存序时、单例双重检查锁定。

### 1.3.4 `memory_order_consume`

比 acquire 更弱，仅约束**依赖原子操作结果的代码**不被重排序到 consume 之前：

```cpp
std::atomic<Data*> ptr{nullptr};

// 线程B
Data* p = ptr.load(std::memory_order_consume);
if (p) {
    cout << p->value;   // 依赖于 p，保证看到最新值
    int x = 1;          // 不依赖 p，可能被重排序到 load 之前
}
```

> C++17 起不建议使用，`acquire` 是更安全的替代。

### 1.3.5 内存序对比总表

| 内存序 | 禁止指令重排 | 线程间同步 | 性能开销 | 推荐度 |
|--------|-------------|-----------|----------|--------|
| `relaxed` | 不禁止 | 无 | 最小 | ⭐⭐ 计数专用 |
| `release/acquire` | release 前不前移，acquire 后不后移 | 建立 happens-before | 中等 | ⭐⭐⭐⭐⭐ 首选 |
| `seq_cst` | 全局全屏障 | 全局总序 | 最大（ARM 上明显） | ⭐⭐⭐⭐ 省事用 |
| `consume` | 仅依赖数据 | 有限 | 小 | ❌ 不推荐 |

## 1.4 原子操作接口选择

`std::atomic` 提供三类接口：

### 1.4.1 重载运算符（`++`/`=`/`--`）

默认使用 `memory_order_seq_cst`，适合简单场景：

```cpp
std::atomic<int> counter{0};
counter++;        // 等价 fetch_add(1, seq_cst)
counter = 42;     // 等价 store(42, seq_cst)
```

### 1.4.2 `load()` / `store()` — 纯读/写

允许显式指定内存序，适合性能敏感或复杂同步：

```cpp
std::atomic<int> data{0};

// 单纯读，使用获取语义
int v = data.load(std::memory_order_acquire);

// 单纯写，使用释放语义
data.store(100, std::memory_order_release);
```

### 1.4.3 `fetch_*()` — 读-改-写

也允许显式指定内存序：

```cpp
int old = counter.fetch_add(1, std::memory_order_relaxed);
int old = counter.exchange(42, std::memory_order_acq_rel);
bool ok = counter.compare_exchange_weak(expected, desired);
```

### 1.4.4 选择指南

| 场景 | 推荐方式 | 原因 |
|------|----------|------|
| 简单计数/赋值 | 重载运算符（`++`/`=`） | 默认强序，简单直观 |
| 自定义内存序 | `fetch_*()` / `load()` / `store()` | 显式指定，优化性能 |
| 复杂同步（生产者-消费者） | `load(acquire)` / `store(release)` | 精确控制读写语义 |
| 性能敏感 | 显式方法 + `relaxed` | 减少不必要的内存屏障 |

> 性能敏感场景中，`seq_cst` 在 x86 上开销与 `acquire/release` 相近，但在 ARM 等弱序架构上有显著差异。

## 1.5 原子屏障 vs 独立 Fence

### 1.5.1 两种屏障

| 类型 | 形式 | 特点 |
|------|------|------|
| **原子自带屏障（隐式）** | `a.store(1, release)` | 和原子变量绑定，仅同步相关读写，安全高效 |
| **独立 fence（显式）** | `std::atomic_thread_fence(release)` | 不绑定变量，作用于整个线程所有内存，底层危险 |

### 1.5.2 功能等价示例

下面两段代码**效果完全相同**：

```cpp
// 方式1：原子自带屏障（推荐）
data = 42;
ready.store(true, std::memory_order_release);

if (ready.load(std::memory_order_acquire))
    cout << data;
```

```cpp
// 方式2：独立 fence（不推荐）
data = 42;
std::atomic_thread_fence(std::memory_order_release);
ready.store(true, std::memory_order_relaxed);

if (ready.load(std::memory_order_relaxed)) {
    std::atomic_thread_fence(std::memory_order_acquire);
    cout << data;
}
```

### 1.5.3 结论

**99% 的场景永远用「原子操作 + 内存序」**，不要用独立 `atomic_thread_fence`。独立 fence：
- 更难写、更难理解
- 没有安全性优势
- 不提供额外功能
- 只在底层架构优化或极少数需要非对称屏障的场景下使用

## 1.6 选择速查

| 需求 | 内存序 |
|------|--------|
| **纯计数**（无同步需求） | `relaxed` |
| **生产者-消费者同步**（传递数据） | `acquire` / `release` |
| **不想思考，省事默认** | `seq_cst` |
| **双向同步**（读-改-写） | `acq_rel` |

**三条规则背下来即可**：

1. **只是计数？** → `relaxed`（最快）
2. **线程间同步数据？** → `acquire/release`（最通用、性能最好）
3. **不想思考、图省事？** → `seq_cst`（默认、安全）

**永远不要**：
- 用独立 `atomic_thread_fence`（除非底层架构工程师）
- 用 `consume`（C++17 已不推荐）
- 混用内存序（如 release 配 seq_cst）
