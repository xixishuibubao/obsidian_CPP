# 1. UML 类图语法（PlantUML）

C++ 面向对象设计的图形表达：类、关系、权限、包；配合 [[G-语言与标记/4. PlantUML学习资料|PlantUML]] 文本化输出。

## 【核心精简提纲】

### 记忆主干

线型：`---` 实线 / `...` 虚线。箭头：`-->` 关联、`--|>` 继承、`--o` 聚合、`--*` 组合、`..>` 依赖。权限：`+` public、`-` private、`#` protected。

### 核心要点清单

- **继承**：`Child --|> Parent`（子→父）
- **关联**：`A --> B` 单向；`A -- B` 双向
- **聚合/组合**：`A o-- B` 空心菱形（整体不销毁局部）；`A *-- B` 实心（整体负责销毁）
- **依赖**：`A ..> B` 临时使用（参数、局部变量、静态调用）
- **标签**：`A "arg1" --> "arg2" B : label`；`note left of Class` 注释
- **别名**：`class "Car" as C`；`abstract`/`static` 修饰
- **包**：`package "com.example" { class Foo }` 作用域

### 学习路径

1. 记六种关系箭头（见 1.3 关联标识符）
2. 画一个 3～5 类的 C++ 继承+组合示例
3. 对照 1.9 OOP 设计原则检查耦合
4. 用 PlantUML 插件渲染并入库

### 关键结论 / 流程

```plain
识别类与职责 → 选关系类型 → PlantUML 代码 → 评审「高内聚低耦合」
设计优先级：聚合 > 组合 > 继承
```

## 【完整拓展详情】

## 1.1 线型与箭头

- 不同 `-` 或 `.` 数量代表长度：`---` 实线、`...` 虚线
- 箭头：`-->` 单箭头、`--|>` 空心（继承）、`--*` 实心菱形（组合）、`--o` 空心菱形（聚合）

## 1.2 权限标识符
1. `+` public
2. `-` private
3. `#` protected

## 1.3 关联标识符
1. 继承关系:`A--|>B` ,由子类引出指向父类,A是B的子类。
2. 弱关联：
	1. 单向关联：一方知道另一方的存在，另一方不知道。`A-->B` AB关联，一般是 `A has B`关系。
	2. 双向关联：互相都知道对方存在(使用对方引用/指针)，`A--B` AB关联，一般是 `A with B`关系。
3. 强关联：
	1. 聚合:`--o` (`小写字母O o`)，`A o-- B`。双方是整体和局部的关系`A has B` ，空心菱形由局部指向整体部,但整体不负责局部资源的销毁。
	2. 组合:`A *-- B`,整体和局部，整体负责局部的销毁，`A has B`。实心菱形由局部指向整体，局部以成员存在，而非引用或指针。
4. 不确定关联：
	1. 虚线箭头`A..>B`,`A use B`，关系是偶然的，临时的。
		- 一般表现为B是A的成员函数参数
		- 或B是A的成员函数的局部变量
		- 或A的成员函数调用了B的静态方法

## 1.4 标签
1. 属性标签:
	- `A "arg1" --> "arg2" B : args`,可以在线的首尾和侧面添加属性标识。
	- 注意`" "`一定是比类更靠近箭头线，否则会报错。
	- `args`可以是箭头，表明双方的所属关系。
2. 注释标签：`note`
	1. `note 位置 of class [func/value]`
	2. `some text about code`
	3. `end note`
	- ```plantuml
	  	class C{  
			- func(str: string):void  
			+ func(str: int):void
			}  
			  
			note left of C  
			hello world  
			end note  
			  
			note left of C::"func(str: string):void"  
			test  
			end note
	  ```
	- 使用`C::"func(str: string):void"`的方式可以识别重载的不同签名的函数。
---

## 1.5 `as` 别名
1. `class "Car" as B { }`,给`Car`取别名为`B` ，首先要对类名使用`" "`，其次是要使用`as`，最后一旦使用了别名，后续就不可以再使用原名了，否则会产生一个新的类。


## 1.6 虚函数

1. `abstract`抽象(虚函数/虚基类)
	1. `{abstract} func():void` ,可以通过`{abstract}`修饰声明函数为虚函数(抽象函数)  
	2. `{static} func():void` ,可以通过`{static}`修饰声明函数为静态函数

## 1.7 注释

1. 单行注释：` ' `(单引号)
	- `func() '注释 `
2. 多行注释：`/* */`
	- `func() /* 兼容C的注释风格 */`

## 1.8 包 `package`

1. 包作用域：
	- `package`声明这是一个包类型。
	- 嵌套关系在头部用`" "`包裹，并在`" "`内部用`.`来连接。
	- 最后使用`{ }`包裹整个作用域。
	- ```plantuml
			@startuml  
			
			package "com.example.service" {  
			class UserService {  
			+ getUserById(id: int): User  
			}  
			class OrderService {  
			+ createOrder(user: User): Order  
			}  
			}  
			  
			package "com.example.model" {  
			class User {  
			- id: int  
			- name: String  
			}  
			class Order {  
			- id: int  
			- userId: int  
			}  
			}  
			' 跨包关联  
			com.example.service.UserService --> com.example.model.User  
			@enduml
	  ```


## 1.9 OOP 设计原则

1. 高内聚，低耦合。
	- 系统尽可能能够独立使用，不严重依赖外部资源。
	- 耦合常常强调的是类与类/模块与模块之间的关系，内聚是内部关系。
2. 单一职责:
	- 不要让类大而全的去完成所有事，一个类(模块)尽可能只做一件事。
	- 如果有多件事要做，可以转而使用继承/组合/聚合来实现。
3. 开闭原则：
	- 面向抽象编程，抽象是稳定的，不要将Base类本身设计的过于具体，这样方便进行替换。
4. 里氏替换：
	- 子类必须可以替代父类。
	- 不要在子类实现同名的非虚函数，覆盖不是好的行为。
5. 接口分离：
	- 和低耦合，高内聚一个思想。不要让类/函数大而全，根据粒度拆分。
	- 尽可能使用 聚合/组合/继承来实现具体的目标，而非按照目标设计一个具体的类。
6. 依赖倒置:
	- 面向接口编程，依赖抽象，而非具体。 
	- 开闭原则是目标，里氏替换是基础，依赖倒置是手段。
7. 迪米特原则(最小知道原则) :
	- 核心就是减少耦合度，在资源和操作者之间可以给定一个代理类。
8. 优先级：
	- 实现时，优先以如下顺序使用设计方法，聚合>组合>继承，以降低耦合度。
