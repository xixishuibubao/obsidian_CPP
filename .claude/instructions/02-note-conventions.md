# 笔记内容规范

## 标题编号体系

所有笔记使用一致的数字编号体系：

```markdown
# 1. 主标题
## 1.1 二级标题
### 1.1.1 三级标题
#### 1.1.1.1 四级标题
```

- 文件名的排序数字（如 `5. 笔记.md` 中的 `5`）与文件内标题编号无关，标题编号始终从 `# 1.` 开始
- 不以 `#` 开头的文件需要补全
- **教程型旧笔记**（如 `1. git.md`、`1. Markdown.md`）允许多个 `# 2.`、`# 3.` 一级章节；**新笔记**优先 `# 1.` + `## 1.1` 层级
- 标题内不混用 `**` 加粗标记
- 不使用自动编号或手动中文编号（一、二、三）
- `###` 三级及以上标题：**编号可选**，允许无编号的语义化标题（如 `### 使用场景`）

## 跨笔记引用

使用 Obsidian wikilink 语法：

```markdown
[[文件名]]                          → 显示为文件名
[[文件名|自定义显示文字]]             → 显示为自定义文字
[[C-Linux生态/04.调试与优化/1. gdb|GDB 笔记]]  → 跨目录引用
```

- `文件名` **不含 `.md` 扩展名** — `[[file.md]]` 是错误写法，Obsidian 无法解析
- **跨模块引用**（A/B/C/D/E/F/G 之间）使用 **vault 相对路径** `[[模块-名称/子目录/文件名|显示文字]]`，路径以 vault 根为起点，不加 `/` 前缀。例如 `[[C-Linux生态/04.调试与优化/1. gdb|GDB 笔记]]`
  - ❌ ~~`C:/Users/.../study_cpp/C-Linux生态/...`~~（绝对路径，不可移植）
  - ✅ `[[C-Linux生态/04.调试与优化/1. gdb|GDB 笔记]]`（相对 vault 根，跨平台通用）
- 同目录内短链 `[[4. 驱动插件与pybind11动态加载]]` 可接受

## Markdown 表格中的 wikilink

GFM / Obsidian 用 `|` 分列。**表格单元格内禁止使用带别名的 wikilink** `[[路径|显示文字]]`——别名前的 `|` 会被当成列分隔符，链接截断、表格列数错乱。

| 场景 | 写法 |
|------|------|
| 表格单元格（推荐） | **纯文本**显示名；可点击 `[[路径\|显示]]` 放在表格外 `### 交叉引用` 或正文列表 |
| 表格内必须可点 | 仅用 `[[路径]]`（无 `\|` 别名；显示为文件名） |
| 索引型导航表 | 优先改为 **列表** + wikilink；若保留表格，第二列用 `[[路径]]` |
| 表格外正文 / 列表 | `[[路径\|显示文字]]` 照常 |

```markdown
| 层级 | 工具 | 本笔记 |
|------|------|--------|
| 编译选项 | g++ | `-Wl,-T` … |

### 交叉引用

- [[B-构建与脚本/01.构建工具/1. g++|g++ 编译与链接]]
```

机械检查（vault-audit **M10**、增量审查第 4 项子规则）：

```bash
grep -rnP '^\|[^\n]*\[\[[^|\]]+\|[^\]]+\]\]' --include='*.md' . \
  | grep -v '\.git/' | grep -v '\.claude/skills/' | grep -v 'CLAUDE.md'
```

## 链接完整性检查

完整性检查（CLAUDE.md 中的一键检查命令）在扫描 wikilinks 时，会误匹配 **shell 代码块中的 `[[ ]]` 条件判断**（如 `[[ $var == x ]]`）。这些不是真正的 wikilinks，可忽略。

如需精确定位真实断链，使用以下命令过滤掉 shell 条件：

```bash
# 精确检查断裂 wikilinks（排除 shell [[ 条件）
grep -roP '\[\[[^$\][]+?\]\]' . --include='*.md' | grep -v '\.git/' | sort -u
```

该命令通过排除包含 `$` 的 wikilink 来跳过 shell 变量引用，只保留真实的文件名引用。

## 代码块

使用 fenced code blocks 并标注语言（**标签统一小写**）：

- `c` — C 语言代码
- `cpp` — C++ 代码（不使用 `C++` 或 `c++`）
- `bash` — Shell 命令（不使用 `Bash`）
- `powershell` — PowerShell 命令
- `makefile` — Makefile
- `go` — Go 代码
- `lua` — Lua 代码
- `python` — Python 代码
- `java` — Java 代码
- `sql` — SQL 语句
- `plain` / `text` — 纯文本（不要使用无标注的缩进代码块）

> 语言标签使用全小写（`bash` 而非 `Bash`，`cpp` 而非 `C++` 或 `c++`），保持全库一致。

## 图片

```markdown
![描述文字](picture/文件名.png)
```

- 统一使用小写 `.png` 引用，即使磁盘上是 `.PNG`
- 图片放在根目录 `/picture/` 下

## 单文件双分区迭代格式（薄→厚→薄）

**同一篇笔记文件（一个 `.md`）**内用两个 `##` 分区承载「薄」与「厚」，**不拆成两个文件**：

| 分区 | 标题 | 用途 |
|------|------|------|
| 上 | `## 【核心精简提纲】` | 厚→薄产出：记忆主干、要点清单、关键结论/流程，**快速查阅** |
| 下 | `## 【完整拓展详情】` | 薄→厚产出：原理、推导、案例、踩坑、方案对比、交叉引用 |

> Obsidian 中仍是一篇笔记、一条 wikilink；分区只是文内标题，不是独立条目。

| 分区 | 标题 | 用途 |
|------|------|------|
| 上 | `## 【核心精简提纲】` | 厚→薄产出：记忆主干、要点清单、关键结论/流程，**快速查阅** |
| 下 | `## 【完整拓展详情】` | 薄→厚产出：原理、推导、案例、踩坑、方案对比、交叉引用 |

### 迭代步骤（均在同一文件内完成）

1. **薄→厚**（扩充下方 `【完整拓展详情】`）：补原理与前置依赖、边界与限制、正反例与踩坑、库内 wikilink、多方案对比、易混概念辨析。
2. **厚→薄**（提炼上方 `【核心精简提纲】`）：删重复与无关铺垫，合并同类观点，抽出主干与清单；提纲不得引入详情区没有的新事实。

### 文档骨架

```markdown
# 1. 主题标题

一句话定位（可选）。

## 【核心精简提纲】

### 记忆主干
…

### 核心要点清单
- …

### 关键结论 / 流程
…

## 【完整拓展详情】

## 1.1 二级章节
…
```

### 细则

- **一个主题 = 一个 `.md` 文件**；`【核心精简提纲】` 与 `【完整拓展详情】` 是该文件内的两个章节，不是另建 `xxx-提纲.md` / `xxx-详情.md`。
- `# 1.` 主标题保留在文件最顶部；双分区用 `##`，详情区内继续 `## 1.1` / `###` 编号。
- **提纲在上、详情在下**——打开笔记先见主干，向下滚动查细节。
- **索引页**（如 E-AI 工具索引）：提纲区放导航表，详情区放规划与背景即可。
- **教程型旧笔记**（多个 `# 2.` 一级章）：优先整体迁入 `【完整拓展详情】`，再统一提炼提纲；大文件可分次迭代。
- 迭代后检查 wikilink 与 [[.claude/instructions/01-repo-structure|目录]] 内相关条目的交叉引用是否补齐。

## 其他约定

- **无 YAML frontmatter**：笔记不使用 `---` 分隔的元数据块、tags 或 aliases
- **无 Obsidian `#tag`**：不使用标签语法
- **无 Obsidian `%%` 注释**：不包含隐藏注释块
- **双语写作**：技术术语用英文，解释说明用中文
- **跨平台发布**：如需发布到飞书/公众号，使用 [markdown.lovejade.cn](https://markdown.lovejade.cn/) 转换（选择"公众号"模式）

## 文件命名

```
数字. 空格 + 名称.md
```

示例：`1. C++基础.md`、`2. 代码规范.md`、`8. 常用指令.md`

目录和文件均遵循此规范。

> **注意**：文件名中的数字仅用于排序，与文件内标题编号无关。文件内标题始终从 `# 1.` 开始编号。
