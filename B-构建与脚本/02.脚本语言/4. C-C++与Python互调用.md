# 1. C/C++ 与 Python 互调用

C/C++ 负责性能与硬件访问，Python 负责编排与快速迭代。互调用本质是 **跨语言边界传递数据、共享进程地址空间或启动子进程**。本笔记梳理常见方案与选型；工程案例见 [[C-Linux生态/07.Telescope1126实战/4. 驱动插件与pybind11动态加载|驱动插件与 pybind11 动态加载]]。

前置：[[C-Linux生态/01.Linux环境/2. 动态库与静态库|动态库与静态库]] · [[B-构建与脚本/02.脚本语言/3. Python|Python 笔记]] · [[B-构建与脚本/01.构建工具/3. CMake入门与实践|CMake 入门与实践]]

## 1.1 两个方向

```plain
┌─────────────────────────────────────────────────────────────┐
│  Python 调 C/C++（Extension / FFI）                          │
│  ─────────────────────────────────────────────────────────  │
│  ctypes / cffi  →  已有 .so，运行时加载                      │
│  pybind11 / C API  →  编译出 .so 扩展模块，import 加载       │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  C/C++ 调 Python（Embedding / 进程外调用）                   │
│  ─────────────────────────────────────────────────────────  │
│  pybind11 embed   →  scoped_interpreter + import / py::exec  │
│  Embedding（Python.h）→  进程内嵌解释器，直接调 PyObject*    │
│  popen / system   →  子进程跑 python 脚本，见 IPC 笔记     │
└─────────────────────────────────────────────────────────────┘
```

| 方向 | 典型场景 | 数据拷贝 | 开发成本 |
|------|----------|----------|----------|
| Python → C/C++ | 调现有 C 库、硬件驱动、数值计算 | FFI 需手动声明类型 | ctypes 低 · pybind11 中 |
| C/C++ → Python | 脚本化配置、插件、批处理 | Embedding 共享堆对象 | pybind11 embed 中 · popen 低 |

## 1.2 方案总览

| 方案 | 语言侧重 | 编译 | 类型安全 | 适用 |
|------|----------|------|----------|------|
| **ctypes** | Python 调 C | 否（仅 C 库） | 弱，手写签名 | 已有 `.so`、快速胶水 |
| **cffi** | Python 调 C | ABI 模式否 · API 模式要 | 中 | 比 ctypes 更易维护 |
| **pybind11** | **双向**：Python 调 C++（扩展模块）· C++ 调 Python（embed） | 是 | 强（模板推导） | **C++ 互调首选** |
| **CPython C API** | 双向 | 是 | 手动 refcnt | 底层扩展、Embedding |
| **Cython** | Python ↔ C | 是 | 中 | 性能热点、渐进式 C 化 |
| **SWIG / nanobind** | 双向 | 是 | 中 | 大型 API 自动生成 |
| **Embedding** | C/C++ 调 Python | 是 | C API | 主程序是 C++，内嵌脚本 |
| **subprocess / popen** | C 调 Python 脚本 | 否 | 进程隔离 | 一次性任务、无共享内存 |

> 本项目驱动层：**pybind11 出 `.so` + Python `importlib`**；FITS 写文件用 **ctypes 调 libcfitsio.so**，见 [[C-Linux生态/07.Telescope1126实战/9. FITS天文图像与后处理|FITS 天文图像与后处理]]。

## 1.3 Python 调用 C：ctypes

**ctypes** 是标准库，通过 `CDLL` / `ctypes.util.find_library` 在运行时 `dlopen` 动态库，无需为 Python 侧单独编译扩展。

```python
from ctypes import CDLL, c_int, c_double, byref, POINTER

lib = CDLL("./libmath.so")

# void add(double a, double b, double *out)
lib.add.argtypes = [c_double, c_double, POINTER(c_double)]
lib.add.restype = None

result = c_double()
lib.add(1.5, 2.5, byref(result))
print(result.value)  # 4.0
```

对应 C 导出函数需 **`extern "C"`**（C++ 库）且符号可见：

```c
// math.c
void add(double a, double b, double *out) {
    *out = a + b;
}
```

```bash
gcc -shared -fPIC math.c -o libmath.so
```

### 1.3.1 ctypes 要点

| 话题 | 说明 |
|------|------|
| 结构体 | `ctypes.Structure` 字段顺序、对齐须与 C 一致 |
| 字符串 | `c_char_p` 传 bytes；C 返回 `char*` 需明确谁负责 free |
| 回调 | `CFUNCTYPE` 包装 Python 函数给 C 函数指针 |
| GIL | ctypes 调用 C 期间 **仍持有 GIL**；长时间 C 计算会阻塞其他 Python 线程 |
| 错误码 | 习惯用返回值 + 出参，Python 侧检查 status |

**优点**：不改 C 源码、不装编译链即可调 `.so`。  
**缺点**：无 C++ 类/重载/模板；签名写错易 segfault；大 API 维护成本高。

## 1.4 Python 调用 C++：pybind11

**pybind11** 是 header-only 库，用 C++11 模板把类、函数、枚举绑成 **Python 扩展模块**（`.so` / `.pyd`），导入方式与普通模块相同。

### 1.4.1 最小示例

`example.cpp`：

```cpp
#include <pybind11/pybind11.h>

namespace py = pybind11;

int add(int a, int b) { return a + b; }

PYBIND11_MODULE(example, m) {
    m.doc() = "pybind11 example";
    m.def("add", &add, "Add two integers");
}
```

`CMakeLists.txt`：

```cmake
cmake_minimum_required(VERSION 3.15)
project(pybind_example LANGUAGES CXX)

find_package(Python3 COMPONENTS Interpreter Development REQUIRED)
find_package(pybind11 CONFIG REQUIRED)

pybind11_add_module(example example.cpp)
```

```bash
mkdir build && cd build
cmake .. -DCMAKE_BUILD_TYPE=Release
cmake --build .
# 生成 example.cpython-312-x86_64-linux-gnu.so
python3 -c "import example; print(example.add(1, 2))"
```

### 1.4.2 绑定 C++ 类

```cpp
PYBIND11_MODULE(camera, m) {
    py::class_<Camera>(m, "Camera")
        .def(py::init<const std::string &>())
        .def("connect", &Camera::connect)
        .def("start_exposure", &Camera::start_exposure,
             py::arg("exposure_ms"), py::arg("raw_path"));
}
```

| 能力 | API 示例 |
|------|----------|
| 构造函数 | `.def(py::init<...>())` |
| 只读属性 | `.def_readonly("width", &Camera::width)` |
| 静态方法 | `.def_static("enumerate", &Camera::enumerate)` |
| 释放 GIL | `py::gil_scoped_release` 包裹长时间 C++ 调用 |
| 持有 Python 回调 | `py::function` 存为成员，注意 GIL 与生命周期 |

### 1.4.3 模块名与 import

- `PYBIND11_MODULE(foo, m)` → Python 侧 `import foo`（文件名 `foo*.so`，**不含** `.so` 后缀）。
- 交叉编译时注意 **Python 版本、ABI tag**（如 `cpython-312`）与目标板一致。
- 部署时把 `.so` 放入 `sys.path` 目录，或用 [[C-Linux生态/01.Linux环境/3. rpath与链接路径|rpath]] / `LD_LIBRARY_PATH` 解决依赖 `.so`。

工程细节：[[C-Linux生态/07.Telescope1126实战/6. 交叉编译与DriverSO部署|交叉编译与 DriverSO 部署]]。

## 1.5 Python 调用 C：cffi（简述）

**cffi** 分两种模式：

| 模式 | 用法 | 特点 |
|------|------|------|
| ABI | `ffi.cdef("int foo(int);")` + `ffi.dlopen("lib.so")` | 类似 ctypes，声明在 `.cdef` 字符串 |
| API | 编译时生成 `_foo.so` 扩展 | 可包 C 源码，类型检查更好 |

适合 ctypes 维护困难、但仍以 C（非 C++ 类）为主的场景。

## 1.6 C/C++ 调用 Python：Embedding

主程序是 C/C++，进程内启动 CPython 解释器，执行 Python 代码或 `import` 模块。C++ 项目 **优先用 pybind11 embed**；需要精细控制 refcnt 或纯 C 项目时再落原生 C API。

| 方式 | 头文件 / 链接 | 特点 |
|------|---------------|------|
| **pybind11 embed** | `#include <pybind11/embed.h>` · `pybind11::embed` | RAII 启停、类型转换、与扩展模块同一套 API |
| **CPython C API** | `#include <Python.h>` · `-lpython3.x` | 底层、全控制、代码冗长 |
| **popen / system** | 无嵌入 | 进程隔离，见 [[C-Linux生态/02.Linux系统编程/8. popen 与 system 的区别|popen 与 system 的区别]] |

### 1.6.1 pybind11 Embedding（C++ 调 Python）

pybind11 除 **扩展模块**（`PYBIND11_MODULE`，Python `import` C++）外，还提供 **embed** 模式：C++ `main` 启动解释器，再调 Python 函数、执行脚本、加载 `.py` 模块。

#### 1.6.1.1 最小示例：import 并调 Python 函数

`calc.py`：

```python
def add(a: int, b: int) -> int:
    return a + b
```

`embed_main.cpp`：

```cpp
#include <pybind11/embed.h>
namespace py = pybind11;

int main() {
    py::scoped_interpreter guard{};  // 等价 Py_Initialize / 析构时 Py_Finalize

    py::module_ sys = py::module_::import("sys");
    sys.attr("path").attr("append")(".");  // 让 import 找到 calc.py

    py::module_ calc = py::module_::import("calc");
    int sum = calc.attr("add")(1, 2).cast<int>();  // 3

    // 直接执行 Python 代码
    py::exec(R"(
        def hello():
            return "from embedded Python"
    )");
    std::string msg = py::globals()["hello"]().cast<std::string>();

    return 0;
}
```

`CMakeLists.txt`：

```cmake
cmake_minimum_required(VERSION 3.15)
project(embed_demo LANGUAGES CXX)

find_package(Python3 COMPONENTS Interpreter Development REQUIRED)
find_package(pybind11 CONFIG REQUIRED)

add_executable(embed_demo embed_main.cpp)
target_link_libraries(embed_demo PRIVATE pybind11::embed)
```

```bash
cmake -B build && cmake --build build
./build/embed_demo
```

| API | 用途 |
|-----|------|
| `py::scoped_interpreter` | RAII 管理解释器生命周期；**全局只能有一个** |
| `py::module_::import("name")` | 等价 `import name` |
| `mod.attr("func")(args...)` | 调模块内函数/方法 |
| `py::eval` / `py::exec` | 执行表达式 / 语句块 |
| `py::globals()` / `py::locals()` | 访问命名空间 |
| `.cast<T>()` | `PyObject*` → C++ 类型（`int`、`std::string`、`std::vector` 等） |

#### 1.6.1.2 扩展模块 vs 嵌入模块：双向同进程

pybind11 在**同一进程**里可同时做两件事：

```plain
┌──────────────────────────────────────────────────────────┐
│  C++ 主程序（Embedding）                                  │
│    py::scoped_interpreter                                 │
│         │                                                 │
│         ├── import 用户 Python 脚本 / 插件 (.py)          │
│         │                                                 │
│         └── import 内嵌 C++ 模块 (PYBIND11_EMBEDDED_MODULE)│
│              或 import 扩展 .so (PYBIND11_MODULE)         │
└──────────────────────────────────────────────────────────┘
```

| 宏 | 方向 | 加载方式 |
|----|------|----------|
| `PYBIND11_MODULE(foo, m)` | C++ → 供 Python 调 | 编译为 `foo*.so`，Python `import foo` |
| `PYBIND11_EMBEDDED_MODULE(bar, m)` | C++ → 供**嵌入进程内** Python 调 | 链进可执行文件，无需单独 `.so` |

**内嵌模块**示例 — C++ 向嵌入的 Python 暴露 `fastlib.add`：

```cpp
#include <pybind11/embed.h>
namespace py = pybind11;

int cpp_add(int a, int b) { return a + b; }

PYBIND11_EMBEDDED_MODULE(fastlib, m) {
    m.def("add", &cpp_add, "C++ add exposed to embedded Python");
}

int main() {
    py::scoped_interpreter guard{};

    py::module_ fl = py::module_::import("fastlib");
    int x = fl.attr("add")(3, 4).cast<int>();  // 7

    // Python 脚本也可 import fastlib
    py::exec(R"(
        import fastlib
        assert fastlib.add(10, 20) == 30
    )");

    return 0;
}
```

> **与本项目的关系**：Telescope1126 采用 **扩展模块** 路径（C++ `.so` + Python 主服务 `import`），主程序是 Python 而非 C++ Embedding。若将来改为 C++ 宿主 + Python 插件，可在此基础上用 `PYBIND11_EMBEDDED_MODULE` 或继续 `import` 独立 `.so`。

#### 1.6.1.3 线程与 GIL

从 **非 Python 线程** 的 C++ 代码调 Python API 前，须获取 GIL：

```cpp
void worker() {
    py::gil_scoped_acquire acquire;  // 进入 Python
    py::module_ m = py::module_::import("myplugin");
    m.attr("on_tick")();
}
```

从 Python 回调进入的 C++ **长时间计算**应释放 GIL（与 1.4 扩展模块侧对称）：

```cpp
PYBIND11_EMBEDDED_MODULE(engine, m) {
    m.def("heavy", [](int n) {
        py::gil_scoped_release release;
        return long_running_cpp(n);
    });
}
```

#### 1.6.1.4 pybind11 embed 与原生 C API 对比

| 维度 | pybind11 embed | CPython C API |
|------|----------------|---------------|
| 初始化 | `py::scoped_interpreter` | `Py_Initialize` / `Py_Finalize` |
| import 模块 | `py::module_::import("m")` | `PyImport_ImportModule` |
| 调函数 | `mod.attr("f")(arg).cast<T>()` | `PyObject_CallObject` + 手动组 tuple |
| 类型转换 | `.cast<T>()` / 自动 pybind 转换器 | 手动 `PyLong_AsLong` 等 |
| C++ 暴露给 Python | `PYBIND11_EMBEDDED_MODULE` | 手写 `PyMethodDef` + `PyModule_Create` |
| 异常 | C++ 捕获 `py::error_already_set` | 检查 `PyErr_Occurred()` |

### 1.6.2 原生 C API（Python.h）

需要与 C 代码混编、或依赖仅提供 C API 的库时使用。

```cpp
#include <Python.h>

int main() {
    Py_Initialize();

    PyRun_SimpleString(
        "print('hello from embedded Python')\n"
        "import sys\n"
        "print(sys.version)\n"
    );

    // 调 Python 函数
    PyObject *pName = PyUnicode_FromString("mymodule");
    PyObject *pModule = PyImport_Import(pName);
    Py_DECREF(pName);

    if (pModule) {
        PyObject *pFunc = PyObject_GetAttrString(pModule, "handler");
        PyObject *pArgs = PyTuple_Pack(1, PyLong_FromLong(42));
        PyObject *pValue = PyObject_CallObject(pFunc, pArgs);
        Py_DECREF(pFunc);
        Py_DECREF(pArgs);
        Py_DECREF(pModule);
        if (pValue) Py_DECREF(pValue);
    }

    Py_Finalize();
    return 0;
}
```

链接：`-lpython3.x`（版本与运行时一致）。

| 话题 | 说明 |
|------|------|
| 引用计数 | `Py_INCREF` / `Py_DECREF`；每个 `PyObject*` 成对释放 |
| GIL | 多线程 C++ 调 Python API 前 `PyGILState_Ensure()` |
| 路径 | `PySys_AppendPath` 或设置 `PYTHONPATH` |
| 与 pybind11 | C++ 新项目优先 embed；本节为 C API 等价写法 |

**适用**：纯 C 主程序、维护遗留 Embedding、或必须直接操作 `PyObject*`。  
**C++ 新项目**：优先 **1.6.1 pybind11 embed**。

轻量替代：`popen("python3 script.py", "r")` — 见 [[C-Linux生态/02.Linux系统编程/8. popen 与 system 的区别|popen 与 system 的区别]]。

## 1.7 数据与 ABI 注意事项

### 1.7.1 内存归属

| 模式 | 谁分配 / 谁释放 |
|------|-----------------|
| ctypes 入参 buffer | Python `create_string_buffer`，C 只写不 free |
| C 返回 `malloc` 指针 | 需提供 `free_xxx()` 或在 Python 用 `ctypes` 调 `free` |
| pybind11 `std::string` / `bytes` | 自动拷贝，边界清晰 |
| NumPy 数组 | pybind11 `py::array_t<T>` 或 buffer protocol |

### 1.7.2 异常与错误

- C 习惯返回错误码 → Python 包装层转成 `raise RuntimeError(...)`。
- pybind11：C++ 异常可映射为 Python 异常（`pybind11::error_already_set`）。
- ctypes：**C 侧 segfault 无法被 Python try/except 捕获**。

### 1.7.3 线程与 GIL

```cpp
// pybind11：长时间阻塞操作应释放 GIL
m.def("heavy_compute", [](int n) {
    py::gil_scoped_release release;
    return do_compute(n);
});
```

CPython 扩展与 ctypes 调用默认持有 GIL；CPU 密集型 C 代码若不释放 GIL，多线程 Python 无法并行。

## 1.8 选型决策

```
需要从 Python 调 C/C++？
  │
  ├── 已有 C .so，API 少、无 C++ 类？
  │     └── ctypes 或 cffi ABI
  │
  ├── C++ 类 / STL / 项目主语言是 C++？
  │     └── pybind11（本项目驱动层）
  │
  ├── 要包装海量 C API 自动生成？
  │     └── SWIG / clang 解析 + 代码生成
  │
  └── 热点在 Python 循环内？
        └── Cython 把 inner loop 编译成 C

需要从 C/C++ 调 Python？
  │
  ├── 偶尔跑脚本、要进程隔离？
  │     └── popen / subprocess
  │
  ├── 主程序是 C++，内嵌脚本、频繁互调？
  │     └── pybind11 embed（scoped_interpreter + import）
  │           同进程双向：PYBIND11_EMBEDDED_MODULE 暴露 C++ 给 Python
  │
  └── 纯 C 或必须原生 C API？
        └── Python.h Embedding（1.6.2）
```

| 场景 | 推荐 |
|------|------|
| 调第三方 C 库（CFITSIO 等） | ctypes |
| 自研 C++ 硬件驱动暴露给 Python | pybind11 扩展模块（`PYBIND11_MODULE`） |
| 主程序 C++，内嵌 Python 脚本/插件 | pybind11 embed（`pybind11::embed`） |
| 主程序 C++，用户写 Python 插件 + 同进程调 C++ | embed + `PYBIND11_EMBEDDED_MODULE` |
| 批处理、CI、一次性转换 | subprocess / CLI |

## 1.9 与插件架构的关系

[[D-系统与架构/02.软件架构设计/5. 插件架构设计|插件架构设计]] 中「主程序 + 动态模块」在 Python 技术栈的常见落地：

1. C++ 实现 → pybind11 编译为 `.so`
2. Python `importlib.import_module("foo")` 加载（底层仍是 `dlopen`）
3. Python 封装类（DriverAPI）统一接口，Fake / 真机可替换

这与 C 侧 `dlopen` / `dlsym` 同构，只是符号解析由 Python 导入系统完成。
