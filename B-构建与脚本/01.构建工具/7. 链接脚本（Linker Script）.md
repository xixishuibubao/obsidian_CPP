# 1. 链接脚本（Linker Script）

GNU `ld` 通过 `.ld` / `.lds` 链接脚本描述 **MEMORY 布局** 与 **SECTIONS 排布**，决定 ELF 各段的 VMA/LMA、入口符号及段与物理存储的映射。编译链末段：`gcc/g++` 调用 `ld`；裸机/嵌入式几乎必写自定义脚本；Linux 用户态通常用默认脚本，交叉编译与特殊内存布局时需显式 `-T`。

> 依据 [Using LD](https://sourceware.org/binutils/docs/ld.html)（GNU Binutils **ld 2.46**）整理；正文已剔除 MIPS/68K/HPPA 等非 ARM/x86 架构章节。语法与旧版 binutils 向后兼容；具体选项以本机 `ld --help` 为准。

## 【核心精简提纲】

### 记忆主干

**链接末步**：`.o`/`.a` → `ld`（+ 脚本）→ 可执行/`.so`/ROM 镜像。**SECTIONS** 必选；**MEMORY** 可选（裸机/Flash+RAM 必用）。**`.`** 为位置计数器；**VMA** 运行地址，**LMA** 加载地址（`AT()` 分离，启动代码负责拷贝）。入口：`-e` > `ENTRY()` > `start` > `.text` 首字节 > `0`。架构增量：**ARM32**（Thumb/stub/errata）→ **AArch64**（BTI/PAC/GCS/MTE）→ **x86 ELF**（CET/IBT/LAM）→ **x86 PE**（DLL/DEF）。

### 核心要点清单

- 驱动：`-T script.ld` / `-Wl,-T,script.ld`；**替换**默认脚本；隐式追加仅 `INPUT`/`GROUP`/`OUTPUT`
- 骨架：`ENTRY(sym)` · `MEMORY { rom(rx): ORIGIN=0, LENGTH=256K … }` · `SECTIONS { .text : { *(.text) } >rom }`
- 收集输入：`*(.text)` · `file.o(.data)` · `/DISCARD/` · `KEEP()` · `OVERLAY`（省 RAM）
- 孤儿段：脚本未显式收集的输入段；`-Wl,--orphan-handling=place|warn|error|discard`；嵌入式推荐 `error` + 脚本补全
- 段属性：`start` 强制 VMA · `AT(ldaddr)` 设 LMA · `(NOLOAD)` · `>region` · `=fill`
- 符号：`symbol = . ;` · `PROVIDE`/`HIDDEN`/`PROVIDE_HIDDEN` · `REGION_ALIAS` · `ASSERT(exp,"msg")`
- 内置函数：`ADDR` · `LOADADDR` · `ALIGN` · `SIZEOF` · `DEFINED` · `MAX`/`MIN`
- 进阶：`PHDRS`（程序头）· `VERSION`（`.so` 符号版本）
- ARM32：`--thumb-entry` · `--stub-group-size` · `--fix-v4bx` · Cortex/STM32 errata
- AArch64：`-z force-bti` · `-z pac-plt` · `-z gcs` · `-z memtag-*`
- x86 ELF：`-z ibt`/`shstk`/`cet-report` · `-z x86-64-v2|v3|v4` · `-z lam-u48|lam-u57`
- x86 PE：`--dll` · `.def` · `--out-implib` · `--export-all-symbols`
- 调试：`readelf -l/-S` · `objdump -h` · `ld -Map=out.map`
- 链接脚本管 **链接期** 布局；**`ld-linux.so`** 管运行时 → [[C-Linux生态/01.Linux环境/2. 动态库与静态库|动态库与静态库]]

### 关键结论 / 流程

```plain
源文件 -c → .o ─┐
                ├→ ld [-T script.ld] [-e entry] … → ELF（段表 + 符号表）
静态库 .a ──────┘
裸机：ROM 中 .data 的 LMA 在 .text 后，启动代码 memcpy(.data) + 清零 .bss
Linux：默认脚本已含 crt 与 libc；自定义脚本多用于 -nostdlib / 固件 / 内核模块
```

## 【完整拓展详情】

## 1.1 ld 与链接脚本的角色

`ld`（GNU linker）合并多个 **object** 与 **archive**，重定位数据、解析符号引用，产出最终可执行文件或共享库。通常由 `gcc`/`g++` 在最后一步隐式调用：

```bash
gcc -o hello hello.o -lc
# 等价于大致：ld -o hello /lib/crt0.o hello.o -lc -dynamic-linker /lib/ld-linux.so ...
```

每个链接都受脚本控制；未指定 `-T` 时使用内嵌默认脚本（`ld --verbose` 可 dump）。当需要 **完全控制** 输出布局（裸机、Bootloader、ROM 镜像、自定义内存映射）时，编写 **Linker Command Language** 脚本，经 `-T` 传给 `ld`。脚本控制：

- 输入文件与格式（`INPUT`/`GROUP`/`OUTPUT_FORMAT`/`OUTPUT_ARCH`）
- 输出段顺序与地址
- 符号赋值与入口点
- COMMON 块放置

## 1.2 命令行：如何指定脚本

| 选项 | 含义 |
|------|------|
| `-T file.ld` / `--script=file.ld` | 用 `file.ld` **替换**默认链接脚本 |
| `-e sym` / `--entry=sym` | 指定入口符号（优先级最高） |
| `-Map=file.map` | 输出链接 map，便于查段地址与符号 |
| `-L dir` | 脚本 `-T` 相对路径时也在 `-L` 目录中搜索 |
| `-nostdlib` | 不链接标准启动文件与默认库（裸机常用） |

经 GCC 转发：

```bash
gcc -nostdlib -T baremetal.ld -Wl,-Map=firmware.map startup.o main.o -o firmware.elf
# 或
g++ ... -Wl,-T,layout.ld,-Map,app.map
```

**注意**：`-T` 指定的是 **完整** 脚本；若某 `.o` 实为只含 `INPUT`/`GROUP`/`OUTPUT` 的脚本片段，则 **追加** 到主脚本，不能重复出现 `SECTIONS`/`MEMORY`（这类命令全局只能一次）。

## 1.3 脚本基本结构

有意义的脚本至少包含 **`SECTIONS`**；嵌入式通常还有 **`MEMORY`** 与 **`ENTRY`**：

```plain
/* 可选：输出格式 / 入口 */
OUTPUT_FORMAT("elf32-littlearm")
ENTRY(_start)

MEMORY
{
    FLASH (rx)  : ORIGIN = 0x08000000, LENGTH = 512K
    RAM   (rwx) : ORIGIN = 0x20000000, LENGTH = 128K
}

SECTIONS
{
    .text :
    {
        KEEP(*(.vectors))
        *(.text*)
        *(.rodata*)
        _etext = .;
    } > FLASH

    .data : AT (ADDR(.text) + SIZEOF(.text))
    {
        _sdata = .;
        *(.data*)
        _edata = .;
    } > RAM

    .bss (NOLOAD) :
    {
        _sbss = .;
        *(.bss*)
        *(COMMON)
        _ebss = .;
    } > RAM
}
```

注释语法与 C 相同：`/* ... */`。可选增强（手册 §3，`ASSERT`/`REGION_ALIAS`/`PROVIDE`）：

```plain
REGION_ALIAS("ROM", FLASH);
PROVIDE(_heap_end = ORIGIN(RAM) + LENGTH(RAM));
ASSERT(_ebss <= ORIGIN(RAM) + LENGTH(RAM), "RAM overflow");
```

## 1.4 MEMORY：描述目标机存储

`MEMORY` 块声明 **命名存储区域**（与 C 符号不同命名空间）：

```plain
MEMORY
{
    name [(attr)] : ORIGIN = expr, LENGTH = expr
}
```

- **ORIGIN**（可写 `org`/`o`）：区域起始物理地址，须在分配前可求值为常数
- **LENGTH**（可写 `len`/`l`）：区域字节长度；可用 `256K`、`4M` 等后缀
- **attr**：`R` 只读 · `W` 读写 · `X` 可执行 · `A`/`I`/`L` 已分配/已初始化 · `!` 取反

示例（ROM 放代码/只读数据，RAM 放可写段）：

```plain
MEMORY
{
    rom (rx)  : ORIGIN = 0,       LENGTH = 256K
    ram (!rx) : org = 0x40000000, l = 4M
}
```

`REGION_ALIAS(alias, region)` 为区域起别名，便于多段共享同一物理区或切换布局模板。

`ld` **不会**自动搬段以塞进区域；段通过 `>region` 指定归属，超出 `LENGTH` 则 **报错**。

## 1.5 SECTIONS：输出段定义

### 1.5.1 最小形式

```plain
SECTIONS {
    secname : { contents }
}
```

`secname` 两侧需有空格；`:` 与 `{}` 必填。无内容的输出段 **不会** 生成（便于可选段如 `.foo { *(.foo) }`）。

### 1.5.2 收集输入段

| 写法 | 含义 |
|------|------|
| `* (.text)` | 所有尚未分配的文件中的 `.text` |
| `start.o (.vectors)` | 指定文件的段 |
| `* (.text .text.*)` | 多个段名 |
| `* (COMMON)` | 未初始化 COMMON 块 |
| `/DISCARD/` | 匹配段丢弃，不进最终输出 |
| `SORT_BY_NAME` / `SORT_BY_ALIGNMENT` | 排序收集 |

经典三元组：

```plain
SECTIONS {
    .text : { *(.text) }
    .data : { *(.data) }
    .bss  : { *(.bss) *(COMMON) }
}
```

通配符规则类似 shell：`*` `?` `[a-z]`；`KEEP()` 防止 `--gc-sections` 丢弃关键段。

### 1.5.3 完整段定义语法

```plain
secname start BLOCK(align) (NOLOAD) : AT (ldaddr)
{
    contents
} > region :phdr = fill
```

除 `secname` 与 `contents` 外均可选：

| 成分 | 作用 |
|------|------|
| `start` | 强制本段 VMA（如 `output 0x40000000:`） |
| `BLOCK(n)` / `ALIGN(n)` / `SUBALIGN(n)` | 段起始或输入段对齐 |
| `(NOLOAD)` | 段进入 ELF 但不加载到内存（BSS、栈堆预留、调试区） |
| `AT (ldaddr)` | LMA；默认 LMA = VMA |
| `> region` | 绑定 `MEMORY` 中的区域 |
| `:phdr` | 关联 `PHDRS` 定义的程序头 |
| `= 0xFF` / `=fill` | 段内未显式写入的间隙填充值 |

### 1.5.4 位置计数器 `.`

特殊符号 **`.`** 表示当前输出位置计数器，只能在 `SECTIONS` 内使用：

```plain
.text : {
    *(.text)
    . = . + 0x1000;   /* 跳过 1KB 空洞 */
    _gap_end = .;
} = 0xFF              /* 空洞填 0xFF */
```

对 `.` 赋值会 **前移** 计数器制造空洞；**不能**向后移动。段内还可写 `SHORT`/`LONG`/`QUAD`/`SQUAD`/`FILL(expr)` 嵌入常量。

### 1.5.5 VMA 与 LMA

- **VMA**（Virtual Memory Address）：程序 **运行** 时使用的地址（`readelf -S` 的 `Addr`）
- **LMA**（Load Memory Address）：段在 **加载镜像**（Flash/ROM/下载文件）中的地址（`readelf -l` 的 `LOAD` 段）

Flash 驻留、RAM 运行的 `.data` 典型写法：

```plain
.text 0x1000 : { *(.text) _etext = .; }
.mdata 0x2000 :
    AT (ADDR(.text) + SIZEOF(.text))
{
    _data = .;
    *(.data);
    _edata = .;
}
.bss 0x3000 : { _bstart = .; *(.bss) *(COMMON); _bend = .; }
```

启动代码（或 `crt0`）需把 ROM 中 `_etext` 起的数据 **拷贝** 到 `_data`…`_edata`，并 **清零** `_bstart`…`_bend`。这与 [[A-编程语言/01.C语言/2. C语言底层原理|C 语言底层原理]] 中程序启动链衔接。

验证：

```bash
readelf -S firmware.elf    # 看各节 VMA
readelf -l firmware.elf    # 看 LOAD 段 LMA/VMA
objdump -h firmware.elf
```

### 1.5.6 OVERLAY

多个输出段 **共享同一 VMA**，LMA 在 ROM 中依次排列，运行前由加载器/启动代码切换——适合大型算法分块加载、省 RAM。

### 1.5.7 孤儿段（Orphan Sections）

**孤儿段**（orphan section）指输入文件（`.o`/`.a`）中存在、但链接脚本 **未显式收集** 的输入段。GNU `ld` 默认 **不会丢弃** 它们，而是按内置策略自动安置；裸机/嵌入式若依赖自定义 `MEMORY` 布局，这种「静默安置」常导致段落入错误存储区（如可写数据进 Flash、调试/异常表占 ROM），且链接 **不报错**。

#### 1.5.7.1 默认安置策略（`--orphan-handling=place`）

依据 [Using LD §3.10.4 Orphan Sections](https://sourceware.org/binutils/docs/ld/Orphan-Sections.html)：

1. 若孤儿段 **名称与已有输出段完全相同** → 追加到该输出段 **末尾**
2. 否则 → **新建** 同名输出段；多个同名孤儿段合并进同一新输出段
3. 新输出段的位置：优先放在 **属性相同** 的已有段之后（代码 vs 数据、可加载 vs `NOLOAD` 等）；目标不支持或无匹配时 → 放在 **文件末尾**

因此 `__attribute__((section(".mybuf")))` 却未写 `*(.mybuf)` 时，`.mybuf` 仍可能出现在最终 ELF 中，但 **VMA 与 `>region` 归属未必符合预期**。

#### 1.5.7.2 `--orphan-handling` 与 `--unique`

| 模式 | 行为 |
|------|------|
| `place`（默认） | 按上节策略安置，无警告 |
| `warn` | 同 `place`，并对每个孤儿段 **打印 warning** |
| `error` | 发现任意孤儿段即 **链接失败** |
| `discard` | 等价于全部送入 `/DISCARD/` |

经 GCC 转发：

```bash
# 开发期：先 warn 找出漏写的段名
gcc ... -Wl,--orphan-handling=warn

# 裸机/固件：强制脚本显式声明每一类段
gcc -nostdlib -T baremetal.ld ... -Wl,--orphan-handling=error
```

`--unique[=SECTION]`：为每个匹配的输入段（省略 `SECTION` 时为 **每个孤儿段**）单独建输出段，禁止同名段合并；会 **覆盖** 脚本中的合并规则，与 orphan 安置策略联动。

#### 1.5.7.3 典型成因

| 来源 | 示例 |
|------|------|
| 自定义段属性 | `__attribute__((section(".fast_ram")))` 未在脚本中 `*(.fast_ram)` |
| 编译器/工具链段 | `.ARM.exidx`、`.ARM.attributes`、`.eh_frame`、`.init_array` |
| 调试信息 | `.debug_*`、`.zdebug_*`（`-g` 时） |
| 插件/库新增段 | 第三方 `.o` 引入脚本未覆盖的节名 |

#### 1.5.7.4 解决思路

**① 在脚本中显式收集**（首选，语义清晰）：

```plain
SECTIONS {
    .text : { *(.text*) *(.rodata*) } > FLASH
    .fast_ram (NOLOAD) : { *(.fast_ram) *(.fast_ram.*) } > RAM
    .myconfig : { KEEP(*(.myconfig)) } > FLASH
}
```

**② 明确丢弃不需要的段**：

```plain
/DISCARD/ : {
    *(.ARM.exidx)
    *(.ARM.attributes)
    *(.comment)
}
```

**③ 链接期策略**（与 ① 配合，不替代脚本）：

- 嵌入式固件：`--orphan-handling=error`，迫使脚本与源码段名 **一一对应**
- 日常开发：先用 `--orphan-handling=warn`，根据 warning/map 补脚本
- 确定无用：`discard` 或 `/DISCARD/`

**④ 启用 `error` 时的常见补全**：除业务段外，若保留 `-g`，须在脚本或默认策略可接受处处理 `.debug_*`；C++ 异常/静态构造需 `.eh_frame`、`.init_array` 等——要么写入脚本，要么在 `-nostdlib`/裸机场景用 `/DISCARD/` 或关闭对应语言特性。

#### 1.5.7.5 排查

```bash
# 1) warn 模式看链接器列出的 orphan 段名
gcc ... -Wl,--orphan-handling=warn 2>&1 | grep -i orphan

# 2) map 中搜索 Unexpected 或查看各 .o 贡献的段
ld -Map=firmware.map -T baremetal.ld ... -o firmware.elf

# 3) 反查输入段是否进入预期输出段
readelf -S firmware.elf
objdump -h firmware.elf
```

Map 文件中若出现 **脚本未定义** 的输出段名，且该段来自自定义 `section` 属性，即为孤儿段被自动安置的结果。

## 1.6 表达式与符号

表达式语法 **同 C**：整数常量、`+ - * / % & | ^ ~`、括号、三元 `?:`；常量后缀 `4K`/`1M`。

**赋值**（须独立一行并以 `;` 结尾）：

```plain
_stack_top = 0x20010000;
.data : { _sdata = .; *(.data); _edata = .; }
```

- 在 `SECTIONS` **外**赋值 → 绝对地址符号
- 在段 **内**赋值 → 相对段基址的可重定位符号
- `ABSOLUTE(.)` 在段内得到绝对地址

**PROVIDE / HIDDEN**：

```plain
PROVIDE(_heap_start = .);              /* 仅在被引用且未定义时提供 */
HIDDEN(_internal = .);                 /* 符号不导出 */
PROVIDE_HIDDEN(_compat_sym = .);       /* 二者组合 */
```

**ASSERT**：表达式为 0 则链接失败：

```plain
ASSERT(_ebss <= ORIGIN(RAM) + LENGTH(RAM), "RAM overflow");
```

常用内置函数：

| 函数 | 含义 |
|------|------|
| `ADDR(section)` / `LOADADDR(section)` | VMA / LMA |
| `ALIGN(n)` / `ALIGNOF(sec)` | 对齐 |
| `SIZEOF(section)` / `SIZEOF_HEADERS` | 段或文件头大小 |
| `DEFINED(sym)` | 符号是否已定义 |
| `MAX(a,b)` / `MIN(a,b)` | 最值 |
| `DATA_SEGMENT_ALIGN` / `DATA_SEGMENT_RELRO_END` | Linux 高级布局 |
| `NEXT(exp)` | 下一可用地址 |

`ld` 对表达式 **惰性求值**：依赖段大小的赋值在分配完成后才算。

## 1.7 入口点 ENTRY

```plain
ENTRY(_start)
```

优先级（高到低）：

1. 命令行 `-e symbol`
2. 脚本 `ENTRY(symbol)`
3. 符号 `start`（若存在）
4. `.text` 第一节的首地址
5. 地址 `0`

裸机常 `ENTRY(Reset_Handler)`；ARM Thumb 可用 `--thumb-entry=sym`（置 LSB 以 BX 进入 Thumb）。Linux 用户态默认 `_start` 在 `crt1.o` 中，见 [[A-编程语言/01.C语言/2. C语言底层原理|自定义入口]]（`-nostartfiles -e`）。

## 1.8 PHDRS 与 VERSION

**PHDRS**：自定义 ELF 程序头；段通过 `:phdr_name` 关联。裸机/特殊 ELF 才需手写；Linux 用户态默认脚本已含 `.interp`、`.dynamic` 等，完整替换 `-T` 时须自行保留，否则动态链接失败。

**VERSION**：共享库符号版本脚本，控制导出/隐藏：

```plain
VERS_1.0 {
  global: foo; bar;
  local: *;
};
```

或最简 `{ global: foo; bar; local: *; };`。与 [[C-Linux生态/01.Linux环境/2. 动态库与静态库|SONAME/版本管理]] 衔接；内核 `vmlinux.lds.S` 属于同类语法的高级应用。

## 1.9 Linux 用户态场景

桌面/服务器程序 **通常不写** 链接脚本：`gcc` 使用工具链自带默认脚本。需要自定义的常见情况：

- `-nostdlib` / `-static` 特殊布局
- 内核模块、Bootloader、initramfs 链接
- 将某段固定到特定 VMA（与 [[C-Linux生态/03.Linux开发/2. 交叉编译优化选项|交叉编译]] 内存规划一致）
- 用 `/DISCARD/` 去掉未用段配合 `--gc-sections`

查看默认脚本：

```bash
ld --verbose 2>&1 | less
# 或
gcc -Wl,--verbose 2>&1 | less
```

最小 **追加** 符号（不能含 `SECTIONS`）的脚本片段示例——强制引用某库：

```plain
/* extra.lds — 由 gcc -Wl,-T,extra.lds 追加时仅适合 INPUT/GROUP */
GROUP( -lgcc -lc -lm )
```

## 1.10 MCU / 裸机场景

### 1.10.1 Flash + RAM 标准布局

见 §1.3 骨架：向量表 `KEEP(*(.vectors))` 防止 `--gc-sections` 删掉；`.data` 的 `AT()` 指向 Flash 中紧跟 `.text` 的位置；`.bss` 常用 `(NOLOAD)` 仅占 VMA。

### 1.10.2 NOLOAD 预留 RAM（代码注入 / 缓冲区）

不占用下载镜像，仅在运行时占地址空间，见 [[D-系统与架构/04.MCU嵌入式/2. ARM32内存注入（UART）|ARM32 内存注入]]：

```plain
SECTIONS {
    .exec_ram (NOLOAD) : ALIGN(4096) {
        _exec_start = .;
        . += 4K;
        _exec_end = .;
    } > DRAM
}
```

C 代码中通过 `_exec_start`…`_exec_end` 获得固定 RWX 缓冲区。

## 1.11 与 gcc / Makefile / CMake 集成

**Makefile**：

```makefile
LDFLAGS += -T $(TOP)/scripts/stm32f4.ld -Wl,-Map=$(BUILD)/firmware.map
$(BUILD)/firmware.elf: $(OBJS) $(LDSCRIPT)
	$(CC) $(OBJS) $(LDFLAGS) -nostdlib -o $@
```

**CMake**：

```cmake
target_link_options(firmware PRIVATE
    "-T${CMAKE_SOURCE_DIR}/linker/stm32f4.ld"
    "LINKER:--Map=${CMAKE_BINARY_DIR}/firmware.map"
)
```

**交叉编译**：使用 `aarch64-linux-gnu-ld` 或 `$(CC) -Wl,-T,...` 确保与 [[B-构建与脚本/01.构建工具/1. g++|g++]] 同一工具链前缀。

## 1.12 调试与验证

```bash
# 链接 map：段、符号、丢弃段
ld -Map=out.map -T app.ld ... -o app.elf

# ELF 段与程序头
readelf -S app.elf
readelf -l app.elf

# 反查 VMA
nm app.elf | grep _etext
objdump -t app.elf | head
```

Map 文件中关注：`Memory Configuration`、`*(.text)` 匹配了哪些 `.o`、LMA/VMA 列。

## 1.13 常见踩坑

| 现象 | 原因 / 对策 |
|------|-------------|
| `region RAM overflowed` | `> RAM` 段总和超过 `MEMORY` 的 `LENGTH`；缩段或扩 RAM |
| `Non constant expression for initial address` | 段起始用了链接时未知大小的表达式 |
| `.data` 运行全零 | 有 `AT()` 但未写启动拷贝；或 LMA 算错 |
| 向量表被 GC 掉 | 对关键段用 `KEEP(*(.isr_vector))` |
| `-T` 后无法动态链接 | 自定义脚本缺 `.dynamic`/`.interp`；用户态慎用完整替换 |
| 脚本当 `.o` 追加却含 `SECTIONS` | 应用 `-T` 完整替换，勿混用 |
| `A-B` vs `A - B` | 前者是符号名，后者是减法；符号名含特殊字符用引号 |
| 与 **动态链接器** 混淆 | 链接脚本 ≠ `ld.so`；运行时库路径见 [[C-Linux生态/01.Linux环境/3. rpath与链接路径]] |
| 自定义段跑错地址/进 Flash | 孤儿段被默认 `place` 静默安置；用 `-Wl,--orphan-handling=warn` 定位，脚本补 `*(.段名)` 或裸机设 `error` |
| `--orphan-handling=error` 仍失败 | 除业务段外漏掉 `.debug_*`、`.ARM.exidx` 等；补收集规则或 `/DISCARD/` |

## 1.14 ARM32 链接选项（GNU ld §6.3）

除通用脚本外，ARM/Thumb 目标还提供下列 **链接期** 命令行行为：

| 选项 | 用途 |
|------|------|
| `-mthumb-interwork`（编译/汇编）+ 默认 stub | ARM↔Thumb 互调；旧库无 interwork 时用 `--support-old-code` |
| `--thumb-entry=sym` | 同 `--entry`，且置位 LSB，以 BX 进入 Thumb |
| `--fix-v4bx` | ARMv4 纯 ARM：将 `BX` 改写为 `MOV PC,rM` |
| `--fix-v4bx-interworking` | ARMv4 且需互操作：BX 替换为 veneer |
| `--use-blx` | ARMv5t+ 用 BLX 优化 Thumb→PLT |
| `--stub-group-size=N` | 远跳转 stub 分组（默认 `+1`）；仅 ARM-EABI 完整支持 farcall |
| `--target1-rel` / `--target1-abs` | `.init_array` 中 `R_ARM_TARGET1` 解释 |
| `--target2=rel\|abs\|got-rel` | `R_ARM_TARGET2` 重定位类型 |
| `--be8` | 大端 BE8：大端数据 + 小端代码 |
| `--pic-veneer` | 互操作 veneer 用 PIC 序列（uClinux `--emit-relocs`） |
| `--long-plt` | 16 字节 PLT（代码空间 >512MB） |

**Silicon errata workaround**（按需启用）：

| 选项 | 目标 |
|------|------|
| `--fix-arm1176` / `--no-fix-arm1176` | ARM1176 |
| `--fix-cortex-a8` / `--no-fix-cortex-a8` | Cortex-A8（Thumb-2） |
| `--fix-cortex-a53-835769` | Cortex-A53 erratum 835769 |
| `--fix-stm32l4xx-629360` | STM32L4xx 总线读 ≥9 word |
| `--vfp11-denorm-fix=scalar\|vector\|none` | VFP11 denorm 硬件缺陷 |

**TrustZone / CMSE**（ARMv8-M）：`--cmse-implib`、`--in-implib=file`；SG veneer 放 `.gnu.sgstubs`，须 `--section-start` 或脚本指定地址。

## 1.15 AArch64 链接选项

AArch64 选项合并在 ARM §6.3，无独立章节。裸机脚本用 `OUTPUT_FORMAT("elf64-littleaarch64")` + 通用 `MEMORY`/`SECTIONS`；下列 `-z` 主要用于 **Linux ELF**：

| 选项 | 用途 |
|------|------|
| `--no-apply-dynamic-relocs` | 不在链接期应用动态 RELA 加数 |
| `-z force-bti` | 校验 BTI 标记，PLT 带 BTI，输出打 BTI property |
| `-z bti-report[=none\|warning\|error]` | 缺失 `GNU_PROPERTY_AARCH64_FEATURE_1_BTI` 报告级别 |
| `-z pac-plt` | PLT 使用 Pointer Authentication |
| `-z gcs[=implicit\|always\|never]` | Guarded Control Stack 标记策略 |
| `-z gcs-report` / `-z gcs-report-dynamic` | 缺失 GCS property 诊断 |
| `-z memtag-mode=none\|sync\|async` | MTE 模式 → `DT_AARCH64_MEMTAG_MODE` |
| `-z memtag-stack` | 栈使用 MTE 指令 |

## 1.16 x86 Linux ELF（`-z` 选项节选）

LD 2.46 中下列 `-z` 支持 **Linux/i386** 和/或 **Linux/x86_64**：

| 选项 | 含义 |
|------|------|
| `-z ibt` / `-z ibtplt` | Intel IBT：`.note.gnu.property` + IBT PLT |
| `-z shstk` | Shadow Stack（CET）property |
| `-z cet-report=none\|warning\|error` | 缺失 IBT/SHSTK property 报告 |
| `-z call-nop=prefix-addr\|suffix-nop\|…` | GOT 间接 call 的 NOP 填充 |
| `-z x86-64-baseline` / `-v2` / `-v3` / `-v4` | 输出 `GNU_PROPERTY_X86_ISA_1_*` ISA 级别 |
| `-z lam-u48` / `-z lam-u57` | Intel LAM（仅 x86_64）及缺失报告 |
| `-z mark-plt` | PLT 动态 tag `DT_X86_64_PLT*`（x86_64） |
| `-z indirect-extern-access` / `noindirect-extern-access` | 禁止 copy reloc 的 canonical 函数指针 |
| `-z noseparate-code` / `-z separate-code` | 可执行代码与只读数据分页隔离 |

## 1.17 x86 PE / WIN32（MinGW/cygwin）

### 1.17.1 生成 DLL 与导出

- `-shared` / `--dll`：输出 DLL（`.dll`）
- `.def` 文件：ASCII 导出列表，作为输入传给链接器；使用 DEF 时默认关闭 auto-export，除非 `--export-all-symbols`
- `--out-implib=file`：生成导入库；`--output-def=file`：反生成 DEF
- 导出途径：**Auto-export**（默认）· **DEF 文件** · 源码 `__declspec(dllexport)` / `__attribute__((dllexport))`

### 1.17.2 PE 专用选项（节选）

| 选项 | 含义 |
|------|------|
| `--enable-stdcall-fixup` | cdecl/stdcall 名模糊匹配 |
| `--kill-at` / `--add-stdcall-alias` | stdcall `@nn` 后缀处理 |
| `--large-address-aware` | >2GB 用户空间 |
| `--file-alignment` / `--section-alignment` | PE 对齐 |
| `--heap reserve,commit` / `--stack` | 堆栈预留 |
| `--subsystem windows\|console\|…` | PE 子系统 |

### 1.17.3 仅资源 DLL 的链接脚本

默认 PE 脚本总会生成 `.text`/`.idata`；**纯资源 DLL** 须自定义脚本：

```plain
OUTPUT_FORMAT(pei-i386)
SECTIONS
{
  . = SIZEOF_HEADERS;
  . = ALIGN(__section_alignment__);
  .rsrc __image_base__ + __section_alignment__ : ALIGN(4)
  {
    KEEP (*(.rsrc))
    KEEP (*(.rsrc$*))
  }
  /DISCARD/ : { *(*) }
}
```

```bash
ld -dll --subsystem windows -e 0 -s rsrc.o -o rsrc.dll -T rsrc.ld
```

x86_64 PE 须将 `OUTPUT_FORMAT` 改为 `pei-x86-64` 等对应格式名。
