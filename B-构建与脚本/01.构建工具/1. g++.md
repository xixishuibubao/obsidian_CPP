# 1. g++ 编译

> `[]` 代表参数可缺省，`<>` 必填参数，`...` 代表重复或可拓展参数。

## 1. gcc 编译命令

- `gcc [-控制参数] <原始文件> -o [目标文件名] [可选附加]`

	1. `gcc -E a.c -o a.i` 生成.i预处理后的文件
	2. `gcc -S a.c -o a.s` 进行预处理和编译，生成.s汇编文件
	3. `gcc -c a.c -o a.o`生成.o目标文件
	4. `gcc a.c -o a `编译并链接，生成可执行程序

可选附加：

- `-Wall warning all` 显示告警
- `-g` 编译携带gdb调试信息
- `-OX` （为0-3，或S） 优化等级

- `-D`就是宏定义
- `-DABC` 等价`#define ABC` 
- `-DABC=123` 等价`#define ABC 123`

## 2. 编译选项

### 2.1 符号可见性

```bash
-fvisibility=hidden
```

对直接编译和编译 `.a` 库：减少符号表大小，不影响调用。对 `.so` 动态库：隐藏后外部无法正常调用。

### 2.2 运行时库路径（rpath）

`-Wl,-rpath` 指定动态链接器在运行时搜索共享库的路径，解决程序找不到 `.so` 的问题。

相关笔记：[[C-Linux生态/01.Linux环境/3. rpath与链接路径|运行时链接路径]]（OS 级概述与搜索优先级）

#### 2.2.1 单路径写法

`-Wl,-rpath` 后跟路径，逗号和等号均合法：

```bash
# 逗号写法（推荐，最通用）
-Wl,-rpath,/usr/lib

# 等号写法
-Wl,-rpath=/usr/lib
```

> `-Wl,xxx` 表示将参数 `xxx` 传递给链接器 `ld`，逗号是 GCC 传递参数到 ld 的分隔符。

#### 2.2.2 多路径写法

**核心规则：多路径用冒号 `:` 分隔。**

```bash
# 单 -rpath + 冒号（主流写法，兼容最好）
-Wl,-rpath,/usr/lib:/usr/local/lib:/usr/lib/rknn

# 单个 -rpath= + 冒号
-Wl,-rpath=/usr/lib:/usr/local/lib:/usr/lib/rknn

# 多次写 -rpath（等效，路径自动合并）
-Wl,-rpath,/usr/lib -Wl,-rpath,/usr/local/lib -Wl,-rpath,/usr/lib/rknn
```

#### 2.2.3 相对路径与 `$ORIGIN`

##### `./` 相对路径（不推荐）

```bash
-Wl,-rpath,./lib
```

`./` 表示**当前工作目录**（用户执行命令的目录），不是程序所在目录。换目录运行就会失效，**不推荐正式产品使用**。

##### `$ORIGIN`（推荐）

`$ORIGIN` = **可执行文件/so 自身所在目录**，移动整个目录也能正常运行，嵌入式/打包程序首选。

```bash
# 程序同目录
-Wl,-rpath,'$ORIGIN'

# 程序目录下的 lib 子目录
-Wl,-rpath,'$ORIGIN/lib'

# 多目录组合
-Wl,-rpath,'$ORIGIN:$ORIGIN/lib:/usr/lib'
```

> **必须加单引号**，防止 shell 将 `$ORIGIN` 解析为空。

##### 交叉编译注意事项

- rpath 路径是**目标板运行时**的路径，不是编译主机的路径
- 常见错误：写入主机上的绝对路径（如 `/home/user/rootfs/usr/lib`），目标板上不存在

#### 2.2.4 验证 rpath

```bash
readelf -d your_app | grep RPATH
```

正确输出示例：
```
0x000000000000000f (RPATH) Library rpath: [$ORIGIN:$ORIGIN/lib]
```

#### 2.2.5 CMake 写法

CMake 中用**分号 `;`** 分隔多路径（不是冒号）：

```cmake
set_target_properties(target PROPERTIES
    BUILD_RPATH "$ORIGIN;$ORIGIN/lib"
    INSTALL_RPATH "$ORIGIN;$ORIGIN/lib"
)
```

#### 2.2.6 常见坑点

| 坑点 | 正确做法 |
|------|----------|
| 多路径用逗号分隔 | ❌ 错误，必须用**冒号 `:`** |
| `$ORIGIN` 没加引号 | ❌ shell 解析为空，编译后 RPATH 缺失 |
| 交叉编译写入主机路径 | ❌ 写入**目标板上的路径** |
| 裸用 `./lib` | ❌ 换目录就失效，优先用 `$ORIGIN` |

### 2.3 交叉编译

指定目标平台架构与标准库路径，编译主机上生成目标板的机器码。

相关笔记：[[C-Linux生态/03.Linux开发/3. 交叉编译优化选项|RK3588 交叉编译实例]]

#### 2.3.1 基本参数

```bash
gcc \
  --sysroot=/opt/fsl-imx-x11/4.1.15-2.0.0/sysroots/cortexa7hf-neon-poky-linux-gnueabi \
  -march=armv7ve \
  -mfpu=neon \
  -mfloat-abi=hard \
  -mcpu=cortex-a7
```

| 参数 | 说明 |
|------|------|
| `--sysroot` | 指定标准库路径（交叉编译必选项） |
| `-march` | 目标架构（指定指令集） |
| `-mfpu` | 浮点单元（仅 ARM32 需显式指定） |
| `-mfloat-abi` | 浮点 ABI 类型（仅 ARM32：soft/hard） |
| `-mcpu` | 硬件型号（省去同时写 -march/-mtune） |

#### 2.3.2 `-march` 核心规则

##### 常见错误

| 写法 | 结果 |
|------|------|
| `-march=native` | **交叉编译禁用**，识别的是编译主机（x86）而非目标板 |
| `armv8.2+` | **不规范**，标准写法必须是 `armv8.2-a` |
| 无效值如 `active` | 编译报错，GCC 无此架构 |

##### ARMv8 SIMD 默认行为

`armv8-a` 及以上版本**默认开启 SIMD（NEON/ASIMD）**，无需手动添加 `+simd`：

| `-march` | SIMD/NEON |
|----------|-----------|
| `armv8-a` | ✅ 默认自带 `+fp +simd` |
| `armv8.1-a` | ✅ 继承，默认保留 |
| `armv8.2-a` | ✅ 继承，默认保留 |

```bash
# 以下效果完全相同，均开启 SIMD
-march=armv8-a
-march=armv8.1-a
-march=armv8.2-a

# 只有显式 +nosimd 才会关闭
-march=armv8.2-a+nosimd   # ❌ 禁用 NEON
```

> `armv8.2-a` 新增的是 FP16、点积指令等，**不改变原有 SIMD/NEON 开关**。

#### 2.3.3 `-mtune` vs `-march` vs `-mcpu`

| 参数 | 控制内容 | 适用场景 |
|------|----------|----------|
| `-march` | **指令集架构**（允许使用的指令） | 代码兼容性关键，目标板最低指令集 |
| `-mtune` | **流水线调度优化**（不影响指令集） | 针对特定微架构做调度优化 |
| `-mcpu` | `-march` + `-mtune` 的简写 | 明确知道 CPU 型号时使用 |

```bash
# 指定指令集 + 调度优化（推荐写法）
-march=armv8-a -mtune=cortex-a76

# -mcpu 等效写法（省事）
-mcpu=cortex-a76
```

> `-mcpu` 相当于同时指定 `-march`（对应 CPU 的架构）和 `-mtune`（对应 CPU 型号）。编译前应先确认目标板 CPU 型号，优先用 `-mcpu` 简化配置。

#### 2.3.4 常用优化组合

```bash
-O3 -march=armv8.2-a -mtune=cortex-a76 -ffast-math -fopenmp -fno-rtti -fno-exceptions
```

| 选项 | 作用 |
|------|------|
| `-O3` | 最高级别优化 |
| `-march=armv8.2-a` | 目标指令集，默认带 SIMD |
| `-mtune=cortex-a76` | 流水线调度优化 |
| `-ffast-math` | 浮点加速（图像/算法场景） |
| `-fopenmp` | OpenMP 并行 |
| `-fno-rtti -fno-exceptions` | 减小体积（嵌入式项目推荐） |

#### 2.3.5 验证 SIMD 是否开启

```bash
# 检查生成的机器码是否包含 SIMD 指令
aarch64-linux-gnu-objdump -d your_program | grep -E "vadd|vmul|ld1"
```

能搜到向量指令说明 SIMD 已正常开启。

## 3. 工具链辅助

### 3.1 调试与符号工具

```bash
objcopy     # 分离调试信息，可在需要时重新链接
objdump -t  # 查看符号表
objdump -d  # 查看汇编
objdump -p  # 查看可执行文件链接的库
readelf -d  # 查看动态段信息
strip       # 去除 unused 和 debug 符号
```

### 3.2 编译缓存

`ccache` 可以缓存中间产物 `.o`，加快重复编译效率。
