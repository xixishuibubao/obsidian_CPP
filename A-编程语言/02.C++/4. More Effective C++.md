# 1. 基础议题

Scott Meyers 风格 C++ 效能与正确性要点：指针/引用、cast、Rule 0/3/5、运算符、异常安全与性能。

## 【核心精简提纲】

### 记忆主干

**指针可空须检 · 引用必初始化不可重绑 · 四 cast 替代 C 风格 · 数组勿多态 · Rule 0/3/5 · `explicit` 转换 · 禁重载 `&&`/`,` · RAII+智能指针 · 析构禁抛异常 · 80/20 先 profiling**

### 核心要点清单

- 指针：可改指向、可为 `nullptr`，解引用前须检查；引用：创建即绑定、不可重绑，垂悬引用属逻辑错误
- 禁用 C 风格 `(T)e`；`static_cast` 基本/上行；`dynamic_cast` 下行须虚继承+RTTI；`const_cast` 去 const；`reinterpret_cast` 慎用
- 勿将基类数组当形参按元素切片；若必须，虚析构+虚函数
- Rule 0：不管理资源用编译器默认；Rule 3：拷贝构造/拷贝赋值/析构；Rule 5：加移动构造/移动赋值
- 显式定义拷贝或析构会抑制移动；显式移动会使拷贝 `= delete`；单移动操作不自动生成另一移动
- 转换运算符加 `explicit`；前缀 `++` 返回引用优于后缀构造临时对象
- 禁重载 `&&`、`|`、`，`（短路语义被破坏）；`new`/`delete` 与 `new[]`/`delete[]` 成对
- 构造异常：已构造成员不自动析构 → `unique_ptr`/`shared_ptr`；析构函数禁止抛异常
- 性能：80% 代码非热点；`+=` 优于 `+`；避免隐式转换临时对象；RVO 可消除返回临时

## 1.1 指针与引用

### 1.1.1 指针

- 指针最大的问题在于，他可能指向一个 `nullptr`，访问 `nullptr` 的行为是 `UB` 的，但好处在于你可以随时修改其指向的目标。
- 因此你在调用指针指向内容的时候总是需要检查其指针合法性（访问空指针是语言特性层面的错误）。

### 1.1.2 引用

- 引用要求你必须创建的同时初始化，且一旦创建了引用，便不再允许你修改引用指向为另一个目标。
- 由于创建必须初始化的特性，引用在使用时几乎不必检查其合法性，因为他总是存在指向的，尽管有时存在垂悬引用的问题，但那是另一个维度（逻辑错误而非语言特性错误）的问题。

## 1.2 避免 C 风格的转换

### 1.2.1 C 风格转换的问题

- 它过于的万能了，稍有不慎就会导致错误的转化，而后产生 `UB`。
- 另一方面，C 风格的转换格式特殊，不能很好的从视觉区分其目的。

### 1.2.2 C++ 的 cast

- `expression` 表达式，`target` 目标。
1. 静态转换：`static_cast<target_type>(expression)`，几乎适用于所有基础类型的转换和派生类对基类指针/引用的安全转换。
2. 动态转换：`dynamic_cast<target_type>(expression)`，可以由基类安全的向下转换为派生类指针/引用。
   - `dynamic_cast` 只能在存在虚函数的派生关系中使用，因为其依赖 `RTTI`，而不包含虚函数的派生关系的 `RTTI` 是缺失的。
   - 对指针转化失败则返回 `nullptr`，对引用转换失败则抛出异常。
3. 常量转换：`const_cast<target_type>(const target_type_expression)`，用来去除原表达式的 `const` 属性。
4. 内存转化：`reinterpret_cast<target_type>(expression)`，常被用来处理非关联关系的对象指针到指针的转换，比较危险，尽可能避免。

## 1.3 不要对数组使用多态

### 1.3.1 数组作为形参的隐患

1. 当你使用基类的数组作为形参传入 `function`，而其内部承载的确是派生类对象，可能会存在一些意想不到的麻烦（按基类大小步进切片）。
2. 应当避免使用基类数组作为形参，确切的说不要直接把他的一切元素当作基类对象使用，如果一定要这么做，虚析构和定义要操作的函数为虚函数是必须的。

## 1.4 类的设计

### 1.4.1 035 法则

1. `0` 法则：如果类不管理任何动态资源（仅包含普通成员），无需显式定义拷贝构造、赋值运算符、析构函数，直接使用编译器生成的缺省函数即可。
2. `3` 法则：如果类管理动态资源（如用 `new` 分配内存），必须显式定义以下 3 个函数。
   3. 拷贝构造函数 `MyClass(const MyClass& oth)`
   4. 赋值运算符重载 `MyClass& operator=(const MyClass& oth)`
   5. 析构函数（如果需要继承，则应当声明为虚函数）`virtual ~MyClass()`
6. `5` 法则：C++11 在 `3` 法则的基础上增加了移动拷贝和移动构造。
   7. 移动构造：`MyClass(MyClass&& oth)`
   8. 移动赋值：`MyClass& operator=(MyClass&& oth)`

### 1.4.2 编译器缺省实现和生成抑制

- 显示定义某些实现会抑制其他缺省实现，以上仅针对：构造函数、析构函数、拷贝构造、拷贝赋值、移动构造、移动赋值。
1. 只要用户显式声明了任意构造函数，默认构造函数不会被隐式生成。
2. 显式声明拷贝构造、拷贝赋值或析构函数，会阻止编译器隐式生成移动构造和移动赋值。
3. 显式声明移动构造或移动赋值，会导致拷贝构造和拷贝赋值被隐式声明为 deleted（即不可用）。
4. 显式声明一个移动操作（如移动构造），不会自动生成另一个移动操作（如移动赋值）。

## 2. 运算符

## 2.1 谨慎的使用类型转换函数

1. 对于实现了自定义的类型转化函数，应当使用 `explicit` 来修饰，避免意料之外的隐式转化。
   2. `explicit operator int() const`
   3. 对于类型转换操作符的重载不需要返回值，函数名本身就代表了返回值类型，同时推荐后置 `const` 来表明不修改源对象。
4. 或通过显式的方法，如 `int ToInt()` 方法显式转换。

## 2.2 自增、自减前后缀差异

### 2.2.1 前缀（`++obj`）形式

1. 前缀自增、自减更为推荐，其直接返回源对象，不必构造新的对象。
2. `MyClass& operator++()`，返回源值引用以便于链式调用

### 2.2.2 后缀（`obj++`）形式

1. 后缀自增、自减不推荐，其直接返回新的对象，有额外构造开销。
2. `MyClass operator++(int)`，返回一个新的对象，这里的 `int` 仅做占位，标识这是后置形式，无实际意义。

## 2.3 禁止重载 `&&` `|` `,` 操作符

1. 这些操作符有着明确的含义，重载他们会使得程序发生不那么容易察觉的意料之外的错误（短路求值语义丢失）。

## 2.4 `new` 与 `delete`

### 2.4.1 `new` 与 `delete` 应当总是成对出现

1. 常规 `new` 与常规 `delete` 用来从堆创建一个对象。
2. `new[cnt]` 与 `delete[]` 用来创建数组。
3. 定位 `new(buf)` 用来在现有内存创建对象，由于是已有内存，故不需要 `delete`，但是要手动调用析构 `obj->~ObjType();`

## 3. 异常与析构

## 3.1 memory leak

### 3.1.1 用析构来避免泄露

1. 将需要持有的资源以指针存入你的管理类，并在析构中去处理他们以实现 RAII，避免忘记手动释放带来的问题。
2. 但当发生异常的时候，未走完构造链的对象不会调用完整析构；局部 `shared_ptr` 可在栈展开时释放已接管资源。
3. 配合使用 `shared_ptr<T>` 来存储局部对象也是一种好的选择，会在生命周期结束自动释放资源。

### 3.1.2 构造中的异常处理

1. 当一个类包含多个资源，在构造的中途出现抛出异常的时候，原有资源是不会被自动释放的，整个类也不会调用自己的析构函数。
2. 我们使用 `try { 要运行的代码 } catch(...) { 异常捕获后处理；throw; }` 的形式来包裹动态申请的资源。
3. 但是对于使用了初始化列表的构造函数异常，我们无法通过使用 `try catch` 的组合来处理。因此我们需要引入智能指针 `shared_ptr/unique_ptr` 来让他们自动处理异常之前的动态资源。

### 3.1.3 禁止析构函数抛出异常

1. 析构抛出异常可能会出现连锁反应，在析构中抛出异常可能出现套娃问题导致程序最终崩溃（`std::terminate`）。

## 3.2 异常传递的细节

### 3.2.1 使用引用

1. `try { } catch (const std::exception& e) { throw; }`，使用引用来捕获抛出的异常。
   1. 避免使用拷贝捕获而出现的派生类异常不识别（对象切片）
   2. 避免抛出指针而出现的悬挂问题或忘记释放带来的泄露
2. 对于嵌套的异常，如果外层无法承载内部抛出的异常就会进入 `unexpected` 状态而导致调用函数终止 `terminate` 进而导致整个程序终止 `abort`。
3. 异常是有开销的，如无必要，避免使用大量异常。

## 4. 效率与性能

## 4.1 82 法则

### 4.1.1 82 法则

1. 大约 80% 的代码对效率和性能几乎没有影响，影响程序效率的关键在于那 20%
2. 提高性能的关键在于找到那 20%，仅凭直觉是无法实现的，应当学会使用性能分析工具。

## 4.2 惰性加载

### 4.2.1 惰性加载

1. 对于不常用或较大数据，我们可以只在其被使用的时候进行加载，以提高性能。
   1. 惰性单例，只在首次使用被初始化。
   2. 惰性计数，拷贝时是共享数据，只在首次修改时才会独立出自己的内存占用（写时拷贝）。
   3. 惰性加载，大的数据集构建时不初始化，只在访问和修改时初始化局部内存，以减小整体开销。

## 4.3 预处理与 cache

### 4.3.1 预处理

1. 对于需要频繁操作且单次求取有规律可循的数据，可以在每次改动数据集时顺便计算，使用时即可用已经预处理好的缓存值。

## 4.4 临时对象的开销

### 4.4.1 临时对象

1. 函数形参隐式转换带来的临时对象开销是性能损耗的重要组成，因此我们应当尽可能避免会发生临时对象构造和拷贝的函数设计。

### 4.4.2 RVO 技术

1. RVO 返回值优化，当返回值是一个构造的时候，编译器会优化掉临时对象构建→拷贝→返回的过程，直接在返回处构建一个目标对象（NRVO 同类）。

## 4.5 重载与隐式转换

### 4.5.1 隐式转换

1. 隐式转换通常会发生在单参数构造函数中，当你使得非显式声明 `explicit` 的类发生隐式转换就会构造临时对象。
2. 通过重载对应的 `operator` 函数的形参，可以很好的解决隐式转换问题，但大多时候是非必要的，且可能因为疏忽重载错误的函数带来一些麻烦。

## 4.6 `+` 与 `+=`

### 4.6.1 `+` 与 `+=`

1. `+=` 的效率比 `+` 更高，因为 `+` 要生成额外的临时对象。
2. 对于 `-` 和 `-=` 同理。
