## 1. goroutine vs pthread

### 1.1 调度模型

- **goroutine**：Go 运行时的 M:N 调度（M 个 goroutine 映射到 N 个 OS 线程），由 Go scheduler 管理，用户态协作 + 抢占。
- **pthread**：1:1 模型，每个线程对应一个内核线程，由 OS 调度器管理，切换需内核态上下文保存/恢复。

### 1.2 栈管理

- goroutine 初始栈约 2KB，可按需动态增长/收缩，支持创建百万级 goroutine。
- pthread 栈固定且较大（默认 ~8MB），创建开销高，通常只能开数千个。

### 1.3 创建与切换成本

- goroutine 创建仅分配用户态数据结构，切换在用户态完成，开销约几十 ns。
- pthread 创建涉及 `clone` 系统调用，切换涉及内核态上下文，开销 µs 级。

```go
go myfunc()          // goroutine 启动
go func() { ... }()  // 匿名函数启动
```

## 2. channel vs pipe

### 2.1 通信语义

- **channel**：Go 的 CSP 通信原语，goroutine-safe，有缓冲/无缓冲两种模式。
- **pipe**：Unix 进程间通信，内核环形缓冲区 + 两个文件描述符（读端/写端），需 read/write 系统调用。

### 2.2 同步机制

- 无缓冲 channel 收发成对阻塞，天然实现 goroutine 间同步。
- 有缓冲 channel 在缓冲区满/空时阻塞，解耦生产消费速率。
- pipe 阻塞与否取决于 O_NONBLOCK 标志，无内置同步语义。

```go
ch := make(chan int)       // 无缓冲
ch := make(chan int, 10)   // 有缓冲
ch <- v                    // 发送
v := <-ch                  // 接收
```

## 3. interface vs 虚继承

### 3.1 多态实现方式

- **interface**：Go 的结构化类型系统，类型隐式满足接口（duck typing），无需显式声明继承。
- **虚继承**：C++ 的显式继承体系，需在类定义中声明 `virtual`，编译器构建虚表（vtable）。

### 3.2 运行时开销

- Go interface 值包含 itab（接口类型表）+ 数据指针，调用需 itab 查找。
- C++ 虚函数通过 vptr 间接寻址 vtable，调用开销为一次间接跳转。

```go
type Stringer interface {
    String() string
}
// 任何实现了 String() 的类型自动满足 Stringer
```

## 4. defer vs RAII

### 4.1 执行机制

- **defer**：将函数调用注册到 goroutine 的延迟栈，在所在函数返回前按 LIFO 顺序执行。
- **RAII**：C++ 的惯用法，资源获取在构造函数中完成，释放在析构函数中，作用域退出自动触发。

### 4.2 适用场景对比

| 维度 | defer | RAII |
|------|-------|------|
| 绑定范围 | 函数级别 | 作用域级别 |
| 执行时机 | return 前 LIFO | 离开作用域 |
| 典型用途 | 解锁、关闭文件、恢复 panic | 智能指针、锁守卫、文件句柄 |

```go
mu.Lock()
defer mu.Unlock()  // 函数返回时自动解锁

f, _ := os.Open("file")
defer f.Close()    // 函数返回时自动关闭
```

## 5. Go 选型建议

### 5.1 适合 Go 的场景

- 高并发网络服务（HTTP/gRPC 服务器、API 网关）
- 云原生工具与 CLI（Docker、Kubernetes、Prometheus 等）
- 快速原型到生产部署（编译快、单一二进制、交叉编译方便）
- 团队以开发效率优先，统一工具链

### 5.2 适合 C++ 的场景

- 极致性能优化（游戏引擎、高频交易、实时渲染）
- 硬实时系统（控制系统、车载嵌入式）
- 底层硬件操作（驱动、固件、内核模块）
- 对二进制体积和内存占用有严格限制
