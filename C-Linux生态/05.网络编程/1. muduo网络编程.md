# 1. muduo 网络编程

Linux 五种 IO 模型辨析、muduo Reactor 架构剖析，以及基于 epoll 的 Socket 编程关键代码模式。

## 【核心精简提纲】

### 记忆主干

**IO 模型**：阻塞 → 非阻塞 → IO 复用（epoll）→ 信号驱动 → 异步（io_uring）。**muduo** = Reactor + one loop per thread + epoll。**Socket 编程**：`socket` → `bind` → `listen` → `epoll_wait` → `accept` → `recv`/`send`。

### 核心要点清单

- `recv` 返回 0：对端正常关闭；`<0` 且 `EAGAIN`：非阻塞暂无数据
- epoll：红黑树存 fd + 就绪链表回调，只返回活跃 fd
- Nginx：多进程 + epoll；muduo：多线程每线程一个 EventLoop
- select 受 `FD_SETSIZE` 限制（1024）；epoll 无此限制
- muduo 模块：`Logger` / `Timestamp` / `InetAddress` / `Buffer` / `Channel` / `EventLoop` / `Poller`

### 五种 IO 对比

| 模型 | 等数据 | 拷数据 | 典型 API |
|------|--------|--------|----------|
| 阻塞 | 阻塞 | 阻塞 | `recv` flags=0 |
| 非阻塞 | 立即返回 | 阻塞 | `fcntl O_NONBLOCK` |
| IO 复用 | epoll_wait | 阻塞 | epoll |
| 信号驱动 | 信号通知 | 同步处理 | 少用 |
| 异步 | 内核完成 | 内核完成 | io_uring（5.1+） |

## 【完整拓展详情】

## 1. IO 模型对比

### 1.1 阻塞与非阻塞

默认 `recv(..., flags=0)` **阻塞**直到有数据。

```plain
ssize_t recv(int sockfd, void* buf, size_t len, int flags);
```

返回值：

| 返回值 | 含义 |
|--------|------|
| `> 0` | 实际字节数 |
| `== 0` | 对端正常关闭 |
| `< 0` | 错误；`errno==EAGAIN/EWOULDBLOCK` 表示非阻塞下暂无数据 |

### 1.2 五种 IO 模型

`recv` 是 `read` 的 socket 特化，本质都是内核态 IO。

#### 1.2.1 阻塞 blocking

- **等数据**：未就绪则线程挂起
- **读写**：就绪后从内核缓冲区拷贝到用户空间（仍可能阻塞）

#### 1.2.2 非阻塞 non-blocking

- **等数据**：未就绪立即返回 `EAGAIN`，需应用层轮询
- **读写**：就绪后拷贝；轮询空转会浪费 CPU

#### 1.2.3 IO 复用

`select` / `poll` / `epoll` 批量监听 fd 就绪事件；读写本身仍可在阻塞或非阻塞模式下进行。

- 核心：**事件通知**——只处理就绪 fd，避免无效 `recv`
- epoll 内核：fd 注册进红黑树 → 就绪回调链入就绪链表 → `epoll_wait` 取活跃 fd

#### 1.2.4 信号驱动

注册 signal handler，数据就绪时内核发信号；handler 内同步读数据。存在信号竞争与多线程安全难题，**服务器后端极少采用**。

#### 1.2.5 异步 async

- **Linux aio**（2.6–4.x）：`libaio`，主要 O_DIRECT 文件 IO，不支持 socket
- **io_uring**（5.1+，Ubuntu 20.04+）：统一 fd，真·内核异步完成
- 区别于业务层「多线程/async 库」——此处指 **内核级** 异步

### 1.3 踩坑

| 易混点 | 澄清 |
|--------|------|
| epoll ET 漏读 | 边缘触发须一次读尽或循环读到 EAGAIN |
| 「异步库」vs io_uring | Node/libuv 多为线程池 + IO 复用，不是内核 aio |
| 非阻塞 + 单线程 | 仍需 epoll 驱动，裸 while 轮询会烧 CPU |

### 1.4 网络服务器设计要素

#### 1.4.1 线程 vs 进程

- CPU 调度单位是线程；IO 密集型应多线程提高阻塞时 CPU 利用率
- **多线程**：Web API、DB 访问，逻辑简单，共享内存通信方便
- **多进程（Nginx）**：worker 隔离，单 worker 崩溃不影响其他；配合 epoll/kqueue

#### 1.4.2 one loop per thread

每个 IO 线程（或进程/协程）运行一个 **EventLoop**，处理注册在其上的 Channel 事件——muduo、Netty 等主流 Reactor 方案。

#### 1.4.3 epoll 数据结构

1. 注册 fd → 红黑树
2. 就绪事件 → 就绪链表
3. 用户态 `epoll_wait` 批量取回

## 2. muduo 架构

### 2.1 模块总览

muduo 采用分层极简设计，核心模块职责如下：

| 类 | 职责 |
|----|------|
| `Logger` | 分级日志（DEBUG/INFO/ERROR/FATAL） |
| `Timestamp` | 时间戳，`now()` / `toString()` |
| `InetAddress` | 封装 `sockaddr_in` / `sockaddr_in6`，隐藏 IPv4/IPv6 差异 |
| `Buffer` | 预分配 + 读/写指针，减少 `recv` 系统调用次数；支持 `append` / `retrieve` |
| `Channel` | 封装 fd + 关注事件 + 回调；不拥有 fd |
| `EventLoop` | 循环 `poll` → 分发 `Channel` 回调；**one loop per thread** |
| `Poller` | 抽象 epoll（策略模式，可换 poll） |

### 2.2 设计哲学

- **Reactor 模式**：事件循环等待 IO 事件，分发到对应回调处理
- **极简主义**："够用即可"，对比 ACE 框架的过度抽象
- **单线程写原则**：一个 EventLoop 只属于一个线程，避免锁竞争

### 2.3 架构核心

- **分层设计**：`muduo::net`（网络层）、`muduo::base`（基础库）
- **性能优化**：无锁队列（`BlockingQueue`）、内存池
- **学习入口**：《Linux 多线程服务器编程：使用 Muduo C++ 网络库》（陈硕）
- **源码**：[chenshuo/muduo](https://github.com/chenshuo/muduo)

## 3. 关键代码模式

### 3.1 Socket 通用流程（阻塞单连接）

```cpp
int listen_fd = socket(AF_INET, SOCK_STREAM, 0);
int opt = 1;
setsockopt(listen_fd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));
struct sockaddr_in serv_addr{};
serv_addr.sin_family = AF_INET;
serv_addr.sin_addr.s_addr = INADDR_ANY;
serv_addr.sin_port = htons(PORT);
bind(listen_fd, (struct sockaddr*)&serv_addr, sizeof(serv_addr));
listen(listen_fd, SOMAXCONN);

struct sockaddr_in cli_addr{};
socklen_t cli_len = sizeof(cli_addr);
int conn_fd = accept(listen_fd, (struct sockaddr*)&cli_addr, &cli_len);

char buf[1024];
ssize_t n = recv(conn_fd, buf, sizeof(buf) - 1, 0);
if (n <= 0) { close(conn_fd); }
```

> 开发调试可开 `SO_REUSEADDR` 避免 `TIME_WAIT` 占端口；生产高并发还需 `SO_REUSEPORT`、非阻塞、`TCP_NODELAY` 等。

### 3.2 epoll 服务器（Linux 推荐）

```cpp
int epoll_fd = epoll_create1(0);
struct epoll_event ev{};
ev.events = EPOLLIN;
ev.data.fd = listen_fd;
epoll_ctl(epoll_fd, EPOLL_CTL_ADD, listen_fd, &ev);

struct epoll_event events[MAX_EVENTS];
int nfds = epoll_wait(epoll_fd, events, MAX_EVENTS, -1);
for (int i = 0; i < nfds; ++i) {
    if (events[i].data.fd == listen_fd) { /* accept + EPOLL_CTL_ADD */ }
    else { /* recv; n<=0 → close + EPOLL_CTL_DEL */ }
}
```

### 3.3 踩坑汇总

| 问题 | 说明 |
|------|------|
| 单线程阻塞 `recv` | 一个慢客户端拖死全体；改非阻塞 + epoll ET/LT |
| 忘记 `EPOLL_CTL_DEL` | fd close 后 epoll 仍可能报告事件（取决于 LT/ET） |
| `listen backlog` 过小 | 高并发 SYN 队列溢出，适当调大 `listen` 第二参数 |
| epoll ET 漏读 | 边缘触发须一次读尽或循环读到 EAGAIN |
| 非阻塞 + 单线程 | 仍需 epoll 驱动，裸 while 轮询会烧 CPU |

### 3.4 关键流程

```plain
listen_fd ──epoll_ctl ADD──► epoll_fd
                │
         epoll_wait 就绪
                ├─ listen_fd 可读 → accept → EPOLL_CTL_ADD conn_fd
                └─ conn_fd 可读 → recv；n<=0 → close + EPOLL_CTL_DEL
```
