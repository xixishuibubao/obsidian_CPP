# 1. mjpg-streamer 视频服务器

**mjpg-streamer** 是经典 **UVC/V4L2 → MJPEG/JPEG → HTTP/UDP** 流媒体方案，插件化架构，适合理解 Linux 摄像头推流与输出插件机制。可与 [[C-Linux生态/02.Linux系统编程/3. 用户态与内核态|用户态 ioctl/V4L2]] 对照阅读。

## 【核心精简提纲】

### 记忆主干

**input 插件采帧 → 核心框架传 `video_buffer` → output 插件分发**；插件为独立 `.so`，运行时 `dlopen`；依赖 V4L2 `mmap` 队列与 MJPEG 或软编码。

### 核心要点清单

- 启动：`-i "input_uvc.so -d /dev/video0 …" -o "output_http.so -p 8080 …"`
- input：UVC/V4L2；output：HTTP multipart、UDP、写文件
- 格式：设备直出 MJPEG 可零编码转发；YUYV 等需 jpeg/ffmpeg 插件
- 与自研 MPP 路径对比：问题域相同（预览推流），实现不同
- 学习入口：`plugins/input_uvc`、`plugins/output_http`

### 关键流程

```plain
/dev/video* → VIDIOC_REQBUFS/mmap → VIDIOC_STREAMON
    → input 插件读帧 → 核心回调链 → output_http（multipart/x-mixed-replace）
```

## 【完整拓展详情】

## 1.1 架构概览

```plain
┌─────────────┐    ┌──────────────┐    ┌─────────────────┐
│ input_uvc   │───►│ 核心框架      │───►│ output_http     │──► 浏览器 MJPEG
│ (V4L2 采集)  │    │ (插件链)      │    │ output_udp      │
└─────────────┘    └──────────────┘    │ output_file       │
                                        └─────────────────┘
```

| 组件 | 作用 |
|------|------|
| **input 插件** | 打开 `/dev/video*`，配置格式/分辨率，循环 `DQBUF` 取帧 |
| **核心** | 解析命令行、加载 `.so`、维护插件链表、分发帧缓冲 |
| **output 插件** | 按协议输出：HTTP MJPEG、RTP/UDP、落盘 |

### 前置依赖

- Linux **V4L2** 用户态 API：`open` → `ioctl(VIDIOC_*)` → `mmap` 或 `read`
- 动态库加载：`dlopen` / `dlsym`，见 [[C-Linux生态/01.Linux环境/2. 动态库与静态库|动态库]]
- 浏览器 MJPEG：HTTP `multipart/x-mixed-replace` 连续 JPEG 边界

## 1.2 与 V4L2 的关系

- 内核 **Video4Linux2**：`VIDIOC_S_FMT` 设格式、`VIDIOC_REQBUFS` 申请缓冲、`VIDIOC_STREAMON` 开流
- **mmap 模式**：减少拷贝，input 插件从映射地址读帧
- 常见像素格式：
  - `MJPEG` / `JPEG`：压缩流，output 可直接封装
  - `YUYV` / `NV12`：原始 YUV，需 **output_jpeg.so** 或 ffmpeg 插件编码

### 与自研硬编路径的差异

| 维度 | mjpg-streamer | 自研 MPP/ISP 栈 |
|------|---------------|-----------------|
| 采集 | 通用 UVC/V4L2 | 定制传感器 + 驱动 |
| 编码 | MJPEG 直传或软编 | Rockchip **MPP** 硬编 |
| 输出 | 插件可选 HTTP/UDP | 自有服务协议 |

问题域一致：**低延迟预览、多客户端拉流**；实现栈不同，可互证设计取舍。

## 1.3 典型启动与参数

```bash
# HTTP 8080 输出 MJPEG，静态页 /www
mjpg_streamer -i "input_uvc.so -d /dev/video0 -f 30 -r 1280x720" \
              -o "output_http.so -p 8080 -w /www"

# 多路 output（同一 input 可挂多个 -o）
mjpg_streamer -i "input_uvc.so -d /dev/video0" \
              -o "output_http.so -p 8080" \
              -o "output_file.so -f /tmp/snap.jpg"
```

| 参数 | 含义 |
|------|------|
| `-i` | input 插件路径及插件私有参数（字符串传入） |
| `-o` | output 插件路径及参数 |
| `-d` | 设备节点（input_uvc） |
| `-f` | 帧率（尽力而为，依赖设备） |
| `-r` | 分辨率 `WxH` |
| `-p` | HTTP 端口（output_http） |

### 踩坑点

- **设备不支持请求分辨率/格式** → `VIDIOC_S_FMT` 失败或静默降级，用 `v4l2-ctl --list-formats-ext` 先查
- **权限**：用户需在 `video` 组，或 root 运行
- **USB 带宽**：高分辨率 MJPEG 多路 output 可能丢帧
- **仅一个客户端占满 USB isoc** 时，检查是否另一进程已 `STREAMON`

## 1.4 插件机制（与 dlopen 同构）

- 各插件编译为独立 `.so`，主程序 `dlopen` 并解析统一 C 接口（如 `init` / `run` / `stop` / `cmd`）
- 与 [[D-系统与架构/02.软件架构设计/5. 插件架构设计|插件架构设计]]、[[3. C-C++与Python互调用|pybind11 互调用]] 同构：**稳定 ABI 边界 + 字符串配置**

```plain
主程序 --dlopen--> input_uvc.so
         --函数指针--> input_init / input_run
帧缓冲 --回调--> 各 output 插件
```

### 典型错误

- 插件 `.so` 与主程序 **架构不一致**（x86 vs arm64）→ `dlopen: wrong ELF class`
- 缺少插件依赖（如 `libjpeg`）→ 加载成功但编码段崩溃

## 1.5 HTTP MJPEG 输出原理（简述）

- `output_http` 监听 TCP，对 `GET /` 返回 `Content-Type: multipart/x-mixed-replace; boundary=...`
- 每帧 JPEG 作为 multipart 一段写入，浏览器 `<img src="...">` 可持续刷新
- 延迟低于完整 H.264 管线，但带宽高于 H.264；适合监控预览
