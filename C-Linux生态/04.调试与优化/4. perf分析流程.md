# 1. 编译与准备

## 【核心精简提纲】

### 记忆主干

perf 基于硬件采样，attach 运行中进程无侵入；`record -g` 录调用栈，`report --source` 到源码行，再配合 FlameGraph 可视化热点。

### 核心要点清单

- 编译：`g++ -g -O2`，勿 strip；可用 `objcopy` 分离 debug 符号部署
- 概览：`perf stat` 看 cycles / cache-misses / branch-misses
- 采样：`perf record -g -p PID --sleep N`；`-u` 只要用户态
- 报告：`perf report -g`；`--source` 显示源码行占比
- 实时：`perf top -p PID` 快速看热点函数
- 火焰图：`perf script | stackcollapse-perf.pl | flamegraph.pl`
- 排错：报告出 `????` → 缺 `-g` 或被 strip

- [[1. gdb|GDB 断点验证热点]]、[[7. 综合调试实战|perf→GDB→coredump]]
- [[8. 分支优化|branch-misses 优化]]、[[10. CPU 高占用排查|CPU 高占用排查流程]]

`perf stat` 定方向 → `record -g` → `report --source` → 火焰图 → gdb 验证

## 【完整拓展详情】

## 1.1 编译选项

热点分析需要函数名、文件名等调试信息，编译时需添加 `-g`：

```bash
g++ -g -O2 -o myprogram myprogram.cpp
# -g：保留调试符号
# -O2：保留优化，同时避免函数过度内联导致无法识别
```

`-g3` 会携带更多调试信息，但带调试信息的版本不适合部署到现场。可通过 `objcopy` 分离调试信息，需要时再链接回去（见下文 5.1 `objcopy`）。

**不要 strip**，否则函数名会丢失，perf report 中显示为 `????`。

## 1.2 安装 perf

大部分 Linux 发行版预装 `perf`，若未安装：

```bash
# Ubuntu/Debian
sudo apt install linux-tools-common linux-tools-$(uname -r)
# CentOS/RHEL
sudo yum install perf
```

# 2. `perf` 采样分析

## 2.1 `stat` — 性能概览

先通过 `stat` 获取程序整体耗时分布，定位瓶颈方向：

```bash
sudo perf stat ./myexe                                                # 程序整体耗时
sudo perf stat -e cache-misses,cache-references ./myexe               # 缓存命中情况
sudo perf stat -e cache-misses,cache-references -p <PID> --sleep 10   # 对运行中的进程采样
sudo perf stat -e branch-misses -p <PID> --sleep 10                   # 分支预测失败
```

## 2.2 `record` — 采样记录

对目标程序进行 CPU 采样，记录函数调用的热点信息：

```bash
sudo perf record -g -T ./myfile              # 直接启动程序并采样
sudo perf record -g -p <PID>                 # 对已运行的程序采样，Ctrl+C 结束
sudo perf record -g -p <PID> --sleep 10      # 采样 10 秒自动停止（推荐）
sudo perf record -g -p <PID> -u --sleep 10   # 只采样用户态代码，排除内核干扰
```

参数说明：
- `-g`：记录函数调用栈（**必须加**，否则无法关联父子函数关系）
- `-T`：记录时间信息
- `-p <PID>`：对已有进程采样，无需重启
- `--sleep N`：采样 N 秒后自动停止（无需手动 Ctrl+C）
- `-u`：只采样用户态代码（排除内核干扰，C++ 程序常用）
- `-F <频率>`：指定采样频率（默认 4000 次/秒，如 `-F 1000` 降低开销）
- `-o <文件>`：保存到指定文件（默认 `perf.data`）

> `perf` 可直接 attach 运行中的进程，**无需重启、无需修改代码、无侵入**。

## 2.3 `report` — 报告分析

```bash
sudo perf report -g graph,<最低阈值> -T       # 标准报告模式
sudo perf report --source                     # 显示源码行号（编译需加 -g）
```

报告按函数 CPU 占用率降序排列：

| 字段 | 含义 |
|------|------|
| `Overhead` | 函数占用的 CPU 百分比（越高越"热"） |
| `Command` | 程序名 |
| `Shared Object` | 函数所在二进制文件（如 `./myprogram` 或系统库） |
| `Symbol` | 函数名（C++ 显示完整签名，如 `MyNamespace::my_function(int)`） |

交互操作：
- `Enter` 进入函数，查看调用链（谁调用了谁）
- `Tab` 切换排序方式
- `s` 切换 C++ 命名空间、模板的显示/折叠
- `↑ ↓` 选择函数
- `q` 退出

### `perf report --source` 查看源码行

若编译时加了 `-g`，`perf report --source` 会直接显示耗时函数的**源码片段**，并标注每一行的耗时百分比。这是定位 C++ 瓶颈**最精准的方法**。

## 2.4 `perf top` — 实时热点

类似 `top`，实时显示函数级 CPU 占用，无需等待采样完成：

```bash
perf top -p <PID>        # 实时监控指定进程的热点函数
```

- 实时显示当前最耗 CPU 的函数（内核态 + 用户态）
- `↑ ↓` 浏览，`q` 退出
- 适合快速定位异常进程中的热点

# 3. 火焰图 `FlameGraph`

火焰图是可视化热点函数的最佳方式，直观展示调用栈和各函数的 CPU 占比。

## 3.1 下载

```bash
git clone --depth=1 --branch master https://github.com/brendangregg/FlameGraph.git
```

对于工具类仓库使用浅克隆快速拉取。

## 3.2 使用步骤

```bash
# 1. 将 perf.data 转为可读中间格式
sudo perf script > perfdata1

# 2. 折叠调用栈
sudo ~/dev/FlameGraph/stackcollapse-perf.pl perfdata1 > perfdata2

# 3. 生成 SVG 火焰图
sudo ~/dev/FlameGraph/flamegraph.pl perfdata2 > flame.svg
```

或通过管道一步完成：

```bash
sudo perf script | ~/FlameGraph/stackcollapse-perf.pl | ~/FlameGraph/flamegraph.pl > flame.svg
```

### 3.3 查看火焰图

用浏览器打开 `flame.svg`：
- **Y 轴**：函数调用栈（上层调用者，下层被调用者）
- **X 轴**：函数执行时间占比（越宽 CPU 占用越高）
- 点击函数放大查看细节，定位最深层的热点函数

# 4. `perf sched` — 线程调度分析

多线程程序中，查看每个线程的运行时间、等待时间、切换次数：

```bash
perf sched record -p <PID> --sleep 10
perf sched latency
```

- 显示每个线程的运行时间、等待时间、切换次数
- 定位线程卡住、等待、调度延迟的原因

# 5. 辅助工具

## 5.1 `objcopy` — 符号表分离

带调试信息的程序不适合直接部署到现场，可通过 `objcopy` 分离/恢复调试信息：

```bash
objcopy --only-keep-debug myexe myexe.debug     # 保存调试信息表
objcopy --strip-all myexe                        # 从源程序剔除符号表
objcopy --add-gnu-debuglink=myexe.debug myexe    # 建立调试符号与源程序的链接
```

链接回去之后 `perf` 可正常观察到函数信息，否则 `report` 可能出现异常或函数名不显示。

## 5.2 `nm` — 查看符号表

```bash
nm -C -S -l ./myexe | grep "funcname"
```

- `-C`：反混淆 C++ 函数名（关键）
- `-S`：显示符号地址
- `-l`：尝试显示行号（可选）

可与 `report` 联合查找热点函数所在的源文件位置。

## 5.3 `addr2line` — 地址转源码位置

当只有崩溃/采样地址（如 `0x00007F123456789A`）时，反查函数名、文件名、行号：

```bash
addr2line -f -C -e ./myexe 0x地址
```

- `-f`：显示函数名
- `-C`：反混淆 C++ 函数名，显示原名

也支持动态库内地址：

```bash
addr2line -e /path/to/libxxx.so -f 0x地址
```

### GDB 反查地址

```gdb
gdb ./myapp
info line *0x地址
```

直接输出 `Line 88 of "test.cpp"`。

### `????` 排查

perf report 中看到 `????`，99% 是以下原因：

| 原因 | 解决 |
|------|------|
| 编译未加 `-g` | 重新编译加 `-g` |
| 程序被 strip 过 | 保留符号表，或重新编译 |
| 地址属于内核/无符号的库 | 使用 `-u` 只采样用户态 |

## 5.4 `ldd` — 查看依赖库

```bash
ldd myexe      # 查看程序依赖的库
ldd myso.so    # 查看动态库的依赖库
```

# 6. C++ 注意事项

## 6.1 函数内联

优化编译（如 `-O2`）可能导致函数被内联，`perf` 可能无法识别原函数名：

- 临时用 `-O1` 编译可减少内联（性能仍接近实际）
- 火焰图中内联函数会合并到调用者，需结合代码分析

## 6.2 命名空间与模板

C++ 的命名空间（如 `Namespace::Func`）和模板函数（如 `std::vector<int>::push_back`）在 `perf` 报告中显示完整符号（可能较长），需结合代码对应。

## 6.3 排除系统库干扰

若热点集中在系统库（如 `libc.so`）：

- 使用 `-u` 只采样用户态代码
- 在 `perf report` 中按 `s` 键折叠 STL
- 火焰图中忽略系统库函数

## 6.4 常见问题排查

| 问题 | 原因 | 解决 |
|------|------|------|
| 看到一堆 `????` 无函数名 | 编译未加 `-g`，或被 strip | 重新编译加 `-g` |
| 全是 STL 函数，看不到自己代码 | 库函数占比高 | 按 `s` 键折叠，或使用 `-u` 只看用户态 |
| 看不到调用链 | `perf record` 没加 `-g` | 加上 `-g` 重新采样 |
| 想精确看到哪一行代码耗时 | 需要源码级分析 | 用 `perf report --source` |

# 7. 完整示例流程

假设分析 `myprogram` 的热点函数：

```bash
# 1. 编译程序（带调试符号）
g++ -g -O2 -o myprogram myprogram.cpp

# 2. 实时查看热点（快速定位）
perf top -p $(pidof myprogram)

# 3. 采样 10 秒生成报告
sudo perf record -g -p $(pidof myprogram) --sleep 10

# 4. 查看文本报告（含源码行）
sudo perf report --source

# 5. 生成火焰图（需 FlameGraph 工具）
sudo perf script | ~/FlameGraph/stackcollapse-perf.pl | ~/FlameGraph/flamegraph.pl > flame.svg

# 6. 浏览器打开火焰图
firefox flame.svg
```
