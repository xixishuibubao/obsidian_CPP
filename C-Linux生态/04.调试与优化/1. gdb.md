# 1. gdb 调试

- 使用 gdb 的先决条件是在编译时使用 `-g` 添加调试信息。
- 一般指令都可以简化为首字母直接使用。

## 1.1 编译和设置入参

- `gdb filename`，使用 gdb 调试可执行程序
- `set args 参数内容`，设置 main 的入口参数，多个参数以空格分隔

## 1.2 查看源码

- `list(l) [文件名:][行号|函数名]`
  - `list 数字`，查看当前行附近源码
  - `list 函数名`，查看函数附近源码
  - `list 文件名:行数/函数名`，查看某个文件的行数或函数附近源码

## 1.3 断点调试

- `break (b) [文件名:][行号|函数名]`
- `info break(i b)`，查看断点列表
- `delete (d) [n]`，删除所有断点或 n 号断点

## 1.4 启动调试

- `run (r)` 启动函数运行

## 1.5 继续运行

- `continue (c)`，继续运行（在断点后继续）

## 1.6 忽略断点

- `ignore N count`，N 是断点序号，count 是忽略次数

## 1.7 单步调试

- `step (s)`，单步运行
- `si`，单步运行机器指令

## 1.8 跳出函数

- `finish`，跳出当前函数

## 1.9 逐行调试

- `next (n)`，调用函数的位置不进入，直接视为语句运行

## 1.10 打印信息

使用 `print` 必须每次手动查看 val 的值：

- `print(p) EXP`，EXP 是表达式
- `print val`，查看变量的值
- `print val=num`，给变量赋值

## 1.11 自动显示

`display` 会自动输出之前 `display` 的全部内容：

- `display EXP`，将 EXP/VAL 添加到列表
- `info display`，查看 display 参数列表
- `undisplay [num]`，删除 display 列表中的某个参数或全部

## 1.12 查看内存

- `x/16x $esp`，esp 是一个 CPU 寄存器，查看其内容
- `x/nFU addr`
  - `x` 查看内存命令
  - `n` 数据类型长度
  - `F` 输出格式（默认 `hex` 16进制）
    - `o` octal 8进制
    - `x` hex 16进制
    - `d` decimal 10进制
    - `u` unsigned decimal 无符号10进制
    - `t` binary 2进制
    - `f` float 浮点数
    - `c` char 单字符
  - `U` 内存单元数据类型（默认 `w` 4bytes）
    - `b` byte 单字节 1byte
    - `h` halfword 半字 2bytes
    - `w` word 字 4bytes
    - `g` giant 双字 8bytes

## 1.13 退出

- `quit (q)`，退出 gdb 调试模式

## 1.14 查看反汇编

- `disass 函数名`，查看反汇编

## 1.15 只调试当前线程

- `set scheduler-locking on/off`

## 1.16 coredump 崩溃分析

coredump（核心转储）是程序异常退出时保存的内存快照（栈、寄存器、堆数据），配合 gdb 可还原崩溃现场。

相关笔记：[[2. coredump|coredump 配置速查]]

### 1.16.1 开启 coredump

```bash
# 查看当前限制
ulimit -a | grep core

# 取消 core 文件大小限制（临时，当前 shell 生效）
ulimit -c unlimited
```

**永久配置 core 文件路径与命名：**

```bash
# 编辑 sysctl 配置
sudo vim /etc/sysctl.conf

# 添加一行（% 为占位符，参见 1.16.4 表格）
kernel.core_pattern = %e_core_%s_%t

# 使配置生效
sudo sysctl -p
```

> 若未设置 `core_pattern`，coredump 文件默认生成在程序当前工作目录，文件名为 `core`。

### 1.16.2 gdb 加载 coredump

程序崩溃生成 coredump 文件后，用 gdb 加载查看崩溃位置：

```bash
gdb 可执行文件名 coredump文件名
```

示例：
```bash
gdb ./myapp core.myapp_11_12345678
```

### 1.16.3 查看崩溃现场

进入 gdb 后，以下命令查看程序崩溃时的状态：

```gdb
bt               # 查看调用栈（backtrace）
frame N          # 切换到 n 号栈帧
info args        # 查看当前帧的参数值
info locals      # 查看当前帧的局部变量
info registers   # 查看寄存器状态
```

`bt` 输出示例：
```
#0  0x0000ffff in crash_func (p=0x0) at main.cpp:15
#1  0x0000aaaa in main () at main.cpp:22
```

### 1.16.4 `core_pattern` 占位符参考

| 占位符 | 含义 | 示例 |
|--------|------|------|
| `%e` | 可执行文件名 | `myapp` |
| `%p` | 进程 PID | `12345` |
| `%s` | 导致终止的信号编号 | `11` (SIGSEGV) |
| `%t` | 转储时间（Unix 时间戳） | `1718000000` |
| `%h` | 主机名 | `ubuntu-arm64` |
| `%%` | 输出一个 `%` | `%` |

## 1.17 多线程调试

### 1.17.1 查看线程信息

```gdb
info threads        # 列出所有线程（简称 i th）
```

输出示例：
```gdb
  Id   Target Id                      Frame
* 1    Thread 0x7ffff7fd8740 (LWP 12345) "myapp" main () at main.cpp:10
  2    Thread 0x7ffff6c8e700 (LWP 12346) "myapp" worker_func (arg=0x...) at worker.cpp:25
  3    Thread 0x7ffff648d700 (LWP 12347) "myapp" worker_func (arg=0x...) at worker.cpp:30
```

- `Id` — GDB 内部线程号（用于 `thread N` 切换）
- `*` 号标记当前活跃线程
- 最后一列为线程当前所在的函数与位置

### 1.17.2 切换线程

```gdb
thread 2            # 切换到线程 2
thread apply 1 2 bt # 查看线程 1 和 2 的调用栈
thread apply all bt # 查看所有线程的调用栈（排查死锁最常用！）
```

`thread apply all bt` 是**多线程调试最高频命令**——一次输出所有线程的调用栈，快速定位死锁（哪个线程在等锁、锁在谁手上）。

### 1.17.3 线程断点

```gdb
# 所有线程都命中
b worker.cpp:25

# 仅特定线程命中（thread 2 是 GDB 内部线程号）
b worker.cpp:25 thread 2

# 线程断点 + 条件
b worker.cpp:25 thread 2 if count > 100
```

**应用场景**：多线程竞争时，只跟踪其中一个线程的流程，避免被其他线程反复中断。

### 1.17.4 线程调度锁定（`scheduler-locking`）

```gdb
set scheduler-locking on      # 单步时只运行当前线程，其他线程暂停
set scheduler-locking off     # 单步时所有线程都运行（默认）
set scheduler-locking step    # step 命令时锁住，next/continue 时不锁
```

| 模式 | 单步(step/next) | 继续(continue) |
|------|:---:|:---:|
| `off` | 所有线程运行 | 所有线程运行 |
| `on` | 仅当前线程运行 | 所有线程运行 |
| `step` | 仅当前线程运行（仅 step） | 所有线程运行 |

> **调试技巧**：单步跟踪多线程代码时开 `on` 或 `step`，否则 `step` 可能跳到其他线程，导致流程不可控。

### 1.17.5 线程特定操作

```gdb
thread apply <thread_id> <cmd>   # 对指定线程执行命令
thread apply all <cmd>           # 对所有线程执行命令
```

常用组合：
```gdb
thread apply all bt              # 全线程调用栈（排查死锁首选）
thread apply all bt full         # 全线程调用栈 + 局部变量
thread apply all frame           # 全线程当前帧信息
```

### 1.17.6 断点同步设置

```gdb
set breakpoint pending on        # 共享库加载后自动设置断点
                                  # 多线程场景下，线程函数尚未加载时有用
```

### 1.17.7 死锁排查流程

1. **挂住程序**：程序卡住无响应时，`Ctrl+C` 中断进 gdb
2. **全栈回溯**：`thread apply all bt` 查看所有线程卡在哪里
3. **切线程查看**：对可疑线程依次 `thread N` → `frame N` → `info locals`
4. **查看锁状态**：
   ```gdb
   p mutex           # 查看 mutex 对象状态
   p pthread_t      # 查看线程 ID
   ```

典型死锁输出：
```gdb
Thread 3 (LWP 12347):
#0  __lll_lock_wait () at ...
#1  _L_lock_812 () at worker.cpp:50    ← 等待锁 B
#2  worker_func (arg=0x...) at worker.cpp:50

Thread 2 (LWP 12346):
#0  __lll_lock_wait () at ...
#1  _L_lock_812 () at worker.cpp:60    ← 等待锁 A  ← 死锁：互等对方持有的锁
#2  worker_func (arg=0x...) at worker.cpp:60
```

线程 2 持有锁 B 等锁 A，线程 3 持有锁 A 等锁 B——经典的 ABBA 死锁。

### 1.17.8 多线程条件变量调试

```gdb
# 在 pthread_cond_wait 处设断点
b pthread_cond_wait

# 在 pthread_cond_signal 处设断点
b pthread_cond_signal
```

结合 `thread apply all bt` 查看哪些线程在等待条件变量、哪些线程在持有锁。

### 1.17.9 记录线程日志

```gdb
set pagination off                # 关闭分页，避免长输出中断
set logging file thread_log.txt   # 输出到文件
set logging on                    # 开启日志记录
thread apply all bt full          # 记录所有线程的完整回溯
set logging off                   # 关闭日志记录（日志保存到文件）
```

### 1.17.10 TUI 模式下的多线程

```gdb
tui enable                        # 开启 TUI 界面
layout src                        # 源码 + 命令布局
```

TUI 模式下 `info threads` 仍有效，配合 `thread N` 切换线程时源码窗口自动跳转。

### 1.17.11 快速参考

| 命令 | 缩写 | 用途 |
|------|------|------|
| `info threads` | `i th` | 列出所有线程 |
| `thread N` | — | 切换到 N 号线程 |
| `thread apply all bt` | — | 查看所有线程调用栈 |
| `thread apply all bt full` | — | 查看所有线程调用栈 + 局部变量 |
| `b func thread N` | — | 仅 N 号线程命中该断点 |
| `b func thread N if cond` | — | 线程断点 + 条件 |
| `set scheduler-locking on` | — | 单步时只跑当前线程 |
| `set scheduler-locking off` | — | 单步时所有线程都跑 |
