# 1. 概述

## 【核心精简提纲】

### 记忆主干

Valgrind 在 IR 层插桩，无需 `-fsanitize` 重编译即可查泄漏、越界、未初始化值；慢 5~20 倍，仅开发/测试用。Memcheck 查内存，Helgrind 查竞态，Callgrind 查热点。

### 核心要点清单

- Memcheck：`valgrind --leak-check=full --show-leak-kinds=all`
- 泄漏级：definitely lost 必修 > possibly lost > still reachable
- vgdb：`--vgdb-error=1` 首错暂停，`gdb` → `target remote | vgdb`
- Callgrind：`Ir` 最高函数即热点；Cachegrind 看 D1mr
- Massif：堆峰值与分配点调用栈
- 局限：不查整数溢出/逻辑错；生产禁用；日常可用 ASan（~2x）

### 交叉引用

- [[1. gdb|gdb / vgdb 联动]]、[[2. coredump|coredump 对比现场崩溃]]
- [[4. perf分析流程|perf 与 Callgrind 互补]]、[[7. 综合调试实战]]

### 关键结论 / 流程

段错误/泄漏 → Memcheck；竞态 → Helgrind；堆峰值 → Massif；首错现场 → vgdb + gdb

## 【完整拓展详情】

```plain
┌─────────────────────────────────────────────┐
│                 Valgrind 框架                │
├─────────┬─────────┬──────┬──────┬────────────┤
│ Memcheck│Callgrind│Cache │Helgrind│  Massif  │
│ 内存错误 │ 性能分析 │grind │ 线程竞 │ 堆分析    │
│         │         │缓存  │ 态检测 │           │
│         │         │分析  │       │           │
└─────────┴─────────┴──────┴──────┴────────────┘
```

**工作原理**：Valgrind 将程序代码编译为中间表示（IR），在 IR 层面插入检测代码，再翻译为机器码执行。这导致程序运行速度**降低 5-20 倍**（取决于工具），**无需重新编译**原程序。

> **何时用 Valgrind**：程序出现段错误、内存泄漏、未定义行为时。不能用于实时系统或生产环境——性能开销太大。

# 2. 工具概览

| 工具 | 命令 | 用途 | 速度损失 |
|------|------|------|---------|

# 4. Memcheck — 内存错误检测

## 4.1 基本用法

```bash
# 最简单的启动方式
valgrind ./myprogram

# 等价于（--tool=memcheck 是默认值）
valgrind --tool=memcheck ./myprogram

# 带程序参数
valgrind ./myprogram --input data.txt --verbose
```

## 4.2 输出解读

## 4.3 错误类型

### 4.3.1 非法读写（Invalid read/write）

访问不属于进程的内存（越界、释放后访问等）：

```plain
Invalid write of size 4
   at 0x10915C: main (leak.c:5)
 Address 0x4a9c068 is 0 bytes after a block of size 40 alloc'd
```

`Invalid write of size N` 表示写入 N 字节到非法地址；`Invalid read of size N` 表示读取 N 字节从非法地址。常见原因：

| 原因 | 典型代码 |
|------|---------|
| 数组越界 | `arr[10]` 但 `arr` 只分配了 10 个元素（索引 0-9） |
| 使用已释放内存 | `free(p); p[0] = 1;` |
| 栈缓冲区溢出 | 局部数组越界赋值 |
| 使用已返回的栈地址 | `int *p = func();`，`func` 返回了局部变量的地址 |

### 4.3.2 使用未初始化值（Conditional jump or move depends on uninitialised value）

```c
int x;
if (x > 0) { ... }    // x 未初始化
```

```plain
==12345== Conditional jump or move depends on uninitialised value(s)
==12345==    at 0x1091A0: main (test.c:8)
```

### 4.3.3 内存泄漏类型

| 类型 | 含义 |
|------|------|
| **definitely lost** | 指针丢失，完全无法释放——**必须修复** |
| **indirectly lost** | 因父节点泄漏导致子节点也无法释放（常见于链表/树结构） |
| **possibly lost** | 指针指向结构体内部而非起始位置，Valgrind 无法确定能否释放 |
| **still reachable** | 指针仍存在但未释放（程序退出前应 clean up，但非严重问题） |
| **suppressed** | 被抑制文件过滤掉的泄漏（通常是第三方库） |

优先级：`definitely lost` > `indirectly lost` > `possibly lost` > `still reachable`。

## 4.4 常用选项

### 报告控制

```bash
# 显示泄漏详情（告诉 Valgrind 退出时打印哪些泄漏）
valgrind --leak-check=full ./myprogram

# 显示"可能泄漏"的详细信息（默认只显示"确定泄漏"）
valgrind --show-leak-kinds=all ./myprogram

# 显示可达泄漏（不清理全局变量时会用到）
valgrind --show-leak-kinds=definite,possible ./myprogram

# 等价于完整模式
valgrind --leak-check=full --show-leak-kinds=all ./myprogram
```

### 错误追踪

```bash
# 每个错误最多显示 N 次（默认 10，设 0 不限）
valgrind --errors-for-leak-kinds=all --num-callers=20 ./myprogram

# 追踪分配时的完整调用栈深度（默认 12，增大可看到更全的调用链）
valgrind --num-callers=30 ./myprogram

# 首次错误时附加 gdb（见第 5 节）
valgrind --vgdb-error=0 ./myprogram
```

### 抑制误报

```bash
# 使用抑制文件过滤已知误报
valgrind --suppressions=my.supp ./myprogram

# 生成抑制信息（用于第三方库的已知泄漏）
valgrind --gen-suppressions=all ./myprogram 2> supp.txt
```

## 4.5 实战示例

### 示例：检测 use-after-free

```c
#include <stdlib.h>
#include <stdio.h>

int main() {
    int *p = malloc(sizeof(int));
    free(p);
    *p = 10;                    // ❌ use-after-free
    printf("%d\n", *p);         // ❌ 读取已释放内存
    return 0;
}
```

```bash
$ gcc -g -o uaf uaf.c && valgrind ./uaf

==12345== Invalid write of size 4
==12345==    at 0x10917C: main (uaf.c:7)
==12345==  Address 0x4a9c040 is 0 bytes inside a block of size 4 free'd
==12345==    at 0x484B3AB: free (vg_replace_malloc.c:987)
==12345==    by 0x109177: main (uaf.c:6)
==12345==  Block was alloc'd at
==12345==    at 0x4848899: malloc (vg_replace_malloc.c:442)
==12345==    by 0x10916E: main (uaf.c:5)
```

Valgrind 清楚标出：被写入的地址曾在 `uaf.c:5` 分配，在 `uaf.c:6` 释放，在 `uaf.c:7` 写入——完整的时间线。

### 示例：检测未初始化变量

```c
#include <stdio.h>

int main() {
    int a[5];
    int i = 5;
    if (a[i] > 0) {       // a[i] 未初始化
        printf("positive\n");
    }
    return 0;
}
```

```plain
==12345== Conditional jump or move depends on uninitialised value(s)
==12345==    at 0x10916E: main (uninit.c:6)
```

> 注意：栈上分配的 `a` 默认不会清零（自动变量），Valgrind 能检测到读取未初始化的栈内容。

### 示例：C++ new/delete 误配

```c
#include <string>
int main() {
    std::string *s = new std::string("hello");
    free(s);                    // ❌ new 配 free，不是 delete
    return 0;
}
```

```bash
$ g++ -g -o mis match.cpp && valgrind ./mis

==12345== Mismatched free() / delete / delete []
==12345==    at 0x484B3AB: free (vg_replace_malloc.c:987)
==12345==    by 0x10923E: main (mis.cpp:5)
==12345==  Address 0x4a9c080 is 0 bytes inside a block of size 32 alloc'd
==12345==    at 0x484A3E5: operator new(unsigned long) (vg_replace_malloc.c:422)
==12345==    by 0x10920E: main (mis.cpp:4)
```

Valgrind 能够检测 `new`/`delete`/`malloc`/`free`/`new[]`/`delete[]` 之间的不匹配。

# 5. 与 GDB 联动（vgdb）

## 5.1 方式一：gdb 附加到 Valgrind

```bash
# 终端 1：用 vgdb 模式启动程序
valgrind --vgdb-error=0 ./myprogram

# 终端 2：启动 gdb 连接
gdb ./myprogram
(gdb) target remote | vgdb
```

`--vgdb-error=0` 表示在**第 0 个错误前**暂停（即立即暂停），等待 gdb 连接。

## 5.2 方式二：Valgrind 中直接启动 gdb

```bash
# 在 Valgrind 检测到非法操作时自动暂停并启动 gdb
valgrind --vgdb-error=1 ./myprogram
```

`--vgdb-error=1` 表示在第一个错误时暂停，显示：
```plain
==12345== (action at error) Starting VGDB...
==12345==    at 0x10917C: main (test.c:7)
==12345== TO DEBUG THIS PROCESS USING GDB: (gdb) target remote | vgdb
```

然后在另一个终端运行 `gdb ./myprogram`，输入 `target remote | vgdb` 即可在错误点调试。

# 6. Callgrind — 性能分析

## 6.1 基本用法

```bash
valgrind --tool=callgrind ./myprogram
```

运行后生成 `callgrind.out.<PID>` 文件。用 `callgrind_annotate` 或可视化工具 `kcachegrind` 查看：

```bash
callgrind_annotate callgrind.out.12345

# GUI 可视化（需安装 kcachegrind）
kcachegrind callgrind.out.12345
```

## 6.2 输出解读

```plain
--------------------------------------------------------------------------------
Profile data file: callgrind.out.12345
Events recorded:  Ir  Dr  Dw  I1mr  D1mr  D1mw  ILmr  DLmr  DLmw
Events shown:     Ir  Dr  Dw  I1mr  D1mr  D1mw  ILmr  DLmr  DLmw
Event sort order:  Ir
Thresholds:        99
Include dirs:
User annotated:
Auto-annotation:  on
--------------------------------------------------------------------------------

       Ir  Dr  Dw
--------------------------------------------------------------------------------
8,500,200,121  1,234,567,890  987,654,321  PROGRAM TOTALS

        Ir  Dr  Dw  file:function
--------------------------------------------------------------------------------
2,100,050,000  300,123,456  250,098,765  myapp.cpp:main
1,800,300,000  400,234,567  300,123,456  myapp.cpp:process_data
  950,200,000  150,100,000  120,050,000  myapp.cpp:compute
```

**事件含义**：

| 缩写 | 含义 |
|------|------|
| `Ir` | 指令读取数（**最常用**——衡量 CPU 工作量） |
| `Dr` | 数据读取数 |
| `Dw` | 数据写入数 |
| `I1mr` | L1 指令缓存未命中 |
| `D1mr` | L1 数据缓存未命中 |
| `D1mw` | L1 数据缓存写未命中 |

**关键指标**：关注 `Ir`（指令数）最高的函数——它们是热点，优化收益最大。

## 6.3 性能优化思路

从 Callgrind 输出中可以：

1. **定位热点**：`Ir` 最高的函数就是最值得优化的地方
2. **检查缓存行为**：`D1mr` / `D1mw` 高的函数可能存在**缓存不友好**的内存访问模式
3. **对比优化前后**：优化后重新 profile，验证 `Ir` 是否下降

示例：将链表遍历改为数组遍历后，`Ir` 和 `D1mr` 预期都会显著下降。

## 7.4 Massif 常用选项

```bash
# 增加快照数量，提高精度
valgrind --tool=massif --time-unit=B ./myprogram

# 监控子进程
valgrind --tool=massif --trace-children=yes ./myprogram

# 设置堆分配阈值（默认 1%，降低可看到更多分配细节）
valgrind --tool=massif --threshold=0.1 ./myprogram

# 查看峰值分配点的更详细调用栈
valgrind --tool=massif --detailed-freq=1 ./myprogram
```

`--time-unit` 可选：
| 单位 | 含义 |
|------|------|
| `i`（默认） | 指令数 |
| `ms` | 毫秒 |
| `B` | 字节（分配/释放次数） |

# 8. Helgrind — 线程竞态检测

## 8.1 基本用法

```bash
valgrind --tool=helgrind ./mythread_program
```

## 8.2 常见检测

### 数据竞争

```c
#include <pthread.h>
int counter = 0;

void *worker(void *arg) {
    for (int i = 0; i < 100000; i++)
        counter++;              // ❌ 非原子操作，无锁保护
    return NULL;
}

int main() {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, worker, NULL);
    pthread_create(&t2, NULL, worker, NULL);
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
    return 0;
}
```

```plain
==12345== Possible data race during write of size 4 at 0x10C040
==12345==    at 0x1091BC: worker (race.c:6)
==12345==  This conflicts with a previous write of size 4
==12345==    at 0x1091BC: worker (race.c:6)
==12345==  Location 0x10C040 is 0 bytes inside counter
```

### 锁顺序冲突（潜在死锁）

```c
pthread_mutex_lock(&A);
pthread_mutex_lock(&B);      // 线程 1: 锁 A → 锁 B
// ...
pthread_mutex_unlock(&B);
pthread_mutex_unlock(&A);

// 另一个线程：
pthread_mutex_lock(&B);
pthread_mutex_lock(&A);      // 线程 2: 锁 B → 锁 A  ← Helgrind 会报警
```

# 9. 高级技巧

## 9.1 抑制文件（Suppression）

第三方库的已知泄漏不想每次看到，可以用抑制文件过滤：

```bash
# 生成抑制信息
valgrind --gen-suppressions=all ./myprogram 2> supp.raw

# 提取抑制记录到文件
# 格式（直接复制 Valgrind 输出的抑制块到 .supp 文件）：
cat supp.raw
# 手动提取 {...} 内的内容到 my.supp

# 使用抑制文件
valgrind --suppressions=my.supp ./myprogram
```

抑制文件格式：
```plain
{
   <抑制器名称>
   Memcheck:Leak
   fun:malloc
   ...
   fun:__libc_start_main
}
```

## 9.2 只跟踪特定内存区域

```bash
# 用客户请求（client request）标记感兴趣的代码段
valgrind --tool=memcheck ./myprogram

# 在源码中插入 Valgrind 客户请求（需包含 valgrind/valgrind.h）
#include <valgrind/valgrind.h>

VALGRIND_MEMPOOL_ALLOC(pool, addr, size);  // 标记内存池分配
VALGRIND_MEMPOOL_FREE(pool, addr);         // 标记内存池释放
```

这需要链接 Valgrind 头文件，通常用于自定义分配器的调试。

## 9.3 日志输出到文件

```bash
# 输出到文件而非终端（程序输出多时方便）
valgrind --log-file=valgrind.log ./myprogram

# 带进程 ID 的日志文件
valgrind --log-file=valgrind.%p.log ./myprogram

# 同时输出到终端和文件
valgrind --log-file=valgrind.log --log-fd=2 ./myprogram
```

# 10. 注意事项与局限

## 10.1 性能开销

- Memcheck/Helgrind：程序运行慢 **10-20 倍**
- Callgrind/Cachegrind：慢 **5-10 倍**
- Massif：慢 **5-10 倍**

**不在生产环境使用**。仅用于开发和测试阶段。

## 10.2 不能检测的

| 问题 | Valgrind 能否检测 |
|------|:---:|
| 数组越界（栈上连续变量） | ⚠️ 取决于布局 |
| 全局变量未初始化 | ✅ |
| 栈上局部变量未初始化 | ✅ |
| 整数溢出 | ❌ |
| 逻辑错误（算法错误） | ❌ |
| 缓冲区溢出（栈上 padding 后） | ⚠️ 可能漏报 |
| 死锁（锁顺序冲突） | ✅（Helgrind） |
| 数据竞争 | ✅（Helgrind/DRD） |

## 10.3 与 AddressSanitizer（ASan）对比

| 特性 | Valgrind (Memcheck) | AddressSanitizer |
|------|:---:|:---:|
| 性能开销 | 10-20x | ~2x |
| 需重新编译 | ❌ | ✅（`-fsanitize=address`） |
| 检测精度 | 高（IR 层面） | 中等（编译插桩） |
| 内存占用 | 高 | 中等 |
| 适用阶段 | 开发/测试 | 开发/CI |
| 未初始化值检测 | ✅ | ❌ |
| 泄漏检测 | ✅ | ✅（+`-fsanitize=leak`） |

- **建议**：日常开发用 ASan（快，适合 CI），遇到 ASan 不能捕捉的未初始化值问题或需要更详细的分析时用 Valgrind。崩溃现场用 [[2. coredump|coredump]] + [[1. gdb|gdb]] 作补充。

# 11. 快速参考

```bash
# 最常用：内存错误 + 泄漏详情
valgrind --leak-check=full --show-leak-kinds=all ./myprogram

# 轻量泄漏检查
valgrind --leak-check=full ./myprogram

# 性能分析（函数热点）
valgrind --tool=callgrind ./myprogram
callgrind_annotate callgrind.out.*

# 堆内存分析
valgrind --tool=massif ./myprogram
ms_print massif.out.*

# 线程竞态
valgrind --tool=helgrind ./myprogram

# 输出到文件
valgrind --log-file=valgrind.log --leak-check=full ./myprogram

# 与 gdb 联动（首个错误暂停）
valgrind --vgdb-error=1 ./myprogram
```
