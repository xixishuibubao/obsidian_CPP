# 1. Coredump

## 【核心精简提纲】

### 记忆主干

异常退出时 OS 将栈、寄存器、堆快照写入 core 文件；配合带 `-g` 的可执行文件用 gdb 离线还原崩溃现场，适合偶现与客户现场问题。

### 核心要点清单

- 开启：`ulimit -c unlimited`（当前 shell）；`core_pattern` 永久配置命名
- 检查：`ulimit -a | grep core`、`cat /proc/sys/kernel/core_pattern`
- 分析：`gdb 可执行文件 core文件` → `bt` / `frame` / `info locals`
- 占位符：`%e` 程序名、`%s` 信号、`%t` 时间戳、`%p` PID
- 前提：编译 `-g`，保留符号表；strip 后只能看地址
- 挂起进程：`gcore PID` 可手动生成 core

`ulimit -c unlimited` → 复现崩溃 → `gdb app core` → `bt full` → 定位行号

## 【完整拓展详情】

## 1.1 为什么需要 coredump

| 场景 | 无 coredump | 有 coredump |
|------|-------------|-------------|
| 段错误 | 只知道崩了，不知道在哪 | 直接定位到行号、参数、寄存器 |
| 偶现崩溃 | 跑一天复现一次 | 保留现场，随时分析 |
| 客户现场崩溃 | 无法复现，干瞪眼 | 拿到 core 文件即可分析 |

> **前提**：编译时必须加 `-g` 保留调试信息。

## 1.2 开启与配置

### 临时开启（当前 shell）

```bash
ulimit -c unlimited    # 取消 core 文件大小限制
```

### 永久配置命名规则

```bash
# 编辑系统配置
sudo vim /etc/sysctl.conf
kernel.core_pattern = %e_core_%s_%t    # 添加此行

# 生效
sudo sysctl -p
```

`%e` = 可执行文件名，`%s` = 信号编号，`%t` = 时间戳。完整占位符见下。

### 快速检查

```bash
ulimit -a | grep core          # 确认 core 大小限制
cat /proc/sys/kernel/core_pattern  # 确认命名规则
```

## 1.3 `core_pattern` 占位符

| 占位符 | 含义 | 示例 |
|--------|------|------|
| `%e` | 可执行文件名 | `myapp` |
| `%p` | 进程 PID | `12345` |
| `%s` | 导致终止的信号编号 | `11` (SIGSEGV) |
| `%t` | 转储时间（Unix 时间戳） | `1718000000` |
| `%h` | 主机名 | `ubuntu-arm64` |
| `%%` | 输出一个 `%` | `%` |

```bash
# 示例：配置为 程序名.core
kernel.core_pattern = %e.core
# 生成文件：myapp.core
```

## 1.4 分析步骤

1. 开启 coredump（1.2 节）
2. 运行程序直到崩溃，生成 core 文件
3. 用 gdb 加载并分析：

```bash
gdb 可执行文件 core文件
```

进入 gdb 后使用 `bt`（backtrace）查看调用栈，`frame N` 切换帧，`info locals/args` 查看变量。

> 完整分析命令参见：[[1. gdb|gdb coredump 分析详解]]

## 1.5 踩坑与边界

- **容器/Docker**：默认 `ulimit -c 0` 且可能禁用 core；需在容器启动参数或 `docker-compose` 中显式放开
- **core 体积**：全内存快照可能数 GB，注意磁盘空间；`core_pattern` 可写到专用目录并配合清理策略
- **setuid 程序**：安全策略可能禁止 setuid 进程写 core，需查 `/proc/sys/fs/suid_dumpable`
- **与 ASan 共存**：AddressSanitizer 崩溃时优先看 ASan 报告；core 分析作补充，符号可能重叠
