# 1. Coredump

Coredump（核心转储）是程序异常退出时，操作系统保存的内存快照（栈、寄存器、堆数据），用于事后还原崩溃现场。

相关笔记：[[1. gdb#1.16 coredump 崩溃分析|gdb: coredump 分析详解]]、[[7. 综合调试实战|perf + GDB + coredump 实战]]

## 1.1 为什么需要 coredump

| 场景 | 无 coredump | 有 coredump |
|------|-------------|-------------|
| 段错误 | 只知道崩了，不知道在哪 | 直接定位到行号、参数、寄存器 |
| 偶现崩溃 | 跑一天复现一次 | 保留现场，随时分析 |
| 客户现场崩溃 | 无法复现，干瞪眼 | 拿到 core 文件即可分析 |

> **前提**：编译时必须加 `-g` 保留调试信息。

## 1.2 开启与配置

### 临时开启（当前 shell）

```bash
ulimit -c unlimited    # 取消 core 文件大小限制
```

### 永久配置命名规则

```bash
# 编辑系统配置
sudo vim /etc/sysctl.conf
kernel.core_pattern = %e_core_%s_%t    # 添加此行

# 生效
sudo sysctl -p
```

`%e` = 可执行文件名，`%s` = 信号编号，`%t` = 时间戳。完整占位符见下。

### 快速检查

```bash
ulimit -a | grep core          # 确认 core 大小限制
cat /proc/sys/kernel/core_pattern  # 确认命名规则
```

## 1.3 `core_pattern` 占位符

| 占位符 | 含义 | 示例 |
|--------|------|------|
| `%e` | 可执行文件名 | `myapp` |
| `%p` | 进程 PID | `12345` |
| `%s` | 导致终止的信号编号 | `11` (SIGSEGV) |
| `%t` | 转储时间（Unix 时间戳） | `1718000000` |
| `%h` | 主机名 | `ubuntu-arm64` |
| `%%` | 输出一个 `%` | `%` |

```bash
# 示例：配置为 程序名.core
kernel.core_pattern = %e.core
# 生成文件：myapp.core
```

## 1.4 分析步骤

1. 开启 coredump（1.2 节）
2. 运行程序直到崩溃，生成 core 文件
3. 用 gdb 加载并分析：

```bash
gdb 可执行文件 core文件
```

进入 gdb 后使用 `bt`（backtrace）查看调用栈，`frame N` 切换帧，`info locals/args` 查看变量。

> 完整分析命令参见：[[1. gdb#1.16 coredump 崩溃分析|gdb coredump 分析详解]]
