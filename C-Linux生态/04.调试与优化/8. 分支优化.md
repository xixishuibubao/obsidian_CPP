# 1. 分支优化（减少流水线停顿）

跳转（分支）指令是导致 CPU 流水线停顿的**头号元凶**。嵌入式 CPU 流水线深度更浅、分支预测能力更弱，分支带来的性能损失比桌面 CPU 更明显。

参见 [[3. 基本调优|基本调优 - 分支预测]]。

## 1.1 底层原理

CPU 流水线是**指令执行的并行化设计**，典型分为 5 级：**取指 → 译码 → 执行 → 访存 → 写回**。理想情况下每个时钟周期完成一条指令。

跳转指令破坏流水线的两个原因：

1. **分支预测失败**：CPU 通过分支预测器猜测跳转方向并提前取指。猜测错误时，CPU 必须**清空流水线**中所有已取的错误指令，重新取指，导致 **3~20 个时钟周期停顿**。
2. **流水线排空**：即使预测成功，跳转指令本身也会导致流水线部分阶段无法并行。

**嵌入式场景更严重**：嵌入式 CPU（如 ARM Cortex-A/R 系列）通常使用**静态分支预测**（向后跳转默认循环，向前跳转默认不执行），预测成功率远低于桌面 CPU 的**动态分支预测**。

## 1.2 核心优化原则

1. **扁平化控制流** — 用顺序执行替代条件分支，减少跳转数量
2. **优化分支结构** — 大概率执行的分支放在前面，提高分支预测成功率
3. **消除冗余分支** — 移除不必要的条件判断
4. **查表法替代分支** — 用数组索引替代 `switch/case` 或 `if/else if`
5. **循环优化** — 减少循环内部的分支，优化循环条件
6. **避免无条件跳转** — 减少不必要的 `goto`、函数调用

## 1.3 实战优化技巧

### 技巧 1：三目运算符替代简单 if/else

**原理**：三目运算符 `?:` 是表达式而非语句，编译器可编译为无分支指令（如 `cmov`）：

```c
// 优化前：产生跳转指令
int func1(int a, int b, int c) {
    int ret;
    if (a > 0) ret = b;
    else       ret = c;
    return ret;
}

// 优化后：无分支，编译器生成无跳转指令
int func2(int a, int b, int c) {
    return (a > 0) ? b : c;
}
```

> 仅适用于简单的条件赋值，勿嵌套复杂逻辑。

### 技巧 2：查表法替代 switch/case

**原理**：用函数指针数组替代 `switch/case`，将分支跳转转化为顺序的访存操作（无分支、利用缓存）。

```c
// 优化前：switch/case 产生多个跳转
int drv_ioctl_old(unsigned int cmd, unsigned long arg) {
    switch (cmd) {
    case CMD_SET_DATA:     return set_data(arg);
    case CMD_GET_DATA:     return get_data(arg);
    case CMD_TRIGGER_IRQ:  return trigger_irq(arg);
    case CMD_CLEAR_STATS:  return clear_stats(arg);
    default:               return -EINVAL;
    }
}

// 优化后：查表法，无分支
typedef int (*cmd_handler_t)(unsigned long arg);

static const cmd_handler_t cmd_handlers[] = {
    [CMD_SET_DATA]    = set_data,
    [CMD_GET_DATA]    = get_data,
    [CMD_TRIGGER_IRQ] = trigger_irq,
    [CMD_CLEAR_STATS] = clear_stats,
};

int drv_ioctl_new(unsigned int cmd, unsigned long arg) {
    if (cmd >= ARRAY_SIZE(cmd_handlers))
        return -EINVAL;                       // 仅一个条件判断
    return cmd_handlers[cmd](arg);             // 无分支查表
}
```

**适用条件**：命令号需为从 0 开始的连续整数；处理函数原型一致。这是驱动开发中 `ioctl` 命令处理、中断分发最常用的优化技巧。

### 技巧 3：循环内部分支外提

**原理**：循环是程序执行的热点，内部分支被执行多次，将其提到循环外部可大幅减少分支数量：

```c
// 优化前：循环内部有分支，执行 n 次
void loop_old(int *a, int *b, int n) {
    for (int i = 0; i < n; i++) {
        if (a[i] > 0)    b[i] = a[i] * 2;
        else             b[i] = a[i] / 2;
    }
}

// 优化后：拆分为两个循环，每个循环内部无分支
void loop_new(int *a, int *b, int n) {
    for (int i = 0; i < n; i++)
        if (a[i] > 0)    b[i] = a[i] * 2;   // 仅判断是否处理，无 else 分支
    
    for (int i = 0; i < n; i++)
        if (a[i] <= 0)   b[i] = a[i] / 2;
}
```

总循环次数增加，但分支预测成功率接近 100%，整体性能提升明显。

**其他循环优化**：
- **循环展开**：小循环展开为顺序语句，减少循环条件判断
- **循环不变量外提**：将 `n`、地址计算等不变量提到循环外
- **避免循环内函数调用**：简单逻辑用 `inline` 内联

### 技巧 4：大概率分支放在前面

**原理**：CPU 分支预测器倾向于预测前面的分支被选中（静态预测）。将大概率分支放在前面可提高预测成功率：

```c
// 优化前：小概率分支在前，预测器大概率猜错
int func_old(void *ptr) {
    if (ptr == NULL)       // 1% 概率，但放在前面
        return -EINVAL;
    else
        return 0;          // 99% 概率，但放在后面
}

// 优化后：大概率分支在前
int func_new(void *ptr) {
    if (ptr != NULL)       // 99% 概率，放在前面
        return 0;
    else
        return -EINVAL;    // 即使猜错也只影响 1%
}
```

适用于：错误处理、边界条件判断、驱动参数检查。

### 技巧 5：数组遍历替代条件遍历

将条件判断提到遍历外部，遍历过程中无分支：

```c
// 优化前
void process_old(int *data, int n) {
    for (int i = 0; i < n; i++) {
        if (data[i] % 2 == 0)    data[i] = data[i] * 2;
        else                     data[i] = data[i] * 3;
    }
}

// 优化后：两次遍历，各无分支
void process_new(int *data, int n) {
    for (int i = 0; i < n; i++)
        if (data[i] % 2 == 0)    data[i] *= 2;
    for (int i = 0; i < n; i++)
        if (data[i] % 2 != 0)    data[i] *= 3;
}
```

### 技巧 6：避免无条件跳转

不必要的 `goto` 和函数调用导致流水线停顿 + 栈操作：

```c
// 优化前：不必要的 goto
int func_old(int a) {
    int ret;
    if (a > 0) { ret = 0; goto out; }
    else       { ret = -EINVAL; goto out; }
out:
    return ret;
}

// 优化后：直接返回
int func_new(int a) {
    if (a > 0) return 0;
    else       return -EINVAL;
}
```

> 驱动中必要的资源释放 `goto`（如 `goto err_free`）不能消除，但应避免无意义的跳转。简单函数用 `static inline` 内联消除调用开销。

## 1.4 编译器与架构辅助

### 编译器选项

| 选项 | 说明 | 适用场景 |
|------|------|----------|
| `-O2` | 消除冗余分支、优化循环、内联小函数 | 内核驱动开发（平衡性能与稳定性） |
| `-O3` | 循环展开、向量化、更激进内联 | 用户态程序 |
| `-fbranch-probabilities` | 基于执行记录优化分支概率 | 有 profiling 数据时 |
| `-finline-functions` | 强制内联函数 | 消除函数调用开销 |

### likely/unlikely 宏

告诉编译器分支执行概率，帮助编译器优化：

```c
// 内核中常用封装
#define likely(x)   __builtin_expect(!!(x), 1)
#define unlikely(x) __builtin_expect(!!(x), 0)

// 使用示例
if (likely(ptr != NULL)) {
    // 正常逻辑（99% 概率）
} else {
    // 错误处理（1% 概率）
}
```

内核驱动中，配合**技巧 4**，在参数检查、错误处理场景收益最大。

## 1.5 实战调优流程

1. **性能分析** — 用 `perf` 找到热点代码和分支预测失败率最高的位置：
   ```bash
   perf stat -e branch-misses,branch-instructions ./your_program
   ```
2. **确定优化目标** — 如执行时间减少 20%、分支预测失败率降低 50%
3. **手动优化** — 按优先级：查表法 > 循环优化 > 分支前置 > 三目运算 > 消除 goto
4. **编译器优化** — 设置合适的编译选项
5. **性能验证** — 再次用 `perf` 验证优化效果
6. **代码审查** — 确保可读性和可维护性
