# 1. 工具分工

## 【核心精简提纲】

### 记忆主干

perf 找哪里慢，GDB 查为什么慢，coredump 还原为什么崩；三者共用 `g++ -g -O2` 且不 strip，形成「热点→断点→崩溃栈」闭环。

### 核心要点清单

- perf：`record -g` → `report --source` 得热点函数与源码行
- GDB：对热点 `b 函数/行` → `run` → `n/s/p` 查变量
- coredump：`ulimit -c unlimited` → `gdb app core` → `bt`
- 经典模式：死循环（perf 99% CPU）、野指针（`this=0x0`）、栈溢出
- 无符号：`addr2line -e app -f 0x地址` 或 gdb `info line *addr`
- 口诀：慢看 perf，逻辑看 gdb，崩看 core

- [[4. perf分析流程]]、[[1. gdb]]、[[2. coredump]]、[[9. Valgrind 内存调试]]
- [[10. CPU 高占用排查]]、[[5. 高性能常用工具]]

### 关键结论 / 流程

CPU 高+崩溃：`perf top` → `record` → 崩溃后 `gdb core` → 对比热点与 `bt` 是否同一函数

## 【完整拓展详情】

| 工具 | 作用 | 解决问题 |
|------|------|----------|
| **perf** | 看性能：哪个函数耗 CPU、缓存命中率、调用链 | 程序慢、CPU 高、耗时函数 |
| **GDB** | 看运行时：单步调试、变量值、内存、崩溃栈 | 逻辑错误、死锁、内存越界 |
| **coredump** | 保存崩溃现场：程序挂掉时的内存/栈/寄存器 | 程序突然崩溃、段错误、野指针 |

核心口诀：**perf 定位哪里慢，GDB 调试为什么慢，coredump 还原为什么崩。**

# 2. 编译标准

三者共用前提——**编译必须加 `-g`**：

```bash
g++ -g -O2 -o myapp main.cpp func.cpp
```

- `-g`：保留调试信息（perf + GDB + coredump 都需要）
- `-O2`：生产优化（不影响调试）

**不要 strip，不要删符号表**，否则函数名和行号全部丢失。

# 3. perf 定位热点

## 3.1 查找热点函数

```bash
perf record -g -p <PID> --sleep 10
perf report
```

得到输出：

```plain
35%  [.] MyClass::calculate()
```

## 3.2 查看热点源码行

```bash
perf report --source
```

直接显示耗时函数的源码片段，标注每一行的耗时百分比。

# 4. GDB 调试热点函数

通过 perf 定位到热点函数后，用 GDB 精准调试该函数：

## 4.1 对热点函数打断点

```bash
gdb ./myapp
```

```gdb
b MyClass::calculate()          # 对热点函数下断点
run
```

或对热点代码行号下断点（配合 `perf report --source` 获得行号）：

```gdb
b main.cpp:88                   # 直接在耗时代码行打断点
```

## 4.2 查看调用链与变量

```gdb
bt                              # 查看调用栈
n                               # 单步执行
s                               # 进入函数
p 变量名                         # 查看变量值
p this                          # 检查对象是否为空
```

# 5. coredump 分析崩溃

## 5.1 开启 coredump

```bash
ulimit -c unlimited             # 允许生成 core 文件
```

程序崩溃后自动生成 `core.<PID>`。

## 5.2 GDB 加载并分析

```bash
gdb ./myapp core.<PID>
```

```gdb
bt                              # 看崩溃栈——哪一行崩溃
f 0                             # 跳转到崩溃所在的栈帧
list                            # 显示崩溃处代码
p this                          # 检查对象是否为空
p 变量名                         # 查看崩溃时的变量值
info registers                  # 查看寄存器
```

# 6. 三者联动实战

## 6.1 完整排查步骤

**场景**：程序 CPU 100% → 过一会直接崩溃

```bash
# 1. perf 抓实时热点
perf top -p <PID>
# 发现 MyClass::calculate() 占用 99% CPU

# 2. 采样生成报告
perf record -g -p <PID> --sleep 10
perf report --source
# 确认热点函数和具体耗时代码行

# 3. 程序崩溃，生成 coredump

# 4. GDB 打开 coredump，看崩溃位置
gdb ./myapp core.<PID>
bt
# 发现崩溃也在 MyClass::calculate()
# 对比 perf 热点 + GDB 崩溃栈 → 热点和崩溃点是同一函数
```

## 6.2 经典问题模式

| 问题 | perf 表现 | coredump 表现 |
|------|-----------|---------------|
| 死循环 | 函数 99% CPU | `bt` 停在循环内 |
| 野指针 / 空指针 | 函数频繁执行 | `this=0x0` 或指针非法 |
| 重复释放 / 越界 | 函数耗 CPU | 崩溃在 `free()` / `delete` |
| 无限递归 | 栈爆炸 | 栈溢出崩溃 |

# 7. 地址反查（只有地址没有符号时）

当 `bt` 显示纯地址（全是 `??()`，只有 `0x...` 数值）时，用 addr2line 反查：

```bash
addr2line -e ./myapp -f 0x地址
```

输出示例：`MyClass::compute()  test.cpp:88`

详细用法见 [[4. perf分析流程|addr2line]]。

# 8. 命令速查

```bash
# perf 找性能
perf top -p <PID>                        # 实时热点
perf record -g -p <PID> --sleep 10       # 采样
perf report --source                     # 看耗时代码行

# GDB 调试函数
gdb ./myapp
b 函数名                                  # 断点
b 文件:行号                                # 精准断点
run
bt                                       # 调用栈
p 变量                                    # 查看值

# coredump 分析崩溃
gdb ./myapp core.<PID>
bt full                                  # 完整崩溃栈
info registers                           # 寄存器
```
