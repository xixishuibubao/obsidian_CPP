# 1. 文件打开标志：O_TRUNC 与 O_APPEND

## 【核心精简提纲】

### 记忆主干

`O_TRUNC` 打开即清空文件，`O_APPEND` 每次写前内核原子定位到末尾；高吞吐异步 I/O 用 Linux 5.1+ 的 io_uring，比 epoll 更适合文件+网络统一场景。

### 核心要点清单

- `O_TRUNC`：覆盖写、重置日志；与 `O_APPEND` 互斥，数据不可恢复
- `O_APPEND`：多进程追加原子；勿用手动 `lseek(SEEK_END)` 代替
- io_uring：SQ/CQ 环形队列，`prep_write` + `submit` + `wait_cqe`
- 需 `liburing`，内核 ≥ 5.1；`IORING_FSYNC_DATASYNC` 只刷数据
- vs epoll：epoll 管 socket 就绪；io_uring 真异步、syscall 更少
- 性能：批量提交 SQE、预分配 buffer、事件驱动处理 CQE

### 交叉引用

- [[4. perf分析流程|perf 定位 IO 热点]]、io_uring vs epoll
- [[C-Linux生态/05.网络编程/2. IO模型与muduo设计|IO 模型与 epoll 设计]]

### 关键结论 / 流程

日志追加用 `O_APPEND`；高并发文件/盘 IO 优先 io_uring；老内核或纯 socket 仍用 epoll

## 【完整拓展详情】
## 1.1 O_TRUNC — 文件截断

`open()` 中指定 `O_TRUNC` 时，若文件已存在则清空其内容，文件大小变为 0 字节。元数据（权限、所有者、时间戳）不变。

### 使用场景

- **覆盖已有文件**：直接丢弃原内容重新写入
- **日志文件初始化**：程序启动时清空旧日志
- **数据重置**：配置文件或数据文件重置为初始状态

### 代码示例

```c
#include <stdio.h>
#include <fcntl.h>
#include <unistd.h>
#include <string.h>

int main() {
    int fd;
    const char *filename = "example.txt";
    const char *data = "Hello, World!\n";

    fd = open(filename, O_WRONLY | O_CREAT | O_TRUNC, 0644);
    if (fd == -1) {
        perror("open failed");
        return 1;
    }

    write(fd, data, strlen(data));
    close(fd);
    return 0;
}
```

### 注意事项

- **数据不可恢复**：直接删除原内容，需谨慎使用
- **与 O_APPEND 互斥**：二者语义矛盾，不能同时使用
- **文件权限**：使用 `O_CREAT` 时需提供第三个参数设置权限

> 如需保留原内容追加数据，应使用 `O_APPEND` 而非 `O_TRUNC`。

## 1.2 O_APPEND — 追加模式

每次写操作前内核自动将文件偏移量定位到文件末尾，多进程写入时保证原子性。

### 典型场景

1. **日志记录**：持续向日志文件添加新事件
2. **数据采集**：传感器数据或用户行为追加到文件
3. **并发写入**：多个进程安全共享同一日志文件

### 代码示例

```c
#include <stdio.h>
#include <fcntl.h>
#include <unistd.h>
#include <string.h>

int main() {
    int fd;
    const char *filename = "appended_data.txt";
    const char *data1 = "First line of appended data.\n";
    const char *data2 = "Second line of appended data.\n";

    fd = open(filename, O_WRONLY | O_CREAT | O_APPEND, 0644);
    if (fd == -1) {
        perror("open failed");
        return 1;
    }

    write(fd, data1, strlen(data1));
    write(fd, data2, strlen(data2));
    close(fd);
    return 0;
}
```

### 关键点

- **自动定位末尾**：无需手动 `lseek()`，内核自动重置偏移量
- **原子性保证**：多进程追加时各 `write()` 原子执行，输出不交错
- **与 O_TRUNC 互斥**：同时指定时 `O_TRUNC` 优先清空文件再启用追加

### ⚠️ 注意事项

1. **追加 vs 手动定位**：手动 `lseek(fd, 0, SEEK_END)` 不保证原子性，多进程可能数据覆盖
2. **追加与读操作**：需同时读写时使用 `O_RDWR | O_APPEND`，读取需手动控制偏移量
3. **文件截断风险**：追加模式下误用 `ftruncate()` 或 `O_TRUNC` 会清空文件

### 其他语言的追加模式

```python
# Python
with open("appended_data.txt", "a") as f:
    f.write("New data line\n")
```

```java
// Java
try (FileWriter writer = new FileWriter("appended_data.txt", true)) {
    writer.write("New data line\n");
}
```

# 2. Linux 异步 I/O：io_uring

**io_uring** 是 Linux 5.1+ 引入的新一代异步 I/O 框架，通过共享内核和用户空间的环形缓冲区，减少系统调用次数和内存拷贝，适合高性能应用。

## 2.1 C 示例：异步写 + 刷盘

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <fcntl.h>
#include <unistd.h>
#include <sys/stat.h>
#include <liburing.h>

#define BUFFER_SIZE 1024
#define QUEUE_DEPTH 2

int main(int argc, char *argv[]) {
    struct io_uring ring;
    struct io_uring_sqe *sqe;
    struct io_uring_cqe *cqe;
    char buffer[BUFFER_SIZE] = "Hello, this data will be asynchronously written and flushed to disk.\n";
    int fd, ret;
    off_t offset = 0;

    // 初始化 io_uring
    ret = io_uring_queue_init(QUEUE_DEPTH, &ring, 0);
    if (ret < 0) { perror("io_uring_queue_init failed"); return 1; }

    // 打开文件
    fd = open("async_file.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
    if (fd < 0) { perror("open failed"); return 1; }

    // ── 写请求 ──
    sqe = io_uring_get_sqe(&ring);
    if (!sqe) { fprintf(stderr, "Failed to get SQE\n"); return 1; }
    io_uring_prep_write(sqe, fd, buffer, strlen(buffer), offset);
    sqe->user_data = 1;

    ret = io_uring_submit(&ring);
    if (ret <= 0) { fprintf(stderr, "io_uring_submit failed: %d\n", ret); return 1; }

    ret = io_uring_wait_cqe(&ring, &cqe);
    if (ret < 0) { perror("io_uring_wait_cqe failed"); return 1; }
    if (cqe->res < 0)
        fprintf(stderr, "Write failed: %d\n", cqe->res);
    else
        printf("Write completed: %d bytes written\n", cqe->res);
    io_uring_cqe_seen(&ring, cqe);

    // ── fsync 刷盘 ──
    sqe = io_uring_get_sqe(&ring);
    if (!sqe) { fprintf(stderr, "Failed to get SQE for fsync\n"); return 1; }
    io_uring_prep_fsync(sqe, fd, IORING_FSYNC_DATASYNC);
    sqe->user_data = 2;

    ret = io_uring_submit(&ring);
    if (ret <= 0) { fprintf(stderr, "io_uring_submit(fsync) failed: %d\n", ret); return 1; }

    ret = io_uring_wait_cqe(&ring, &cqe);
    if (ret < 0) { perror("io_uring_wait_cqe(fsync) failed"); return 1; }
    if (cqe->res < 0)
        fprintf(stderr, "Fsync failed: %d\n", cqe->res);
    else
        printf("File successfully flushed to disk\n");
    io_uring_cqe_seen(&ring, cqe);

    close(fd);
    io_uring_queue_exit(&ring);
    return 0;
}
```

### 编译运行

```bash
gcc -o async_fsync async_fsync.c -luring
./async_fsync
```

### 流程说明

1. `io_uring_queue_init()` — 创建异步 I/O 队列
2. `open()` — 以写模式打开文件
3. `io_uring_prep_write()` + `io_uring_submit()` — 准备写操作并提交
4. `io_uring_wait_cqe()` — 等待写完成，检查结果
5. `io_uring_prep_fsync()` + `io_uring_submit()` — 准备刷盘
6. `io_uring_wait_cqe()` — 等待刷盘完成
7. `close()` + `io_uring_queue_exit()` — 清理资源

### 注意事项

- 编译需安装 `liburing` 开发库
- `IORING_FSYNC_DATASYNC` 仅刷新文件数据而非元数据，提升性能
- 实际应用通常处理多个请求，用循环管理 CQE / SQE
- 要求 Linux 内核 ≥ 5.1

## 2.2 C++ 示例：多文件并发处理管道

以下示例通过 io_uring 异步并行处理多个文件（读取 → 转为大写 → 写入 → fsync）。

### 完整代码

```cpp
#include <iostream>
#include <string>
#include <vector>
#include <algorithm>
#include <fcntl.h>
#include <unistd.h>
#include <liburing.h>
#include <string.h>
#include <sys/stat.h>
#include <filesystem>

namespace fs = std::filesystem;

constexpr size_t BUFFER_SIZE = 1024 * 1024;  // 1MB
constexpr unsigned QUEUE_DEPTH = 128;

// I/O 操作上下文
struct IoContext {
    enum Operation { READ, WRITE, FSYNC };
    Operation op;
    int fd;
    char* buffer;
    size_t size;
    off_t offset;
    std::string filename;
};

class IoUring {
public:
    IoUring(unsigned entries) {
        if (io_uring_queue_init(entries, &ring_, 0) != 0)
            throw std::runtime_error("Failed to initialize io_uring");
    }
    ~IoUring() { io_uring_queue_exit(&ring_); }

    void addRead(int fd, char* buffer, size_t size, off_t offset, void* user_data) {
        auto* sqe = io_uring_get_sqe(&ring_);
        if (!sqe) throw std::runtime_error("Failed to get SQE");
        io_uring_prep_read(sqe, fd, buffer, size, offset);
        io_uring_sqe_set_data(sqe, user_data);
    }

    void addWrite(int fd, char* buffer, size_t size, off_t offset, void* user_data) {
        auto* sqe = io_uring_get_sqe(&ring_);
        if (!sqe) throw std::runtime_error("Failed to get SQE");
        io_uring_prep_write(sqe, fd, buffer, size, offset);
        io_uring_sqe_set_data(sqe, user_data);
    }

    void addFsync(int fd, void* user_data) {
        auto* sqe = io_uring_get_sqe(&ring_);
        if (!sqe) throw std::runtime_error("Failed to get SQE");
        io_uring_prep_fsync(sqe, fd, IORING_FSYNC_DATASYNC);
        io_uring_sqe_set_data(sqe, user_data);
    }

    int submit() { return io_uring_submit(&ring_); }

    bool waitForCompletion(struct io_uring_cqe** cqe) {
        int ret = io_uring_wait_cqe(&ring_, cqe);
        if (ret < 0) throw std::runtime_error("Failed to wait for CQE");
        return true;
    }

    void markCompletionSeen(struct io_uring_cqe* cqe) {
        io_uring_cqe_seen(&ring_, cqe);
    }

private:
    struct io_uring ring_;
};

// 数据处理：大写转换
void processData(char* buffer, size_t size) {
    for (size_t i = 0; i < size; ++i)
        if (buffer[i] >= 'a' && buffer[i] <= 'z')
            buffer[i] -= 32;
}

int main(int argc, char* argv[]) {
    if (argc < 2) {
        std::cerr << "Usage: " << argv[0] << " <input_directory>" << std::endl;
        return 1;
    }

    const std::string inputDir = argv[1];
    const std::string outputDir = inputDir + "_processed";

    if (!fs::exists(outputDir))
        fs::create_directory(outputDir);

    try {
        IoUring ring(QUEUE_DEPTH);
        std::vector<IoContext> contexts;
        std::vector<char*> buffers;

        // 收集文件
        std::vector<std::string> inputFiles;
        for (const auto& entry : fs::directory_iterator(inputDir))
            if (entry.is_regular_file())
                inputFiles.push_back(entry.path());

        // 预分配 context 容量，避免后续 push_back 导致指针失效
        contexts.reserve(inputFiles.size());

        // 提交读请求
        for (const auto& file : inputFiles) {
            int fd = open(file.c_str(), O_RDONLY);
            if (fd < 0) { std::cerr << "Failed to open: " << file << std::endl; continue; }

            struct stat st;
            if (fstat(fd, &st) < 0) { close(fd); continue; }

            auto* buffer = new char[BUFFER_SIZE];
            buffers.push_back(buffer);

            IoContext ctx;
            ctx.op = IoContext::READ;
            ctx.fd = fd;
            ctx.buffer = buffer;
            ctx.size = std::min<size_t>(st.st_size, BUFFER_SIZE);
            ctx.offset = 0;
            ctx.filename = file;
            contexts.push_back(ctx);
            ring.addRead(fd, buffer, ctx.size, 0, &contexts.back());
        }

        int submitted = ring.submit();
        std::cout << "Submitted " << submitted << " read requests" << std::endl;

        // 事件循环
        int completed = 0;
        while (completed < submitted) {
            struct io_uring_cqe* cqe;
            if (ring.waitForCompletion(&cqe)) {
                auto* ctx = static_cast<IoContext*>(cqe->user_data);
                if (cqe->res < 0) {
                    std::cerr << "Operation failed: " << ctx->filename
                              << ", res=" << cqe->res << std::endl;
                } else {
                    switch (ctx->op) {
                    case IoContext::READ: {
                        processData(ctx->buffer, ctx->size);
                        std::string outFile = outputDir + "/"
                                            + fs::path(ctx->filename).filename().string();
                        int outFd = open(outFile.c_str(),
                                         O_WRONLY | O_CREAT | O_TRUNC, 0644);
                        if (outFd < 0) {
                            std::cerr << "Failed to open output: " << outFile << std::endl;
                        } else {
                            ctx->fd = outFd;
                            ctx->op = IoContext::WRITE;
                            ring.addWrite(outFd, ctx->buffer, ctx->size, 0, ctx);
                            ring.submit();
                        }
                        break;
                    }
                    case IoContext::WRITE: {
                        ctx->op = IoContext::FSYNC;
                        ring.addFsync(ctx->fd, ctx);
                        ring.submit();
                        break;
                    }
                    case IoContext::FSYNC: {
                        close(ctx->fd);
                        completed++;
                        break;
                    }
                    }
                }
                ring.markCompletionSeen(cqe);
            }
        }

        for (auto* buf : buffers) delete[] buf;
        std::cout << "Processed " << completed << " files successfully" << std::endl;

    } catch (const std::exception& e) {
        std::cerr << "Exception: " << e.what() << std::endl;
        return 1;
    }
    return 0;
}
```

### 编译运行

```bash
g++ -std=c++17 -o io_uring_demo io_uring_demo.cpp -luring -O3 -Wall
./io_uring_demo /path/to/input/directory
```

### 性能优化要点

- **批量提交**：一次性提交多个 I/O 请求减少系统调用
- **事件驱动**：通过完成队列异步处理结果
- **内存池**：预先分配缓冲区减少动态分配开销
- **并行处理**：同时处理多个文件读写
- **IORING_FSYNC_DATASYNC**：仅同步数据而非元数据

> 可根据需要修改 `processData()` 实现更复杂的数据处理逻辑。

# 3. io_uring 与 epoll 的关系

**核心：io_uring 是 epoll 的下一代、全场景升级版，二者是替代关系，不是互补关系。**

## 3.1 各自定位

| | epoll | io_uring |
|--|-------|----------|
| **引入** | Linux 2.6+（2002） | Linux 5.1+（2019） |
| **定位** | I/O 多路复用器，专用于网络 I/O（socket） | 通用异步 I/O 框架，统一网络 + 文件 + 磁盘 |
| **工作方式** | 监听 fd 就绪事件 | 共享环形缓冲区，零系统调用 |
| **核心局限** | 文件 I/O 仍阻塞；频繁内核交互 | — |

## 3.2 对比

| 维度 | epoll | io_uring |
|------|-------|----------|
| **I/O 类型** | 仅网络（socket） | 网络 + 文件 + 磁盘 + 所有 I/O |
| **I/O 模型** | 同步非阻塞（等事件就绪） | 真正异步（提交后干别的） |
| **系统调用开销** | 高 | 极低（几乎无系统调用） |
| **内存拷贝** | 有（频繁拷贝） | 无（共享内存） |
| **线程模型** | 单线程可并发，但文件 I/O 仍阻塞 | 单线程全异步，无阻塞 |
| **使用复杂度** | 较高（ET/LT、事件注册） | 更简洁（统一接口） |
| **性能** | 网络场景优秀，文件拉胯 | 全场景超越（延迟低 50%+、吞吐翻倍） |

## 3.3 直观理解

- **epoll** 像餐厅叫号机：你必须不停向内核询问"有没有就绪的 I/O？"，且只能管网络"订单"
- **io_uring** 像全自动传送带：你和内核共用一条传送带，扔上任务内核自己处理，全程零沟通成本

## 3.4 选型建议

| 场景 | 推荐 | 原因 |
|------|------|------|
| 老项目 / Linux < 5.1 | epoll | 兼容性限制（nginx、Redis 传统版本） |
| 新项目 / Linux ≥ 5.10 | **io_uring** | 云原生网关、数据库、存储、高并发服务 |
| 老代码迁移 | io_uring 兼容 | `IORING_OP_EPOLL_CTL` 可模拟 epoll 行为 |

## 3.5 总结

- **关系**：io_uring 是 epoll 的官方升级版、全场景替代品，不是补充工具
- **定位**：epoll 专用于网络 I/O，io_uring 统一所有 I/O
- **性能**：io_uring 无系统调用、真正异步，全面碾压 epoll
- **选型**：新系统用 io_uring，老系统兼容用 epoll

> **epoll 是过去的王者，io_uring 是现在和未来的标准。**

### 3.6 踩坑与边界

- **内核版本**：io_uring 在 5.1 引入，生产建议 ≥ 5.10；老内核勿强上
- **O_APPEND 误用**：同时指定 `O_TRUNC | O_APPEND` 时先清空再追加，日志轮转场景易丢数据
- **性能验证**：IO 瓶颈用 [[4. perf分析流程|perf]] 看 `read/write` 热点，勿仅凭理论选型
