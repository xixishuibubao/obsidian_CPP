# 1. Linux 常用指令

日常 Linux 运维与开发中的文件传输、查找、归档、远程登录、编辑与 I/O 重定向命令速查。

## 【核心精简提纲】

### 记忆主干

按场景串联：**传输**（scp 推拉远端）→ **查找**（find 定位 + -exec/xargs 批量处理）→ **归档**（tar 打包压缩）→ **远程登录**（SSH 密钥免密）→ **编辑与输出**（Vim / tee / Shell 重定向组合 FD 0/1/2）。

### 核心要点清单

- scp：`目标 源`，远端格式 `user@host:path`；目标是远端则推送，源是远端则拉取
- find 批量：有空格风险优先 `-exec {} +`；需管道/并行用 `find -print0 | xargs -0`
- tar：`-c` 创建 / `-x` 解压 / `-z` gz；常用 `czvf` / `xzvf`
- SSH 免密：`ssh-keygen` → `ssh-copy-id` → 远程 `~/.ssh` 权限 700/600
- tee：stdin 分叉到终端与文件；`sudo tee` 解决 `sudo > /etc/...` 权限失败
- 重定向：`command > file 2>&1` 顺序不能反；管道只传 stdout

### 关键结论 / 流程

```
传输 → 查找 → 归档 → 免密 SSH → 编辑 / 日志留存（tee + 2>&1）
```

# 2. 查找
## 2.1 查找文件
### 2.1.1 `find`
1. `find . -type d -exec chmod 755 {} \;`
	1. `find . -type d`, 在当前目录查找子目录，并获取名字。
	2. `-exec`，执行后续指令。
	3. `{}` 占位符，自动替换前面`find`的结果到`{}`。
	4. `\;`,对分号的转义，避免语法错误。
2. `find mydir -type l ! -exec test -e {} \; -delete`, 删除失效的链接。
3. `find mydir -type l ! -exec test -e {} \; -print`,打印失效的链接。
### 2.1.2 `xargs`

#### 2.1.2.1 生成式 vs 逐次式

`xargs` 是一个参数生成器（arguments builder），它从标准输入（`stdin`）读取数据，拼成**一条命令**执行：

```bash
find . -name "*.log" | xargs rm
```

与 `find -exec` 的区别：

| 特性 | `-exec` | `xargs` |
|------|---------|---------|
| 执行方式 | 每个结果执行一次（`\;`）或一批一次（`+`） | 默认**批量拼接到一条命令** |
| 数据来源 | `find` 输出 | 任意管道传来的 stdin |
| 效率 | 逐条调用时慢，但安全 | 一次调用多个参数，效率高 |
| 特殊字符 | `find` 自带处理 | 默认空格分隔，需 `-print0` + `-0` |
| 独立性 | 不依赖外部命令 | 依赖管道 |

#### 2.1.2.2 常用选项

```bash
# 限制每批参数个数（避免命令行过长）
find . -type f | xargs -n 10 rm    # 每 10 个文件调一次 rm

# 处理文件名含空格（必须与 find -print0 配对！）
find . -name "*.log" -print0 | xargs -0 rm

# 占位符指定位置（-I：替换标记）
find . -name "*.txt" | xargs -I {} cp {} /backup/

# 并行执行（-P：一次跑 N 个进程）
find . -name "*.gz" | xargs -P 4 -n 1 gunzip
```

> **安全原则**：文件名可能含空格时，永远用 `find ... -print0 \| xargs -0` 配对，否则空格会被当作分隔符。

#### 2.1.2.3 `-exec` 补充：加号模式

`find -exec` 末尾用 `+` 而非 `\;` 也能**批量**执行：

```bash
find . -type f -name "*.c" -exec grep "main" {} +   # 所有结果拼成一条命令
```

行为类似 `xargs`（一次传多个文件），效率远高于逐条 `\;`，且**没有空格分裂问题**。

#### 2.1.2.4 场景对比

```bash
# 场景一：简单批量删除 → 优先 -exec
find . -name "*.tmp" -delete                        # 👍 find 内置，最简
find . -name "*.tmp" -exec rm {} +                  # 👍 安全，无空格问题

# 场景二：后续还要挂管道 → 用 xargs
find . -name "*.log" -print0 | xargs -0 tar czf logs.tar.gz

# 场景三：需要并发加速 → xargs -P 完胜
find . -name "*.jpg" -print0 | xargs -0 -P 8 -n 1 convert -resize 50%

# 场景四：少量文件、逐条确认 → -exec \; 直观
find . -type d -empty -exec rmdir {} \;
```

**选型建议**：
- **有空格风险 + 无需管道后续** → `-exec {} +`
- **需要并行/管道后续/处理非 find 来源** → `xargs`
- **需要逐条确认/精确控制每次调用** → `-exec {} \;`
- **文件含空格时 xargs 必须配 `-print0` + `-0`**

### 2.1.3 grep

2. `grep -nr str ./`，以当前目录为起点，递归查找所有文件，并标识包含`str`字符的行号和文件。
3. 详细选项与组合用法见 [[5. grep 文本搜索|grep 文本搜索]]。

# 4. SSH 免密登录

## 4.1 背景

每次通过 `ssh user@host` 登录远程服务器都需要输入密码，频繁操作时很麻烦。
`ssh-copy-id` 可以将本机的 SSH **公钥** 追加到远程服务器的 `~/.ssh/authorized_keys` 中，
此后 SSH 登录将自动使用密钥对完成身份验证，不再需要手动输入密码。

## 4.2 前置条件

- 本机已生成 SSH 密钥对（公钥 + 私钥）
  - 检查是否已有：`ls ~/.ssh/`，看是否存在 `id_rsa`（或 `id_ed25519`）和 `id_rsa.pub`（对应 `.pub` 文件）
  - 如果没有，先生成：`ssh-keygen -t ed25519 -C "your_email@example.com"`
    - `-t ed25519`：指定使用 Ed25519 算法（更安全、更高效）；也可用 `-t rsa -b 4096`
    - `-C`：注释标签，通常填邮箱便于识别
- 远程服务器已开启 SSH 服务，且当前知道密码（首次仍需密码来安装公钥）

## 4.3 基本用法

```bash
ssh-copy-id user@remote_host
```

- 首次执行会提示输入远程用户的密码
- 之后 `ssh user@remote_host` 即免密登录

### 指定端口

```bash
ssh-copy-id -p 2222 user@remote_host
```

### 指定使用的公钥文件

```bash
ssh-copy-id -i ~/.ssh/mykey.pub user@remote_host
```

- 不指定 `-i` 时，`ssh-copy-id` 会自动尝试 `~/.ssh/` 下默认的密钥

## 4.4 手动等价操作

`ssh-copy-id` 本质就是做了以下操作：

```bash
cat ~/.ssh/id_ed25519.pub | ssh user@remote_host "mkdir -p ~/.ssh && chmod 700 ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"
```

所以即使目标机器没有 `ssh-copy-id` 命令，也可以用这行管道命令手动完成。

## 4.5 原理简述

```
┌──────────────┐          ┌─────────────────┐
│   本机       │          │   远程服务器     │
│              │   SSH    │                 │
│  id_ed25519  │◄────────►│  authorized_keys │
│  (私钥)      │          │  (存放公钥)      │
│              │          │                 │
│  id_ed25519  │          │  匹配则免密登录  │
│  .pub (公钥) │          │                 │
└──────────────┘          └─────────────────┘
```

- 私钥（`id_ed25519`）**留在本机，绝不外传**
- 公钥（`id_ed25519.pub`）可以安全地复制到任意服务器
- SSH 通过**挑战-响应机制（challenge-response）**验证：服务器用公钥加密一个随机数，客户端用私钥解密并返回，证明持有私钥

## 4.6 常见问题

### 4.6.1 免密登录仍提示输入密码

检查文件和目录权限（SSH 对权限要求严格）：

```bash
# 远程服务器上执行
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
chmod 700 ~          # 家目录也不能对其他用户可写
```

### 4.6.2 多个密钥的管理

编辑 `~/.ssh/config` 为不同主机指定不同密钥：

```
Host myserver
    HostName 192.168.1.100
    User root
    Port 2222
    IdentityFile ~/.ssh/mykey
```

之后直接 `ssh myserver` 即可。

### 4.6.3 `ssh-copy-id` 未安装

macOS / 多数 Linux 发行版自带。若没有：
- **Ubuntu/Debian**: `apt install openssh-client`
- **CentOS/RHEL**: `yum install openssh-clients`

也可直接使用第 4.4 节的手动管道命令。

## 4.7 安全注意事项

- **私钥必须保护好**：设置文件权限 `chmod 600 ~/.ssh/id_ed25519`
- 可以考虑为私钥设置密码（passphrase）：`ssh-keygen -p`
  - 搭配 `ssh-agent` 可避免每次使用都输入 passphrase：`eval "$(ssh-agent -s)" && ssh-add ~/.ssh/id_ed25519`
- 离职或密钥泄露后，及时从远程服务器的 `authorized_keys` 中移除对应公钥
- 如遇 SSH 安全加固（如禁止密码登录），务必在确认密钥登录可用后再关闭密码认证

# 6. `tee` — 双向输出（同时输出到终端与文件）

## 6.1 简介

`tee` 从标准输入读取数据，**同时**写入标准输出（终端）和一个或多个文件。得名于管道中的 T 型三通接头——一路输入，两路输出。

```
┌─────────┐     stdin     ┌─────┐     stdout     ┌──────────────┐
│  命令源  │─────────────►│ tee │─────────────►│   终端显示    │
│(cmd / ls)│              └──┬──┘               └──────────────┘
└─────────┘                  │
                             │ 写入文件
                             ▼
                       ┌──────────┐
                       │  file.log │
                       └──────────┘
```

## 6.2 基本用法

```bash
command | tee [选项] 文件名
```

### 输出到终端 + 写文件

```bash
ls -la | tee output.txt
```
将 `ls -la` 的结果同时显示在屏幕上并保存到 `output.txt`。

### 追加而非覆盖（`-a` / `--append`）

```bash
ls -la | tee -a output.txt     # 追加到文件末尾，不覆盖原有内容
```

### 多个输出文件

```bash
ls -la | tee file1.txt file2.txt file3.txt   # 一次写入多个文件
```

### 忽略中断信号（`-i` / `--ignore-interrupts`）

```bash
command | tee -i output.txt   # 忽略 SIGINT，防止 Ctrl+C 中断
```

## 6.3 实用场景

### 场景一：编译日志存档

```bash
make 2>&1 | tee build.log
```
编译的同时看到实时输出，`build.log` 留存完整日志便于排查。

### 场景二：命令管道分支

```bash
cat access.log | tee raw.log | grep "ERROR" | tee errors.log | wc -l
```
通过多个 `tee` 形成管道分支，保留中间结果的同时继续向下传递数据：
- `cat access.log` → 原始日志
- `tee raw.log` → 保存完整日志到文件，继续传递
- `grep "ERROR"` → 过滤错误行
- `tee errors.log` → 保存错误行到文件，继续传递
- `wc -l` → 统计错误行数并输出

### 场景三：逐步写入 + 管道处理

```bash
# 边处理边保存中间结果 —— 搭配管道逐步过滤
find /var/log -name "*.log" | tee all-files.txt | head -20 | tee first20.txt | xargs grep "error"
```

### 场景四：sudo 下写入受限文件

```bash
echo "some config" | sudo tee /etc/nginx/nginx.conf > /dev/null
```

Shell 重定向 `>` 以当前用户权限打开文件，无权写入 `/etc/`；借助 `sudo tee` 让 `tee` 以 root 身份打开文件，同时用 `> /dev/null` 屏蔽终端回显（不需要 stdout 输出时）。

也可以利用 `tee` 支持**多文件**的特性，同时写入配置并保留备份：

```bash
echo "new config" | sudo tee /etc/nginx/nginx.conf /etc/nginx/nginx.conf.bak > /dev/null
```

> 对比：`sudo echo "config" > /etc/nginx/nginx.conf` 会因权限失败——`sudo` 只提升 `echo`，不提升重定向的文件打开权限。这是 **`sudo tee` 的核心使用动机**。

### 场景五：保留输出同时传递到下一个命令

```bash
# 统计行数并保留原始输出到文件
ps aux | tee process-list.txt | wc -l
```

## 6.4 常用选项汇总

| 选项 | 含义 |
|------|------|
| `-a` / `--append` | 追加模式，不覆盖已有文件内容 |
| `-i` / `--ignore-interrupts` | 忽略中断信号 |
| `-p` / `--output-error` | 设置对写错误的处理行为（`warn`/`warn-nopipe`/`exit`/`exit-nopipe`） |

## 6.5 配合 `>>` 的对比

| 方式 | 效果 | 适用场景 |
|------|------|----------|
| `cmd >> file` | 仅写入文件，终端无输出 | 静默后台任务 |
| `cmd \| tee file` | 终端显示 + 写入文件 | 需要实时观察输出 |
| `cmd \| tee -a file` | 终端显示 + 追加到已有文件 | 日志追加 |
| `cmd \| tee files > /dev/null` | 静默写入一个或多个文件 | 只想存文件，不关心回显 |

## 6.6 注意事项

- **管道阻塞**：`tee` 的输出端若有某个文件写入慢或挂起，可能阻塞整个管道——`-p` 选项可控制此行为
- **缓冲区**：当 `tee` 作为长流水线的一环时（如持续写入日志），注意磁盘 IO 可能成为瓶颈
- **与 `>>` 不等价**：`cmd >> file` 只写不显示；`cmd \| tee` 同时显示和写入，二者不是同一件事

# 7. Shell 重定向

## 7.1 概述

Shell 重定向控制命令的输入输出流向。Linux 中每个进程启动时默认打开三个文件描述符（file descriptor, FD）：

```
  ┌─────────────┐
  │   进程      │
  │             │
  │ 0 stdin ◄───┤──── 键盘 / 管道输入
  │ 1 stdout───►├──── 终端显示（正常输出）
  │ 2 stderr───►├──── 终端显示（错误信息）
  └─────────────┘
```

重定向的本质是**替换文件描述符指向的目标**——将输出指向文件而不是终端，或将文件内容作为输入喂给命令。

## 7.2 基本操作符

### `>` — 输出重定向（覆盖）

```bash
command > file     # 将 stdout 写入 file（覆盖原有内容）
```

等价于 `command 1> file`。如果文件不存在则创建，存在则截断。

```bash
ls -la > output.txt
```

### `>>` — 输出重定向（追加）

```bash
command >> file    # 将 stdout 追加到 file 末尾（不覆盖）
```

### `<` — 输入重定向

```bash
command < file     # 将 file 内容作为 command 的 stdin
```

```bash
wc -l < /etc/passwd       # 统计文件行数，不显示文件名
sort < unsorted.txt       # 从文件读取内容并排序
```

### `2>` — 错误输出重定向

```bash
command 2> file            # 仅重定向 stderr（错误信息）
```

```bash
grep "root" /etc/shadow 2> error.log    # 权限报错写入日志，不显示在终端
```

### `&>` / `2>&1` — 合并 stdout 与 stderr

```bash
# 方式一（推荐，Bash 4+）
command &> file            # stdout + stderr 同写入 file

# 方式二（通用，兼容所有 sh）
command > file 2>&1        # 将 stderr 重定向到 stdout 的目标

# 方式三（追加合并）
command &>> file           # 追加模式合并输出
command >> file 2>&1
```

> **`>&1` 不是文件名**：`2>&1` 意思是"把 FD 2 重定向到 FD 1 当前指向的位置"。`1` 前面的 `&` 告诉 Shell 这是文件描述符而非文件名。如果写成 `2>1`，Shell 会创建名为 `1` 的文件。

### `&> /dev/null` — 丢弃所有输出

```bash
command &> /dev/null       # 静默执行，不产生任何输出
```

`/dev/null` 是 Linux 的**空设备**（bit bucket / black hole），任何写入它的数据都会被丢弃。常用于丢弃不需要的输出。详见 [[6. Linux 目录结构#2.3 dev — 设备文件|Linux 目录结构：/dev]]。

### `>` 与 `>>` 对比

| 场景 | 命令 |
|------|------|
| 每次全新写入 | `cmd > file` |
| 追加到已有日志 | `cmd >> file` |
| 只存错误 | `cmd 2> error.log` |
| 全部保存 | `cmd &> all.log` |
| 全部追加保存 | `cmd &>> all.log` |
| 错误显示+只存正常 | `cmd > output.log`（stderr 仍显示） |
| 只存错误，正常丢弃 | `cmd > /dev/null 2> error.log` |
| 全丢弃 | `cmd &> /dev/null` |

## 7.3 进阶操作符

### Here Document (`<<`) — 内联输入

将一段文本直接嵌入脚本或命令中，作为标准输入：

```bash
command << delimiter
文本内容
...
delimiter
```

```bash
# 示例：多行输入传给 cat
cat << EOF
Hello World
This is a here document
EOF

# 输出：
# Hello World
# This is a here document
```

常见分隔符惯例：`EOF`、`END`、`EOL`。只要结束行以分隔符开头即可。

**禁止变量展开**：分隔符加引号，其中的 `$变量` 不会被替换：

```bash
cat << 'EOF'
当前用户: $USER     # 原样输出 $USER，不展开
EOF
```

### Here String (`<<<`) — 行内字符串当作输入

```bash
command <<< "string"
```

```bash
# 等价于 echo "string" | command
grep "error" <<< "no error found"        # 直接在字符串中搜索
bc <<< "2^10"                            # 计算 1024

# 比管道更简洁——少一个子进程开销
echo "2^10" | bc                         # 管道写法
bc <<< "2^10"                            # here string 写法
```

### `|` — 管道（Pipeline）

```bash
command1 | command2       # 将 command1 的 stdout 接到 command2 的 stdin
```

管道是重定向的扩展概念——它不再指向文件，而是指向另一个进程的输入口。

```bash
# 链式管道
cat log.txt | grep "ERROR" | sort | uniq -c | sort -rn | head -10
```

## 7.4 文件描述符操作

### 自定义文件描述符

```bash
exec 3> output.txt        # 打开 FD 3 用于写入 output.txt
echo "hello" >&3          # 写入 FD 3（即写入 output.txt）
exec 3>&-                 # 关闭 FD 3
```

```bash
exec 4< input.txt         # 打开 FD 4 用于读取 input.txt
cat <&4                   # 从 FD 4 读取
exec 4<&-                 # 关闭 FD 4
```

### 交换 stdout 与 stderr

```bash
command 3>&1 1>&2 2>&3 3>&-   # 经典 FD 交换技巧
```

分步拆解：
1. `3>&1` — FD 3 指向 FD 1 的位置（终端）
2. `1>&2` — FD 1 指向 FD 2 的位置（也是终端，但语义互换）
3. `2>&3` — FD 2 指向 FD 3 的位置（原来的 FD 1）
4. `3>&-` — 关闭 FD 3

结果：原本输出的内容去了错误通道，原本报错的内容去了输出通道。

### 复制文件描述符

```bash
command 2>&1              # FD 2 复制 FD 1 的指向（非 1 的数值）
command > file 2>&1       # 先让 FD 1 指向 file，再让 FD 2 指向 FD 1（= file）

# 顺序重要！以下不等价：
command 2>&1 > file       # 先让 FD 2 指向终端（此时 FD 1 仍是终端），再让 FD 1 指向 file
                          # 结果：stdout → file，stderr → 终端 ❌
```

## 7.5 实用组合

### 日志记录 — 同时保存并查看

```bash
# 在 tee 之前，常用方式：
command 2>&1 | tee -a app.log

# 或只记录错误到日志，正常输出仍显示
command 2>&1 | tee -a app.log | grep "FATAL"
```

### 条件销毁

```bash
# 只保留错误信息（正常输出丢弃）
command > /dev/null 2> error.log

# 对比命令差异，忽略报错行
diff file1 file2 2>/dev/null
```

### 以空输入运行交互命令

```bash
command < /dev/null       # 从空设备读取输入，命令立即读到 EOF
                          # 常用于非交互模式运行 ssh / mysql 等
```

### 批量检查文件是否存在

```bash
for f in a b c; do
  if [[ -f "$f" ]]; then
    echo "$f exists"
  fi
done > /dev/null 2>&1     # 整个循环的输出全部丢弃
```

## 7.6 与 `tee` 的分工

| 需求 | 方案 |
|------|------|
| 只存输出 | `cmd > file` / `cmd >> file` |
| 存输出 + 终端显示 | `cmd \| tee file` |
| 存全部（含错误）+ 终端显示 | `cmd 2>&1 \| tee file` |
| 一个输出到多个地方 | `tee file1 file2`（无法用裸重定向实现） |
| 权限提升写入 | `sudo tee file`（`sudo > file` 不可行） |
| 丢弃输出 | `cmd > /dev/null` |
| 只存错误 | `cmd 2> file` |

## 7.7 常见陷阱

### 陷阱一：`2>&1` 放错位置

```bash
# ✅ 正确：stdout 和 stderr 都进 file
command > file 2>&1

# ❌ 错误：只有 stdout 进 file，stderr 仍进终端
command 2>&1 > file
```

因为重定向从右到左解析——`2>&1` 在 `> file` 之前，此时 `1` 仍指向终端。

### 陷阱二：管道中的错误丢失

```bash
command1 | command2        # 只有 command1 的 stdout 进入管道
                           # command1 的 stderr 仍输出到终端
                           # command2 的错误也不会传到父 shell
```

完整捕获需要显式合并：
```bash
command1 2>&1 | command2
```

### 陷阱三：`>` 覆盖已有文件

```bash
# 危险操作——直接覆盖，不可撤销
command > important-config.conf
```

建议：养成先 `cp` 备份的习惯，或使用 `set -o noclobber`（阻止 `>` 覆盖已有文件，但仍可用 `>|` 强制覆盖）。

### 陷阱四：`>` vs `|` 的目标

```bash
# `>` 前面是文件，`|` 后面是命令
command > file             # ✅ 写入文件
command | command2         # ✅ 传给下一个命令
command > command2         # ❌ 语法没错，但会创建名为 command2 的文件
```
