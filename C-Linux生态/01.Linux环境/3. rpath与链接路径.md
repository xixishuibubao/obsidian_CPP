# 1. 运行时链接路径

Linux 动态链接器（`ld-linux.so`）在运行时按优先级搜索共享库的路径机制。

相关笔记：[[2. 动态库与静态库|动态库与静态库]]、[[B-构建与脚本/01.构建工具/1. g++|g++ 编译选项（rpath）]]、[[E-杂项/05.杂项/2. 加载与劫持|加载与劫持（dlopen）]]

## 1.1 rpath（编译时嵌入）

rpath（run-path / runtime library search path）在**编译时**通过链接器选项将路径写入 ELF 文件的 `.dynamic` 段。

```bash
# 查看已嵌入的 rpath
readelf -d your_app | grep -E 'RPATH|RUNPATH'
```

```bash
# 输出示例
0x000000000000000f (RPATH) Library rpath: [$ORIGIN:$ORIGIN/lib]
0x000000000000001d (RUNPATH) Library runpath: [$ORIGIN/lib]
```

> **DT_RPATH vs DT_RUNPATH**：两者功能相似，但搜索优先级不同（参见 1.4 节）。GCC 默认写入 `DT_RPATH`；链接器 `--enable-new-dtags` 或 `-Wl,--enable-new-dtags` 会改为写入 `DT_RUNPATH`。

`-Wl,-rpath` 的具体编译选项写法、`$ORIGIN` 用法、多路径、CMake 配置等详见：[[B-构建与脚本/01.构建工具/1. g++#2.2 运行时库路径（rpath）|g++: rpath 编译选项详解]]

### 1.1.1 rpath 的局限

- 只在编译时固定，运行时无法方便修改
- 若二进制来自第三方且未设 rpath，无法事后补救（除非用 `patchelf`）
- 交叉编译时路径容易写错（误用主机路径）

### 1.1.2 patchelf 修正 rpath

```bash
# 查看
patchelf --print-rpath your_app

# 修改（动态链接器会匹配新路径）
patchelf --set-rpath '/usr/lib:/usr/local/lib' your_app

# 删除
patchelf --delete-rpath your_app
```

## 1.2 `LD_LIBRARY_PATH`（运行时环境变量）

**运行时动态设置**，优先级最高（参见 1.4 节搜索顺序）。

```bash
# 临时设置（当前终端生效）
export LD_LIBRARY_PATH=/usr/local/lib:$LD_LIBRARY_PATH

# 仅本次运行生效
LD_LIBRARY_PATH=/opt/lib ./your_app
```

> **安全限制**：对于 `setuid`/`setgid` 程序，Linux 会忽略 `LD_LIBRARY_PATH`。

### 坑点

- 环境变量是进程级设置，不持久化到程序本身
- 影响范围过大（导出后**所有**程序都会去搜索该路径），可能引入版本冲突
- 调试时方便，生产部署不推荐

## 1.3 `ldconfig` 与缓存（系统级）

系统级搜索路径管理，通过 `/etc/ld.so.conf` 配置文件 + `ldconfig` 命令维护缓存。

```bash
# 查看配置
cat /etc/ld.so.conf

# 配置文件中 include 的目录列表
cat /etc/ld.so.conf.d/*.conf
```

```bash
# 更新缓存（需要 root 权限）
sudo ldconfig

# 查看当前缓存中的库
ldconfig -p | grep xxx
```

### 工作流程

1. 系统启动 / 安装库后运行 `ldconfig`
2. `ldconfig` 扫描 `/etc/ld.so.conf` 中配置的目录
3. 解析 `.so` 文件名中的 SONAME，生成映射缓存 `/etc/ld.so.cache`
4. 动态链接器在 1.4 节所述步骤中命中缓存

### 添加自定义路径

```bash
# 创建配置文件
echo '/usr/local/myapp/lib' | sudo tee /etc/ld.so.conf.d/myapp.conf

# 更新缓存
sudo ldconfig
```

### 坑点

- `ldconfig` 只在系统级路径和配置目录中生效，对 `LD_LIBRARY_PATH` 和 rpath 无影响
- 嵌入式开发中常忘记跑 `ldconfig`，导致 `.so` 存在于目录但找不到

## 1.4 搜索优先级（完整顺序）

动态链接器按照以下顺序搜索共享库（**高优先级优先**）：

| 优先级 | 机制 | 说明 |
|--------|------|------|
| **1（最高）** | `DT_RPATH` | ELF 中硬编码的搜索路径（旧式，**已弃用**） |
| **2** | `LD_LIBRARY_PATH` | 环境变量，运行时设置 |
| **3** | `DT_RUNPATH` | ELF 中硬编码的搜索路径（新式，推荐） |
| **4** | `/etc/ld.so.cache` | `ldconfig` 缓存的系统库 |
| **5（最低）** | `/lib`、`/usr/lib` | 默认系统库目录 |

> **关键区别 DT_RPATH vs DT_RUNPATH**：
> - `DT_RPATH` 优先级高于 `LD_LIBRARY_PATH`
> - `DT_RUNPATH` 优先级**低于** `LD_LIBRARY_PATH`，允许环境变量覆盖
> - 现代 Linux 发行版倾向于使用 `DT_RUNPATH`（通过 `-Wl,--enable-new-dtags`）

### 优先级对调试的影响

```bash
# 若程序有 DT_RUNPATH，以下环境变量会覆盖它
LD_LIBRARY_PATH=/debug/lib ./your_app

# 若程序有 DT_RPATH（而非 RUNPATH），LD_LIBRARY_PATH 无效
# 必须 patchelf 或在链接时解决
```

## 1.5 dlopen 的搜索路径

`dlopen()` 加载动态库时的搜索规则（与普通程序的加载略有不同）：

```c
void* handle = dlopen("libfoo.so", RTLD_LAZY);
```

1. 若路径包含 `/` → 直接按路径加载（相对路径相对于当前工作目录）
2. 若不包含 `/` → **不搜索 DT_RPATH/DT_RUNPATH**，只搜索：
   - `LD_LIBRARY_PATH`
   - `/etc/ld.so.cache`
   - `/lib`、`/usr/lib`

> 若希望 `dlopen` 使用 rpath 路径，需在调用 `dlopen("libfoo.so")` 前先通过 `LD_LIBRARY_PATH` 或系统库缓存中包含该路径。

相关笔记：[[E-杂项/05.杂项/2. 加载与劫持|加载与劫持（dlopen/dlsym 详解）]]

## 1.6 验证方法

```bash
# 查看程序依赖哪些库
ldd your_app

# 查看 ELF 动态段（含 RPATH/RUNPATH/NEEDED）
readelf -d your_app

# 查看动态链接器搜索过程（调试）
LD_DEBUG=libs your_app
```

### `ldd` 注意事项

- `ldd` 会**实际执行**目标程序或动态链接器，对不可信程序有安全隐患
- 安全替代方案：`objdump -p your_app | grep NEEDED` 或 `readelf -d your_app`
- 若 ldd 输出显示 `not found`，按 1.4 节优先级逐一排查

## 1.7 方案选择指南

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 自编译程序/库 | **rpath + `$ORIGIN`** | 自我包含，移动目录不影响 |
| 调试/开发 | **`LD_LIBRARY_PATH`** | 无需重新编译，灵活切换 |
| 系统级库安装 | **`ldconfig`** | 全局生效，标准做法 |
| 第三方二进制需修复 | **`patchelf`** | 不改代码，修改 ELF 的 rpath |
| 嵌入式/打包部署 | **rpath + `$ORIGIN`** | 避免依赖 `ldconfig` 和环境变量 |
