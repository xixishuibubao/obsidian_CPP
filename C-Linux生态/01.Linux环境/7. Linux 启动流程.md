# 1. Linux 启动流程概述

## 1.1 启动阶段总览

Linux 系统的启动过程是一系列阶段递进的过程，从按下电源到用户登录界面，主要分为以下阶段：

| 阶段 | x86_64 | ARM64 |
|------|--------|-------|
| **Firmware** | BIOS / UEFI | ATF (Trusted Firmware) / UEFI |
| **Bootloader** | GRUB 2 / syslinux | U-Boot / GRUB / TF-A (BL2) |
| **Kernel** | bzImage (vmlinuz) | Image / Image.gz |
| **Init** | systemd / initramfs | systemd / initramfs |

## 1.2 通用启动流程

Source: [kernel.org — Linux Booting Process Overview]

```text
Power On
   │
   ▼
Firmware Initialization
   │
   ▼
Bootloader (Stage 1 → Stage 2)
   │
   ▼
Kernel Decompression & Setup
   │
   ▼
Initramfs → Root Filesystem
   │
   ▼
Init (PID 1) → Userspace
```

x86_64 和 ARM64 在 Firmware 和 Bootloader 阶段有本质差异，而在内核初始化之后的流程趋于一致。

# 2. x86_64 启动流程

## 2.1 Firmware 初始化

### 2.1.1 Legacy BIOS 启动

BIOS (Basic Input/Output System) 是传统的 x86 固件接口：

1. **Power-On Self Test (POST)** — 硬件自检，检测 CPU、内存、基本 I/O
2. **BIOS Interrupt Vector Table** — 建立实模式中断向量表（IVT）
3. **启动设备扫描** — 按 BIOS 设置的启动顺序扫描磁盘
4. **Master Boot Record (MBR)** — 读取磁盘第一个扇区（512 bytes），加载到 `0x7C00`
5. **MBR 代码执行** — MBR 中 446 bytes 的引导代码跳转到 active partition 的 Volume Boot Record (VBR)

BIOS 的局限：
- 16 位实模式，仅访问 1 MiB 地址空间
- MBR 分区表仅支持 2 TiB 以下磁盘
- 无可扩展的固件运行时服务

### 2.1.2 UEFI 启动

UEFI (Unified Extensible Firmware Interface) 是现代 x86_64 的标准固件接口：

1. **SEC (Security Phase)** — 初始化临时内存（cache-as-RAM），验证固件签名
2. **PEI (Pre-EFI Initialization)** — 发现并初始化系统内存
3. **DXE (Driver Execution Environment)** — 加载 EFI 驱动，初始化 PCIe、SATA、NVMe 等设备
4. **BDS (Boot Device Selection)** — 扫描 EFI System Partition (ESP)，读取 NVRAM 中的 `Boot####` 变量
5. **EFI Application 加载** — 加载 bootloader（如 `\EFI\GRUB\grubx64.efi`）

UEFI 关键特性：
- 32/64 位保护/长模式
- GPT 分区表支持 2 TiB+
- Secure Boot — 签名验证 chain of trust
- Runtime Services — `ExitBootServices()` 后仍可调用

## 2.2 Bootloader 阶段

### 2.2.1 GRUB 2

GRUB 2 (Grand Unified Bootloader v2) 是目前 Linux x86_64 最主流的 bootloader：

1. **boot.img** — 嵌入 MBR 或 GPT 的专用分区，跳转到 core.img
2. **core.img** — 包含文件系统驱动（ext4、btrfs、xfs），读取 `/boot/grub/grub.cfg`
3. **grub.cfg** — 展示启动菜单，选择 kernel + initramfs
4. **Linux Kernel 加载** — 调用 `kexec` 系统调用风格的 `boot` 命令，将控制权交给内核

GRUB 执行的关键操作：
```bash
# grub.cfg 中启动 Linux 的典型条目
menuentry 'Linux' {
    linux   /vmlinuz-6.1.0 root=/dev/sda2 ro quiet
    initrd  /initramfs-6.1.0.img
}
```

### 2.2.2 x86 Linux Boot Protocol

Kernel 在 `Documentation/x86/boot.rst` 中定义了 boot protocol：

1. **实模式头部 (Real-Mode Header)** — 内核镜像前 512 bytes 包含 setup header
2. **setup.bin** — 实模式初始化代码（`arch/x86/boot/header.S`）
3. **压缩内核 (vmlinuz)** — 从 `setup` 之后的偏移量开始，GZIP/XZ/ZSTD 压缩

内核启动约定的寄存器状态：
```c
// x86 Boot Protocol 关键字段 (boot_params)
struct boot_params {
    __u8  screen_info[64];      // 屏幕参数
    __u8  apm_bios_info[20];    // APM BIOS 信息
    __u8  efi_info[32];         // UEFI 信息
    __u64 efi_systab;           // EFI System Table
    __u32 efi_signature;        // EFI 签名（"ELiT"）
    // ... 更多字段
};
```

## 2.3 内核初始化 (x86_64)

### 2.3.1 实模式 → 保护模式 → 长模式

内核入口点在 `arch/x86/boot/header.S`，代码执行路径：

1. **`_start` (实模式)** — 16 位代码，设置段寄存器
2. **`startup_32`** — 切换到保护模式（32-bit），启用分段和分页
3. **`startup_64`** — 切换到 IA-32e 长模式（64-bit），设置 4 级页表
4. **`extract_kernel`** — 解压内核到 `CONFIG_PHYSICAL_START` 地址
5. **`x86_64_start_kernel`** — 64 位内核入口点

### 2.3.2 核心内核初始化

```c
// init/main.c — 内核入口点
start_kernel()
├── setup_arch()          // 架构特定初始化
│   ├── early_idt_setup() // 早期中断描述符表
│   ├── e820__memory_setup() // 物理内存布局
│   ├── init_mem_mapping()   // 初始化内核页表
│   ├── x86_init.pci.arch_init() // PCI 初始化
│   ├── apic_boot_setup()    // APIC 初始化
│   └── acpi_boot_init()     // ACPI 表解析
├── trap_init()           // 中断向量表安装
├── mm_init()             // 内存管理子系统
├── sched_init()          // 调度器初始化
├── time_init()           // 时钟子系统
├── console_init()        // 控制台初始化
├── vfs_caches_init()     // VFS 缓存初始化
├── rest_init()           // 收尾：启动 PID 1
│   ├── kernel_thread(kernel_init) // 创建 init 进程
│   └── cpu_startup_entry()       // idle 循环
└── ...
```

## 2.4 Init 进程与用户态

`kernel_init()` 执行：
1. 加载 initramfs（initrd/initramfs 内存文件系统）
2. 执行 `/init` 脚本或二进制
3. initramfs 加载必要驱动和根文件系统
4. `switch_root` — 切换到真实根文件系统
5. 启动 PID 1 — 通常是 systemd

```text
kernel_init() → initramfs → switch_root → systemd
                                              ├── default.target
                                              ├── basic.target
                                              ├── multi-user.target
                                              └── graphical.target
```

# 3. ARM64 启动流程

## 3.1 Firmware 阶段

### 3.1.1 ARM Trusted Firmware (TF-A)

ARM64 的固件架构基于 ARM 的 **Power State Coordination Interface (PSCI)**，由 TF-A 实现：

| Exception Level | 名称 | 功能 |
|:---:|------|------|
| **EL3** | Secure Monitor | TF-A BL1/BL2/BL31 — 安全监控、PSCI |
| **EL2** | Hypervisor | KVM/ Xen — 虚拟化扩展 |
| **EL1** | OS Kernel | Linux 内核 — 特权 OS |
| **EL0** | Userspace | 应用程序 — 非特权 |

TF-A 启动序列（ARM Trusted Board Boot）：

```text
Boot ROM (芯片内置)
   │
   ▼
BL1 (AP Trusted ROM) — 初始信任根，验证 BL2
   │
   ▼
BL2 (Trusted Boot Firmware) — 平台初始化，加载 BL31 + BL33
   │
   ▼
BL31 (EL3 Runtime Firmware) — PSCI 运行时服务，SMC 处理
   │
   ▼
BL33 (Non-Trusted Firmware) — UEFI / U-Boot / 直接 Kernel
```

> BL33 即 **Non-Trusted World Bootloader**，可以是 UEFI、U-Boot 甚至直接是 Linux Kernel。

### 3.1.2 UEFI on ARM64

ARM64 同样支持 UEFI（特别是 Server 领域和 Microsoft 认证设备）：

- **UEFI + ACPI** — 企业/服务器标准（如 Ampere Altra、AWS Graviton）
- **UEFI + Device Tree** — 嵌入式标准（如 Raspberry Pi 的 UEFI 固件）

ARM64 UEFI 启动需遵守 **UEFI 2.x + ACPI 6.x** 规范，与 x86 共享 EFI 运行时服务。

## 3.2 Bootloader 阶段

### 3.2.1 U-Boot

U-Boot (Das U-Boot) 是嵌入式 ARM64 最常用的 bootloader：

```text
U-Boot 执行流程：
1. SPL (Secondary Program Loader) — 初始化 DDR，加载完整 U-Boot
2. U-Boot proper — 驱动初始化（MMC、NAND、网络）
3. bootcmd — 执行启动命令
4. booti — 启动 ARM64 Linux 内核
```

U-Boot 典型启动配置：
```bash
# U-Boot env
setenv kernel_addr_r 0x80080000
setenv fdt_addr_r 0x88000000
setenv bootargs root=/dev/mmcblk0p2 console=ttyS0,115200

# 启动 ARM64 Kernel
load mmc 0:1 ${kernel_addr_r} Image
load mmc 0:1 ${fdt_addr_r}    board.dtb
booti ${kernel_addr_r} - ${fdt_addr_r}
```

> `booti` 是 ARM64 专用的内核启动命令（对应 ARM32 的 `bootz`）。

### 3.2.2 ARM64 Linux Boot Protocol

ARM64 内核的 boot protocol 定义在 `Documentation/arm64/booting.rst`：

**进入内核前的寄存器约定**：
```c
// x0 — Device Tree Blob (DTB) 物理地址（必须）
// x1 — 保留（通常为 0）
// x2 — 保留（通常为 0）
// x3 — 保留（通常为 0）
// PC — 内核入口地址

// 系统寄存器要求：
//  MMU = off（关闭）
//  D-cache = off（数据缓存关闭）
//  I-cache = on 或 off（指令缓存任意）
//  x0 指向 DTB 内存区域（8-byte aligned）
```

**内核放置约束**：
- 内核必须在 2 MiB 对齐的地址
- 内核 Image 被直接加载到内存（非压缩格式是纯二进制）
- 如果是 Image.gz，bootloader 需先解压

## 3.3 内核初始化 (ARM64)

```c
// arch/arm64/kernel/head.S — 内核早期入口
primary_entry()
├── preserve_boot_args()    // 保存 x0-x3（DTB 地址等）
├── el2_setup()             // 检测并配置 EL2→EL1 降级
├── __cpu_setup()           // 缓存/MMU 配置
├── __primary_switch()      // 启用 MMU，跳转 C 代码
├── start_kernel()          // 架构无关内核入口
│   ├── setup_arch()        // ARM64 特定
│   │   ├── setup_machine_fdt()     // 解析 DTB
│   │   ├── acpi_boot_table_init()  // ACPI 初始化（若存在）
│   │   ├── paging_init()           // 页表初始化
│   │   ├── cpu_read_bootcpu_ops()  // CPU 拓扑检测
│   │   └── arm64_memblock_init()   // 内存布局
│   ├── trap_init()         // 异常向量表
│   ├── mm_init()
│   ├── sched_init()
│   ├── ...
│   └── rest_init()
```

ARM64 特有初始化要点：
- **`el2_setup`** — 检测是否从 EL2（虚拟化层）启动，初始化 Hypervisor 配置
- **`setup_machine_fdt`** — 解析 Flattened Device Tree（FDT），获取硬件拓扑
- **`acpi_boot_table_init`** — 如果固件提供 ACPI 表，可选择使用 ACPI 替代 DTB

## 3.4 硬件描述方式

ARM64 有两个并行的硬件描述机制，这是与 x86_64 最显著的差异之一。

### 3.4.1 Device Tree (DTB)

Device Tree 是嵌入式 ARM64 的标准硬件描述方式：

```dts
/dts-v1/;

/ {
    compatible = "vendor,board-name";
    #address-cells = <2>;
    #size-cells = <2>;

    memory@80000000 {
        device_type = "memory";
        reg = <0x0 0x80000000 0x0 0x80000000>;  // 2 GiB @ 2 GiB
    };

    chosen {
        bootargs = "console=ttyS0,115200 root=/dev/mmcblk0p2";
        stdout-path = "serial0:115200n8";
    };

    cpus {
        #address-cells = <1>;
        #size-cells = <0>;

        cpu@0 {
            compatible = "arm,cortex-a72";
            device_type = "cpu";
            reg = <0x0>;
            enable-method = "psci";
        };
    };
};
```

- DTB 由内核直接解析，无需 ACPI 表
- Linux 维护了一个巨大的 `arch/arm64/boot/dts/` 设备树仓库
- UEFI 提供 Device Tree 作为 ACPI 的替代（`EFI_DT_FIXUP_PROTOCOL`）

### 3.4.2 ACPI on ARM64

ARM64 的 ACPI 源自 UEFI 规范（ACPI 6.x 开始支持 ARM）：
- **DSDT/SSDT** — 描述设备拓扑和配置
- **MADT (Multiple APIC Description Table)** — CPU、GIC 中断控制器信息
- **GTDT (Generic Timer Description Table)** — 通用定时器信息
- **IORT (I/O Remapping Table)** — IOMMU/SMMU 配置

ACPI 主要用于 ARM64 服务器领域，嵌入式场景仍以 Device Tree 为主。

# 4. x86_64 vs ARM64 关键差异对比

## 4.1 固件与硬件初始化

| 维度 | x86_64 | ARM64 |
|:-----|:-------|:------|
| **初始代码位置** | BIOS/UEFI ROM（主板 Flash） | Boot ROM（SoC 片内 ROM） |
| **内存初始化** | BIOS PEI 阶段自动探测 SPD | Bootloader/固件完成 DDR 初始化 |
| **标准固件** | UEFI（必须） | TF-A + UEFI / U-Boot（可选） |
| **调试手段** | BIOS POST 码、串口（少见） | JTAG、串口（常见）、TrustZone |
| **厂商定制** | 少（标准化程度高） | 多（SoC 厂商高度定制） |

## 4.2 硬件发现机制

| 维度 | x86_64 | ARM64 |
|:-----|:-------|:------|
| **标准发现** | ACPI（必须） | Device Tree（嵌入式） / ACPI（服务器） |
| **PCIe 枚举** | ACPI + PCI 配置空间 MMIO | ACPI IORT + PCIe ECAM / DTB PCIe 节点 |
| **中断模型** | APIC (I/O APIC + Local APIC) | GIC (v2/v3/v4) — Distributor + Redistributor |
| **定时器** | LAPIC Timer + HPET + TSC | Generic Timer (CNTFRQ) |
| **电源管理** | ACPI S-states / C-states / P-states | PSCI（通过 SMC 调用）+ SCP firmware |

## 4.3 安全架构

| 维度 | x86_64 | ARM64 |
|:-----|:-------|:------|
| **安全启动** | UEFI Secure Boot | Trusted Board Boot (TF-A) |
| **信任根** | UEFI Key Enrollment Key (KEK) | Boot ROM（不可修改） |
| **运行时安全** | Boot Guard / TXT (SMX) | TrustZone (EL3/S-EL1) |
| **虚拟化安全** | IOMMU (VT-d) | SMMU (System MMU) |
| **密钥存储** | TPM (fTPM / dTPM) | OP-TEE / TrustZone |

## 4.4 特权级模型

```text
x86_64 Rings                  ARM64 Exception Levels

Ring 0 (Kernel)      ←→    EL1 (OS Kernel)
Ring 1 (Drivers)     ←→    (无对应，驱动在 EL1)
Ring 2 (???)
Ring 3 (Userspace)   ←→    EL0 (Userspace)
(无对应)             ←→    EL2 (Hypervisor)
(无对应)             ←→    EL3 (Secure Monitor)
```

**关键差异**：
- ARM64 的 EL2（虚拟化）和 EL3（安全监控）是 x86 没有的硬件特权级
- x86 的 Ring 1/2 在现代 Linux 中几乎不使用
- ARM64 的 Secure World（S-EL1/S-EL0）提供硬件隔离的可信执行环境

## 4.5 启动协议

| 维度 | x86_64 | ARM64 |
|:-----|:-------|:------|
| **内核镜像格式** | bzImage (setup + compressed vmlinux) | Image (PE/COFF) 或 Image.gz |
| **入口要求** | 实模式头部 → 保护模式 → 长模式 | 直接进入 EL1（或 EL2→EL1） |
| **参数传递** | `boot_params` 结构体 | DTB 物理地址 (x0) |
| **解压方式** | 内核自带解压代码 (arch/x86/boot/compressed/) | 可选 Image.gz → bootloader 解压 |
| **内存位置** | 约定 1 MiB 以下（real mode）/ 1 MiB+（protected） | 任意 2 MiB 对齐地址 |
| **页表状态** | 结束时 MMU on → off → on | 入口时 MMU off → 内核自行开启 |

## 4.6 中断与异常处理

| 维度 | x86_64 | ARM64 |
|:-----|:-------|:------|
| **异常向量表** | IDT (Interrupt Descriptor Table) | VBAR_EL1 (Vector Base Address Register) |
| **中断控制器** | APIC (xAPIC / x2APIC) | GIC (Generic Interrupt Controller) |
| **中断类型** | IRQ / NMI / SMI | IRQ (Group 0/1) / FIQ / SGI |
| **IPI 实现** | ICR (Interrupt Command Register) | SGI / PPI via GIC |
| **SMC/HVC** | VMMCALL (VMX Exit) | SMC (EL3) / HVC (EL2) |

# 5. 实用调试与观察

查看当前系统启动日志：

```bash
# 查看内核日志（dmesg 可观察启动过程）
dmesg | grep -E 'Command line|Kernel command line'
dmesg | grep -E 'ACPI.*tables|ACPI.*MADT|ACPI.*GTDT'

# 查看启动时间（systemd-analyze）
systemd-analyze
systemd-analyze blame

# 查看 UEFI 变量（x86_64 + ARM64 UEFI）
efibootmgr -v    # 查看启动管理器条目
ls -l /sys/firmware/efi/

# 查看 Device Tree（ARM64）
ls /sys/firmware/devicetree/base/
cat /sys/firmware/devicetree/base/model

# 查看 GIC 版本（ARM64）
cat /proc/interrupts | head -5
dmesg | grep GIC
```

判断系统架构：
```bash
uname -m                      # x86_64 / aarch64
cat /sys/firmware/efi         # 是否存在（判断是否 EFI 启动）
ls /sys/firmware/             # 查看固件类型（efi / devicetree）
```

# 6. 总结

x86_64 和 ARM64 的 Linux 启动流程在**内核初始化之后高度一致**，差异集中在：

1. **固件层** — x86 依赖标准化 UEFI，ARM64 则存在多种固件方案（TF-A + U-Boot / UEFI / 纯 Device Tree）
2. **硬件描述** — x86 强制 ACPI，ARM64 按场景选择 Device Tree（嵌入式）或 ACPI（服务器）
3. **特权模型** — ARM64 额外提供 EL2（虚拟化）和 EL3（安全监控）硬件特权级
4. **安全启动** — x86 的 UEFI Secure Boot vs ARM64 的 Trusted Board Boot + TrustZone
5. **中断架构** — x86 的 APIC 演进体系 vs ARM64 的 GIC 分布式中断控制器
6. **启动协议** — x86 的实模式→保护模式→长模式三段跳转 vs ARM64 的直接 EL1 入口

随着 ARM64 服务器（Ampere、Graviton、Grace）的普及和 UEFI + ACPI 在 ARM64 上的标准化，两者的启动流程正在趋同。UEFI + ACPI 已成为 AArch64 服务器的事实标准，与 x86_64 共享大量固件基础设施。
