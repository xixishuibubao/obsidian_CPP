# 1. FITS 天文图像与后处理

## 【核心精简提纲】

### 记忆主干

曝光完成后 RAW 经 `raw_to_fits` 写入带标准 Header 的 FITS 科学文件，可选经外部二进制生成预览 JPG。寄存器换算（Excel/V4L2）与 FITS Header 元数据是不同层。

### 核心要点清单

- 数据流：`.so` 写 BG12 RAW → notify → `raw_to_fits` → 可选 JPG 后处理 → IPC 通知 App
- FITS = Header 键值对 + 2D 像素；App/后期软件依赖 `EXPTIME`、`GAIN`、`DATE-OBS` 等
- `RawFitsFileBuffer` 五槽环形缓冲，避免每次曝光路径冲突
- `libcfitsio.so` 经 ctypes `CDLL` 调用，非 pip 包，须交叉编译进 DriverSO
- JPG 后处理用 `subprocess.Popen` + 文件锁，同一相机同时只跑一个
- `wait_for_device_event` 等 datagram notify 的 `Finished` 状态
- 注意仓库中 `after_exposure_process` 部分被注释，JPG 推送可能未启用

### 关键结论 / 流程

`SingleExposure`：`run_sync` 曝光 → `wait_for_device_event` → `raw_to_fits`（`save=True` 时写正式路径）→ event 上报。

## 1.1 FITS 是什么

**FITS**（Flexible Image Transport System）是天文学常用的图像容器：

- **Header**：键值对（曝光时间、增益、日期、望远镜名…）
- **Data**：通常是 2D 像素数组（本项目 BG12 → 16bit 存盘）

App/后期软件（PixInsight、Siril 等）依赖标准 Header 字段。

## 1.2 本项目数据流

```plain
CMOS285 .so  start_exposure(ms, raw_path)
    → 写 BG12 RAW 文件
    → notify Executor

SingleExposure.run()
    → raw_to_fits()           RAW + Header → .fits
    → (可选) after_exposure_process()  → raw2jpg 二进制 → .jpg
    → IPC event 通知 App
```

核心代码：

| 模块 | 文件 |
|------|------|
| 任务编排 | `APIServer/Eexcutor/FixedScripts/SingleExposure.py` |
| RAW→FITS | `.../Functions/FitsRelatedProcess.py` |
| CFITSIO 封装 | `SupportLib/fits.py` |
| 路径/Header 类型 | `SharedParameter/AfterProcessDatatype.py` |
| JPG 后处理 | `SupportLib/AfterExposureProcessPopen.py` |

## 1.6 libcfitsio.so 与 ctypes

Python 不直接解析 RAW，而是：

1. **`save_to_fits`**（`SupportLib/fits.py`）用 **ctypes** 调 NASA CFITSIO：

```python
cfitsio = CDLL('DriverSO/libcfitsio.so')
cfitsio.ffinit(...)   # 创建 FITS
cfitsio.ffpky(...)   # 写 Header 关键字
cfitsio.ffppr(...)   # 写像素数据
cfitsio.ffclos(...)
```

2. RAW 像素可能经 **raw2fits 专用 .so**（`SupportLib/pyfits.py` / DriverSO 内模块）读取二进制再写入 FITS。

交叉编译 cfitsio 见 `DriverSO/ReadMe.md` 与 [[6. 交叉编译与DriverSO部署]]。

## 1.7 JPG 后处理（子进程 + 锁）

```python
# AfterExposureProcessPopen.py
subprocess.Popen([
    str(raw_2_jpg_command), camera_type, lock_file, raw_path, jpg_path, stretch
], start_new_session=True)
```

| 机制 | 作用 |
|------|------|
| `process_{camera_type}.lock` | 同一相机同时只跑一个后处理 |
| `open('x')` 原子创建 | 防重复 Popen |
| `start_new_session=True` | 子进程独立于父进程信号 |

`server.ini` 里 `raw_2_jpg` / `fits_2_jpg` 指向 **SupportLibReplace** 编译的二进制（见 DriverSO ReadMe）。

## 1.9 与天文软件生态

| Header 键 | 含义 |
|-----------|------|
| `EXPTIME` / `EXPOSURE` | 曝光秒数 |
| `GAIN` / `OFFSET` | 传感器参数 |
| `BAYERPAT` | 彩色相机拜耳模式 |
| `DATE-OBS` | UTC 观测时间 |

规格与 Excel「V2修正hts」负责 **曝光寄存器**；FITS Header 负责 **文件元数据**，二者不同层。

## 1.11 典型错误与边界

| 误区 | 后果 |
|------|------|
| 混淆 V4L2 寄存器与 FITS Header | 曝光秒数写对但文件元数据缺字段 |
| `save=False` 却期望正式目录有 FITS | 只写环形缓冲 `fits_file_path` 槽位 |
| 未部署 `raw_2_jpg` 二进制 | JPG 后处理静默失败 |

**反例**：在 PC 用 pip `astropy` 替代板端 `libcfitsio.so` — 板端无 pip 环境，须 ctypes 调交叉编译库。

**正例**：板端 `grep raw_to_fits executor.log` + PC 上 `astropy.io.fits` 验 Header。
