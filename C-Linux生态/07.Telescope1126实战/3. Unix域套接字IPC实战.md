# 1. Unix 域套接字 IPC 实战（Telescope1126 案例）

本项目用 **Unix Domain Socket（UDS）** 做两类 IPC，是对 [[C-Linux生态/02.Linux系统编程/6. 进程间通信|进程间通信]] 中「Unix 域套接字」行的直接工程化应用。

前置：[[2. Server双进程架构]] · 源码：`Telescope1126Server/APIServer/ipc_protocol.py`

## 1.1 两条 Socket 通道

| 路径 | 类型 | 方向 | 用途 |
|------|------|------|------|
| `/tmp/executor_ipc.sock` | **Stream**（SOCK_STREAM） | 双向 | 主服务 ↔ Executor：注册、心跳、下发任务、事件上报 |
| `/tmp/executor_notify.sock` | **Datagram**（SOCK_DGRAM） | 单向通知 | 相机 `.so` / 驱动 → Executor：曝光完成 notify |

```python
# ipc_protocol.py
IPC_SOCK_PATH           = "/tmp/executor_ipc.sock"
DEVICE_NOTIFY_SOCK_PATH = "/tmp/executor_notify.sock"
```

为什么用 `/tmp/`：板端可写、重启可清理；**Windows 无 UDS**，故本地 Windows 只能读代码，不能完整跑 IPC。

## 1.2 Stream 通道：JSON-Lines 协议

**帧格式**：一行一个 JSON，以 `\n` 结尾（JSON Lines / NDJSON）。

```python
def make_msg(msg_type: str, **kwargs) -> bytes:
    payload = {"type": msg_type, "ts": time.time(), **kwargs}
    return (json.dumps(payload) + "\n").encode()
```

### 1.2.1 消息方向

**Executor → 主服务**

| type | 常量 | 含义 |
|------|------|------|
| `register` | `REGISTER` | 子进程启动注册 |
| `heartbeat` | `HEARTBEAT` | 周期心跳（约 5s） |
| `event` | `EVENT` | 任务生命周期事件 |
| `sync_snapshot` | `SYNC_SNAPSHOT` | 重连后状态同步 |

**主服务 → Executor**

| type | 常量 | 含义 |
|------|------|------|
| `cmd` | `CMD` | 如下发 `start_task` |
| `startup` | `STARTUP` | 启动参数 |
| `heartbeat` | `HB_REQ` | 请求心跳（同名不同向） |

**event 子类型**（`EV_*`）：`task_started`、`task_progress`、`task_finished`、`task_failed`、`task_exposure_update` 等。

### 1.2.2 典型时序

```plain
Executor 启动
    → register（带 snapshot、queue_depth）
    → loop: heartbeat 每 5s
主服务 TaskStartHandler
    → cmd start_task
Executor TaskRunner
    → event task_started
    → … 曝光 …
    → event task_finished
主服务 StateStore 更新 + WsHub 广播给客户端
```

主服务 `HorivisionAppServer._hb_watchdog`：**15 秒**无心跳则标记 `executor_online = False`。

## 1.3 Datagram 通道：设备异步通知

科学曝光在 C++ 层可能是 **非阻塞**：`start_exposure()` 立即返回，完成后向 datagram socket 发 JSON。

```python
def make_device_notify(device_id: str, event_type: str, **kwargs) -> bytes:
    payload = {
        "state": event_type,      # Finished / Failed / Connected ...
        "device_id": device_id,
        "ts": time.time(),
        **kwargs
    }
    return json.dumps(payload).encode()
```

`executor.py` 中 **DeviceNotifyListener** 绑定 `DEVICE_NOTIFY_SOCK_PATH`，收到后唤醒等待中的任务逻辑。

```plain
CMOS285CAMERAMain.so  start_exposure()
    → V4L2 抓 RAW 完成
    → sendto(/tmp/executor_notify.sock, JSON)
    → Executor DeviceNotifyListener
    → TaskRunner 继续 FITS 后处理
```

这与 [[C-Linux生态/02.Linux系统编程/6. 进程间通信#1.2 IPC 机制全景|IPC 全景表]] 中 UDS **字节流 vs 消息/datagram** 的区分一致。

## 1.4 与 vault 通用 IPC 笔记的对照

| 通用概念 | 本项目实例 |
|----------|------------|
| `socket(AF_UNIX, SOCK_STREAM)` | `IpcServer` / `IpcSession` |
| `socket(AF_UNIX, SOCK_DGRAM)` | 设备 notify |
| 应用层组帧 | JSON + 换行符 |
| 心跳 / 看门狗 | HEARTBEAT + 15s timeout |
| 状态同步 | REGISTER / SYNC_SNAPSHOT |

未使用：共享内存传大图（RAW 走文件路径）；管道（单向且不适合双向 RPC）。

## 1.5 最小 Python 示例（理解 UDS Stream）

```python
# 仅 Linux 可运行；帮助理解 ipc_protocol 传输层
import socket
import json

SOCK = "/tmp/demo_ipc.sock"

# --- server ---
def run_server():
    srv = socket.socket(socket.AF_UNIX, socket.SOCK_STREAM)
    srv.bind(SOCK)
    srv.listen(1)
    conn, _ = srv.accept()
    line = conn.recv(4096).decode()
    print("recv:", json.loads(line))
    conn.sendall((json.dumps({"type": "ack"}) + "\n").encode())
    conn.close()
    srv.close()

# --- client ---
def run_client():
    cli = socket.socket(socket.AF_UNIX, socket.SOCK_STREAM)
    cli.connect(SOCK)
    cli.sendall((json.dumps({"type": "hello", "ts": 1.0}) + "\n").encode())
    print("reply:", cli.recv(4096).decode())
    cli.close()
```

本项目在 Tornado/asyncio 层用 **非阻塞** 方式读写同一抽象。
