# 1. Server 双进程架构

**Telescope1126Server** 采用「**主服务 + 任务执行器**」双进程模型：主进程负责网络 I/O 与状态广播，子进程 **Executor** 串行执行拍摄任务（硬件同一时刻只能跑一个曝光）。

前置：[[1. 1126B项目总览与学习路线]] · 关联：[[3. Unix域套接字IPC实战]] · [[C-Linux生态/02.Linux系统编程/1. 进程管理|进程管理]]

## 1.1 两个入口文件

| 文件 | 角色 | 监听/通信 |
|------|------|-----------|
| `HoriVisionDriverServer.py` | 主服务（Tornado Application） | HTTP/WebSocket **:8999** |
| `executor.py` | 任务 worker（子进程） | Unix Socket → `/tmp/executor_ipc.sock` |

主进程启动时会由 `ExecutorSupervisor`（在 `APIServer/ServerState.py`）**拉起并监控** `executor.py`，类似「看门狗 + 重启策略」。

## 1.2 主服务内部模块

```plain
HoriVisionDriverServer.py
    │
    ├── HorivisionAppServer
    │       ├── StateStore          内存状态（Executor 在线、任务历史、曝光结果）
    │       ├── WsHub               WebSocket 广播
    │       ├── IpcServer           Unix Socket 服务端（收 Executor 消息）
    │       └── ExecutorSupervisor  管理 executor 子进程
    │
    ├── APIServer/http_interfaces/  REST API（OTA、文件、任务、启动）
    ├── APIServer/WebsocketHandler.py   /ws/device/
    ├── APIServer/HVDevices/        设备 connect/disconnect、驱动路由
    └── APIServer/BeforeStartScript.py  启动前按 hardware.ini 连接相机
```

## 1.3 为什么要把 Executor 拆成子进程

| 原因 | 说明 |
|------|------|
| **硬件互斥** | 相机曝光、RAW 写入、后处理占满时间；必须 **FIFO 串行**（`TaskRunner`） |
| **故障隔离** | 驱动崩溃时可重启 Executor，而不必整站重启 Tornado |
| **阻塞隔离** | 底层 `.so` 可能阻塞；放在子进程避免卡住 HTTP 事件循环 |
| **清晰边界** | 网络层（Tornado）与硬件控制层（Executor + Driver）职责分离 |

这与 [[D-系统与架构/03.并发与内存/3. 线程与异步|线程与异步]] 中「I/O 与计算分离」的思路一致，这里用 **进程** 而不是线程，隔离更强。

## 1.4 任务从哪里来

```plain
客户端 POST /executor/task/start/
    → TaskStartHandler（APIServer/http_interfaces/StatusExecutorAPI.py）
    → IpcServer 向 Executor 发送 CMD 消息
    → executor.py TaskRunner 入队
    → FixedScripts/SingleExposure.py 或 ContinuousExposure.py
    → DriverAPI 相机驱动 → DriverSO/*.so
```

任务类型注册在 `APIServer/Eexcutor/FixedScripts/`：

| 类 | 文件 | 用途 |
|----|------|------|
| `SingleExposure` | `SingleExposure.py` | 单次曝光 |
| `ContinuousExposure` | `ContinuousExposure.py` | 连续曝光 |

## 1.5 设备管理的两条路径

**路径 A — 主进程 WebSocket 控制（实时指令）**

```plain
/ws/device/ → WebsocketHandler → HVDevices → HVDriverAPI/CameraDriver → DriverAPI → .so
```

**路径 B — Executor 任务内调用（拍摄流程）**

```plain
executor.py → ExecutorDeviceDrivers（executor_driver.py）→ 同样一套 HVDriverAPI
```

两套路径最终都落到 **DriverAPI + DriverSO**，只是调用上下文不同。

## 1.7 Tornado 与 asyncio

Tornado 6 基于 **asyncio** 事件循环：

```python
# HoriVisionDriverServer.py 入口模式
async def main():
    app = HorivisionAppServer(HoriVisionDevices)
    await app.startup()
    app.listen(8999)
    await shutdown_event.wait()
    await app.shutdown()

asyncio.run(main())
```

要点：

- `async def` Handler 可 `await` 非阻塞 I/O
- `PeriodicCallback` / `asyncio.create_task` 用于心跳、IPC
- **不要在 Handler 里直接做长时间阻塞曝光** — 应交给 Executor

## 1.9 自测问题

- 主进程和 Executor 各监听什么？（端口 vs Unix Socket 路径）
- `StateStore.executor_online` 由什么更新？（心跳超时见 `HB_TIMEOUT = 15.0`）
- 拍摄任务为什么必须串行？（`TaskRunner` 注释中的 hardware resource constraint）
- WebSocket 改增益和 Executor 曝光任务会不会冲突？（同一驱动实例需注意状态，coordinator 在 C 层协调预览/科学曝光）
