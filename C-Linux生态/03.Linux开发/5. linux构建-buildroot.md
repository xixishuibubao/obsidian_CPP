# 1. linux 构建 — Buildroot

**Buildroot** 通过 Kconfig + Makefile 从源码交叉编译完整嵌入式 Linux（工具链、内核、bootloader、rootfs 镜像）。

## 【核心精简提纲】

### 记忆主干

**`make <board>_defconfig` → `make menuconfig` 选包 → `make -j` → `output/images/`**；不是运行时包管理器，软件须在构建前选入；与 Yocto 比学习曲线低、适合固定固件。

### 核心要点清单

- 核心目录：`package/` 规则、`configs/` defconfig、`output/images/` 镜像
- 构建阶段：下载 → 解压打补丁 → 配置 → 编译 → install-target → 打包 rootfs
- 自定义：`rootfs_overlay`、post-build/post-image 脚本、`BR2_EXTERNAL`
- 增量：`make clean` 清 target 保留 build；`ccache` + 共享 `BR2_DL_DIR` 加速
- vs Yocto：Buildroot 快且简单；Yocto 灵活、适合发行版级定制

### 关键流程

```
defconfig → menuconfig(选架构/包/FS格式) → make -j
    → output/target/ 根 FS → output/images/rootfs.ext4 等
```

# 1. Buildroot 概述

**Buildroot** 是一个简单、高效且高度可定制的嵌入式 Linux 构建系统。它通过 **Kconfig 配置 + Makefile 自动化**的方式，从源码交叉编译完整的 Linux 系统（内核、根文件系统、bootloader、工具链、应用软件），输出可直接烧录或挂载的镜像。

```
Buildroot 构建流程

源代码（内核 / busybox / 库 / 应用）
    ↓
  下载（DL_DIR） → 解压 → 打补丁 → 配置 → 编译
    ↓
  构建根文件系统根目录（target/）
    ↓
  打包为多种镜像格式（rootfs.tar, rootfs.ext4, rootfs.squashfs ...）
    ↓
  内核 + bootloader + rootfs → 完整系统镜像
```

| 特性 | 说明 |
|------|------|
| **定位** | 嵌入式 Linux 系统的完整构建工具 |
| **源码** | 约 3000+ 软件包的支持（busybox, dropbear, Qt, GStreamer, ...） |
| **输出** | 工具链 + 内核 + bootloader + 根文件系统镜像 |
| **配置** | Kconfig 风格（`make menuconfig`、`make xconfig`、`make nconfig`） |
| **语言** | Makefile 为主，少量 Python 脚本 |
| **许可证** | GPL-2.0 |

## 1.1 Buildroot vs Yocto

| 维度 | Buildroot | Yocto |
|------|-----------|-------|
| **学习曲线** | 低（1-2 天上手） | 高（1-2 周入门） |
| **构建速度** | 快（首次约 30-60 分钟） | 慢（首次约 2-6 小时） |
| **灵活性** | 中（适合固定配置） | 极高（layer 机制，可无限定制） |
| **包管理** | ❌ 无运行时包管理 | ✅ 支持 RPM/IPK 包 |
| **工具链** | 自动编译或外部工具链 | 自动编译（buildtools） |
| **生态规模** | 约 3000+ 包 | 约 10000+ 层/配方 |
| **典型场景** | 产品固件、快速原型 | 复杂 Linux 发行版定制 |

> 简而言之：**Buildroot 适合产品固件（单一镜像、固定配置），Yocto 适合需要持续集成和包管理的 Linux 发行版级项目。**

## 1.2 Buildroot vs QEMU + busybox 手工构建

[[4. 容器化与系统构建#6. QEMU — 运行自定义系统（busybox + 内核 + rootfs）|手工构建方式]] 可深入理解内核/rootfs 的每一个细节，但每次修改配置都需要手动操作。Buildroot 将整个过程自动化：

- **手工方式**：手动下载内核 → 配置 → 编译 → 手动构建 busybox → 手动打包 rootfs → 手动组织启动脚本
- **Buildroot**：`make menuconfig` 选包 → `make` → 产出镜像，**一条命令完成全部**

## 1.3 Buildroot 与发行版包管理器的区别

| 对比项 | Buildroot | apt/yum/pacman |
|--------|-----------|----------------|
| **构建目标** | 交叉编译到目标机 | 本机直接安装 |
| **输出** | 完整系统镜像 | 单个软件包 |
| **依赖管理** | 构建时依赖，静态链接进镜像 | 运行时依赖，动态解析 |
| **更新方式** | 整机更新（新镜像烧录） | 逐包更新 |

> Buildroot **不设计为发行版操作系统**，而是一个**嵌入式系统构建工具**。不能在 Buildroot 构建的系统上用 `apt install` 装软件——要装什么必须在构建前配好。

# 2. 目录结构

Buildroot 源码目录布局如下：

```
buildroot/
├── arch/              # 架构相关的配置（ARM / x86 / MIPS / RISC-V ...）
├── board/             # 板级支持（配置文件、内核补丁、启动脚本）
├── boot/              # bootloader（uboot / grub / atf ...）
├── configs/           # 预置 defconfig（raspberrypi4_defconfig, qemu_x86_64_defconfig ...）
├── docs/              # 文档
├── fs/                # 根文件系统格式支持（ext4 / squashfs / cpio / tar ...）
├── linux/             # Linux 内核的构建规则
├── output/            # 构建产物（自动生成的目录）
│   ├── build/         # 每个软件包的解压和编译目录
│   ├── host/          # 主机端工具（交叉工具链、mke2fs、mkimage ...）
│   ├── images/        # 最终镜像文件（内核、rootfs、bootloader）
│   ├── staging/       # 头文件和库（供其他包交叉编译时引用）
│   └── target/        # 目标根文件系统
├── package/           # 所有软件包的构建规则（.mk + Config.in）
├── support/           # 辅助脚本和工具
├── system/            # 系统配置（设备表、shell 配置）
├── toolchain/         # 工具链相关的构建规则
├── Config.in          # 顶层 Kconfig 配置入口
└── Makefile           # 顶层 Makefile
```

## 2.1 核心目录详解

| 目录 | 作用 |
|------|------|
| `package/` | **所有软件包的构建规则**，每个包一个子目录（如 `package/busybox/`、`package/openssh/`） |
| `configs/` | **预置配置**，执行 `make <board>_defconfig` 加载 |
| `board/` | 板级支持文件（设备树、补丁、rc 脚本、genimage 配置） |
| `output/` | 构建产物（首次构建自动生成） |
| `fs/` | 根文件系统镜像格式（`fs/ext4/`、`fs/squashfs/`、`fs/cpio/` 等） |

## 2.2 output/ 产物详解

```bash
output/
├── build/          # 每个软件包的编译工作目录
│   ├── linux-6.6/          # 内核源码（已编译）
│   ├── busybox-1.36.1/     # busybox 源码（已编译）
│   ├── uboot-2024.01/      # U-Boot 源码（已编译）
│   └── ...                  # 其他所有包
├── host/           # 主机端工具
│   ├── bin/                 # 交叉编译器（aarch64-linux-gcc, arm-linux-g++ ...）
│   ├── aarch64-buildroot-linux-gnu/sysroot/   # 工具链 sysroot
│   └── sbin/                # mkfs.ext4, mkimage, mkenvimage ...
├── images/         # 最终镜像 ⭐
│   ├── bzImage              # 内核镜像
│   ├── rootfs.ext4          # 根文件系统（ext4）
│   ├── rootfs.cpio.gz       # 根文件系统（initramfs）
│   ├── sdcard.img           # 完整 SD 卡镜像
│   └── uboot.env            # U-Boot 环境变量
├── staging/        # 头文件 + 库的符号链接（指向 host/ 下的 sysroot）
└── target/         # 目标的根文件系统目录（构建完成后 populating 至此）
    ├── bin/
    ├── dev/
    ├── etc/
    ├── lib/
    ├── proc/
    ├── sbin/
    ├── tmp/
    └── usr/
```

> **注意**：`output/target/` 是构建产物，通过 `make clean` 会清空，但不会删除 `output/build/`，因此重新构建很快。

# 3. 快速上手

## 3.1 获取 Buildroot

```bash
# 方法一：下载稳定发布版（推荐）
wget https://buildroot.org/downloads/buildroot-2024.02.tar.gz    # LTS 版本
tar xf buildroot-2024.02.tar.gz
cd buildroot-2024.02/

# 方法二：克隆 git 仓库（最新开发版，可能不稳定）
git clone https://git.buildroot.net/buildroot
cd buildroot
```

> 建议使用 **LTS 版本**（每年 2 月发布，如 2024.02、2025.02），生产环境稳定可靠。

## 3.2 配置

Buildroot 使用 Kconfig 进行配置，入口命令：

```bash
# 加载预置配置（先查看 configs/ 目录有哪些可用）
ls configs/ | grep -E 'qemu|x86|arm'

# 以 QEMU x86_64 为例
make qemu_x86_64_defconfig

# 也可以从零开始配置
make menuconfig
```

`make menuconfig` 主要配置项：

| 配置路径 | 说明 |
|---------|------|
| `Target options` | 目标架构、CPU 类型、浮点 ABI |
| `Build options` | 编译选项（共享/静态库、调试符号、ccache） |
| `Toolchain` | 工具链类型（Buildroot 内部编译 / 外部工具链） |
| `System configuration` | 系统配置（hostname、密码、init 系统、rc 脚本） |
| `Kernel` | Linux 内核版本和配置 |
| `Target packages` | **选包**——要装哪些软件（busybox、openssh、Qt、...） |
| `Filesystem images` | **镜像格式**——输出 ext4/squashfs/cpio/tar 等 |
| `Bootloaders` | bootloader（uboot / grub2 / barebox） |
| `Host utilities` | 主机工具（genimage、mkpasswd 等） |

## 3.3 构建

```bash
# 开始编译（-jN 建议使用 CPU 核心数 + 1）
make -j$(nproc)

# 或者一步步来（调试用）
make source        # 仅下载所有源码包
make -j$(nproc)    # 正常编译
```

> **首次构建**需要下载大量源码包（~30-60 分钟取决于网络和 CPU），后续增量构建只需几秒到几分钟。

## 3.4 启动验证

```bash
# 对 QEMU x86_64 目标，构建完成后直接运行
make qemu_run

# 或手动启动 QEMU
qemu-system-x86_64 \
    -kernel output/images/bzImage \
    -drive file=output/images/rootfs.ext2,format=raw,if=virtio \
    -append "root=/dev/vda console=ttyS0" \
    -nic user,hostfwd=tcp::2222-:22 \
    -nographic
```

# 4. 构建流程详解

Buildroot 的构建流程可以分为以下几个阶段：

## 4.1 阶段一：下载（Download）

```
配置选择的所有包
    ↓
检查 DL_DIR 中是否已有源码包
    ↓ 缺失
从官方镜像或自定义源下载（.tar.gz / .tar.xz / git 仓库）
    ↓ 已有
跳过
```

```bash
# DL_DIR 默认是 $(TOP_DIR)/dl/（构建目录下的 dl/）
# 也可在 menuconfig → Build options → Download dir 中修改

# 分享 DL_DIR 到多个项目（推荐）
make menuconfig  # 设置 BR2_DL_DIR=/shared/downloads

# 或通过环境变量
make BR2_DL_DIR=/shared/downloads
```

## 4.2 阶段二：解压 + 打补丁（Extract & Patch）

```
DL_DIR 中的源码包
    ↓
解压到 output/build/<package>-<version>/
    ↓
依次打补丁（按文件名顺序）：
  1. package/<pkg>/<version>/ 下的补丁
  2. board/<vendor>/<board>/patches/<pkg>/ 下的补丁
```

补丁文件命名规范：

```bash
# 补丁以数字前缀确保顺序
board/mycompany/myboard/patches/linux/
├── 0001-enable-usb-driver.patch
├── 0002-fix-ethernet-irq.patch
└── 0003-add-custom-dts.patch
```

## 4.3 阶段三：配置（Configure）

每个软件包有自己的配置步骤：

```bash
# 对于 autotools 包：./configure --host=... --target=...
# 对于 cmake 包：cmake -DCMAKE_TOOLCHAIN_FILE=...
# 对于 busybox：make -C $(BUSYBOX_DIR) menuconfig

# 配置的缓存结果存放在 output/build/<pkg>-<version>/.config
```

## 4.4 阶段四：编译（Build）

```bash
# 编译单个包
make <package>

# 常用编译目标
make busybox                  # 只编译 busybox
make linux                    # 只编译内核
make uboot                    # 只编译 U-Boot
make busybox-menuconfig       # 配置 busybox（交互式）
make linux-menuconfig         # 配置内核（交互式）
make uboot-menuconfig         # 配置 U-Boot
```

## 4.5 阶段五：安装（Install）

安装分为三步，由 Buildroot 基础设施自动完成：

| 步骤 | 目标目录 | 作用 |
|------|---------|------|
| `install-staging` | `output/staging/` | 安装头文件和静态库（供其他包交叉编译引用） |
| `install-target` | `output/target/` | 安装到目标根文件系统（实际运行的文件） |
| `install-host` | `output/host/` | 安装主机端工具（仅对 host 类包） |

## 4.6 阶段六：打包根文件系统（RootFS Image）

所有包安装到 `output/target/` 后，Buildroot 会：

```bash
# 1. 执行设备表（makedev）
#    创建 /dev 下的静态设备节点

# 2. 执行 post-build 脚本
#    用户自定义的最终修改（参见第 6 章）

# 3. 生成根文件系统镜像
#    根据 menuconfig → Filesystem images 中的选择：
#    - rootfs.tar      # tar 打包
#    - rootfs.ext2/4   # ext2/ext4 磁盘镜像
#    - rootfs.squashfs # SquashFS 只读压缩镜像
#    - rootfs.cpio.gz  # initramfs（内核内嵌）
#    - rootfs.ubifs    # UBIFS（NAND Flash）

# 4. 执行 post-image 脚本
#    对整个镜像做最终处理（参见第 6 章）
```

## 4.7 阶段七：生成完整镜像（Final Image）

```bash
# genimage 工具根据配置将 内核 + bootloader + rootfs + 设备树 组装为最终镜像

# 配置示例（board/mycompany/myboard/genimage.cfg）：
image sdcard.img {
    hdimage {
        gpt = true
    }
    partition boot {
        image = "boot.vfat"
        partition-type-uuid = U
    }
    partition rootfs {
        image = "rootfs.ext4"
        partition-type-uuid = L
    }
}
```

# 5. 核心配置详解

## 5.1 目标选项（Target Options）

```bash
Target options
    → Target Architecture = AArch64 (ARM 64-bit)
    → Target Architecture Variant = cortex-a72
    → Target ABI = LP64
    → Floating point strategy = NEON
```

## 5.2 工具链配置

Buildroot 支持三种工具链方式：

| 方式 | 优点 | 缺点 | 推荐场景 |
|------|------|------|---------|
| **内置工具链** | 自动编译，配置灵活 | 构建慢（首次约 30 分钟） | 首次使用、有充足编译时间 |
| **外部工具链** | 开箱即用，编译快 | 固定版本，不易定制 | CI/生产环境 |
| **Linux 内核工具链** | 无额外二进制 | 功能受限 | 原型开发 |

```bash
# 使用 Linaro 外部工具链（ARM 官方）
Toolchain → Toolchain type = External toolchain
         → Toolchain = Linaro AArch64
```

## 5.3 系统配置（System Configuration）

```bash
System configuration
    → System hostname = mydevice
    → System banner = Welcome to MyDevice
    → Init system = BusyBox                 # busybox init / systemd
    → Root password = root123               # root 密码（留空则无需密码）
    → Path to the login scripts = board/mycompany/mydevice/rootfs_overlay/etc/init.d
```

关于 init 系统的选择：

| Init 系统 | 适合场景 |
|-----------|---------|
| **BusyBox init** | 小型系统（< 32MB Flash），功能简单 |
| **systemd** | 复杂系统（>= 256MB RAM），需服务管理 |
| **OpenRC** | 介于两者之间（Gentoo 风格） |
| **NONE（自定义）** | 极简系统，自己实现 init |

## 5.4 内核配置

```bash
Kernel
    → Linux Kernel = [*]
    → Kernel version = Custom version → 6.6.30
    → Kernel configuration = Using an in-tree defconfig → defconfig_name = qemu
    → Kernel binary format = bzImage / Image / zImage
```

内核配置可以通过 Buildroot 统一管理：

```bash
# 配置内核
make linux-menuconfig

# 保存内核配置到 Buildroot（diffconfig 方式）
make linux-savedefconfig

# 生成的 defconfig 会被保存到 board/ 目录
# 然后在 Kernel configuration 中指向它：
# → Kernel configuration = Using a custom (def)config file
# → Configuration file path = board/mycompany/mydevice/linux.config
```

## 5.5 文件系统镜像配置

```bash
Filesystem images
    → ext2/3/4 variant = ext4
    → ext4 size = 256M
    → squashfs = [*]                     # 只读 rootfs
    → tar the root filesystem = [*]
    → cpio the root filesystem = [*]
```

常见根文件系统方案：

| 方案 | 格式 | 优点 | 缺点 |
|------|------|------|------|
| **可写 rootfs** | ext4 | 可读写，方便调试 | 需要可写存储（eMMC/SD） |
| **只读 rootfs** | squashfs | 节省空间，防篡改 | 运行时无法修改 |
| **initramfs** | cpio | 无需存储设备 | 占用内存 |
| **overlayfs** | squashfs + overlay | 只读基础 + 可写覆盖 | 需要两层存储 |

## 5.6 实用缓存配置

```bash
# ccache 加速重复构建
Build options → Enable compiler cache = [*]

# 共享下载目录（多个项目复用）
Build options → Download dir = /opt/buildroot/dl

# 共享工具链（多个项目复用，需配置只读）
Build options → Enable build per-package directory = [*]
```

# 6. 自定义与扩展

## 6.1 Rootfs Overlay（根文件系统覆盖层）

这是最常用的自定义方式——在 `board/` 下创建一个目录，构建时合并到 `output/target/`：

```
board/mycompany/mydevice/rootfs_overlay/
├── etc/
│   ├── init.d/
│   │   ├── S40network    # 网络启动脚本
│   │   └── S50sshd       # SSH 服务启动
│   ├── inittab           # 自定义 inittab
│   ├── fstab             # 挂载表
│   └── network/
│       └── interfaces    # 网络配置
├── usr/
│   └── local/
│       ├── bin/          # 自定义二进制
│       └── lib/          # 自定义库
└── app/
    └── myapp.conf        # 应用配置
```

配置使能：

```bash
System configuration
    → Root filesystem overlay directories = board/mycompany/mydevice/rootfs_overlay
```

## 6.2 Post-Build 脚本（构建后处理）

在打包 rootfs 之前对 `output/target/` 做最终修改：

```bash
# board/mycompany/mydevice/post-build.sh
#!/bin/bash
set -e
TARGET_DIR="$1"

# 删除不必要的文件（节省空间）
rm -f "${TARGET_DIR}/usr/share/man/"*
rm -f "${TARGET_DIR}/usr/share/doc/"*
rm -f "${TARGET_DIR}/usr/share/locale/"*

# 设置特殊权限
chmod 4755 "${TARGET_DIR}/usr/bin/busybox"

# 生成运行时配置文件
echo "mydevice-v1.0" > "${TARGET_DIR}/etc/version"

exit 0
```

配置使能：

```bash
System configuration
    → Custom scripts to run before creating filesystem images
    → = board/mycompany/mydevice/post-build.sh
```

## 6.3 Post-Image 脚本（镜像后处理）

在生成镜像后对镜像文件做进一步处理：

```bash
# board/mycompany/mydevice/post-image.sh
#!/bin/bash
set -e
IMAGE_DIR="$1"   # output/images/

# 计算镜像哈希
cd "${IMAGE_DIR}"
sha256sum sdcard.img > sdcard.img.sha256

# 生成版本文件
echo "build-$(date +%Y%m%d-%H%M%S)" > version.txt

# 拷贝到发布目录
cp sdcard.img /srv/releases/mydevice/

exit 0
```

配置使能：

```bash
System configuration
    → Custom scripts to run after creating filesystem images
    → = board/mycompany/mydevice/post-image.sh
```

## 6.4 BR2_EXTERNAL（外部包管理）

当需要维护项目专属的包而不修改 Buildroot 核心时，使用 `BR2_EXTERNAL` 机制：

```bash
# 1. 创建外部目录结构
mkdir -p mycompany-external/{board,configs,package}

# 2. 创建外部配置入口
cat > mycompany-external/external.mk << 'EOF'
include $(sort $(wildcard $(BR2_EXTERNAL_MYCOMPANY_HOOKS)/package/*/*.mk))
EOF

cat > mycompany-external/external.desc << 'EOF'
name: MYCOMPANY_HOOKS
desc: MyCompany custom packages and boards
EOF

# 3. 使用外部目录
make BR2_EXTERNAL=../mycompany-external/ qemu_x86_64_defconfig
# 或写入配置永久保存：
# → Build options → External toolchain path = .../mycompany-external/
```

`BR2_EXTERNAL` 的优势：

- 项目代码与 Buildroot 源码完全分离
- 升级 Buildroot 版本不影响外部配置
- 可在多个项目间共享外部配置

## 6.5 自定义软件包

在 `package/` 目录中添加自定义包（或通过 BR2_EXTERNAL）：

```
package/myapp/
├── Config.in              # Kconfig 配置项
└── myapp.mk               # Makefile 构建规则
```

### Config.in

```kconfig
config BR2_PACKAGE_MYAPP
    bool "myapp"
    depends on BR2_TOOLCHAIN_HAS_THREADS
    select BR2_PACKAGE_LIBEVENT
    help
      My custom application for demonstration.
      https://github.com/mycompany/myapp
```

### myapp.mk

```makefile
MYAPP_VERSION = 1.0.0
MYAPP_SITE = https://github.com/mycompany/myapp.git
MYAPP_SITE_METHOD = git
MYAPP_LICENSE = GPL-2.0
MYAPP_LICENSE_FILES = LICENSE
MYAPP_DEPENDENCIES = libevent openssl

$(eval $(generic-package))
```

Buildroot 提供了几种包基础设施：

| 基础设施 | 适用场景 |
|---------|---------|
| `generic-package` | 任意构建方式（自己写 build logic） |
| `autotools-package` | 使用 autotools 的项目（`./configure && make`） |
| `cmake-package` | 使用 CMake 的项目 |
| `meson-package` | 使用 Meson 构建系统的项目 |
| `python-package` | Python 包（setuptools） |
| `cargo-package` | Rust 项目（Cargo） |

# 7. 项目实战：从零构建一个嵌入式系统

以 Raspberry Pi 4 为示例，展示完整的项目配置流程：

```bash
# 1. 配置
make raspberrypi4_64_defconfig

# 2. 自定义配置
make menuconfig
# 在 Target packages 中选中：
#   → Networking → openssh（SSH 服务）
#   → Networking → dhcpcd（DHCP 客户端）
#   → System tools → htop
#   → Debugging → gdb（调试用）

# 3. 创建板级支持目录
mkdir -p board/mycompany/rpi4/rootfs_overlay/etc/init.d

# 4. 创建启动脚本
cat > board/mycompany/rpi4/rootfs_overlay/etc/init.d/S50sshd << 'EOF'
#!/bin/sh
# 启动 SSH 服务
/usr/sbin/sshd
EOF
chmod +x board/mycompany/rpi4/rootfs_overlay/etc/init.d/S50sshd

# 5. 创建 genimage 配置
cat > board/mycompany/rpi4/genimage.cfg << 'EOF'
image sdcard.img {
    size = 2G
    boot.vfat {
        files = {
            "bcm2711-rpi-4-b.dtb",
            "Image",
            "boot.scr"
        }
    }
    rootfs.ext4 {}
}
EOF

# 6. 构建
make -j$(nproc)

# 7. 输出
ls output/images/
# boot.scr  bcm2711-rpi-4-b.dtb  Image  rootfs.ext4  sdcard.img
```

# 9. 最佳实践

## 9.1 工作流建议

```
首次开发 →
  1. 选择最接近的 defconfig 为基础
  2. make menuconfig 调整（选包、改内核参数）
  3. make -j$(nproc) 完整构建
  4. 验证（QEMU 模拟或烧录到开发板）

迭代开发 →
  1. 修改配置 / 代码
  2. make -j$(nproc)（增量构建，仅需几秒到几分钟）
  3. 更新 rootfs 或镜像
  4. 测试验证
```

## 9.2 加速构建

```bash
# 1. 启用 ccache
#    menuconfig → Build options → Enable compiler cache
#    第二次编译速度提升 50-80%

# 2. 共享 DL_DIR
#    将 DL_DIR 设为所有项目的共用下载缓存目录
export BR2_DL_DIR=/opt/buildroot/dl

# 3. 使用外部工具链（跳过工具链编译）
#    menuconfig → Toolchain → Toolchain type = External toolchain

# 4. per-package 目录（隔离增量编译）
#    menuconfig → Build options → Enable per-package directory
```

## 9.3 配置管理

```bash
# 开发板的 defconfig 应该纳入版本管理
make savedefconfig                  # 保存当前配置到 configs/myboard_defconfig
# 或手动备份：
cp .config configs/myboard_defconfig

# 内核配置同样管理
make linux-savedefconfig
cp output/build/linux-custom/defconfig board/mycompany/myboard/linux.config
```

## 9.4 常见问题

| 问题 | 解决方案 |
|------|---------|
| **源码下载失败** | 设置 `BR2_PRIMARY_SITE` 为内网镜像 |
| **磁盘空间不足** | `output/build/` 中删除不需要的包源码 |
| **编译错误** | `make <package>-dirclean` 后重新编译 |
| **rootfs 太大** | 使用 squashfs、删除 locale/man、strip 二进制 |
| **内核启动崩溃** | 检查设备树、内核配置、rootfs 类型 |
| **网络不通** | 配置 `/etc/network/interfaces` 或使用 dhcpcd 自动获取 IP |
