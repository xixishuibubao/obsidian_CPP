# 1. 容器化与系统构建

从文件系统隔离到完整嵌入式系统构建的学习路径：chroot → bind mount → systemd-nspawn → QEMU 手工 rootfs。与 [[C-Linux生态/03.Linux开发/5. linux构建-buildroot|Buildroot]] 自动化构建对照。

## 【核心精简提纲】

### 记忆主干

**隔离强度递增**：chroot（仅换根）< nspawn（namespace）< Docker（命名空间+cgroup+镜像）；**手工理解启动链**用 QEMU+内核+busybox+cpio；**量产固件**用 Buildroot/Yocto。

### 核心要点清单

- chroot：只隔离 FS，不隔离进程/网络；root 可突破，需配合 mount proc/dev/sys
- bind mount：VFS 层目录绑定，`--bind -o ro` 只读共享宿主机路径
- systemd-nspawn：无守护进程，`-D` 根 FS，`--network-veth` / `--bind`
- QEMU：`-kernel bzImage -initrd rootfs.cpio.gz -append "console=ttyS0" -nographic`
- busybox 静态编译可免去拷贝动态库

### 选型速查

| 需求 | 工具 |
|------|------|
| 临时换根跑程序 | chroot |
| 完整发行版调试/CI | systemd-nspawn |
| 镜像分发与编排 | Docker |
| 理解启动/教学 | QEMU 手工 rootfs |
| 产品固件 | Buildroot |

# 2. chroot — 根目录切换

## 2.1 原理

`chroot`（change root）将一个进程及其子进程的根目录（`/`）切换到指定目录，使其无法访问该目录之外的文件系统。

```plain
正常系统：                         chroot /myroot 后：
/                                  /myroot/
├── bin/       ← 可见              ├── bin/       ← 视为 /
├── etc/       ← 可见              ├── etc/       ← 可见
├── home/      ← 可见              └── ...        
└── /myroot/   ← 原始位置          实际 /home、/etc 不可见
```

> **核心限制**：chroot 只隔离文件系统，**不隔离进程、网络、用户**。`chroot` 内的进程如果拥有 root 权限且突破 chroot，仍可访问宿主机。

## 2.2 基本用法

```bash
# 创建 chroot 环境目录
mkdir -p /srv/chroot/myenv

# 在 chroot 内创建最小文件系统（需手动放置 bin/lib 等）
# 最简方式——直接解压一个 busybox 或 alpine 镜像

# 进入 chroot
sudo chroot /srv/chroot/myenv /bin/sh
```

## 2.3 构建最小 chroot 环境

chroot 需要目标目录内至少包含一个可执行文件及其依赖的库：

```bash
# 1. 创建目录结构
mkdir -p /srv/chroot/myenv/{bin,lib,lib64,dev,etc,tmp}

# 2. 拷贝 busybox（静态编译版本，零依赖）
cp /bin/busybox /srv/chroot/myenv/bin/

# 3. 如果 busybox 是动态编译，需拷贝依赖库
ldd /bin/busybox
# linux-vdso.so.1
# libm.so.6 → /lib/x86_64-linux-gnu/libm.so.6
# libc.so.6 → /lib/x86_64-linux-gnu/libc.so.6

# 拷贝需要的库
cp /lib/x86_64-linux-gnu/libm.so.6 /srv/chroot/myenv/lib/
cp /lib/x86_64-linux-gnu/libc.so.6 /srv/chroot/myenv/lib/
cp /lib64/ld-linux-x86-64.so.2 /srv/chroot/myenv/lib64/

# 4. 确保 busybox 能用
cd /srv/chroot/myenv && chroot . /bin/busybox --list

# 5. 建立常用命令的符号链接
chroot . /bin/busybox sh -c "for cmd in \$(busybox --list); do ln -s /bin/busybox /bin/\$cmd; done"

# 6. 进入 chroot 环境
sudo chroot /srv/chroot/myenv /bin/sh
```

> **提示**：更简单的做法是直接下载 Alpine Linux 的 minirootfs，解压后即可 chroot。

# 3. bind mount — 挂载绑定

## 3.1 原理

`bind mount`（绑定挂载）将文件系统的某个目录/文件**挂载到另一个路径**，使其在目标位置可见。不同于符号链接，bind mount 在内核 VFS 层面生效，对进程完全透明。

```
bind mount 前：                    bind mount 后：
/home/user/code/                  /srv/chroot/myenv/code/  （挂载点）
└── project/                      └── project/
                                    └── （内容来自 /home/user/code/project）
```

## 3.2 基本用法

```bash
# 将宿主机目录绑定到 chroot 内
sudo mount --bind /home/user/code /srv/chroot/myenv/code

# 将宿主机 /proc 挂载到 chroot 内（让 chroot 能看进程信息）
sudo mount --bind /proc /srv/chroot/myenv/proc

# 将宿主机 /dev 挂载到 chroot 内（让 chroot 能访问设备）
sudo mount --bind /dev /srv/chroot/myenv/dev

# 将宿主机 /sys 挂载到 chroot 内
sudo mount --bind /sys /srv/chroot/myenv/sys
```

## 3.3 只读绑定

```bash
# 以只读方式绑定（chroot 内不能修改宿主机的文件）
sudo mount --bind -o ro /home/user/share /srv/chroot/myenv/share
```

## 3.4 chroot + bind mount 完整示例

```bash
# 准备 chroot 环境
sudo mkdir -p /srv/chroot/{myenv/{bin,lib,lib64,dev,proc,sys,code,tmp},share}

# 挂载必要的文件系统
sudo mount --bind /proc /srv/chroot/myenv/proc
sudo mount --bind /dev  /srv/chroot/myenv/dev
sudo mount --bind /sys  /srv/chroot/myenv/sys

# 挂载共享目录（只读）
sudo mount --bind -o ro /srv/share /srv/chroot/myenv/share

# 拷贝 busybox 和库（略，参考 2.3）

# 进入
sudo chroot /srv/chroot/myenv /bin/sh
```

> **注意事项**：
> - 退出时记得卸载：`sudo umount /srv/chroot/myenv/{proc,dev,sys,share,code}`
> - `mount --bind` 仅当前会话有效，重启后消失
> - 若要永久生效，写入 `/etc/fstab`：`/original/path /target/path none bind 0 0`

## 3.5 查看当前所有 bind mount

```bash
# 查看所有挂载，过滤 bind 类型
mount | grep bind

# 或查看 /proc/mounts
grep bind /proc/mounts
```

# 4. systemd-nspawn — 轻量容器管理器

## 4.1 原理

`systemd-nspawn` 是 systemd 提供的容器管理器，基于 Linux **命名空间（namespace）** 实现：

```
systemd-nspawn 容器

┌──────────────────────────┐
│  文件系统命名空间         │  ← 文件系统隔离（类似 chroot）
│  进程命名空间             │  ← PID 独立编号（PID 1 是 init）
│  网络命名空间             │  ← 独立网络栈（可配置）
│  IPC 命名空间             │  ← 进程间通信隔离
│  UTS 命名空间             │  ← 独立 hostname
│  用户命名空间             │  ← 用户 ID 映射
└──────────────────────────┘
```

与 Docker 的对比：

| 特性 | `systemd-nspawn` | Docker |
|------|:---:|:---:|
| 守护进程 | ❌ 无守护进程 | ✅ dockerd |
| 镜像管理 | ❌ 无内置（直接操作目录） | ✅ 镜像仓库 |
| 网络管理 | 手动配置 | 内置网络桥接 |
| 资源限制 | systemd cgroup | 内置 cgroup |
| 适用场景 | 系统调试/测试/开发 | 生产部署/微服务 |

## 4.2 安装

```bash
# Ubuntu/Debian
sudo apt install systemd-container

# RHEL/Fedora（默认已安装）
sudo dnf install systemd-container
```

## 4.3 基本用法

### 启动容器

```bash
# 进入一个目录作为容器根文件系统
sudo systemd-nspawn -D /srv/container/myroot

# 启动并启动 init 系统（不指定命令时）
sudo systemd-nspawn -bD /srv/container/myroot
# -b = --boot，启动 systemd 作为 PID 1
```

### 常用选项

```bash
# 指定网络（使用宿主机网络）
sudo systemd-nspawn -D /srv/container/myroot --network-veth

# 设置主机名
sudo systemd-nspawn -D /srv/container/myroot --hostname mycontainer

# 挂载目录（宿主机目录到容器内）
sudo systemd-nspawn -D /srv/container/myroot --bind=/home/user/code:/mnt/code

# 只读挂载
sudo systemd-nspawn -D /srv/container/myroot --bind-ro=/usr/share/doc:/usr/share/doc

# 分配内存限制
sudo systemd-nspawn -D /srv/container/myroot -U --memory-max=512M
# -U = 用户命名空间映射
```

### 在容器内执行命令

```bash
# 单次执行（不进入容器）
sudo systemd-nspawn -D /srv/container/myroot -- ps aux

# 传递标准输入（脚本执行）
echo "echo hello from container" | sudo systemd-nspawn -D /srv/container/myroot -- /bin/sh
```

## 4.4 实操示例：用 Alpine minirootfs 启动容器

```bash
# 1. 下载 Alpine minirootfs
wget https://dl-cdn.alpinelinux.org/alpine/latest-stable/releases/x86_64/alpine-minirootfs-3.20.3-x86_64.tar.gz

# 2. 创建并解压到容器目录
sudo mkdir -p /srv/alpine
sudo tar xzf alpine-minirootfs-*.tar.gz -C /srv/alpine

# 3. 启动容器（带 init 系统）
sudo systemd-nspawn -bD /srv/alpine
```

## 4.5 容器管理

```bash
# 列出运行中的 nspawn 容器
machinectl list

# 查看容器状态
machinectl status mycontainer

# 登录到运行中的容器
machinectl login mycontainer

# 关闭容器
machinectl poweroff mycontainer

# 通过 SSH 连接（容器内需运行 sshd）
ssh root@container-ip
```

## 4.6 网络配置

### 方式一：虚拟网卡（veth）

```bash
# 宿主机侧会创建一个 ve-xxx 接口，容器内为 host0
sudo systemd-nspawn -D /srv/container --network-veth
```

容器内需手动配置 IP：
```bash
ip addr add 10.0.0.2/24 dev host0
ip link set host0 up
ip route add default via 10.0.0.1
```

### 方式二：共享宿主机网络

```bash
# 直接使用宿主机的网络栈（不隔离网络）
sudo systemd-nspawn -D /srv/container --share-system
```

> **生产建议**：使用 `--network-veth` + 网桥（bridge）+ 自定 DHCP，实现容器间互通。

# 5. chroot vs systemd-nspawn vs Docker 选型

| 维度 | chroot | systemd-nspawn | Docker |
|------|--------|---------------|--------|
| **隔离性** | 仅文件系统 | 文件+进程+网络+IPC+UTS | 完整命名空间 + cgroup |
| **安全性** | 低（可突破） | 中（需用户映射） | 高（多种安全机制） |
| **启动速度** | 即时 | ~1s | ~1s（镜像已缓存） |
| **镜像大小** | 任意 | 任意 | 分层存储 |
| **配置复杂度** | 低 | 中 | 中高 |
| **适用场景** | 临时修复/测试 | 系统调试/CI 环境 | 微服务/生产环境 |

**选型建议**：
- 只想换个根目录运行程序 → **chroot**
- 需要完整隔离运行一个 Linux 发行版做测试 → **systemd-nspawn**
- 需要镜像分发、编排、生产部署 → **Docker**

# 6. QEMU — 运行自定义系统（busybox + 内核 + rootfs）

## 6.1 概述

从源码分别编译 **内核（kernel）**、**根文件系统（rootfs）**、**busybox**，然后用 QEMU 启动这个自定义 Linux 系统。这是深入理解 Linux 启动过程的最佳实践。

```
编译产物：                    QEMU 启动：
kernel/arch/x86/boot/bzImage ────┐
rootfs/rootfs.cpio.gz       ────┤
                                ▼
                         ┌──────────────┐
                         │   QEMU       │
                         │ (虚拟机)      │
                         └──────────────┘
```

## 6.2 编译内核

```bash
# 1. 下载内核源码
wget https://cdn.kernel.org/pub/linux/kernel/v6.x/linux-6.6.tar.xz
tar xf linux-6.6.tar.xz
cd linux-6.6

# 2. 配置内核（使用默认 x86_64 配置）
make defconfig

# 3. 启用必要的驱动（如果需要）
#   通常 defconfig 已包含 QEMU 所需的 VirtIO 驱动
#   如需自定义：make menuconfig

# 4. 编译（-jN 中的 N 为 CPU 核心数）
make -j$(nproc)

# 5. 确认编译产物
ls arch/x86/boot/bzImage
# 或更通用的位置（通常可在此找到）
ls vmlinux
```

> **关键配置项**（`make menuconfig` 中确认）：
> - `CONFIG_VIRTIO_PCI=y` — VirtIO PCI 驱动（QEMU 默认设备）
> - `CONFIG_VIRTIO_BLK=y` — VirtIO 块设备
> - `CONFIG_VIRTIO_NET=y` — VirtIO 网络设备
> - `CONFIG_EXT4_FS=y` — ext4 文件系统支持（rootfs 用）
> - `CONFIG_9P_FS=y` / `CONFIG_9P_VIRTIO=y` — 9P 共享文件系统（可选）

## 6.3 编译 busybox

```bash
# 1. 下载 busybox
wget https://busybox.net/downloads/busybox-1.36.1.tar.bz2
tar xf busybox-1.36.1.tar.bz2
cd busybox-1.36.1

# 2. 配置——编译为静态链接（最省事，无需拷贝动态库）
make defconfig
make menuconfig
# → Settings → Build static binary (no shared libs) → [*] 选中

# 3. 编译并安装到 _install 目录
make -j$(nproc)
make install
# 安装后 busybox 和所有工具在 busybox-1.36.1/_install/ 下
ls _install/
# bin/  linuxrc  sbin/  usr/
```

## 6.4 构建根文件系统（rootfs）

### 方式一：initramfs（cpio 归档，最简方式）

内核启动时直接加载 cpio 归档作为根文件系统，无需磁盘镜像：

```bash
# 1. 创建 rootfs 目录
mkdir -p rootfs/{bin,sbin,etc,dev,proc,sys,tmp,root}

# 2. 拷贝 busybox 安装产物
cp -a busybox-1.36.1/_install/* rootfs/

# 3. 创建设备节点（必要设备）
sudo mknod -m 666 rootfs/dev/null c 1 3
sudo mknod -m 666 rootfs/dev/console c 5 1

# 4. 创建 init 脚本
cat > rootfs/init << 'EOF'
#!/bin/sh
mount -t proc none /proc
mount -t sysfs none /sys
echo "Hello from custom Linux!"
echo "Kernel: $(uname -r)"
echo "Hostname: $(hostname)"
exec /bin/sh
EOF
chmod +x rootfs/init

# 5. 打包为 cpio 归档（内核可直接加载的格式）
cd rootfs
find . | cpio -o -H newc | gzip > ../rootfs.cpio.gz
cd ..
```

### 方式二：ext4 磁盘镜像（完整文件系统）

如果需要持久化存储和更大的文件系统：

```bash
# 1. 创建空白磁盘镜像（100MB）
dd if=/dev/zero of=rootfs.ext4 bs=1M count=100

# 2. 格式化为 ext4
mkfs.ext4 rootfs.ext4

# 3. 挂载并拷贝文件
sudo mount rootfs.ext4 /mnt
sudo cp -a rootfs/* /mnt/
# 复制 busybox 安装产物
sudo cp -a busybox-1.36.1/_install/* /mnt/
sudo mkdir -p /mnt/{dev,proc,sys,tmp}
sudo mknod -m 666 /mnt/dev/null c 1 3
sudo mknod -m 666 /mnt/dev/console c 5 1
sudo cp rootfs/init /mnt/init
sudo chmod +x /mnt/init

# 4. 卸载
sudo umount /mnt
```

## 6.5 QEMU 启动

### 启动 initramfs 方式

```bash
# 最简 QEMU 启动命令
qemu-system-x86_64 \
    -kernel linux-6.6/arch/x86/boot/bzImage \
    -initrd rootfs/rootfs.cpio.gz \
    -append "console=ttyS0" \
    -nographic
```

参数说明：

| 参数 | 含义 |
|------|------|
| `-kernel` | 内核镜像路径 |
| `-initrd` | initramfs（cpio 归档）路径 |
| `-append` | 内核启动参数（传给内核的命令行） |
| `-nographic` | 无图形界面，使用当前终端（串口输出） |

### 启动 ext4 磁盘镜像方式

```bash
qemu-system-x86_64 \
    -kernel linux-6.6/arch/x86/boot/bzImage \
    -drive file=rootfs.ext4,format=raw,if=virtio \
    -append "root=/dev/vda console=ttyS0" \
    -nographic
```

参数差异：

| 参数 | 含义 |
|------|------|
| `-drive file=...,format=raw,if=virtio` | 挂载磁盘镜像，使用 VirtIO 接口 |
| `root=/dev/vda` | 告诉内核根文件系统在 `/dev/vda`（VirtIO 磁盘） |

### 启用网络

```bash
qemu-system-x86_64 \
    -kernel linux-6.6/arch/x86/boot/bzImage \
    -initrd rootfs/rootfs.cpio.gz \
    -append "console=ttyS0" \
    -nic user,hostfwd=tcp::2222-:22 \
    -nographic
```

`-nic user` 启用用户模式网络（NAT），`hostfwd=tcp::2222-:22` 将宿主机的 2222 端口转发到虚拟机内的 22 端口（SSH）。

### 启用 9P 共享目录（宿主机 ↔ 虚拟机文件交换）

```bash
# 启动时挂载共享目录
qemu-system-x86_64 \
    -kernel linux-6.6/arch/x86/boot/bzImage \
    -initrd rootfs/rootfs.cpio.gz \
    -append "console=ttyS0" \
    -virtfs local,path=/host/share,security_model=passthrough,mount_tag=shared \
    -nographic

# 在虚拟机内挂载
mkdir -p /mnt/shared
mount -t 9p -o trans=virtio shared /mnt/shared
```

## 6.6 实操：完整的一条龙脚本

将以上步骤整合为可复用的构建脚本：

```bash
#!/bin/bash
# build-and-run.sh — 编译内核+busybox+rootfs 并用 QEMU 启动
set -e

ROOTFS_DIR=$(mktemp -d)
BUILD_DIR=$(pwd)

# === 配置 ===
KERNEL_VERSION="6.6"
BUSYBOX_VERSION="1.36.1"
JOBS=$(nproc)

# === Step 1: 编译内核 ===
if [ ! -f linux-${KERNEL_VERSION}/arch/x86/boot/bzImage ]; then
    echo "[1/4] 编译内核..."
    tar xf linux-${KERNEL_VERSION}.tar.xz
    cd linux-${KERNEL_VERSION}
    make defconfig
    make -j${JOBS}
    cd ${BUILD_DIR}
fi

# === Step 2: 编译 busybox ===
if [ ! -f busybox-${BUSYBOX_VERSION}/_install/bin/sh ]; then
    echo "[2/4] 编译 busybox..."
    tar xf busybox-${BUSYBOX_VERSION}.tar.bz2
    cd busybox-${BUSYBOX_VERSION}
    make defconfig
    # 启用静态编译
    sed -i 's/.*CONFIG_STATIC.*/CONFIG_STATIC=y/' .config
    make -j${JOBS}
    make install
    cd ${BUILD_DIR}
fi

# === Step 3: 构建 rootfs ===
echo "[3/4] 构建 rootfs..."
mkdir -p ${ROOTFS_DIR}/{bin,sbin,etc,dev,proc,sys,tmp,root}
cp -a busybox-${BUSYBOX_VERSION}/_install/* ${ROOTFS_DIR}/
sudo mknod -m 666 ${ROOTFS_DIR}/dev/null c 1 3 2>/dev/null || true
sudo mknod -m 666 ${ROOTFS_DIR}/dev/console c 5 1 2>/dev/null || true

cat > ${ROOTFS_DIR}/init << 'INIT'
#!/bin/sh
mount -t proc none /proc
mount -t sysfs none /sys
echo "=== 自定义 Linux 启动成功 ==="
echo "内核版本: $(uname -r)"
echo "架构: $(uname -m)"
echo "============================="
exec /bin/sh
INIT
chmod +x ${ROOTFS_DIR}/init

cd ${ROOTFS_DIR}
find . | cpio -o -H newc | gzip > ${BUILD_DIR}/rootfs.cpio.gz
cd ${BUILD_DIR}

# === Step 4: QEMU 启动 ===
echo "[4/4] 启动 QEMU..."
echo "退出方法: Ctrl+A 然后按 X"
echo ""
qemu-system-x86_64 \
    -kernel linux-${KERNEL_VERSION}/arch/x86/boot/bzImage \
    -initrd rootfs.cpio.gz \
    -append "console=ttyS0" \
    -nographic \
    -m 256M

# 清理
rm -rf ${ROOTFS_DIR}
```

## 6.7 调试技巧

### 内核启动参数调试

```bash
# 启用内核调试输出
-append "console=ttyS0 earlyprintk initcall_debug loglevel=8"
```

### GDB 调试内核

```bash
# QEMU 开启 GDB 服务器（等待 gdb 连接）
qemu-system-x86_64 \
    -kernel linux-6.6/arch/x86/boot/bzImage \
    -initrd rootfs/rootfs.cpio.gz \
    -append "console=ttyS0 nokaslr" \
    -nographic \
    -s -S
# -s = -gdb tcp::1234（简写）
# -S = 启动时暂停 CPU，等待 GDB 连接

# 另一个终端运行 GDB
gdb linux-6.6/vmlinux
(gdb) target remote :1234
(gdb) break start_kernel
(gdb) continue
```

> 需要 `nokaslr` 禁用内核地址空间随机化，否则断点可能不命中。

### 查看内核日志

```bash
# 启动后用 dmesg（如果 busybox 支持）
dmesg

# 或在 QEMU 中按 Ctrl+A 然后按 H 查看帮助
```

## 6.8 常用选项速查

```bash
# 分配更多内存
qemu-system-x86_64 ... -m 512M

# 多核 CPU
qemu-system-x86_64 ... -smp 4

# 启用 KVM 加速（大幅提升性能，需 CPU 支持）
qemu-system-x86_64 ... -enable-kvm

# 启用网络并转发端口
qemu-system-x86_64 ... -nic user,hostfwd=tcp::2222-:22

# 挂载额外的磁盘
qemu-system-x86_64 ... -drive file=data.img,format=raw,if=virtio

# 不使用图形界面，使用串口
qemu-system-x86_64 ... -nographic

# 从 UEFI 启动（而非传统的 BIOS）
qemu-system-x86_64 ... -bios /usr/share/ovmf/OVMF.fd
```
