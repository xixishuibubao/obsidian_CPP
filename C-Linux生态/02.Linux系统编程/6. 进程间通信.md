# 1. 进程间通信概述

## 【核心精简提纲】

### 记忆主干

IPC 解决独立地址空间进程间的数据传递与同步。按需求选型：**字节流** → pipe/FIFO；**大数据零拷贝** → 共享内存 + 信号量；**结构化消息** → 消息队列；**异步通知** → signal/eventfd；**复杂双向** → Unix 域套接字（UDS）。

### 核心要点清单

- pipe：fork 继承 fd，单向，~64KiB 内核缓冲，读写阻塞
- FIFO：文件系统路径标识，无血缘进程可用，默认 open 阻塞
- 共享内存：最快，须配信号量/互斥锁防竞态
- 消息队列：有类型边界，可优先级读取，需 `IPC_RMID` 清理
- UDS：同主机比 TCP 快，Stream/Dgram 两种；配合 epoll
- 工程案例：Telescope1126 Server 用 UDS 连 Tornado 与 Executor

### 关键结论 / 流程

```plain
简单流 → pipe │ 无关进程 → FIFO/UDS │ 高频大数据 → shm+sem │ 复杂 C/S → UDS Stream
```

## 【完整拓展详情】


**进程间通信**（Inter-Process Communication, IPC）指操作系统提供的、用于协调和交换数据的**跨进程数据传输机制**。由于每个进程拥有独立的虚拟地址空间，一个进程无法直接访问另一个进程的内存（受 MMU 保护），因此需要内核或共享介质作为中介。

## 1.1 IPC 动机

| 场景 | 说明 |
|------|------|
| **数据传递** | 一个进程产生数据，另一个进程消费（如管道过滤器） |
| **共享数据** | 多个进程读写同一份数据（如共享内存 + 信号量） |
| **事件通知** | 一个进程向另一个进程发送异步信号（如 `SIGINT`、`SIGTERM`） |
| **资源共享** | 多个进程互斥访问同一资源（如文件锁、信号量） |
| **进程控制** | 调试器控制被调试进程（如 `ptrace`） |

## 1.2 IPC 机制全景

| 机制 | 类型 | 通信方向 | 数据模型 | 内核介入 |
|------|------|----------|----------|----------|
| **匿名管道 `pipe`** | 字节流 | 单向 | 无格式字节流 | 全程 |
| **命名管道 `FIFO`** | 字节流 | 单向 | 无格式字节流 | 全程 |
| **System V 消息队列** | 消息 | 双向 | 结构化消息 | 发送/接收 |
| **System V 共享内存** | 内存 | 双向 | 共享内存区 | 创建/附接 |
| **System V 信号量** | 同步 | N/A | 计数器 | 每次操作 |
| **POSIX 消息队列** | 消息 | 双向 | 结构化消息 | 发送/接收 |
| **POSIX 共享内存** | 内存 | 双向 | 共享内存区 | 创建/映射 |
| **POSIX 信号量** | 同步 | N/A | 计数器 | 仅当竞争 |
| **信号 `signal`** | 异步 | 单向 | 信号编号 | 递送 |
| **Unix 域套接字** | 字节流/消息 | 双向 | 字节流/消息 | 传输 |
| **`eventfd`** | 事件 | 单向 | 计数器 | 通知 |

> 关联笔记：[[2. CPU亲和性|CPU 亲和性]]、[[1. 进程管理|进程管理]] 讨论了进程调度与资源绑定，IPC 是多进程协作的基础。

# 2. 管道 (Pipe)

## 2.1 匿名管道

匿名管道是最古老的 IPC 形式，由 `pipe()` 系统调用创建，返回一对文件描述符 `fd[0]`（读端）和 `fd[1]`（写端）。数据流经内核缓冲区，遵循 **FIFO** 顺序。

**特性：**
- 只存在于血缘关系进程之间（fork 继承 fd）
- 单向（半双工），典型用法：父进程写、子进程读
- 内核缓冲区大小通常为 64 KiB（`/proc/sys/fs/pipe-max-size`）
- 读写均阻塞：读空管道阻塞，写满管道阻塞

### 2.1.1 C 示例 — 管道通信

```cpp
// pipe_example.cpp
// 演示匿名管道在父子进程间传递数据
// 编译: g++ -std=c++17 -Wall -o pipe_example pipe_example.cpp
// 运行: ./pipe_example

#include <cstdlib>
#include <cstring>
#include <iostream>
#include <string>
#include <unistd.h>

// Google 命名规范: 函数 CamelCase, 变量 snake_case

// 常量: k 前缀 + CamelCase
static constexpr size_t kBufferSize = 1024;
static constexpr const char* kMessage = "Hello from parent process!";

// 读取管道直到 EOF，返回读取的完整字符串
std::string ReadFromPipe(int fd) {
  char buffer[kBufferSize];
  std::string result;
  while (true) {
    ssize_t bytes_read = read(fd, buffer, sizeof(buffer) - 1);
    if (bytes_read == -1) {
      std::perror("read");
      break;
    }
    if (bytes_read == 0) break;  // EOF: 写端关闭
    buffer[bytes_read] = '\0';
    result.append(buffer);
  }
  return result;
}

int main() {
  int pipe_fd[2];  // fd[0]=读端, fd[1]=写端

  if (pipe(pipe_fd) == -1) {
    std::perror("pipe");
    return EXIT_FAILURE;
  }

  pid_t pid = fork();
  if (pid == -1) {
    std::perror("fork");
    return EXIT_FAILURE;
  }

  if (pid == 0) {
    // 子进程: 关闭写端，从读端读取
    close(pipe_fd[1]);
    std::string received = ReadFromPipe(pipe_fd[0]);
    std::cout << "[Child] Received: " << received << std::endl;
    close(pipe_fd[0]);
    return EXIT_SUCCESS;
  }

  // 父进程: 关闭读端，向写端写入
  close(pipe_fd[0]);
  write(pipe_fd[1], kMessage, std::strlen(kMessage));
  close(pipe_fd[1]);  // 关闭后子进程 read 返回 EOF

  wait(nullptr);  // 等待子进程结束
  std::cout << "[Parent] Child finished." << std::endl;
  return EXIT_SUCCESS;
}
```

## 2.2 命名管道 (FIFO)

FIFO 通过 `mkfifo()` 创建文件系统中的一个**特殊文件**，**无血缘关系的进程**也能通过它通信。

**特性：**
- 以文件路径名标识，用 `open()` / `read()` / `write()` 操作
- 半双工，但可用两个 FIFO 实现双向通信
- 内核缓冲区语义与匿名管道相同
- 默认阻塞模式：读端未打开时写端阻塞，反之亦然

### 2.2.1 C 示例 — FIFO 通信

```c
// fifo_example.c
// 演示命名管道在两个独立进程间通信（需分别运行读写端）
// 编译: gcc -std=c11 -Wall -o fifo_writer fifo_example.c -DWRITER
//        gcc -std=c11 -Wall -o fifo_reader fifo_example.c
// 运行: 终端1 ./fifo_writer , 终端2 ./fifo_reader

#include <fcntl.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/stat.h>
#include <unistd.h>

// Google C 代码规范: 函数 snake_case, 类型/变量 snake_case

static const char* kFifoPath = "/tmp/example_fifo";
static const size_t kBufferSize = 1024;

#ifdef WRITER
int main() {
  // 创建 FIFO
  if (mkfifo(kFifoPath, 0666) == -1) {
    perror("mkfifo");
    // EEXIST 表示已存在，继续执行
    if (access(kFifoPath, F_OK) == -1) return EXIT_FAILURE;
  }

  printf("[Writer] Opening FIFO for writing (will block until reader opens)...\n");
  int fd = open(kFifoPath, O_WRONLY);
  if (fd == -1) {
    perror("open");
    unlink(kFifoPath);
    return EXIT_FAILURE;
  }

  const char* messages[] = {
      "Message 1: Process communication via FIFO",
      "Message 2: FIFO supports unrelated processes",
      "Message 3: EOF when writer closes",
  };
  size_t count = sizeof(messages) / sizeof(messages[0]);

  for (size_t i = 0; i < count; ++i) {
    write(fd, messages[i], strlen(messages[i]));
    sleep(1);  // 模拟间隔
  }

  printf("[Writer] Done, closing FIFO.\n");
  close(fd);
  unlink(kFifoPath);  // 清理 FIFO 文件
  return EXIT_SUCCESS;
}
#else   // READER
int main() {
  printf("[Reader] Waiting for FIFO to be opened by writer...\n");
  int fd = open(kFifoPath, O_RDONLY);
  if (fd == -1) {
    perror("open");
    return EXIT_FAILURE;
  }

  char buffer[kBufferSize];
  ssize_t bytes_read;
  printf("[Reader] FIFO opened, reading messages:\n");
  while ((bytes_read = read(fd, buffer, sizeof(buffer) - 1)) > 0) {
    buffer[bytes_read] = '\0';
    // 逐行打印（消息间隔写入，可能合并，此处简化）
    printf("[Reader] %s\n", buffer);
  }
  printf("[Reader] EOF received (writer closed).\n");
  close(fd);
  return EXIT_SUCCESS;
}
#endif  // WRITER
```

# 3. 共享内存 (Shared Memory)

共享内存是**最高效**的 IPC 形式：多个进程将同一块物理内存映射到各自的虚拟地址空间，数据无需在内核和用户空间之间拷贝。

**代价：** 需要同步机制（信号量、互斥锁）配合使用，否则会出现竞态条件。

## 3.1 System V 共享内存

使用 `shmget()` 创建/获取共享内存段，`shmat()` 附接到进程地址空间，`shmdt()` 分离。

### 3.1.1 C 示例 — 共享内存 + 信号量协调

```c
// shm_sem_example.c
// 共享内存 + System V 信号量实现生产者消费者
// 编译: gcc -std=c11 -Wall -o shm_sem_example shm_sem_example.c
// 运行: ./shm_sem_example

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/ipc.h>
#include <sys/sem.h>
#include <sys/shm.h>
#include <sys/types.h>
#include <unistd.h>

// --- 常量 & 结构体 ---

// Google C 规范: 宏 ALL_CAPS, 常量 kCamelCase
#define SHM_KEY 0x1234
#define SEM_KEY 0x5678
#define SHM_SIZE 4096

// 联合体: semctl 所需（glibc 要求自行声明）
union semun {
  int val;
  struct semid_ds* buf;
  unsigned short* array;
};

// 共享内存中的数据结构
struct shared_data {
  int counter;             // 当前数据编号
  char message[256];       // 数据内容
};

// --- 信号量辅助函数 ---

// 初始化信号量值为 1（二进制信号量/互斥锁）
static void SemInit(int sem_id) {
  union semun arg;
  arg.val = 1;
  if (semctl(sem_id, 0, SETVAL, arg) == -1) {
    perror("semctl SETVAL");
    exit(EXIT_FAILURE);
  }
}

// P 操作：等待信号量（减 1），阻塞直到资源可用
static void SemWait(int sem_id) {
  struct sembuf sb = {.sem_num = 0, .sem_op = -1, .sem_flg = 0};
  if (semop(sem_id, &sb, 1) == -1) {
    perror("semop wait");
    exit(EXIT_FAILURE);
  }
}

// V 操作：释放信号量（加 1）
static void SemSignal(int sem_id) {
  struct sembuf sb = {.sem_num = 0, .sem_op = 1, .sem_flg = 0};
  if (semop(sem_id, &sb, 1) == -1) {
    perror("semop signal");
    exit(EXIT_FAILURE);
  }
}

int main() {
  // 创建共享内存
  int shm_id = shmget(SHM_KEY, SHM_SIZE, IPC_CREAT | 0666);
  if (shm_id == -1) {
    perror("shmget");
    return EXIT_FAILURE;
  }

  // 附接共享内存
  struct shared_data* data = (struct shared_data*)shmat(shm_id, NULL, 0);
  if (data == (void*)-1) {
    perror("shmat");
    return EXIT_FAILURE;
  }

  // 创建信号量
  int sem_id = semget(SEM_KEY, 1, IPC_CREAT | 0666);
  if (sem_id == -1) {
    perror("semget");
    return EXIT_FAILURE;
  }

  SemInit(sem_id);
  data->counter = 0;

  pid_t pid = fork();
  if (pid == -1) {
    perror("fork");
    return EXIT_FAILURE;
  }

  if (pid == 0) {
    // --- 子进程: 消费者 ---
    for (int i = 0; i < 5; ++i) {
      SemWait(sem_id);  // 上锁
      if (data->counter > 0) {
        printf("[Consumer] Read #%d: %s\n", data->counter, data->message);
        data->counter = 0;  // 标记已消费
      }
      SemSignal(sem_id);  // 解锁
      sleep(1);
    }
    shmdt(data);
    exit(EXIT_SUCCESS);
  }

  // --- 父进程: 生产者 ---
  for (int i = 0; i < 5; ++i) {
    SemWait(sem_id);
    data->counter = i + 1;
    snprintf(data->message, sizeof(data->message),
             "Shared memory message #%d (PID=%d)", i + 1, getpid());
    SemSignal(sem_id);
    printf("[Producer] Wrote #%d\n", i + 1);
    sleep(1);
  }

  wait(NULL);

  // 清理
  shmdt(data);
  shmctl(shm_id, IPC_RMID, NULL);
  semctl(sem_id, 0, IPC_RMID);

  printf("[Parent] Cleanup done.\n");
  return EXIT_SUCCESS;
}
```

# 4. 信号量 (Semaphore)

信号量是一种用于**同步**的计数器，本身不传递数据，而是协调对共享资源的互斥访问。

| 类型 | 创建方式 | 特点 |
|------|----------|------|
| **POSIX 有名信号量** | `sem_open()` | 文件系统路径标识，适合不同进程 |
| **POSIX 无名信号量** | `sem_init()` | 共享内存或线程间使用 |
| **System V 信号量** | `semget()` | 可同时操作多个信号量（集合） |

## 4.1 POSIX 有名信号量示例

```c
// posix_sem_example.c
// POSIX 有名信号量实现进程互斥
// 编译: gcc -std=c11 -Wall -o posix_sem_example posix_sem_example.c -lpthread
// 运行: ./posix_sem_example

#include <fcntl.h>
#include <semaphore.h>
#include <stdio.h>
#include <stdlib.h>
#include <sys/stat.h>
#include <unistd.h>

static const char* kSemName = "/example_sem";

// 访问临界区的模拟工作
static void CriticalWork(const char* process_name, int work_id) {
  // sleep 模拟实际 I/O 或计算
  printf("[%s] Entering critical section #%d\n", process_name, work_id);
  sleep(1);
  printf("[%s] Leaving critical section #%d\n", process_name, work_id);
}

int main() {
  // 创建命名信号量，初始值 = 1（二进制信号量）
  sem_t* sem = sem_open(kSemName, O_CREAT, 0666, 1);
  if (sem == SEM_FAILED) {
    perror("sem_open");
    return EXIT_FAILURE;
  }

  // 先删除名字，避免子进程退出后名字残留（已打开的 sem_t* 仍然有效）
  sem_unlink(kSemName);

  pid_t pid = fork();
  if (pid == -1) {
    perror("fork");
    return EXIT_FAILURE;
  }

  if (pid == 0) {
    // 子进程
    for (int i = 0; i < 3; ++i) {
      sem_wait(sem);
      CriticalWork("Child", i + 1);
      sem_post(sem);
      sleep(2);  // 非临界区延迟
    }
    sem_close(sem);
    exit(EXIT_SUCCESS);
  }

  // 父进程
  for (int i = 0; i < 3; ++i) {
    sem_wait(sem);
    CriticalWork("Parent", i + 1);
    sem_post(sem);
    sleep(1);
  }

  wait(NULL);
  sem_close(sem);
  printf("[Main] Done.\n");
  return EXIT_SUCCESS;
}
```

# 5. 消息队列 (Message Queue)

消息队列在内核中以**链表形式**维护消息，每个消息有类型标识，支持按类型**优先级读取**，避免了管道的字节流边界问题。

## 5.1 System V 消息队列

```c
// msg_queue_example.c
// System V 消息队列 — 父子进程发送/接收不同类型消息
// 编译: gcc -std=c11 -Wall -o msg_queue_example msg_queue_example.c
// 运行: ./msg_queue_example

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/ipc.h>
#include <sys/msg.h>
#include <sys/types.h>
#include <unistd.h>

#define MSG_KEY 0x2468

// 消息结构体: 首字段必须是 long msgtype
struct msgbuf {
  long msgtype;         // 消息类型（>0）, 用于按类型读取
  char mtext[256];      // 消息正文
};

static void SendMessage(int msg_id, long msgtype, const char* text) {
  struct msgbuf msg;
  msg.msgtype = msgtype;
  snprintf(msg.mtext, sizeof(msg.mtext), "%s", text);
  if (msgsnd(msg_id, &msg, strlen(msg.mtext) + 1, 0) == -1) {
    perror("msgsnd");
  }
}

static void ReceiveMessage(int msg_id, long msgtype, const char* label) {
  struct msgbuf msg;
  ssize_t len = msgrcv(msg_id, &msg, sizeof(msg.mtext), msgtype, 0);
  if (len == -1) {
    perror("msgrcv");
    return;
  }
  printf("[%s] Received (type=%ld): %s\n", label, msgtype, msg.mtext);
}

int main() {
  // 创建消息队列
  int msg_id = msgget(MSG_KEY, IPC_CREAT | 0666);
  if (msg_id == -1) {
    perror("msgget");
    return EXIT_FAILURE;
  }

  pid_t pid = fork();
  if (pid == -1) {
    perror("fork");
    return EXIT_FAILURE;
  }

  if (pid == 0) {
    // 子进程: 发送 type=2 消息, 接收 type=1 消息
    SendMessage(msg_id, 2, "Hello from child!");
    ReceiveMessage(msg_id, 1, "Child");
    exit(EXIT_SUCCESS);
  }

  // 父进程: 发送 type=1 消息, 接收 type=2 消息
  SendMessage(msg_id, 1, "Hello from parent!");
  ReceiveMessage(msg_id, 2, "Parent");

  wait(NULL);

  // 清理
  msgctl(msg_id, IPC_RMID, NULL);
  printf("[Main] Message queue removed.\n");
  return EXIT_SUCCESS;
}
```

**与管道的区别：**

| 特性 | 管道 | 消息队列 |
|------|------|----------|
| 数据模型 | 无格式字节流 | 有类型结构消息 |
| 消息边界 | 由应用程序自行分割 | 内核维护消息边界 |
| 优先级 | 不支持 | 支持按消息类型（`msgtyp`）优先读取 |
| 生命周期 | 随进程（关闭即销毁） | 持久（需显式 `IPC_RMID`） |

# 6. 信号 (Signal)

信号是 **异步事件通知** 机制，内核或进程向目标进程发送一个整数编号的信号，目标进程注册的 handler 在**信号上下文**中执行。

**常用信号：**

| 信号 | 默认行为 | 常见触发场景 |
|------|----------|-------------|
| `SIGINT` (2) | 终止进程 | Ctrl+C |
| `SIGTERM` (15) | 终止进程 | `kill <PID>` |
| `SIGKILL` (9) | 终止进程（不可捕获） | `kill -9 <PID>` |
| `SIGSEGV` (11) | 终止 + coredump | 段错误 |
| `SIGUSR1` (10) | 终止 | 用户自定义 |
| `SIGPIPE` (13) | 终止进程 | 写已关闭的管道 |

### 6.1 C 示例 — 信号处理

```c
// signal_example.c
// 演示信号注册、发送与处理，包括 sigaction 高级用法
// 编译: gcc -std=c11 -Wall -o signal_example signal_example.c
// 运行: ./signal_example & 然后 kill -USR1 <PID>

#include <signal.h>
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

// 原子标志: 信号 handler 中设置，主循环中检查
static volatile sig_atomic_t g_usr1_received = 0;
static volatile sig_atomic_t g_sigint_received = 0;

// 信号处理函数: 只做最小工作（写入 volatile 标志）
static void HandleSigusr1(int sig) {
  (void)sig;
  g_usr1_received = 1;
}

static void HandleSigint(int sig) {
  (void)sig;
  g_sigint_received = 1;
}

int main() {
  // 使用 sigaction（优于 signal()，行为可移植且可控）
  struct sigaction sa_usr1 = {0};
  sa_usr1.sa_handler = HandleSigusr1;
  sigemptyset(&sa_usr1.sa_mask);
  sa_usr1.sa_flags = 0;  // 不设置 SA_RESTART，使 read/pause 可被中断
  if (sigaction(SIGUSR1, &sa_usr1, NULL) == -1) {
    perror("sigaction SIGUSR1");
    return EXIT_FAILURE;
  }

  struct sigaction sa_int = {0};
  sa_int.sa_handler = HandleSigint;
  sigemptyset(&sa_int.sa_mask);
  sa_int.sa_flags = 0;
  if (sigaction(SIGINT, &sa_int, NULL) == -1) {
    perror("sigaction SIGINT");
    return EXIT_FAILURE;
  }

  printf("[PID=%d] Waiting for SIGUSR1 or Ctrl+C...\n", getpid());

  // 主循环: 轮询标志而非在 handler 中做复杂工作（异步信号安全原则）
  while (!g_sigint_received) {
    if (g_usr1_received) {
      g_usr1_received = 0;
      printf("[Main] Received SIGUSR1 — performing deferred work.\n");
    }
    pause();  // 挂起等待信号（被信号中断后返回 -1，errno=EINTR）
  }

  printf("[Main] Received SIGINT, exiting gracefully.\n");
  return EXIT_SUCCESS;
}
```

> **⚠️ 异步信号安全**：信号处理函数中只能调用 **async-signal-safe** 函数（见 `signal-safety(7)`）。`printf`、`malloc`、`mutex_lock` 等均不安全。可靠做法：handler 只写 `volatile sig_atomic_t` 标志，主循环中检查并执行实际工作。

# 7. Unix 域套接字 (Unix Domain Socket)

Unix 域套接字（UDS）用**文件系统路径**作为地址，在同一主机上提供比 TCP 套接字**更高性能**（跳过网络协议栈）的双向通信。

**两种类型：**
- **流式 (SOCK_STREAM)**：类似 TCP，面向连接，可靠字节流
- **数据报 (SOCK_DGRAM)**：类似 UDP，保留消息边界，可靠（不丢包，同一主机）

### 7.1 C 示例 — UDS Stream 通信

```c
// uds_stream_example.c
// Unix 域套接字 (SOCK_STREAM) 实现双向 IPC
// 编译: gcc -std=c11 -Wall -o uds_stream_example uds_stream_example.c
// 运行: ./uds_stream_example

#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/socket.h>
#include <sys/un.h>
#include <unistd.h>

static const char* kSocketPath = "/tmp/uds_example.sock";
static const size_t kBufferSize = 1024;

int main() {
  // 清理可能残留的 socket 文件
  unlink(kSocketPath);

  int server_fd = socket(AF_UNIX, SOCK_STREAM, 0);
  if (server_fd == -1) {
    perror("socket");
    return EXIT_FAILURE;
  }

  struct sockaddr_un addr;
  memset(&addr, 0, sizeof(addr));
  addr.sun_family = AF_UNIX;
  strncpy(addr.sun_path, kSocketPath, sizeof(addr.sun_path) - 1);

  if (bind(server_fd, (struct sockaddr*)&addr, sizeof(addr)) == -1) {
    perror("bind");
    close(server_fd);
    return EXIT_FAILURE;
  }

  listen(server_fd, 1);

  pid_t pid = fork();
  if (pid == -1) {
    perror("fork");
    close(server_fd);
    return EXIT_FAILURE;
  }

  if (pid == 0) {
    // --- 子进程: 客户端 ---
    int client_fd = socket(AF_UNIX, SOCK_STREAM, 0);
    if (client_fd == -1) {
      perror("client socket");
      exit(EXIT_FAILURE);
    }

    // 连接服务端
    struct sockaddr_un server_addr;
    memset(&server_addr, 0, sizeof(server_addr));
    server_addr.sun_family = AF_UNIX;
    strncpy(server_addr.sun_path, kSocketPath, sizeof(server_addr.sun_path) - 1);

    if (connect(client_fd, (struct sockaddr*)&server_addr, sizeof(server_addr)) == -1) {
      perror("connect");
      exit(EXIT_FAILURE);
    }

    // 发送数据
    const char* request = "Hello from UDS client!";
    write(client_fd, request, strlen(request) + 1);

    // 读取响应
    char buffer[kBufferSize];
    ssize_t len = read(client_fd, buffer, sizeof(buffer) - 1);
    if (len > 0) {
      buffer[len] = '\0';
      printf("[Client] Server response: %s\n", buffer);
    }

    close(client_fd);
    exit(EXIT_SUCCESS);
  }

  // --- 父进程: 服务端 ---
  struct sockaddr_un client_addr;
  socklen_t client_len = sizeof(client_addr);
  int conn_fd = accept(server_fd, (struct sockaddr*)&client_addr, &client_len);
  if (conn_fd == -1) {
    perror("accept");
    close(server_fd);
    return EXIT_FAILURE;
  }

  char buffer[kBufferSize];
  ssize_t len = read(conn_fd, buffer, sizeof(buffer) - 1);
  if (len > 0) {
    buffer[len] = '\0';
    printf("[Server] Received: %s\n", buffer);
    const char* response = "Hello from UDS server!";
    write(conn_fd, response, strlen(response) + 1);
  }

  close(conn_fd);
  close(server_fd);
  unlink(kSocketPath);

  wait(NULL);
  printf("[Server] Done.\n");
  return EXIT_SUCCESS;
}
```

# 8. eventfd / signalfd / timerfd

Linux 还提供了几种现代化的**文件描述符就绪通知机制**，与 `epoll` 配合使用可以统一 I/O 事件循环：

| 接口 | 用途 | 核心函数 |
|------|------|----------|
| **`eventfd`** | 用户态事件通知计数器 | `eventfd()` → `read()`/`write()` |
| **`signalfd`** | 将信号转换为 fd 读取 | `signalfd()` |
| **`timerfd`** | 将定时器转换为 fd 读取 | `timerfd_create()` |

这些机制的共同优势是可以被 `epoll` 统一监视，避免信号处理函数中的异步安全问题。

# 9. IPC 机制选型指南

| 场景 | 推荐机制 | 理由 |
|------|----------|------|
| 简单数据流（过滤器/管道） | 匿名管道 | 零配置，shell 天然支持 |
| 无关进程间文件传递 | FIFO / 文件 | 持久化，无血缘限制 |
| 大量数据高频共享 | 共享内存 | 零拷贝，吞吐最高 |
| 小消息 / 优先级调度 | 消息队列 | 类型过滤，无需同步 |
| 异步事件通知 | 信号 / eventfd | 轻量，适合事件驱动 |
| 复杂双向通信（C/S 模式） | Unix 域套接字 | 全双工，select/epoll 友好 |
| 进程互斥 / 资源计数 | 信号量 / 文件锁 | 仅同步需求，不传数据 |
| 统一事件循环 | eventfd + epoll | 避免信号安全问题 |

**性能参考（ping-pong 模式，单机）：**

| 机制 | 延迟（近似） | 带宽 |
|------|-------------|------|
| Pipe | ~1–5 µs | 中等 |
| FIFO | ~2–6 µs | 中等 |
| 共享内存 | ~0.1–0.5 µs | **最高** |
| 消息队列 | ~3–10 µs | 中低 |
| Unix 域套接字 | ~1–3 µs | 中高 |
| TCP 本地回环 | ~5–15 µs | 中 |

> 数据为单次小消息往返延迟的近似范围，硬件/内核版本不同会有所浮动。

## 9.1 常见踩坑

- **pipe 未 close 无用 fd**：fork 后父/子各关一端，否则 read 永不 EOF
- **FIFO 打开顺序**：只 open 写端会阻塞到读端出现（反之亦然）→ 双端就绪或 `O_NONBLOCK`
- **共享内存无锁**：数据竞态 → 必须配信号量/mutex
- **System V IPC 残留**：进程崩溃未 `IPC_RMID` → `ipcs` 查看并手动清理
- **SIGPIPE**：写已关闭的 pipe → 默认终止进程；网络代码常 `signal(SIGPIPE, SIG_IGN)`
- **UDS 路径残留**：`bind` 前 `unlink(sockpath)`；见 [[C-Linux生态/07.Telescope1126实战/3. Unix域套接字IPC实战|UDS 实战]]
