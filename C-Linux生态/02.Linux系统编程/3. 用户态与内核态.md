# 1. 用户态与内核态

## 【核心精简提纲】

### 记忆主干

用户态（Ring 3）与内核态（Ring 0）地址空间隔离，用户程序**不能直接调用内核函数**，唯一合法路径是**系统调用**（C 库封装 → 软中断 → 内核 `sys_*`）。自定义驱动通信用 `ioctl`/`sysfs`/`netlink`；`ioctl` 的 cmd/struct 须用户态与内核态**手动保证 ABI 一致**。

### 核心要点清单

- 隔离原因：地址空间分离、权限分离（特权指令）、稳定性（用户错误不拖垮内核）
- 系统调用链：`open()` 等是 glibc 封装，内核侧实现为 `sys_open()` 等
- `ioctl`：接口标准、参数自定义 → 公共头文件 + `_IO/_IOR/_IOW` 宏
- 数据拷贝：必须 `copy_to_user`/`copy_from_user`，禁止直接解引用用户指针
- 机制选型：简单参数 → sysfs；自定义命令 → ioctl；内核推送 → netlink
- 标准库 ABI 由 OS/编译器保证；驱动 ABI 由开发者保证

### 关键结论 / 流程

```plain
用户 open/read/ioctl → 系统调用 → 驱动 file_operations → copy_*_user 安全拷贝
```

## 【完整拓展详情】


用户程序**不能直接调用内核函数**，必须通过 **系统调用（System Call）** 间接调用内核功能。这是操作系统保证稳定性、安全性、可维护性的核心设计原则。

## 1.1 为什么用户程序不能直接调用内核函数

### 1.1.1 地址空间隔离

每个用户进程有独立的**虚拟地址空间**（32 位系统通常 0~3GB），内核有独立的**内核地址空间**（32 位系统通常 3~4GB）。内核地址空间对用户进程**不可见、不可访问**。

如果用户程序直接以函数地址调用内核函数，该地址在用户空间是**无效的或指向用户程序自身的内存**，会触发段错误（`SIGSEGV`）。

### 1.1.2 权限隔离

| 级别 | 权限 | 运行内容 |
|------|------|----------|
| **Ring 3（用户态）** | 普通权限 | 用户程序，无法执行特权指令 |
| **Ring 0（内核态）** | 最高权限 | 内核，可执行所有特权指令（修改页表、访问硬件、开关中断） |

内核函数包含大量**特权指令和敏感操作**。如果用户程序可以直接调用内核函数，等于赋予了用户程序最高权限，恶意程序可以轻易修改内核数据、控制硬件、窃取信息。

### 1.1.3 稳定性隔离

用户程序的错误**只影响自身**（如内存越界只导致自己崩溃），而内核的错误**会影响整个系统**（内核崩溃 → Kernel Panic → 系统死机）。允许用户程序直接调用内核函数，会让用户程序的任何错误都可能导致内核崩溃。

## 1.2 系统调用：用户态与内核态的唯一桥梁

系统调用是内核暴露给用户态的**标准化接口**，是用户程序调用内核功能的**唯一合法方式**。

### 1.2.1 工作流程

以 `open()` 为例：

```plain
用户程序调用 open()         ← 标准 C 库的封装函数
    ↓
封装函数触发软中断           ← x86: int 0x80 / syscall 指令
    ↓                       ← CPU 从 Ring 3 切换到 Ring 0
内核系统调用处理函数          ← 根据系统调用号分发
    ↓
内核函数 sys_open() 执行     ← 真正实现功能的内核函数
    ↓
结果逐级返回至用户程序
```

```c
#include <fcntl.h>       // 声明 open() — 系统调用的封装函数，非内核函数

int main() {
    int fd = open("/dev/multi_func_drv", O_RDWR);  // 调用封装函数
    return 0;
}
```

> `fcntl.h` 中声明的 `open()` 不是内核函数，而是 C 库提供的**系统调用封装函数**，其内部通过软中断指令切换到内核态。真正实现功能的是内核中的 `sys_open()`。

### 1.2.2 系统调用的标准化

内核保证系统调用的 **ABI（Application Binary Interface）稳定性和兼容性**——只要系统调用的接口不变，用户程序可在不同内核版本上正常运行。标准系统调用（`open()`、`read()`、`write()`）的 ABI 已由操作系统标准化，开发者无需手动定义。

## 1.3 ioctl 与手动定义 ABI

### 1.3.1 ioctl 的特殊性

`ioctl` 是一种**特殊的系统调用**：

- **标准化接口**：函数原型 `int ioctl(int fd, unsigned int cmd, unsigned long arg)`
- **自定义参数**：`cmd`（命令）和 `arg`（参数）由驱动开发者定义，内核无法预知

因此内核无法为 `ioctl` 提供标准化的 ABI，必须由开发者**手动定义**并保证用户态与内核态一致。

### 1.3.2 必须手动定义的 ABI 内容

```c
// 公共头文件（用户态和内核态共用，保证二进制兼容）
#define DRV_IOCTL_MAGIC 'M'
#define DRV_IOCTL_SET_DATA _IOW(DRV_IOCTL_MAGIC, 0, struct drv_data)
#define DRV_IOCTL_GET_DATA _IOR(DRV_IOCTL_MAGIC, 1, struct drv_data)

struct drv_data {
    int id;              // 4 字节
    char buf[64];        // 64 字节
    unsigned int len;    // 4 字节
};
```

| 需保证一致的内容 | 不一致的后果 |
|----------------|-------------|
| `ioctl` 命令宏（`_IO`/`_IOR`/`_IOW`）| 内核无法识别命令，返回 `-EINVAL` |
| 共享数据结构（成员顺序、大小、对齐） | 数据解析错误，甚至内核崩溃 |
| 宏定义和常量（设备号、错误码、标志位） | 逻辑错误 |

### 1.3.3 为什么标准库不需要手动定义 ABI

| 层面 | 保证机制 |
|------|----------|
| **编译器** | GCC 等保证标准数据类型（`int`/`long`/`char`）的大小和对齐在用户态和内核态一致 |
| **操作系统** | Linux 内核保证标准系统调用的 ABI 跨版本、跨平台一致 |
| **标准库** | glibc 等保证库函数 ABI 跨版本兼容 |

因此调用标准库函数（`printf()`、`malloc()`）**不需要手动定义 ABI**，这些已由系统标准化。

## 1.4 内核态-用户态通信机制对比

内核提供 4 种标准化机制供用户态与内核态通信：

| 机制 | 适用场景 | 实现难度 | 灵活性 | 推荐指数 |
|------|----------|----------|--------|----------|
| **`ioctl`** | 自定义命令 + 双向数据传输（设置/读取参数、执行功能） | 中 | 中 | ★★★★★ |
| **`sysfs`** | 暴露简单只读/只写参数（电压、状态、计数） | 低 | 低 | ★★★★☆ |
| **`netlink`** | 内核主动推送数据 + 高频双向通信 | 高 | 高 | ★★★☆☆ |
| **`procfs`** | 暴露调试信息 | 低 | 低 | ★★☆☆☆（不推荐新驱动使用） |

### 1.4.1 `sysfs`——最简单

在 `/sys/class/` 下创建属性文件，实现 `show`（读）和 `store`（写）函数。用户态用 `cat`/`echo` 操作，无需编写复杂用户程序。

```c
// 驱动端：创建 /sys/class/charge_gateway/param
static ssize_t param_show(struct device *dev, struct device_attribute *attr, char *buf) {
    return sprintf(buf, "%d\n", g_charge_param);  // 用户态 cat 读取
}
static ssize_t param_store(struct device *dev, struct device_attribute *attr,
                           const char *buf, size_t count) {
    sscanf(buf, "%d", &g_charge_param);           // 用户态 echo 写入
    return count;
}
static DEVICE_ATTR_RW(param);
```
```bash
# 用户态操作
cat /sys/class/charge_gateway/param       # 读取参数
echo 123 > /sys/class/charge_gateway/param # 设置参数
```

### 1.4.2 `netlink`——最灵活

基于套接字的通信机制，支持**内核态主动推送数据**到用户态（如充电状态变化通知）：

- 驱动创建 netlink 套接字，注册回调接收用户态消息
- 用户程序创建 netlink 套接字，通过 `sendto()` 发送命令
- 驱动通过 `netlink_unicast()` 主动推送数据到用户程序

适合高频、低延迟的双向通信。

### 1.4.3 `procfs`——最传统

在 `/proc/` 下创建文件，实现 `read`/`write` 函数。语法简单但已不推荐在新驱动中使用（`sysfs` 是更好的替代）。

### 1.4.4 `ioctl`——最常用（推荐）

完整用法和 Demo 参见 [[4. 多功能字符设备驱动|多功能字符设备驱动]]。以下为精简示例：

**驱动端（`charge_gateway.c`）**：
```c
#define MY_IOCTL_MAGIC 'C'
#define MY_IOCTL_GET_PARAM _IOR(MY_IOCTL_MAGIC, 0, int)
#define MY_IOCTL_SET_PARAM _IOW(MY_IOCTL_MAGIC, 1, int)
#define MY_IOCTL_EXEC_FUNC  _IO(MY_IOCTL_MAGIC, 2)

static int g_charge_param = 0;

static long my_ioctl(struct file *filp, unsigned int cmd, unsigned long arg) {
    int __user *user_buf = (int __user *)arg;
    switch (cmd) {
    case MY_IOCTL_GET_PARAM:
        if (copy_to_user(user_buf, &g_charge_param, sizeof(int))) return -EFAULT;
        break;
    case MY_IOCTL_SET_PARAM:
        if (copy_from_user(&g_charge_param, user_buf, sizeof(int))) return -EFAULT;
        break;
    case MY_IOCTL_EXEC_FUNC:
        // 执行功能代码
        break;
    default: return -EINVAL;
    }
    return 0;
}
```

**用户态测试**：
```c
int fd = open("/dev/my_driver", O_RDWR);
int param = 123;
ioctl(fd, MY_IOCTL_SET_PARAM, &param);      // 注入参数
ioctl(fd, MY_IOCTL_EXEC_FUNC);              // 执行功能
ioctl(fd, MY_IOCTL_GET_PARAM, &param);      // 提取参数
printf("param = %d\n", param);
close(fd);
```

### 1.4.5 安全注意事项

| 原则 | 说明 |
|------|------|
| **安全拷贝** | `copy_to_user()` / `copy_from_user()`，禁止直接指针访问用户态内存 |
| **参数验证** | 验证缓冲区大小、命令有效性、参数合法性 |
| **内核宏定义命令** | 必须用 `_IO`/`_IOR`/`_IOW`/`_IOWR` 宏，保证跨平台兼容性 |

## 1.5 实战建议

1. **使用标准化接口**：优先用 `ioctl`、`sysfs`、`netlink` 等内核提供的标准化接口实现交互
2. **公共头文件**：将 `ioctl` 命令、数据结构、宏定义放在**公共头文件**中，用户态和内核态**共用同一份**
3. **使用内核宏**：定义 `ioctl` 命令时必须用 `_IO`、`_IOR`、`_IOW`、`_IOWR` 宏，保证唯一性和跨平台兼容
4. **数据拷贝**：用户态与内核态交互时使用 `copy_to_user()` / `copy_from_user()`，**禁止直接访问内核内存**
5. **严格参数检查**：驱动的 `ioctl` 处理函数中必须检查命令有效性、数据合法性、地址安全性

## 1.6 常见踩坑

| 现象 | 原因 | 处理 |
|------|------|------|
| `ioctl` 返回 `-EINVAL` | cmd 宏或 magic 不一致 | 用户态/内核态共用同一份头文件 |
| 数据错乱或内核 oops | struct 对齐/大小不一致 | `#pragma pack` 慎用；32/64 位混编须固定宽度类型 |
| 用户指针访问崩溃 | 直接 `*(int*)arg` | 一律 `copy_from_user` / `copy_to_user` |
| `sysfs` 写入无效 | 权限或 `store` 未实现 | 检查 `chmod`；确认 attribute 为 RW |
| 32 位用户态 + 64 位内核 | `unsigned long arg` 传指针截断 | 用 `_IOC` 宏或 `compat_ioctl` |
