# 1. CPU亲和性（核心绑定）

CPU 亲和性（Affinity）用于**限制进程或线程在指定 CPU 核心上运行**，是嵌入式 Linux 多核调度与性能优化的核心手段。

## 1.1 基本概念

| 术语 | 定义 | 通俗理解 |
|------|------|----------|
| **CPU Affinity**（CPU亲和性） | 进程/线程**倾向于在某个/某些核心上运行**的属性 | 给进程贴标签：**尽量**在这些核心上跑 |
| **Core Binding**（核心绑定） | **强制**进程/线程**只能在指定核心上运行**的操作 | 把进程「钉死」在指定核心，**不准**调度到其他核心 |
| **CPU Core Group**（CPU核心组） | 将多个物理核心划分为逻辑组，用于**分组调度、隔离任务** | 给核心「分班组」：组0跑实时任务，组1跑普通任务 |

**关键关系**：

- **核心绑定**是 CPU 亲和性的**强约束版本**——普通亲和性是建议性的（内核可以不遵守），核心绑定是强制性的（内核必须遵守）。
- **CPU 核心组**基于 CPU 亲和性实现——给不同组的进程设置不同的亲和性掩码，让它们只能在自己组的核心上运行。

### 1.1.1 软亲和性与硬亲和性

| 特性 | 硬亲和性（核心绑定） | 软亲和性（默认调度） |
|------|-------------------|-------------------|
| **约束强度** | 强制约束，任务只能在指定核心上运行 | 建议约束，任务倾向于在指定核心上运行，但内核可根据负载调度到其他核心 |
| **优点** | 缓存命中率高、实时性强、任务隔离好 | 负载均衡好、整体利用率高、容错性好 |
| **缺点** | 负载失衡、资源浪费、容错性差 | 有跨核心迁移风险、实时性无法保证 |
| **设置方式** | `taskset -p <mask> <pid>`、`sched_setaffinity` | 内核自动实现；可通过 `sched_setattr` 调整策略 |
| **适用场景** | 实时任务、驱动中断、缓存敏感型任务 | 普通任务、对实时性要求不高但希望减少迁移的任务 |

### 1.1.2 亲和性掩码（Affinity Mask）

CPU 亲和性用**位图（bitmask）** 表示，每一位对应一个核心：

- `0b0001`（`0x01`）→ 仅核心 0
- `0b0010`（`0x02`）→ 仅核心 1
- `0b0101`（`0x05`）→ 核心 0 和核心 2
- `0b1111`（`0x0f`）→ 所有核心（默认状态）

```bash
# 掩码计算小技巧：十六进制转二进制统计核心数
echo "obase=2; ibase=16; 0x05" | bc
> 101  # 2个1 → 2个核心
```

## 1.2 跨核心迁移的性能影响

**跨核心迁移（Cross-Core Migration）** 是内核为平衡负载，将进程/线程从核心 A 转移到核心 B 执行的过程。这是亲和性要解决的核心痛点。

### 1.2.1 性能杀手一：私有缓存完全失效

多核 CPU 的缓存分层结构（以 ARM Cortex-A 为例）：

| 缓存层级 | 属性 | 容量 | 延迟（时钟周期） |
|----------|------|------|-----------------|
| L1 Cache | 每个核心**私有**（指令+数据） | 32KB~64KB | 1~2 |
| L2 Cache | 通常每个核心**私有** | 256KB~512KB | 5~10 |
| L3 Cache | 所有核心**共享** | 1MB~8MB | 20~30 |
| 内存 | 所有核心共享 | — | **100~200** |

跨核心迁移时，进程在原核心的 L1/L2 私有缓存全部失效，新核心需要从共享缓存或内存重新加载数据，**数据访问延迟飙升 10~200 倍**。如果进程被频繁迁移，它**永远无法建立有效的缓存**，性能被严重拉垮。

### 1.2.2 性能杀手二：上下文重建开销

- **流水线完全清空**：新核心恢复进程时，流水线中所有指令被清空，需要重新取指、译码、执行
- **分支预测器历史失效**：新核心没有该进程的分支历史，预测准确率大幅下降，导致更多流水线停顿
- **直接开销**：保存/恢复寄存器需要数十到数百个时钟周期

### 1.2.3 与已有知识点的关联

```
CPU 流水线满负荷运行
    ↓ 遇到分支指令
分支预测 → 预测失败 → 流水线清空（倍数级损失）
    ↓ 跨核心迁移
缓存失效 + 流水线清空 + 分支预测历史失效（数量级损失）
    ↓ 解决方案
CPU 亲和性 / 核心绑定
```

性能调优的**性价比排序**：先解决跨核心迁移（数量级收益）→ 再优化分支指令和流水线（倍数级收益）。

嵌入式场景中跨核心迁移的危害更大：缓存更小、分支预测能力更弱、实时性要求更高。

## 1.3 单核与多核环境下的行为差异

| 场景 | 亲和性是否有概念？ | 是否有实际作用？ |
|------|------------------|----------------|
| **物理单核 CPU**（如 STM32、老旧 x86） | 概念存在，内核仍维护掩码 | 完全无效（只有 1 个核心可选） |
| **逻辑单核**（多核 CPU 屏蔽其他核心） | 概念存在 | 无效（与物理单核一致） |
| **多核 CPU** | 概念存在 | 有效，可避免跨核心迁移 |

### 1.3.1 单核环境下亲和性掩码强制归一化

无论通过 `taskset` 还是 `sched_setaffinity` 设置何种掩码，内核都会自动归一化为**仅包含唯一核心的掩码**：

```bash
taskset -p 0xf 1234   # 尝试设置为全核心
taskset -p 1234       # 查询：pid 1234's current affinity mask: 1
```

### 1.3.2 系统调用不会报错的陷阱

单核环境下调用 `sched_setaffinity`、`pthread_setaffinity_np` **不会返回错误**——内核自动忽略不存在的核心，将掩码调整为可用核心：

```c
cpu_set_t mask;
CPU_ZERO(&mask);
CPU_SET(1, &mask);                           // 尝试绑定核心 1（不存在）
int ret = sched_setaffinity(0, sizeof(mask), &mask);
printf("%d\n", ret);                         // 输出 0（成功），不会报错

CPU_ZERO(&mask);
sched_getaffinity(0, sizeof(mask), &mask);
printf("%d\n", CPU_COUNT(&mask));           // 输出 1（被归一化）
```

## 1.4 绑定粒度：线程 > 进程 > 整个程序

粒度从细到粗：**线程 → 进程 → 整个程序**。优先级：**线程级绑定 > 进程级绑定**。

| 绑定粒度 | 本质 | 优先级 | 适用场景 |
|----------|------|--------|----------|
| **线程** | 系统支持的最小粒度，给单个线程设置掩码 | 最高（覆盖进程设置） | 多线程精细化调度：计算线程绑核心 0~1，IO 线程绑核心 2~3 |
| **进程** | 给主线程设置默认掩码，子线程未单独设置则继承 | 中（被线程覆盖） | 单线程程序整体绑定；多线程程序的默认调度范围 |
| **整个程序** | 工程概念，通过脚本/启动参数/统一函数批量设置 | 最低（本质是批量设置） | 部署阶段统一配置；系统级资源隔离 |

### 1.4.1 进程与线程的亲和性继承关系

**进程是资源的容器，线程是调度的基本单位**。CPU 亲和性是调度属性，本质上是线程的属性。

- 若**线程未单独设置亲和性**，则**继承进程主线程的亲和性掩码**
- 若**线程单独设置了亲和性**，则**使用自身掩码，与进程默认设置无关**

```c
#include <stdio.h>
#include <pthread.h>
#include <sys/sched.h>

void *thread_func(void *arg) {
    cpu_set_t thread_mask, proc_mask;
    CPU_ZERO(&thread_mask);
    pthread_getaffinity_np(pthread_self(), sizeof(thread_mask), &thread_mask);
    CPU_ZERO(&proc_mask);
    sched_getaffinity(0, sizeof(proc_mask), &proc_mask);
    printf("线程核心数 = %d，进程核心数 = %d\n",
           CPU_COUNT(&thread_mask), CPU_COUNT(&proc_mask));
    printf("继承进程设置：%s\n",
           CPU_COUNT(&thread_mask) == CPU_COUNT(&proc_mask) ? "是" : "否");
    pthread_exit(NULL);
}

int main() {
    cpu_set_t proc_mask;
    CPU_ZERO(&proc_mask);
    CPU_SET(0, &proc_mask);
    sched_setaffinity(0, sizeof(proc_mask), &proc_mask);

    pthread_t t1, t2;
    pthread_create(&t1, NULL, thread_func, NULL);       // 继承进程设置
    pthread_join(t1, NULL);

    pthread_create(&t2, NULL, thread_func, NULL);
    cpu_set_t t2_mask;
    CPU_ZERO(&t2_mask);
    CPU_SET(1, &t2_mask);
    pthread_setaffinity_np(t2, sizeof(t2_mask), &t2_mask);  // 覆盖进程设置
    pthread_join(t2, NULL);
    return 0;
}
```

```bash
gcc affinity_demo.c -o affinity_demo -lpthread
./affinity_demo
```
```
进程：已绑定到核心 0，核心数 = 1
线程：自身核心数 = 1，进程核心数 = 1    ← 继承
继承进程设置：是
线程：自身核心数 = 1，进程核心数 = 1    ← 覆盖
继承进程设置：否
```

## 1.5 核心绑定的优缺点与选型

### 1.5.1 优点

| 维度 | 原理 | 收益 |
|------|------|------|
| **缓存命中率** | 进程驻留固定核心，L1/L2 私有缓存持续有效 | 命中率从 <50% 提升到 >90%，延迟从内存级（100~200周期）降为 L1 级（1~2周期），**性能提升 10~100 倍** |
| **实时性** | 不被其他核心任务干扰，流水线/TLB/分支预测状态持续有效 | 响应时间抖动从毫秒级降到微秒级，满足 1ms 控制周期等硬实时需求 |
| **任务隔离** | 不同类型任务绑定到不同核心，实现物理级隔离 | 后台任务卡死不影响实时任务；日志打印不影响驱动响应 |
| **流水线与分支预测** | 流水线不被清空，分支预测器积累完整执行历史 | 循环/分支密集型任务执行效率提升 20%~50% |

### 1.5.2 缺点

| 维度 | 风险 | 说明 |
|------|------|------|
| **负载失衡** | 绑定核心过载时内核无法将任务调度到其他空闲核心 | CPU 密集型批量任务**绝不能**绑定到单个核心 |
| **资源浪费** | 绑定核心专用化，即使任务休眠也无法被其他任务利用 | 嵌入式核心数少（2/4/8核），滥用绑定会导致普通任务资源不足 |
| **容错性差** | 绑定核心硬件故障或中断风暴时，所有绑定任务受影响 | 关键任务（看门狗、紧急停机）**必须绑定到多个核心**或不绑定 |
| **调试困难** | 问题与特定核心强相关，需针对性调试 | 绑定代码必须添加清晰注释，注明核心号、目的、风险 |

### 1.5.3 使用决策树

1. **是否是实时任务？** → 是 → **必须绑定核心（硬亲和性）**
2. **是否是缓存敏感型任务？** → 是 → **建议绑定核心（硬亲和性）**
3. **是否是 CPU 密集型批量任务？** → 是 → **绑定到多个核心，绝不绑单核心**
4. **是否是关键任务？** → 是 → **绑定到多个核心（提高容错性）**
5. **是否是普通/后台任务？** → 是 → **不绑定，使用默认调度策略**

**驱动开发场景**：
- 中断处理函数、工作队列 → 绑定核心（硬亲和性）
- 辅助性后台任务 → 不绑定

## 1.6 判断任务的运行核心范围

### 1.6.1 命令行工具

#### `taskset` —— 直接查看亲和性掩码（金标准）

```bash
taskset -p <pid>

# 输出示例
# pid 1234's current affinity mask: 1    → 单核（0b0001）
# pid 5678's current affinity mask: 5    → 多核（0b0101，核心 0+2）
# pid 9012's current affinity mask: f    → 全核心（0b1111）
```

二进制中仅 1 位为 1 → 单核运行；≥2 位为 1 → 多核运行。

#### `top` —— 实时观察运行核心

`top` 进入后按 `f` → 选中 `P = Last Used Cpu` → 按 `q` 返回。`P` 列显示任务最后一次运行的核心号：
- `P` 列值始终不变 → 大概率单核绑定
- `P` 列值频繁变化 → 大概率多核调度

> 这是间接判断，单核任务 `P` 值不变，但多核任务可能短期不变，需结合 `taskset` 验证。

#### `htop` —— 直观图形化工具

`F2` → `Columns` → 添加 `CPU affinity` 和 `CPU` 列。前者显示亲和性掩码（如 `0,2`），后者显示当前运行核心。

### 1.6.2 代码检测

#### 进程级：`sched_getaffinity`

```c
#include <stdio.h>
#include <sys/sched.h>
#include <unistd.h>

int check_process_affinity(pid_t pid) {
    cpu_set_t mask;
    CPU_ZERO(&mask);
    if (sched_getaffinity(pid, sizeof(mask), &mask) == -1) {
        perror("sched_getaffinity failed");
        return -1;
    }
    int core_count = CPU_COUNT(&mask);
    printf("Process %d: core count = %d → %s\n",
           pid, core_count,
           core_count >= 2 ? "多核运行" : "单核运行");
    return core_count;
}
```

#### 线程级：`pthread_getaffinity_np`

```c
#include <stdio.h>
#include <pthread.h>

void *thread_func(void *arg) {
    cpu_set_t mask;
    CPU_ZERO(&mask);
    pthread_getaffinity_np(pthread_self(), sizeof(mask), &mask);
    int core_count = CPU_COUNT(&mask);
    printf("Thread %lu: core count = %d → %s\n",
           (unsigned long)pthread_self(), core_count,
           core_count >= 2 ? "多核运行" : "单核运行");
    pthread_exit(NULL);
}
```

### 1.6.3 内核态检测（驱动开发）

在驱动中检测进程/线程的运行核心范围：

```c
#include <linux/sched.h>
#include <linux/cpumask.h>

struct task_struct *task = current;
cpumask_t mask = task_cpu_possible_mask(task);
int core_count = cpumask_weight(&mask);

if (core_count >= 2)
    printk(KERN_INFO "Process %s is multicore (%d cores)\n",
           task->comm, core_count);
else
    printk(KERN_INFO "Process %s is single-core\n", task->comm);
```

### 1.6.4 运行核心范围 vs 并行度

两个容易混淆的概念：

- **运行核心范围**：任务**可以在哪些核心上运行**（由亲和性掩码决定）→ 调度器的约束
- **并行度**：任务**同时在多少个核心上运行**（由线程数决定）→ 任务自身的属性

**示例**：

| 场景 | 亲和性掩码 | 运行核心范围 | 并行度 |
|------|-----------|-------------|--------|
| 单线程任务，未绑定 | `0b1111`（4核） | 多核范围 | 1（只能同时跑在 1 个核心） |
| 4 线程任务，未绑定 | `0b1111`（4核） | 多核范围 | 4（可同时跑在 4 个核心） |
| 4 线程任务，绑核心0 | `0b0001`（1核） | 单核范围 | 1（4线程串行执行） |

## 1.7 代码兼容性陷阱

### 陷阱 1：依赖单核亲和性无效性，移植多核后性能下降

单核环境开发的代码未设置亲和性，移植到多核后任务被频繁调度到不同核心，导致缓存失效：

```c
void set_affinity_if_multicore() {
    int core_num = sysconf(_SC_NPROCESSORS_ONLN);
    if (core_num > 1) {
        cpu_set_t mask;
        CPU_ZERO(&mask);
        CPU_SET(0, &mask);
        sched_setaffinity(0, sizeof(mask), &mask);
        printf("多核环境：绑定到核心 0\n");
    } else {
        printf("单核环境：无需设置\n");
    }
}
```

### 陷阱 2：硬编码掩码，限制多核并行性

单核开发时硬编码 `0b0001`，移植到多核后任务被限制在核心 0：

```c
void set_multithread_affinity() {
    int core_num = sysconf(_SC_NPROCESSORS_ONLN);
    cpu_set_t mask;
    CPU_ZERO(&mask);

    if (core_num > 1) {
        for (int i = 0; i < core_num; i++)
            CPU_SET(i, &mask);
        printf("多核环境：允许在所有 %d 个核心上运行\n", core_num);
    } else {
        CPU_SET(0, &mask);
        printf("单核环境：绑定到核心 0\n");
    }
    sched_setaffinity(0, sizeof(mask), &mask);
}
```

## 1.8 嵌入式开发实践建议

### 1.8.1 推荐绑定策略

| 场景 | 推荐绑定粒度 | 原因 |
|------|-------------|------|
| 单线程程序 | 进程级绑定 | 简单方便，效果与线程级一致 |
| 多线程程序（计算密集型） | 线程级绑定，各线程绑不同核心 | 最大化并行性 |
| 多线程程序（IO密集型） | 不绑定或软亲和性 | IO 等待时内核可切换线程 |
| 驱动开发 | 线程级绑定 | 中断处理线程必须单独设置 |
| 实时任务调度 | 线程级绑定 | 保证响应时间可预测 |
| 部署阶段 | 整个程序级绑定 | 统一配置，系统级资源隔离 |

### 1.8.2 5 条实战准则

1. **只绑热点任务，不绑冷任务**：用 `perf` 找到热点任务（高执行频率、高缓存缺失率）后再绑定，冷任务无意义。
2. **绑定到空闲核心**：用 `top`/`htop` 观察各核心负载，选择负载最低的核心绑定。
3. **批量任务绑多核**：CPU 密集型批量任务绑定到多个核心实现负载均衡（如任务绑核心 0~3）。
4. **关键任务绑多核**：看门狗、紧急停机等关键任务绑定到多个核心提高容错性。
5. **添加监控与回退机制**：监控核心负载和任务响应时间，过载时自动取消绑定恢复默认调度。

### 1.8.3 特殊场景：单核带超线程

物理单核但逻辑双核（Hyper-Threading），亲和性**有实际作用**——可限制任务在哪个逻辑核心上运行。应**视为多核环境**处理。

### 1.8.4 验证核心绑定效果

```bash
# 统计缓存缺失率
perf stat -e cache-misses,cache-references ./your_program

# 统计跨核心迁移次数
perf stat -e migrations ./your_program

# top 中查看 Last Used Cpu（按 f 键添加 P 列）
top
```

**核心原则**：
- **最小粒度原则**：尽量用线程级绑定，实现最精细化的调度控制
- **优先级原则**：线程级绑定 > 进程级绑定
- **工程实用原则**：根据程序实际需求平衡灵活性和方便性
