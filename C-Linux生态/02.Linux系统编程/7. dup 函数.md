# 1. 概述

## 【核心精简提纲】

### 记忆主干

`dup`/`dup2`/`dup3` 复制 FD，使两个描述符指向同一内核文件表项（**共享文件偏移量**）。Shell 重定向与管道的底层：`dup2(fd, 1)` 实现 `>`；`dup2(1, 2)` 实现 `2>&1`；`pipe` + `dup2` + `fork` + `exec` 实现 `cmd1 | cmd2`。

### 核心要点清单

- `dup(oldfd)`：分配到最小可用 FD；`dup2(oldfd, newfd)`：指定编号，先关 newfd（原子）
- `dup3`：同 dup2 + 可设 `O_CLOEXEC`（多线程安全，防 exec 泄漏）
- 共享偏移：一个 FD read/write 后，dup 出的 FD 偏移同步变化
- 重定向标准做法：`dup2(file_fd, STDOUT_FILENO); close(file_fd);`
- 管道：`dup2(pipefd[1], STDOUT_FILENO)` 写端；子进程 `dup2(pipefd[0], STDIN_FILENO)` 读端
- 备份恢复：`saved = dup(1); ... dup2(saved, 1); close(saved);`

### 关键结论 / 流程

```plain
> file：open → dup2(fd,1) → close(fd) │ 2>&1：dup2(1,2) │ 管道：pipe → dup2 → fork → exec
```

## 【完整拓展详情】


`dup`、`dup2`、`dup3` 是 Linux 系统编程中用于**复制文件描述符**的系统调用。它们允许创建一个新的文件描述符，指向与原有描述符相同的**内核文件表项**（即共享同一个文件偏移量、打开模式、状态标志等）。

```plain
        进程文件描述符表            内核文件表
        ┌─────────┐            ┌──────────────┐
   FD 3 ┤─────────┤───────────►│  文件状态     │
        │         │            │  文件偏移量   │
   FD 5 ┤─────────┤───────────►│  打开模式     │
        │         │    共享    │   vnode 指针  │
        └─────────┘            └──────┬───────┘
                                      │
                                      ▼
                               ┌──────────────┐
                               │   inode       │
                               │  （具体文件）  │
                               └──────────────┘
```

> **核心要点**：`dup` 复制的是**文件描述符本身**而非底层的文件对象。两个 FD 指向同一个内核文件表项，共享偏移量——一个读/写了，另一个的偏移量也跟着变。

# 2. 函数原型

```c
#include <unistd.h>

int dup(int oldfd);
int dup2(int oldfd, int newfd);

#define _GNU_SOURCE
#include <unistd.h>

int dup3(int oldfd, int newfd, int flags);
```

| 函数 | 返回值 | 行为 |
|------|--------|------|
| `dup(oldfd)` | 返回**最小的未用 FD** | 复制 oldfd 到空闲位 |
| `dup2(oldfd, newfd)` | 返回 newfd | 将 newfd 指向 oldfd 的表项；若 newfd 已打开，先关闭 |
| `dup3(oldfd, newfd, flags)` | 返回 newfd | 同 `dup2`，增加 `flags` 控制 |

### 参数说明

- `oldfd` — 要复制的源文件描述符
- `newfd` — 指定的新文件描述符编号（仅 `dup2` / `dup3`）
- `flags` — 仅 `dup3`，支持：
  - `O_CLOEXEC` — 在 `exec` 时自动关闭新 FD（避免执行子进程后泄漏）

### 返回值

- 成功：返回新的文件描述符编号
- 失败：返回 -1，设置 `errno`

### 常见错误

| errno | 原因 |
|-------|------|
| `EBADF` | `oldfd` 不是已打开的有效 FD |
| `EMFILE` | 进程已达到 `RLIMIT_NOFILE` 上限 |
| `EINTR` | 被信号中断 |
| `EINVAL` | `dup3` 中 `flags` 包含无效值 |
| `EBUSY` | `dup2` / `dup3` 中 `oldfd == newfd` 且不可执行（仅某些实现） |

# 3. 行为对比

```c
int fd = open("file.txt", O_RDWR);

int new_fd1 = dup(fd);           // 新 FD = 当前最小可用（如 4）
int new_fd2 = dup2(fd, 10);      // 新 FD = 10（若 10 已打开，先关 10）
int new_fd3 = dup3(fd, 20, O_CLOEXEC); // 新 FD = 20，且设置 close-on-exec
```

图示 `dup(fd)` 的效果：

```plain
dup() 前:
  FD 3 ──→  ┌─────────────────┐
             │  文件 A          │  偏移量: 0
             │  /tmp/data.txt   │
             └─────────────────┘

dup(3) 后:
  FD 3 ──→  ┌─────────────────┐
             │  文件 A          │  偏移量: 0（共享）
  FD 4 ──→  │  /tmp/data.txt   │
             └─────────────────┘

read(3, buf, 100)  → 偏移量变 100
read(4, buf, 100)  → 从 100 开始读（不是从头！）
```

# 4. 经典使用场景

## 4.1 场景一：重定向标准输出（实现 shell 的 `>`）

最常见的用途——将 `stdout` 重定向到文件，等价于 shell 的 `cmd > file`：

```c
#include <unistd.h>
#include <fcntl.h>
#include <stdio.h>

int main() {
    int fd = open("output.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
    if (fd == -1) {
        perror("open");
        return 1;
    }

    close(STDOUT_FILENO);         // 关闭标准输出（FD 1）
    dup(fd);                      // FD 1 现在指向 output.txt
    close(fd);                    // 原 FD 不再需要

    // 以下 printf 输出到文件，而非终端
    printf("这行会写入 output.txt\n");
    printf("而不是显示在屏幕上\n");

    return 0;
}
```

**分步解析**：

| 步骤 | FD 0 (stdin) | FD 1 (stdout) | FD 3 (open) |
|------|-------------|--------------|-------------|
| 初始 | 终端读 | 终端写 | — |
| `open("output.txt")`→3 | 终端读 | 终端写 | `output.txt` |
| `close(1)` | 终端读 | **空** | `output.txt` |
| `dup(3)` → 分配到 1 | 终端读 | `output.txt`（共享偏移） | `output.txt` |
| `close(3)` | 终端读 | `output.txt` | — |

### 用 `dup2` 简化

上述 `close + dup` 组合可以一步实现：

```c
dup2(fd, STDOUT_FILENO);   // 原子操作：关闭 FD 1，再让 FD 1 指向 fd 的同一表项
close(fd);
```

`dup2` 相比 `close + dup` 的优势：
1. **原子性** —— 关闭和重定向之间不会被信号或线程打断
2. **不需要知道最小可用 FD** —— 直接指定目标 FD 编号

## 4.2 场景二：重定向标准错误（实现 `2>&1`）

shell 的 `2>&1` 语义即"将 stderr 重定向到 stdout 当前指向的位置"：

```c
// 让 stderr（FD 2）指向 stdout（FD 1）的位置
dup2(STDOUT_FILENO, STDERR_FILENO);

// 之后 stdout 和 stderr 都到终端（或同一个重定向目标）
printf("普通输出\n");
fprintf(stderr, "错误输出\n");  // 也到 stdout 的地方
```

分步示意：

```plain
初始状态：
  FD 1 ──→  终端（写）
  FD 2 ──→  终端（写）

dup2(1, 2) 后：
  FD 1 ──→  终端（写）
  FD 2 ──→  ┘  （指向 FD 1 的表项 → 同一终端）

重定向 stdout：
  dup2(file_fd, 1)  或  close(1); dup(file_fd)

  FD 1 ──→  output.txt
  FD 2 ──→  output.txt（跟着 FD 1 走！）
```

## 4.3 场景三：管道实现（`pipe` + `dup2`）

实现 shell 管道 `cmd1 | cmd2` 的底层机制（结合 `fork` + `exec`）：

```c
#include <unistd.h>
#include <stdio.h>
#include <sys/wait.h>

int main() {
    int pipefd[2];
    pipe(pipefd);              // pipefd[0] = 读端，pipefd[1] = 写端

    pid_t pid = fork();

    if (pid == 0) {
        // 子进程：执行 cmd2（从管道读）
        close(pipefd[1]);      // 关闭写端
        dup2(pipefd[0], STDIN_FILENO);  // 读端 → stdin
        close(pipefd[0]);      // 关闭原 FD
        execlp("wc", "wc", "-l", NULL);
    } else {
        // 父进程：执行 cmd1（写入管道）
        close(pipefd[0]);      // 关闭读端
        dup2(pipefd[1], STDOUT_FILENO); // 写端 → stdout
        close(pipefd[1]);      // 关闭原 FD
        execlp("ls", "ls", "-la", NULL);
    }

    wait(NULL);
    return 0;
}
```

**流程示意**：

```plain
pipe() 后：
  pipefd[0] (读端) ──→  管道缓冲区
  pipefd[1] (写端) ──→  管道缓冲区

子进程 dup2(pipefd[0], STDIN_FILENO)：
  FD 0 (stdin) ──→  管道缓冲区（读端）
  pipefd[0]  ← 可关

父进程 dup2(pipefd[1], STDOUT_FILENO)：
  FD 1 (stdout) ──→  管道缓冲区（写端）
  pipefd[1]  ← 可关

最终：ls 写入 stdout → 管道 → wc 从 stdin 读
```

## 4.4 场景四：套接字服务中的标准 IO 重定向

网络服务中，将客户端连接的文件描述符重定向为子进程的 stdin/stdout，使 CGI 等服务无需感知网络层：

```c
// accept 返回 client_fd（如 FD 4）
int client_fd = accept(server_fd, NULL, NULL);

if (fork() == 0) {
    // 子进程
    dup2(client_fd, STDIN_FILENO);   // 客户端连接 → 标准输入
    dup2(client_fd, STDOUT_FILENO);  // 标准输出 → 客户端连接

    close(client_fd);                // FD 1, 0 已指向它，原 FD 可关

    // 之后子进程的 printf / scanf 直接与客户端通信
    printf("HTTP/1.1 200 OK\r\n\r\nHello from CGI!\n");

    // 等价于用 sock_fd 直接 write/read，但子进程不需要知道 socket 的存在
}
```

## 4.5 场景五：保存和恢复文件描述符

重定向前备份原 FD，用完后恢复：

```c
// 备份 stdout
int saved_stdout = dup(STDOUT_FILENO);
if (saved_stdout == -1) {
    perror("dup");
    exit(1);
}

// 重定向 stdout 到文件
int fd = open("log.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
dup2(fd, STDOUT_FILENO);
close(fd);

printf("这行写入 log.txt\n");    // 到文件

// 恢复 stdout
dup2(saved_stdout, STDOUT_FILENO);
close(saved_stdout);

printf("这行回到终端\n");        // 回到终端
```

> 注意：保存时用 `dup()` 获取新 FD（操作系统自动分配），恢复时用 `dup2()` 指定回到 `STDOUT_FILENO`。

## 4.6 场景六：`O_CLOEXEC` 防止 FD 泄漏（`dup3`）

在 `fork` + `exec` 场景中，子进程会继承父进程的全部 FD。如果不希望某些 FD 泄漏到执行的新程序中，在创建时设置 `O_CLOEXEC`（close-on-exec）：

```c
// 普通打开并设置 CLOEXEC
int fd = open("secret.key", O_RDONLY | O_CLOEXEC);

// 复制 FD 并同时设置 CLOEXEC
int new_fd = dup3(old_fd, new_fd, O_CLOEXEC);
// 等价于：
int new_fd = dup2(old_fd, new_fd);
fcntl(new_fd, F_SETFD, FD_CLOEXEC);  // 需额外系统调用
```

`dup3` 的优势是一次系统调用同时完成复制和设置，避免竞态条件（多线程环境中，`dup2` 后再 `fcntl` 之间可能已被 `fork`）。

> **判断是否需要 `O_CLOEXEC`**：思考"这个 FD 在 `exec` 后是否仍应存在"——如果否，一律加 `O_CLOEXEC`。这是安全编程的基本实践。

# 5. 与 `fcntl` 的关系

`dup` 家族的功能也可以用 `fcntl` 实现：

```c
#include <fcntl.h>

// dup(oldfd) 等价于：
fcntl(oldfd, F_DUPFD, 0);           // 返回 ≥ 0 的最小未用 FD

// dup2(oldfd, newfd) 等价于：
fcntl(oldfd, F_DUPFD, newfd);       // 返回 ≥ newfd 的最小未用 FD
// 但 fcntl 版本的 F_DUPFD 不等价于 dup2——它不关闭 newfd，
// 而是返回 ≥ newfd 的最小可用 FD，不一定是 newfd 本身

// dup3(oldfd, newfd, O_CLOEXEC) 等价于：
fcntl(oldfd, F_DUPFD_CLOEXEC, newfd);
```

| 方式 | 关闭已有 newfd | 指定 exact FD | CLOEXEC 控制 |
|------|:---:|:---:|:---:|
| `dup(oldfd)` | — | 否 | 否 |
| `dup2(oldfd, newfd)` | ✅ | ✅ | 否 |
| `dup3(oldfd, newfd, O_CLOEXEC)` | ✅ | ✅ | ✅ |
| `fcntl(oldfd, F_DUPFD, start)` | — | 否（≥ start） | 否 |
| `fcntl(oldfd, F_DUPFD_CLOEXEC, start)` | — | 否（≥ start） | ✅ |

# 6. 常见错误与陷阱

## 6.1 陷阱一：未考虑共享偏移量

```c
int fd = open("file", O_RDWR);
int fd2 = dup(fd);

read(fd, buf, 100);          // 读前 100 字节，偏移 → 100
read(fd2, buf, 100);         // ⚠️ 从 100 开始读，不是从头！
```

通过 `dup` 复制的 FD **共享文件偏移量**。如果需要独立偏移量，应使用 `open` 再次打开文件，而非 `dup`。

## 6.2 陷阱二：`dup2` 的原子性误解

```c
// 正确做法（原子操作）
dup2(fd, STDOUT_FILENO);

// 错误做法（非原子，存在竞态窗口）
close(STDOUT_FILENO);        // 此时 FD 1 空闲
dup(fd);                     // 期望分配到 FD 1
```

`close` + `dup` 之间如果发生信号处理或线程切换，可能被其他代码占用了 FD 1。而 `dup2` 保证关闭和重定向一气呵成。

## 6.3 陷阱三：忘记关闭原始 FD

每次 `dup` 会增加一个指向同一文件表项的引用计数。不关闭不再需要的原始 FD 会导致 FD 泄漏：

```c
int fd = open("file", O_RDONLY);
dup2(fd, STDIN_FILENO);
// ⚠️ 忘记 close(fd)——现在 FD 3 和 FD 0 都指向同一文件

// 原则：dup/dup2 后用不着的原 FD 立即关闭
dup2(fd, STDIN_FILENO);
close(fd);   // ✅ 原 FD 不再需要
```

## 6.4 陷阱四：`dup2` 时 `oldfd == newfd`

当 `oldfd == newfd` 时，`dup2` 直接返回 `newfd` 而**不关闭**它。此时检查返回值可能踩坑：

```c
int ret = dup2(1, 1);   // oldfd == newfd == stdout
// ret == 1，实际上什么也没做

// 而 dup3 在 oldfd == newfd 时返回 EINVAL
int ret = dup3(1, 1, O_CLOEXEC);
// ret == -1, errno == EINVAL
```

## 6.5 陷阱五：忽略 `O_CLOEXEC` 导致安全漏洞

```c
int fd = open("passwd", O_RDONLY);
// ...
pid_t pid = fork();
if (pid == 0) {
    // 子进程
    dup2(fd, STDIN_FILENO);
    close(fd);
    execlp("some_program", NULL);   // ⚠️ 如果 fd 没设 CLOEXEC
                                     // 且 fork 前没关闭，some_program
                                     // 可能遍历 /proc/self/fd 读密码文件
}
```

**修复**：在 `open` 时加 `O_CLOEXEC`，或用 `dup3` 而非 `dup2`。

# 7. 简易 shell 重定向实现

综合 `dup2`、`pipe`、`fork`，可以模拟 shell 的重定向：

```c
// 简易 shell: ls > output.txt
int fd = open("output.txt", O_WRONLY | O_CREAT | O_TRUNC, 0644);
pid_t pid = fork();
if (pid == 0) {
    dup2(fd, STDOUT_FILENO);   // stdout → output.txt
    close(fd);
    execlp("ls", "ls", NULL);
}
wait(NULL);

// 简易 shell: cat < input.txt
int fd = open("input.txt", O_RDONLY);
pid_t pid = fork();
if (pid == 0) {
    dup2(fd, STDIN_FILENO);    // input.txt → stdin
    close(fd);
    execlp("cat", "cat", NULL);
}
wait(NULL);
```

# 8. `dup` vs `dup2` vs `dup3` 选型指南

| 函数 | 适用场景 |
|------|---------|
| `dup` | 只需要一个新 FD，不关心编号；或配合 `close` 模拟重定向（不推荐） |
| `dup2` | 最常见的重定向——把 oldfd 复制到指定 newfd（如重定向到 0/1/2）；**原子操作** |
| `dup3` | 需要在复制的同时设置 `O_CLOEXEC`（多线程 / 安全敏感场景） |

**最佳实践**：
- 日常重定向用 `dup2`
- 多线程场景或编写安全程序用 `dup3` + `O_CLOEXEC`
- 避免手动 `close + dup` 组合——竞态风险不值得的
- 复制后立即关闭原 FD 防止泄漏
