# 1. 概述

## 【核心精简提纲】

### 记忆主干

`system` 与 `popen` 都经 `/bin/sh -c` 执行外部命令（fork+exec），但目标不同：**system** = 阻塞等待 + 返回退出码，I/O 继承终端；**popen** = 返回 `FILE*` 管道，可读 stdout 或写 stdin，须 `pclose` 回收子进程。

### 核心要点清单

- system：全程阻塞，自动 wait，屏蔽 SIGINT/SIGQUIT
- popen `"r"`：读子进程 stdout；`"w"`：写子进程 stdin；单向
- pclose：关流 + waitpid + 返回退出状态；忘记 → 僵尸进程
- popen 创建不阻塞，fgets/pclose 阻塞；可在等待期间做其他事
- 二者均有 shell 注入风险；不需 shell 特性时用 fork+execvp
- 不需捕获输出 → system；需编程处理输出 → popen("r")

### 关键结论 / 流程

```
只要退出码 → system │ 要读/写命令 I/O → popen + pclose │ 双向/无 shell → pipe+fork+exec
```

## 【完整拓展详情】


`popen` 和 `system` 是 POSIX 系统中两个用于**执行外部命令**的标准库函数。它们都基于 `fork` + `exec` + `wait` 机制，但设计目标和能力截然不同：

| 维度 | `system()` | `popen()` |
|------|-----------|-----------|
| 核心用途 | "跑一下，等结果" | "跑起来，我要读写" |
| I/O 交互 | ❌ 无法读取/写入命令的 I/O | ✅ 通过 `FILE*` 读输出或写输入 |
| 返回值 | 命令退出状态码（int） | I/O 流指针（`FILE*`） |
| 调用者阻塞 | ✅ 全程阻塞直到命令结束 | ❌ 立即返回流；读/写时阻塞 |
| 资源清理 | 自动（内部完成 `wait`） | 需要调用 `pclose()` |
| 信号影响 | 自动屏蔽 `SIGINT`/`SIGQUIT` | 不处理信号，继承调用者状态 |

```plain
                   ┌─────────────────────┐
                   │    调用进程          │
                   │  (fork + exec)      │
                   └─────┬───────────────┘
                         │
              ┌──────────┴──────────┐
              ▼                     ▼
       ┌────────────┐       ┌────────────┐
       │ system()   │       │ popen()    │
       │            │       │            │
       │ 执行命令   │       │ 执行命令   │
       │ 阻塞等待   │       │ 返回流指针 │
       │ 返回状态码 │       │ 通过流 I/O │
       └────────────┘       └────────────┘
              │                     │
              ▼                     ▼
        知道"成功与否"        可以"读输出/写输入"
```

# 2. 函数原型与头文件

```c
#include <stdlib.h>

int system(const char *command);

#include <stdio.h>

FILE *popen(const char *command, const char *type);
int pclose(FILE *stream);
```

| 函数 | 参数 | 返回值 |
|------|------|--------|
| `system(cmd)` | 命令字符串（传给 `/bin/sh -c`） | 命令退出状态码（详见下文） |
| `popen(cmd, type)` | 命令 + 模式（`"r"` / `"w"`） | `FILE*` 流指针，失败返回 `NULL` |
| `pclose(stream)` | `popen` 返回的流指针 | 命令退出状态码，失败返回 -1 |

## 2.1 type 参数详解

`popen` 的 `type` 决定了调用者与子进程间的数据流向：

```
popen("cmd", "r") :  子进程标准输出 → 调用者读取
  调用进程 ──┐           ┌──► 调用进程的 FILE*
             ├──pipe──►──┤
  子进程 stdout ──┘     └──► 子进程写入管道

popen("cmd", "w") :  调用者写入 → 子进程标准输入
  调用进程写入 FILE* ──┐   ┌──► 子进程 stdin
                       ├──►──┤
                       └─  └──► 管道 → 子进程
```

> **注意**：标准 POSIX 只保证支持 `"r"` 和 `"w"`。有些实现（如 Linux glibc）支持 `"re"` / `"we"` 额外设置 `O_CLOEXEC`，但不可移植。

# 3. 核心区别对比

## 3.1 I/O 能力

**`system()`** — 无法与子进程通信：

```c
#include <stdlib.h>
#include <stdio.h>

int main() {
    int ret = system("ls -la");
    // ✅ 输出直接打到终端（继承父进程的 stdout）
    // ❌ 无法在程序中捕获或处理这些输出
    printf("Exit status: %d\n", ret);
    return 0;
}
```

**`popen()`** — 可以读取子进程输出：

```c
#include <stdio.h>
#include <stdlib.h>

int main() {
    FILE *fp = popen("ls -la", "r");
    if (!fp) {
        perror("popen");
        exit(1);
    }

    char buf[1024];
    while (fgets(buf, sizeof(buf), fp)) {
        // ✅ 逐行处理子进程的输出
        printf("[CAPTURED] %s", buf);
    }

    int ret = pclose(fp);
    printf("Command exit: %d\n", ret);
    return 0;
}
```

也可以用 `popen("cmd", "w")` 向子进程写入数据：

```c
FILE *fp = popen("grep error", "w");
if (!fp) { /* error */ }

fprintf(fp, "line 1: no error\n");
fprintf(fp, "line 2: critical error\n");  // grep 会输出 "line 2: critical error"
fclose(fp);  // 等价于 pclose() 发送 EOF 到管道

// 注意：这里用 fclose 而非 pclose——因为不需要获取退出状态
// 但更规范的做法始终是 pclose()
```

## 3.2 返回值与状态码

```c
// ─── system() ───
int ret = system("some_command");

if (ret == -1) {
    // system() 本身失败（fork 失败等）
} else if (WIFEXITED(ret)) {
    int code = WEXITSTATUS(ret);   // 命令正常退出
} else if (WIFSIGNALED(ret)) {
    int sig = WTERMSIG(ret);       // 命令被信号终止
}

// ─── popen() + pclose() ───
FILE *fp = popen("some_command", "r");
if (!fp) {
    // popen 失败（fork 或 pipe 失败）
}

// ... 读写操作 ...

int status = pclose(fp);

if (status == -1) {
    // pclose 自身出错
} else if (WIFEXITED(status)) {
    // 命令正常退出
} else if (WIFSIGNALED(status)) {
    // 命令被信号终止
}
```

**关键区别**：

| 行为 | `system()` | `popen()` + `pclose()` |
|------|-----------|----------------------|
| 执行失败（fork 失败） | 返回 -1 | `popen` 返回 `NULL` |
| 命令正常退出 | 返回退出状态（WIFEXITED 宏处理） | `pclose` 返回退出状态 |
| 命令被信号终止 | 返回信号相关状态码 | `pclose` 返回被信号终止状态 |
| shell 不可执行 | 返回 127 | `popen` 返回 `NULL` 或通过 pclose 返回 127 |
| 获取退出码 | 直接通过返回值 | 必须通过 `pclose()` 返回值 |

## 3.3 阻塞行为

```c
// ─── system(): 全程阻塞 ───
puts("Before system");
int ret = system("sleep 5");   // ⏳ 阻塞 5 秒
puts("After system");          // 5 秒后才执行

// ─── popen(): 立即返回流 ───
puts("Before popen");
FILE *fp = popen("sleep 5; echo done", "r");
// ✅ 立即返回，不阻塞
puts("After popen — this prints immediately!");  // ✅ 马上执行

// 但是读操作会阻塞（等待数据可用）
char buf[256];
fgets(buf, sizeof(buf), fp);  // ⏳ 阻塞约 5 秒，等 "done"
printf("Got: %s", buf);

pclose(fp);  // ⏳ 也会阻塞等待子进程结束
```

> `popen` 返回流**本身不阻塞**，但后续的 `fread`/`fgets`/`fwrite` 以及 `pclose` 仍然会阻塞。只是给了调用者在**阻塞前做其他事情**的机会（例如用 `select`/`poll` 非阻塞检查）。

## 3.4 信号处理

```c
// ─── system() 的信号屏蔽行为 ───
// 在 system() 执行期间：
//   - SIGINT （Ctrl+C） → 被屏蔽，不会中断 system()
//   - SIGQUIT（Ctrl+\） → 被屏蔽
//   - SIGCHLD           → 被忽略（防止干扰父进程 wait 逻辑）
//
// 这确保了 system("vi") 这样的调用不会被用户意外中断。

// ─── popen() 的信号行为 ───
// 不处理任何信号！子进程继承父进程的信号处置。
// 如果父进程设置了 SIG_IGN 或 SIG_DFL，子进程也一样。
// Ctrl+C 可以中断 popen 的子进程。
```

**为什么 `system()` 要屏蔽信号？**

假设用户调用 `system("long_running_task")`，如果用户在终端按 Ctrl+C：
- 信号发给前台进程组（包括父进程和子进程）
- 如果不屏蔽 `SIGINT`，父进程的信号处理函数可能在 `wait` 期间被调用，导致状态混乱
- `system()` 选择屏蔽，让子进程自己处理中断

**`popen()` 为什么不屏蔽？**

因为 `popen()` 的设计是"轻量管道连接"，调用者通常希望自己控制子进程的信号行为。如果调用者在管道中运行一个需要交互的程序，屏蔽信号反而会破坏交互。

## 3.5 资源生命周期

```c
// ─── system(): 自包含 ───
int ret = system("cmd");    // fork → exec → wait → 清理
// 此时子进程已完全结束，无残留

// ─── popen(): 需要显式清理 ───
FILE *fp = popen("cmd", "r");
// 子进程在后台运行，只有 pclose() 后才会 wait()
if (!fp) { /* error */ }

// ... 如果中途 return 或 exit ...
// ⚠️ 未 pclose() → 子进程变成僵尸（zombie）

// 正确做法：
int status = pclose(fp);    // 阻塞等待 + wait + 关闭流
```

`pclose()` 做了三件事：

1. **关闭** `FILE*` 流（发送 EOF 到管道）
2. **等待** 子进程结束（内部调用 `waitpid`）
3. **返回** 子进程退出状态

忘记 `pclose()` 会导致：
- 子进程变成僵尸进程（`/proc` 中可见 `<defunct>`）
- 管道资源泄漏
- 子进程可能永远挂起（如果管道缓冲区满且无人读取）

# 4. 内部实现对比

两者底层都基于 `fork()` + `exec()`，但编排方式不同：

```
system("cmd"):
  1. fork()
  2. 子进程: exec("/bin/sh", "-c", "cmd", (char *)NULL)
  3. 父进程: waitpid(child, &status, 0)    ← 阻塞
  4. 父进程: 恢复信号屏蔽，返回状态码

popen("cmd", "r"):
  1. pipe(pipefd)                         ← 创建管道
  2. fork()
  3. 子进程:
     - close(pipefd[0])                   ← 关闭读端
     - dup2(pipefd[1], STDOUT_FILENO)     ← 写端挂到 stdout
     - close(pipefd[1])
     - exec("/bin/sh", "-c", "cmd", NULL)
  4. 父进程:
     - close(pipefd[1])                   ← 关闭写端
     - fdopen(pipefd[0], "r")             ← 读端包装为 FILE*
     - 返回 FILE*                         ← 不阻塞

popen("cmd", "w"):
  1. pipe(pipefd)                         ← 创建管道
  2. fork()
  3. 子进程:
     - close(pipefd[1])                   ← 关闭写端
     - dup2(pipefd[0], STDIN_FILENO)      ← 读端挂到 stdin
     - close(pipefd[0])
     - exec("/bin/sh", "-c", "cmd", NULL)
  4. 父进程:
     - close(pipefd[0])                   ← 关闭读端
     - fdopen(pipefd[1], "w")             ← 写端包装为 FILE*
     - 返回 FILE*
```

# 5. 选型指南

## 5.1 用 `system()` 当且仅当：

- ✅ 只需知道命令"跑成功没"（退出码）
- ✅ 命令的输出直接给用户看（打到终端即可）
- ✅ 不需要处理或捕获命令的输出
- ✅ 不需要向命令的 stdin 写数据
- ✅ 简单的一次性调用

```c
// ✅ 典型 system() 场景
if (system("which gcc") != 0) {
    fprintf(stderr, "GCC not found, installing...\n");
    system("apt install -y gcc");
}

// ✅ 检查命令是否存在
if (system("git --version") == 0) {
    printf("Git is available\n");
} else {
    printf("Git not found\n");
}
```

## 5.2 用 `popen()` 当：

- ✅ 需要**读取命令的标准输出**到程序中处理
- ✅ 需要**向命令的标准输入写入**数据
- ✅ 希望在程序中使用命令的结果（而非展示给用户）
- ✅ 需要在命令执行期间做其他工作（配合非阻塞 I/O）

```c
// ✅ popen("r"): 读取命令输出
// 获取当前用户名
FILE *fp = popen("whoami", "r");
char user[64];
fgets(user, sizeof(user), fp);
pclose(fp);
// user 中即当前用户名，可编程使用

// ✅ popen("w"): 写入命令标准输入
// 用 mail 命令发送邮件
FILE *mail = popen("mail -s 'Report' admin@example.com", "w");
if (mail) {
    fprintf(mail, "Daily report:\n");
    fprintf(mail, "  - CPU: 45%%\n");
    fprintf(mail, "  - Memory: 2.1GB/8GB\n");
    pclose(mail);  // 发送
}
```

## 5.3 决策流程图

```
需要执行外部命令吗？
  │
  ├── 只需退出码？──────→ 用 system()
  │
  ├── 需要读取命令输出？──→ popen("cmd", "r")
  │
  ├── 需要写入命令输入？──→ popen("cmd", "w")
  │
  ├── 需要双向 I/O？────→ ❌ popen 不适用
  │                       考虑 pipe() + fork() 手动实现
  │                       或使用进程间通信（socketpair 等）
  │
  └── 只需纯执行（不经过 shell）？
                         → 用 fork() + execvp() 直接调用
                           避免 shell 注入风险
```

# 6. 安全注意事项

## 6.1 Shell 注入风险

`popen` 和 `system` 都通过 `/bin/sh -c` 执行命令——这意味着输入中包含 shell 元字符会导致**命令注入**：

```c
// ⚠️ 危险！如果 filename 来自用户输入
char filename[256];
// 用户输入: "; rm -rf / #"
sprintf(cmd, "cat %s", filename);  // cmd = "cat ; rm -rf / #"
system(cmd);                        // 💥 灾难！

// ✅ 安全做法：验证输入
if (strchr(filename, ';') || strchr(filename, '|')
    || strchr(filename, '&') || strchr(filename, '$')) {
    fprintf(stderr, "Invalid filename\n");
    exit(1);
}

// 更安全的做法：用 execvp 直接执行，不经过 shell
// 但 popen/system 内部走 sh -c，无法绕过
// 如果不需要 shell 特性，用 fork+exec
```

## 6.2 `popen` + 不可预测的输出

```c
// ⚠️ 如果命令产生大量输出（数 GB），且不读取：
FILE *fp = popen("dd if=/dev/zero bs=1M count=4096", "r");
// ... 不读 ...
// 管道缓冲区满后，子进程阻塞在 write()
// 如果此时忘记 pclose()，子进程永远挂起

// ✅ 要么读取，要么不要用 popen("r")
// ✅ 或者用非阻塞方式设置管道
```

## 6.3 `pclose` 错误处理

```c
FILE *fp = popen("some_command", "r");
// ... 读写 ...

int status = pclose(fp);
if (status == -1) {
    perror("pclose");
    // pclose 自身失败（如 waitpid 被信号中断）
} else if (WIFEXITED(status) && WEXITSTATUS(status) != 0) {
    printf("Command exited with code %d\n", WEXITSTATUS(status));
} else if (WIFSIGNALED(status)) {
    printf("Command killed by signal %d\n", WTERMSIG(status));
}
```

# 7. 完整对比表

| 维度 | `system()` | `popen()` |
|------|-----------|-----------|
| 头文件 | `<stdlib.h>` | `<stdio.h>` |

| Shell 注入风险 | ⚠️ | ⚠️ |
| 双向通信 | ❌ | ❌（单向，只能选 r 或 w） |
| 标准兼容 | POSIX.1 | POSIX.1 |
| 适合场景 | "跑命令，看结果" | "跑命令，用输出" |

# 9. 常见踩坑补充

- **popen 后用 fclose 代替 pclose**：无法获取退出码且 wait 行为不确定 → 始终 `pclose`
- **system 返回值未用 WIFEXITED 解析**：退出码 256 被误读为 1 → 用宏解析
- **EINTR 中断 wait**：pclose/system 可能被信号打断 → 循环重试或检查 errno
- **大量输出不读**：popen("r") 子进程写满管道后阻塞 → 持续 fread 或不用 popen
