# 1. 内核模块开发实践

内核模块开发的核心实践：**符号可见性管控**（`static` / `EXPORT_SYMBOL` / 编译单元作用域）和**多文件组织**（`.h` 头文件跨文件交互）。

相关笔记：[[12. 用户态与内核态|用户态与内核态]]、[[13. 多功能字符设备驱动|多功能字符设备驱动]]

## 1.1 内核中 static 与符号可见性

### 1.1.1 核心误区：非 static ≠ 全局可见

用户态代码中不加 `static` 的函数/变量默认是**全局可见的**（其他文件可通过 `extern` 声明调用）。但在 Linux 内核中**完全不同**：

| 符号修饰 | 用户态 | 内核态 |
|----------|--------|--------|
| `static` | 当前文件可见 | 当前文件可见 |
| 非 `static`（未导出） | 全局可见（默认） | **仅编译单元内可见** |
| 非 `static` + `EXPORT_SYMBOL` | — | 全局可见（唯一方式） |

> **内核中，非 `static` ≠ 全局可见**。全局可见的唯一方式是显式符号导出。

### 1.1.2 static 的用法

`static` 修饰的函数/变量作用域限制在**当前 `.c` 文件内**。

```c
// charge_gateway.c
#include <linux/module.h>

// static 变量：仅当前文件可见
static int g_local_state = 0;

// static 函数：仅当前文件可见
static int __charge_crc_check(const u8 *data, int len) {
    return 0;  // CRC 校验逻辑，仅当前文件内调用
}

// 非 static 函数：编译单元内可见（但未导出，不是全局可见）
int charge_data_process(const u8 *data, int len) {
    if (__charge_crc_check(data, len) != 0)
        return -1;
    g_local_state = 1;
    return 0;
}
```

### 1.1.3 编译单元与符号作用域

内核使用**「编译单元 + 符号导出」**机制管控符号：

1. **编译单元**：一个或多个 `.c` 文件编译成一个 `.o` 文件（由 `Makefile` 中的 `obj-y` / `obj-m` 决定）
2. **默认可见性**：不加 `static` 的符号，可见性限制在**当前编译单元内**
3. **全局可见**：必须用 `EXPORT_SYMBOL` / `EXPORT_SYMBOL_GPL` 显式导出

### 1.1.4 全局可见：EXPORT_SYMBOL

将符号暴露给**整个内核的所有编译单元**，必须显式导出：

```c
// charge_global.c
#include <linux/module.h>

// 定义符号（不加 static）
int charge_global_process(const u8 *data, int len) {
    return 0;  // 全局核心处理逻辑
}

// 导出为全局可见
EXPORT_SYMBOL(charge_global_process);

// 导出变量同理
int g_global_param = 0;
EXPORT_SYMBOL(g_global_param);
```

其他驱动模块调用：

```c
// other_driver.c（不同编译单元）
extern int charge_global_process(const u8 *data, int len);
extern int g_global_param;

static int __init other_driver_init(void) {
    u8 data[] = {0x01, 0x02, 0x03};
    charge_global_process(data, sizeof(data));  // 调用全局符号
    g_global_param = 123;
    return 0;
}
module_init(other_driver_init);
MODULE_LICENSE("GPL");  // 若调用 EXPORT_SYMBOL_GPL 导出的符号，必须用 GPL
```

### 1.1.5 最佳实践速查

| 符号类型 | 作用域 | static | 导出 | 示例 |
|----------|--------|--------|------|------|
| 函数/变量 | 仅当前文件 | ✅ 必须加 | ❌ | 辅助函数、模块内私有状态 |
| 函数/变量 | 编译单元内跨文件 | ❌ 不加 | ❌ | 同一 `obj-y` 的多 `.c` 文件之间调用 |
| 函数/变量 | 全局可见 | ❌ 不加 | ✅ `EXPORT_SYMBOL[_GPL]` | 内核核心函数、模块间 API |
| 驱动对外接口 | 用户态通信 | ❌ 不加 | ❌ 不需导出 | `file_operations`、`ioctl` — 通过字符设备等标准机制暴露给用户态，不通过符号导出 |

> 特别注意：驱动向用户态暴露接口（`ioctl`、`sysfs` 等）**不需要 `EXPORT_SYMBOL`**。内核符号导出用于 **内核态内部模块间通信**，用户态通信通过标准机制（字符设备、sysfs、netlink）实现，两者是不同的概念。

## 1.2 多文件组织：编译单元内交互

内核模块（驱动）通常由多个 `.c` 文件组成，通过自定义 `.h` 头文件进行交互，属于同一个编译单元。

### 1.2.1 核心模式

```
驱动模块/
├── Makefile               # obj-m += xxx.o; xxx-objs := a.o b.o c.o
├── module_a.h             # 声明非 static 共享符号
├── module_a.c             # 定义非 static 共享符号 + static 私有符号
├── module_b.h             # 声明非 static 共享符号
├── module_b.c             # 定义非 static 共享符号 + static 私有符号
└── main.c                 # 驱动入口，调用各模块
```

- 共享函数/变量**不加 `static`**，在 `.h` 中声明，在 `.c` 中定义
- 私有函数/变量**加 `static`**，不在 `.h` 中声明
- `Makefile` 中所有 `.c` 文件加入同一个 `obj-m` 的 `-objs`

### 1.2.2 实战示例：充电桩网关驱动

```
charge_gateway/
├── Makefile
├── charge_utils.h + charge_utils.c    # 工具函数层
├── charge_core.h + charge_core.c      # 核心功能层
└── charge_main.c                      # 驱动入口
```

**工具函数层**：

```c
// charge_utils.h — 声明共享符号
#ifndef CHARGE_UTILS_H
#define CHARGE_UTILS_H
#include <linux/types.h>

u16 charge_utils_crc16(const u8 *data, u32 len);      // 编译单元内可见
void charge_utils_convert(u8 *src, u8 *dst, u32 len);
#endif
```

```c
// charge_utils.c — 定义共享符号 + 私有符号
#include "charge_utils.h"

u16 charge_utils_crc16(const u8 *data, u32 len) {     // 非 static，共享
    u16 crc = 0xFFFF;
    for (u32 i = 0; i < len; i++) {
        crc ^= data[i];
        for (u32 j = 0; j < 8; j++)
            if (crc & 0x0001) crc = (crc >> 1) ^ 0xA001;
            else crc >>= 1;
    }
    return crc;
}

void charge_utils_convert(u8 *src, u8 *dst, u32 len) {
    for (u32 i = 0; i < len; i++) dst[i] = src[i] ^ 0x55;
}

static void charge_utils_private(void) { }  // static：仅当前文件可见
```

**核心功能层**：

```c
// charge_core.h
#ifndef CHARGE_CORE_H
#define CHARGE_CORE_H
#include <linux/types.h>

int charge_core_process(const u8 *data, u32 len);
extern int charge_core_param;              // 共享变量声明
#endif
```

```c
// charge_core.c — 调用 utils 层的共享函数
#include "charge_core.h"
#include "charge_utils.h"

int charge_core_param = 0;                 // 共享变量定义

int charge_core_process(const u8 *data, u32 len) {
    if (!data || len == 0) return -1;

    u16 crc = charge_utils_crc16(data, len);   // 调用另一文件的函数
    if (crc != 0x0000) return -2;

    u8 dst[len];
    charge_utils_convert((u8 *)data, dst, len);
    charge_core_param = dst[0];
    return 0;
}
```

**驱动入口层**：

```c
// charge_main.c — 调 core 层和 utils 层
#include <linux/module.h>
#include "charge_core.h"
#include "charge_utils.h"

static int __init charge_gateway_init(void) {
    u8 data[] = {0x01, 0x02, 0x03, 0x04};
    int ret = charge_core_process(data, sizeof(data));
    if (ret != 0) return ret;

    printk(KERN_INFO "param = %d, crc = 0x%04X\n",
           charge_core_param, charge_utils_crc16(data, sizeof(data)));
    return 0;
}

static void __exit charge_gateway_exit(void) { }

module_init(charge_gateway_init);
module_exit(charge_gateway_exit);
MODULE_LICENSE("GPL");
```

**Makefile**：

```makefile
obj-m += charge_gateway.o
charge_gateway-objs := charge_main.o charge_core.o charge_utils.o

KERNELDIR ?= /lib/modules/$(shell uname -r)/build
PWD := $(shell pwd)

all:
	$(MAKE) -C $(KERNELDIR) M=$(PWD) modules
clean:
	$(MAKE) -C $(KERNELDIR) M=$(PWD) clean
```

`charge_gateway-objs` 将所有 `.c` 文件归入同一个编译单元，使非 `static` 符号在文件间互相可见。

### 1.2.3 与用户态多文件编程的对比

| 对比项 | 用户态 | 内核态 |
|--------|--------|--------|
| 交互基础 | `.h` 声明，`.c` 定义 | 完全一致 |
| 私有符号 | `static` | 完全一致 |
| 共享符号（默认） | 不加 `static`，全局可见 | 不加 `static`，**编译单元内可见** |
| 编译单元组织 | `gcc a.c b.c -o app` | `Makefile` 中 `xxx-objs := a.o b.o` |
| 跨编译单元交互 | 静态库/动态库 | `EXPORT_SYMBOL` 导出 |

**核心一致点**：头文件（`.h`）是多文件交互的基础，`static` 语义一致，模块化设计思想一致。

**核心差异**：内核对共享符号的作用域管控更严格——不加 `static` 仅编译单元内可见，全局可见必须显式导出。
